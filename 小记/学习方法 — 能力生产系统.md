# 🧠 学习方法 — 把陌生能力变成可复用能力的生产系统

> **一句话**：不是更快地输入，而是更系统地建模 → 更频繁地输出 → 更及时地反馈 → 更长期地检索 → 更主动地迁移。
> 重要自省:一定要把问题想到可以问再问!!!

---

## 一、学任何新技术，先套「五不变量」

不用从头读文档，先用这个模板建骨架，细节才有地方挂：

| 维度 | 问题 | 举例：Redis |
|------|------|------------|
| **数据模型** | 它操作的核心对象是什么？ | string/list/hash/set/zset/stream |
| **执行模型** | 它怎么运行？ | 单线程事件循环 + IO 多路复用 |
| **状态模型** | 状态放在哪里、怎么持久？ | 内存 + RDB/AOF |
| **扩展模型** | 加能力怎么加？ | 模块、Lua、集群 |
| **失败模型** | 坏了怎么表现、怎么恢复？ | 穿透/击穿/雪崩、主从延迟、淘汰 |

> 五不变量是可迁移的。下次学 Kafka、PaiFlow、任何新框架，套同一个模板。

---

## 二、学习卡片：每学一个技术填一张

```
技术名：
它解决什么问题：
不用它会怎样：
核心抽象：
一次主链路：
关键数据结构：
最容易出错的点：
我做过的最小 demo：
我做过的一次改造：
面试 1 分钟讲法：
还能迁移到哪里：
```

**填不出的地方 = 没学透的地方。** 这个方法朴素但狠——逼你从「看过」变成「可表达、可迁移、可验证」。

---

## 三、学习深度：四个循环，别停在第二层

```
理解循环：我知道它是什么         ← 大部分人在这里
应用循环：我能用它做东西         ← 很多人到这里就停了
诊断循环：它坏了我能查          ← 面试和工作真正看的
迁移循环：换个场景我还能用       ← 高手和普通人的分界线
```

**检验标准**：学完一个项目不能只问「怎么跑起来」，还要问：

- 如果数据库连不上，哪里会报错？
- 如果 SSE 断了，前端怎么感知？
- 如果我要把 TTS 从讯飞换成小米，抽象边界在哪里？

---

## 四、检索练习：输出不是结果，是手段

> 学习不只发生在输入时，更多发生在**主动从脑子里往外拿**的时候。

```text
看 20 分钟 → 关掉资料 → 白板讲 5 分钟
读一个模块 → 关掉 IDE → 画调用链
学一个概念 → 写 3 个面试问答
改一段代码 → 写一段 PR 说明
```

**说不出来的地方，就是下一轮学习的入口。**

---

## 五、个人学习流水线（8 步）

```text
1. 定义目标     — 面试、做项目、修 bug、还是设计系统？
2. 建立地图     — 画模块、数据、链路（五不变量）
3. 最小闭环     — 跑一个最小的 demo 或请求
4. 深挖主路径   — 只研究最高频、最核心的调用链
5. 主动输出     — 画图、写解释、讲 5 分钟
6. 制造故障     — 故意让它失败，练排查
7. 间隔检索     — 隔 1 天 / 3 天 / 7 天不看资料复述
8. 迁移改造     — 换一个需求，用同一套思想解决
```

---

## 六、你现在的升级方向

| 你已经在做     | 升级为                        |
| --------- | -------------------------- |
| 以输出为目标    | 以**可迁移表现**为目标              |
| 先搞清楚需求    | 先搞清楚**问题空间 + 因果模型**        |
| 通过项目带动知识点 | 通过**最小闭环 → 故障 → 反馈**形成稳定能力 |
|           | 通过**复述 + 间隔检索**固化长期记忆      |

**最终能力**（秋招和实习真正值钱的）：

> 看到陌生系统 → 快速建模 → 找主链路 → 跑通闭环 → 定位风险 → 做出改造 → 讲清楚取舍
