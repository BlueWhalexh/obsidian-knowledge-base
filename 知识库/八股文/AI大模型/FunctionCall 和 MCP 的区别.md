# FunctionCall 和 MCP 的区别

## 一、你如何理解大模型

### 1.1 Transformer 架构

大模型（Large Language Model, LLM）的核心架构是 **Transformer**，由 Google 在 2017 年论文《Attention Is All You Need》中提出。

**核心组件**：

- **自注意力机制（Self-Attention）**：计算输入序列中每个 token 与其他所有 token 的关联度，捕获长距离依赖关系
- **多头注意力（Multi-Head Attention）**：多组注意力并行计算，从不同子空间学习不同的依赖模式
- **前馈神经网络（FFN）**：对注意力输出进行非线性变换
- **位置编码（Positional Encoding）**：注入位置信息，因为 Transformer 本身不感知顺序
- **层归一化（Layer Norm）**：稳定训练过程

**Encoder-Decoder 结构**：

- **Encoder-only**：BERT 系列，擅长理解任务（分类、NER）
- **Decoder-only**：GPT 系列、Claude、DeepSeek，擅长生成任务（对话、写作）
- **Encoder-Decoder**：T5、BART，擅长翻译、摘要等序列到序列任务

当前主流大模型几乎都是 **Decoder-only** 架构。

### 1.2 预训练（Pre-training）

**核心思想**：在海量无标注文本上进行自监督学习，让模型学会语言的通用规律。

**预训练任务**：

- **因果语言建模（CLM）**：给定前文预测下一个 token（GPT 系列使用）
- **掩码语言建模（MLM）**：随机遮盖部分 token，让模型预测被遮盖的内容（BERT 使用）

**预训练数据**：

- 互联网文本（Common Crawl）
- 书籍、论文、代码
- 维基百科、新闻
- 数据量从几 TB 到几十 TB 不等

**训练规模**：

| 模型 | 参数量 | 训练数据量 | 训练算力 |
|------|--------|-----------|---------|
| GPT-3 | 175B | 300B tokens | ~3640 PF-days |
| LLaMA 2 | 70B | 2T tokens | ~1843 PF-days |
| DeepSeek-V3 | 671B | 14.8T tokens | ~2048 GPUs |

### 1.3 微调（Fine-tuning）

**为什么需要微调？** 预训练模型虽然有很强的语言能力，但在特定任务上表现不够好。微调可以让模型适应特定领域或任务。

**微调方式**：

| 方式 | 说明 | 数据量 | 效果 |
|------|------|--------|------|
| 全量微调 | 更新所有参数 | 大 | 最好，但成本高 |
| LoRA | 低秩适配，只训练少量参数 | 中 | 接近全量微调（推荐） |
| QLoRA | 量化 + LoRA | 中 | 适合资源有限场景 |
| Prompt Tuning | 只训练 prompt embedding | 小 | 效果一般 |
| RLHF | 人类反馈强化学习 | 中 | 对齐人类偏好 |
| DPO | 直接偏好优化 | 中 | RLHF 的简化版 |

**RLHF 流程**：

```
1. 监督微调（SFT）：用高质量的对话数据微调基座模型
2. 训练奖励模型（RM）：人类标注哪个回答更好，训练一个打分模型
3. 强化学习优化（PPO）：用奖励模型的打分作为奖励信号，通过 PPO 算法优化模型
```

### 1.4 推理（Inference）

**推理过程**：给定输入 prompt，模型自回归地生成每个 token。

```
输入："今天天气"
→ 模型预测下一个 token："真"
→ 输入变为"今天天气真"
→ 模型预测下一个 token："好"
→ 输入变为"今天天气真好"
→ ...直到生成 EOS（结束标记）
```

**推理优化技术**：

| 技术 | 说明 |
|------|------|
| KV Cache | 缓存已计算的 Key/Value，避免重复计算 |
| 量化（INT8/INT4） | 减少模型精度，降低内存和计算量 |
| 投机采样 | 用小模型先生成草稿，大模型验证 |
| vLLM/TGI | 高性能推理框架，PagedAttention |
| 批处理（Batching） | 多个请求共享计算资源 |
| 模型并行 | 将大模型分布到多张 GPU 上 |

## 三、大模型面临的关键问题

### 3.1 幻觉（Hallucination）

模型生成的内容看起来合理但实际上是不真实的。详见下一节。

### 3.2 成本

- **训练成本**：GPT-3 训练成本约 460 万美元，GPT-4 估计超过 1 亿美元
- **推理成本**：每次 API 调用都有费用，token 数越多越贵
- **GPU 成本**：高端 GPU（A100/H100）供不应求

### 3.3 延迟

- 长文本生成需要逐 token 输出，延迟较高
- 首 token 延迟（TTFT）和每 token 延迟（TPOT）都需要优化
- 实时对话场景对延迟敏感

### 3.4 隐私

- 数据上传到云端 API 存在隐私泄露风险
- 企业敏感数据不适合发送到外部 API
- 解决方案：本地部署、联邦学习、差分隐私

### 3.5 可控性

- 模型可能生成有害、偏见、不合规的内容
- 难以精确控制输出格式和内容
- 需要安全对齐（Safety Alignment）和内容过滤

## 四、大模型的幻觉问题

### 4.1 什么是幻觉

大模型生成的内容看起来流畅、合理，但实际上是不真实的、虚构的，或者与事实不符。

**幻觉类型**：

| 类型 | 说明 | 示例 |
|------|------|------|
| 事实性幻觉 | 生成不存在的事实 | "爱因斯坦在 1921 年获得了诺贝尔化学奖"（实际是物理学奖） |
| 忠实性幻觉 | 生成与输入不一致的内容 | 摘要中添加了原文没有的信息 |
| 推理幻觉 | 推理过程看似合理但结论错误 | 数学题解题步骤正确但计算结果错误 |
| 引用幻觉 | 引用不存在的文献/链接 | 生成一个不存在的论文引用 |

### 4.2 幻觉产生的原因

1. **训练数据**：训练数据中包含错误或过时的信息
2. **概率生成**：模型根据概率选择 token，可能选择了"看似合理但不正确"的 token
3. **知识截止**：模型的知识有截止日期，无法获取最新信息
4. **过度泛化**：模型将已学习的模式错误地推广到新场景
5. **暴露偏差（Exposure Bias）**：训练时看到的是正确序列，推理时可能偏离

### 4.3 解决幻觉的方法

**方法一：RAG（检索增强生成）**

```
用户提问 → 检索相关文档 → 将文档作为上下文 → 模型基于上下文生成回答
```

RAG 让模型的回答基于检索到的真实文档，而非完全依赖模型的记忆。

```java
// RAG 流程伪代码
public String rag(String question) {
    // 1. 检索相关文档
    List<Document> docs = vectorStore.similaritySearch(question, 5);
    
    // 2. 构建 prompt
    String context = docs.stream()
        .map(Document::getContent)
        .collect(Collectors.joining("\n"));
    
    String prompt = "基于以下参考资料回答问题：\n" + context 
        + "\n\n问题：" + question;
    
    // 3. 调用大模型
    return llm.generate(prompt);
}
```

**方法二：RLHF（人类反馈强化学习）**

通过人类标注者评估模型回答的质量，训练奖励模型，再用强化学习优化模型，使其生成更真实、更有帮助的回答。

**方法三：提示工程**

```
请基于以下事实回答，如果不确定请说"我不确定"：
1. 只使用提供的参考资料中的信息
2. 不要编造信息
3. 如果找不到答案，明确说明
```

**方法四：事实核查（Fact Checking）**

生成回答后，使用另一个模型或工具对回答中的事实进行核查。

**方法五：思维链（Chain of Thought）**

让模型展示推理过程，便于发现和纠正推理中的错误。

### 4.4 各方法对比

| 方法 | 效果 | 成本 | 实时性 | 适用场景 |
|------|------|------|--------|---------|
| RAG | 好 | 中 | 高 | 知识问答（推荐） |
| RLHF | 好 | 高 | - | 模型训练阶段 |
| 提示工程 | 中 | 低 | 高 | 快速改善 |
| 事实核查 | 好 | 中 | 中 | 高精度场景 |
| 思维链 | 中 | 低 | 高 | 推理任务 |

## 六、如何看待 DeepSeek

### 6.1 DeepSeek 的背景

DeepSeek（深度求索）是一家中国 AI 公司，推出了多个开源大模型，在推理能力上表现出色。

### 6.2 核心特点

| 特点 | 说明 |
|------|------|
| 国产大模型 | 中国团队研发，对中文理解优秀 |
| 开源 | DeepSeek-V2/V3、DeepSeek-R1 等模型开源，可本地部署 |
| 推理能力强 | DeepSeek-R1 在数学、代码等推理任务上接近 GPT-4 |
| 成本优势 | 训练和推理成本远低于同等水平的闭源模型 |
| MoE 架构 | 采用混合专家（Mixture of Experts）架构，降低推理成本 |

### 6.3 技术亮点

1. **MoE（Mixture of Experts）**：DeepSeek-V3 有 671B 参数，但每次推理只激活 37B，大幅降低计算成本
2. **Multi-head Latent Attention（MLA）**：创新的注意力机制，减少 KV Cache 的内存占用
3. **FP8 训练**：使用 FP8 混合精度训练，降低训练成本
4. **DeepSeek-R1**：通过强化学习训练的推理模型，能展示详细的思维链

### 6.4 行业影响

1. **降低门槛**：开源且低成本，让更多公司能使用高质量大模型
2. **推动竞争**：打破 OpenAI/Anthropic 的垄断，促进技术进步
3. **本地部署**：企业可以在自己的服务器上部署，解决数据隐私问题
4. **生态建设**：开源模型可以被二次开发和微调，形成丰富的生态

### 6.5 局限性

1. 在某些创意写作和复杂对话场景上仍不如 Claude/GPT-4
2. API 服务的稳定性和速度有时不如头部厂商
3. 多模态能力相对较弱（视觉、语音等）

## 七、FunctionCall 和 MCP 的区别

## 问题
Function Call 和 MCP（Model Context Protocol）有什么区别？

## 答案

**Function Call**：大模型调用预定义函数的能力。模型输出函数名和参数，应用层执行函数后把结果返回给模型继续对话。比如 ChatGPT 调用 `get_weather(city)`，应用层执行后返回天气数据。是模型层面的一种输出格式。

**MCP（Model Context Protocol）**：Anthropic 提出的开放协议，定义了 AI 模型与外部工具、数据源之间的标准通信方式。MCP 是协议层的标准，解决"AI 如何统一接入各种工具"的问题。一个 MCP Server 可以暴露资源、提示、工具三种能力，任何支持 MCP 的 AI 客户端都能直接调用。

**区别**：Function Call 是大模型的能力，让模型能输出结构化调用指令；MCP 是协议标准，让工具提供方能用统一格式暴露能力，AI 应用方能用统一方式接入。MCP 解决的是生态碎片化问题，Function Call 解决的是模型输出问题。
