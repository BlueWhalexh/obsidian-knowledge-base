# CMS 与 G1 垃圾回收器对比

## 问题
CMS 和 G1 垃圾回收器有什么区别？各自的回收流程是怎样的？如何选择垃圾回收器？

## 答案

### CMS 垃圾回收器

**设计目标**：响应时间优先，减少 STW 时间，适用于要求低延迟、低停顿的系统。针对**老年代**的垃圾回收器。

#### CMS 回收的 4 个阶段

```
初始标记(STW) → 并发标记 → 重新标记(STW) → 并发清理
```

| 阶段 | 是否 STW | 说明 |
|------|---------|------|
| **初始标记** | 是 | 标记 GC Roots 直接关联的对象，速度很快 |
| **并发标记** | 否 | 从 GC Roots 出发遍历整个对象图，标记所有可达对象。与用户线程并发执行 |
| **重新标记** | 是 | 处理并发标记阶段用户线程导致的引用变化，修正漏标对象 |
| **并发清理** | 否 | 清理未标记的垃圾对象，与用户线程并发执行 |

**达到响应时间优先的实现**：
1. 并发标记和并发清理阶段允许垃圾回收线程和用户线程同时工作
2. 采用**标记-清除算法**，速度快

#### CMS 的弊端

**1. 内存碎片问题**
- 原因：标记-清除算法不整理内存，长期运行后老年代会积累碎片
- 影响：即使老年代总空间足够，大对象分配也会失败，触发 Full GC

**2. 浮动垃圾问题**
- 定义：并发清理阶段用户线程继续产生的垃圾
- 影响：浮动垃圾过多导致老年代空间不足，并发模式失败，触发 Full GC

**3. 退化问题**
CMS 发生 Full GC 时会退化为 **Serial Old** 垃圾回收器（单线程、标记-整理算法、全程 STW），导致长时间停顿。

### G1 垃圾回收器

**设计目标**：全年代垃圾回收器，可设置每次垃圾回收 STW 的暂停时间目标，适用于超大堆内存、高吞吐量、低延迟的系统。JDK9 设为默认垃圾回收器。

#### Region 分区

G1 不再将堆简单分为新生代和老年代，而是将堆划分为若干个大小相等的 **Region**：

```
堆内存
├── Eden Region（动态数量）
├── Survivor Region（动态数量）
├── Old Region（动态数量）
└── Humongous Region（存放大对象）
```

"分区"取代"分代"，支持更灵活的回收策略。每个 Region 类型和数量动态变化。

#### G1 可预测 STW 的原理

G1 可以设置暂停时间目标（如 `-XX:MaxGCPauseMillis=200`）。实现原理：
- 每个 Region 都会产生垃圾，但垃圾量不同
- 回收时选择性回收**垃圾较多的 Region**（Garbage First 名称由来）
- 范围缩小节省时间，同时回收了较多垃圾
- 由此使 STW 时间接近设定的暂停时间目标

#### G1 回收的 4 个阶段

```
初始标记(STW) → 并发标记 → 最终标记(STW) → 混合回收(STW)
```

| 阶段 | 是否 STW | 说明 |
|------|---------|------|
| **初始标记** | 是 | 标记 GC Roots 直接关联的对象 |
| **并发标记** | 否 | 遍历整个对象图，标记所有可达对象，与用户线程并发执行 |
| **最终标记** | 是 | 处理并发标记阶段引用变化，确保标记准确性 |
| **混合回收** | 是 | 制定回收计划，选择新生代、老年代、大对象中的部分 Region 进行回收（复制存活对象到空 Region，清空原 Region） |

### CMS vs G1 对比表

| 维度 | CMS | G1 |
|------|-----|-----|
| **作用范围** | 老年代 | 全年代 |
| **内存结构** | 传统分代（新生代+老年代） | Region 分区 |
| **回收算法** | 标记-清除 | 标记-复制（整体看是标记-整理） |
| **内存碎片** | 有碎片问题 | 无碎片（复制算法整理） |
| **STW 控制** | 不可预测 | 可设置暂停时间目标 |
| **适用场景** | 中小堆、低延迟 | 超大堆（4G+）、可预测停顿 |
| **Full GC 退化** | 退化为 Serial Old | 无退化问题 |
| **JDK 版本** | JDK8 常用 | JDK9+ 默认 |

### 垃圾回收器选择指南

| 场景 | 推荐回收器 |
|------|-----------|
| 单核机器 / 小内存 | Serial GC |
| 批处理任务，注重吞吐量 | Parallel GC（JDK8 默认） |
| 低延迟、低停顿应用 | CMS（JDK8）或 G1（JDK9+） |
| 超大堆（8G+），可预测停顿 | G1 |
| 极低延迟（<10ms） | ZGC（JDK11+）或 Shenandoah（JDK12+） |
