# JIT 编译器与编译解释混合

## 问题
Java 是编译型还是解释型语言？javac 和 JIT 的区别是什么？什么是热点代码？C1 和 C2 编译器有什么区别？

## 答案

### 一、Java 是编译型还是解释型

Java 是**混合型**语言，既有编译过程也有解释过程：

```
.java 源码 → javac 编译 → .class 字节码 → JVM 运行
                                    ↓
                          ┌─────────────────────┐
                          │   解释器（逐条执行）   │ ← 简单代码、启动阶段
                          │   JIT 编译器（编译后执行）│ ← 热点代码、长期运行
                          └─────────────────────┘
```

| 阶段 | 工具 | 输入 | 输出 | 说明 |
|------|------|------|------|------|
| **编译期** | javac（Java 编译器） | .java 源码 | .class 字节码 | 静态编译，将源码编译为平台无关的字节码 |
| **运行期** | JVM（解释器 + JIT） | .class 字节码 | 机器码（JIT）/ 直接执行（解释） | 动态执行 |

所以严格来说，Java 源码到字节码是"编译"，字节码到机器码既有"解释"也有"编译"。

### 二、编译期：javac 编译器

#### 2.1 javac 的作用

javac 是 Java 的**前端编译器**，负责将 `.java` 源文件编译为 `.class` 字节码文件。

```bash
javac HelloWorld.java  # 编译
java HelloWorld         # 运行（JVM 加载 .class 字节码）
```

#### 2.2 javac 的编译过程

```
.java 源码
    │
    ▼
1. 词法分析 → Token 流
    │
    ▼
2. 语法分析 → 抽象语法树（AST）
    │
    ▼
3. 语义分析 → 填充符号表、类型检查、解语法糖
    │
    ▼
4. 字节码生成 → .class 文件
```

| 阶段 | 说明 |
|------|------|
| **词法分析** | 将源码字符流解析为 Token（关键字、标识符、运算符等） |
| **语法分析** | 根据 Token 流构建抽象语法树（AST） |
| **语义分析** | 类型检查、自动装箱/拆箱、泛型擦除、条件编译等 |
| **字节码生成** | 将 AST 转换为字节码指令，写入 .class 文件 |

#### 2.3 字节码（Bytecode）

字节码是一种中间表示（IR, Intermediate Representation），平台无关，可以在任何安装了 JVM 的平台上运行。

```
字节码示例（javap -c 反编译）：
public static void main(java.lang.String[]);
  Code:
     0: getstatic     #2    // 获取 System.out
     3: ldc           #3    // 加载常量 "Hello World"
     5: invokevirtual #4    // 调用 println 方法
     8: return
```

### 三、运行期：解释执行与 JIT 编译

#### 3.1 解释器（Interpreter）

解释器**逐条读取字节码指令并执行**，不生成本地机器码。

**优点**：
- 启动速度快（无需等待编译）。
- 内存占用小（不需要存储编译后的机器码）。
- 可以动态加载新的字节码。

**缺点**：
- 执行速度慢（每条指令都需要解释）。
- 无法进行编译优化。

**适用场景**：程序启动初期、简单应用、代码只执行一两次。

#### 3.2 JIT 编译器（Just-In-Time Compiler）

JIT 编译器将**热点代码**（Hot Spot Code，即被频繁执行的代码）编译为本地机器码，后续执行时直接运行机器码，无需再次解释。

**优点**：
- 执行速度快（直接运行机器码）。
- 可以进行各种编译优化。

**缺点**：
- 需要预热（热点代码需要执行一定次数后才会被编译）。
- 编译本身需要时间和内存。

**适用场景**：长期运行的服务、热点代码。

#### 3.3 解释器与 JIT 的协同工作

```
字节码
  │
  ├──→ 解释器 ──→ 执行（慢，但立即可用）
  │         │
  │         ▼
  │     热点代码检测
  │         │
  │         ▼
  └──→ JIT 编译器 ──→ 本地机器码 ──→ 执行（快，已编译）
```

JVM 启动后，先通过解释器执行字节码。当某段代码被检测为"热点代码"时，JIT 编译器将其编译为本地机器码并缓存。后续执行该代码时直接运行编译后的机器码。

#### 3.4 编译优化手段

JIT 编译器可以进行多种优化，远超 javac 的能力：

| 优化手段 | 说明 |
|----------|------|
| **方法内联** | 将被调用方法的代码直接展开到调用处，减少方法调用开销 |
| **逃逸分析** | 分析对象是否逃逸，实现栈上分配、标量替换、锁消除 |
| **循环展开** | 减少循环控制指令的开销 |
| **公共子表达式消除** | 避免重复计算相同的表达式 |
| **死代码消除** | 删除永远不会执行的代码 |
| **空值检查消除** | 如果编译器能证明某处不会为空，消除空值检查 |
| **范围检查消除** | 如果编译器能证明数组访问不越界，消除边界检查 |
| **寄存器分配** | 将热点变量分配到 CPU 寄存器中 |
| **指令重排序** | 在不改变语义的前提下优化指令顺序 |

### 四、JIT 编译器：C1 与 C2

HotSpot JVM 内置了两个 JIT 编译器：

#### 4.1 C1 编译器（Client Compiler）

| 特性 | 说明 |
|------|------|
| **设计目标** | 快速编译，较少优化 |
| **编译速度** | 快 |
| **优化程度** | 低（简单优化） |
| **适用场景** | 客户端应用、GUI 程序 |
| **优化手段** | 方法内联、去虚拟化、冗余消除等简单优化 |

C1 编译器注重**编译速度**，适合启动时间敏感的应用。

#### 4.2 C2 编译器（Server Compiler / Opto）

| 特性 | 说明 |
|------|------|
| **设计目标** | 深度优化，最大化运行性能 |
| **编译速度** | 慢（需要更多分析时间） |
| **优化程度** | 高（激进优化） |
| **适用场景** | 服务端应用、长期运行的服务 |
| **优化手段** | 逃逸分析、标量替换、循环展开、向量化等深度优化 |

C2 编译器注重**运行性能**，适合长期运行的服务端应用。

#### 4.3 C1 与 C2 的对比

| 维度 | C1（客户端） | C2（服务端） |
|------|-------------|-------------|
| **编译速度** | 快 | 慢 |
| **优化程度** | 低 | 高 |
| **编译阈值** | 较低（更容易触发编译） | 较高 |
| **适合场景** | GUI 应用、短生命周期程序 | 服务端、长生命周期程序 |
| **内联深度** | 较浅 | 较深 |
| **逃逸分析** | 不支持（或有限支持） | 完整支持 |
| **运行时性能** | 较低 | 较高 |

### 五、分层编译（Tiered Compilation）

JDK 8 默认开启了**分层编译**，将 C1 和 C2 的优势结合起来：

```bash
# 开启分层编译（默认开启）
-XX:+TieredCompilation

# 关闭分层编译
-XX:-TieredCompilation
```

#### 5.1 分层编译的 5 个层级

| 层级 | 执行方式 | 说明 |
|------|----------|------|
| **0** | 解释执行 | 不编译，收集性能监控数据（Profile） |
| **1** | C1 编译（无 Profile） | 简单编译，不收集性能数据 |
| **2** | C1 编译（有限 Profile） | 编译并收集有限的性能数据（方法调用和回边计数） |
| **3** | C1 编译（完整 Profile） | 编译并收集完整的性能数据 |
| **4** | C2 编译 | 深度优化编译 |

#### 5.2 分层编译的工作流程

```
代码首次执行 → 层级0（解释执行 + 收集Profile）
                    │
                    ▼
              达到阈值 → 层级3（C1编译 + 完整Profile）
                              │
                              ▼
                        达到阈值 → 层级4（C2深度优化编译）
```

**工作流程**：
1. 代码首次执行时，通过解释器（层级 0）执行，同时收集性能数据（Profile）。
2. 当执行次数达到 C1 编译阈值时，C1 编译器使用收集到的 Profile 进行初步编译（层级 3）。
3. 当执行次数达到 C2 编译阈值时，C2 编译器使用更完整的 Profile 进行深度优化编译（层级 4）。
4. 最终热点代码以 C2 编译的机器码运行。

#### 5.3 分层编译的优势

- **启动快**：初期用解释执行和 C1 快速编译，不需要等 C2 慢慢编译。
- **性能高**：长期运行后，热点代码由 C2 深度优化编译。
- **Profile 数据共享**：C1 编译时收集的 Profile 数据可以传递给 C2，避免 C2 重复收集。

### 六、热点代码检测

#### 6.1 什么是热点代码

热点代码是指被**频繁执行**的代码，分为两种：
1. **被多次调用的方法**：方法级别的热点。
2. **被多次执行的循环体**：循环级别的热点（即使方法本身调用次数不多，但循环次数很多）。

#### 6.2 热点代码的检测方式

**1. 基于采样的热点探测**

- JVM 周期性地检查各个线程的调用栈。
- 如果某个方法经常出现在栈顶，则认为它是热点方法。
- 优点：实现简单，可以获取方法调用关系。
- 缺点：受线程阻塞等因素影响，不够精确。

**2. 基于计数器的热点探测（HotSpot 使用）**

HotSpot JVM 为每个方法维护两个计数器：

| 计数器 | 作用 | 阈值（C2） |
|--------|------|-----------|
| **方法调用计数器**（Invocation Counter） | 记录方法被调用的次数 | 10000 次 |
| **回边计数器**（Back Edge Counter） | 记录循环体中代码的执行次数（回边 = 循环跳回开始处） | 10700 次 |

**方法调用计数器的工作流程**：

```
方法被调用
    │
    ▼
方法调用计数器 + 1
    │
    ├── 达到阈值 → 提交 JIT 编译请求 → C1/C2 编译
    │
    └── 未达到阈值 → 继续解释执行

注意：方法调用计数器统计的是方法的调用频率（一段时间内的调用次数）
      超过一定时间未达到阈值，计数器会减半（热度衰减）
```

**回边计数器的工作流程**：

```
循环回边执行
    │
    ▼
回边计数器 + 1
    │
    ├── 达到阈值 → 提交 OSR 编译请求（On-Stack Replacement）
    │
    └── 未达到阈值 → 继续解释执行
```

#### 6.3 OSR（On-Stack Replacement）

OSR 是一种特殊的编译方式：在方法**执行过程中**进行编译替换。

当一个循环体被频繁执行时，不需要等整个方法再次被调用，而是在**当前方法的栈帧中**，将正在解释执行的循环体替换为编译后的机器码继续执行。

```
方法执行中 → 进入循环 → 回边计数器达到阈值
    │
    ▼
OSR 编译循环体 → 用编译后的代码替换当前执行位置 → 继续执行
```

#### 6.4 热点代码编译的相关参数

```bash
# 方法调用计数器阈值（分层编译时由 JVM 内部管理）
-XX:CompileThreshold=10000  # 默认 10000 次

# 回边计数器阈值
-XX:OnStackReplacePercentage=140  # 默认值

# 是否使用热度衰减（默认开启）
-XX:+UseCounterDecay

# 热度衰减周期（默认不衰减，分层编译时由 JVM 管理）
-XX:CounterHalfLifeTime=<time>
```

### 七、AOT 编译（Ahead-Of-Time Compilation）

除了 JIT，JDK 9 还引入了 **AOT 编译**（Graal 编译器）：

| 维度 | JIT | AOT |
|------|-----|-----|
| **编译时机** | 运行时 | 编译期/安装时 |
| **启动速度** | 慢（需要预热） | 快（直接运行机器码） |
| **运行性能** | 高（基于运行时 Profile 的优化） | 中（缺少运行时信息） |
| **适用场景** | 长期运行的服务 | 启动敏感的应用、微服务、Serverless |

```bash
# JDK 9+ 使用 jaotc 工具
jaotc --output libHelloWorld.so HelloWorld.class
```

### 八、总结

```
Java 执行流程：
.java → javac → .class → JVM
                          ├── 解释器：逐条执行（启动快，性能低）
                          ├── C1 JIT：快速编译（中等优化）
                          ├── C2 JIT：深度编译（最大优化）
                          └── 分层编译：解释 → C1 → C2（最佳平衡）
```

| 概念 | 说明 |
|------|------|
| **javac** | 前端编译器，源码→字节码 |
| **解释器** | 逐条执行字节码，启动快 |
| **JIT** | 将热点代码编译为机器码，运行快 |
| **C1** | 客户端编译器，编译快，优化少 |
| **C2** | 服务端编译器，编译慢，优化多 |
| **分层编译** | 结合 C1 和 C2 的优势 |
| **热点代码** | 方法调用计数器 + 回边计数器检测 |
| **OSR** | 方法执行中替换为编译后的代码 |
