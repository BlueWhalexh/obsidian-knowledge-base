# JVM 调优实战经验

## 问题
JVM 可以调哪些参数？为什么要调优？Eden 和 Survivor 大小有什么影响？实际调优案例如何？

## 答案

### 可调参数

| 参数类别 | 具体参数 |
|---------|---------|
| 堆内存 | 初始大小（`-Xms`）、最大大小（`-Xmx`） |
| 年轻代 | 初始大小（`-Xmn`）、最大大小 |
| 分代比例 | 年轻代与老年代比例（默认 1:2） |
| Eden/Survivor 比例 | Eden:S0:S1（默认 8:1:1） |
| 元空间 | 初始大小（`-XX:MetaspaceSize`）、最大大小（`-XX:MaxMetaspaceSize`） |
| 垃圾回收器 | 更换 GC（如 CMS、G1、ZGC） |
| 暂停时间目标 | `-XX:MaxGCPauseMillis`（G1） |

### 为什么需要调优

区域的内存大小会直接影响程序运行效果：

#### Eden 区大小的影响

| 情况 | 影响 |
|------|------|
| Eden **很小** | YoungGC **频繁发生**（空间小，存不下东西，频繁清理） |
| Eden **很大** | YoungGC **耗时很久**（空间大，垃圾多，收集耗时） |

YoungGC 是全程 STW 的，GC 期间外界请求无法响应。例如请求处理到 5ms 时发生 YoungGC（耗时 50ms），则响应时间为 60ms。需要通过调整 Eden 区大小来优化。

#### Survivor 区大小的影响

| 情况 | 影响 |
|------|------|
| Survivor **很小** | 存活对象直接晋升到老年代，不该晋升的晋升了，最终导致 **Full GC** |
| Survivor **很大** | Survivor 区被一分为二（一半始终不用），浪费内存 |

### 面试时如何回答

**1. 默认参数一般不需要调优**
JVM 默认参数是官方 JVM 团队长期试验的结果，大多数场景下无需调优。

**2. 配置监控告警**
对 JVM 核心指标配置监控告警，当出现波动时人为介入分析评估：
- GC 耗时过长 → 告警
- GC 次数过多 → 告警

这是大厂规范的执行标准：线上服务稳定性建设，配置监控告警。

**3. 实际调优案例**

---

#### 案例一：CMS 内存碎片导致 Full GC

**环境**：ParNew + CMS + JDK8

**问题现象**：C 端核心业务每个接口请求耗时限制在 100ms 以内，高峰期服务器发生 Full GC（耗时可达 1s），导致部分请求超时报错。

**原因分析**：CMS 使用标记-清除算法，老年代随着应用运行会碎片化。碎片过多影响对象分配，虽然老年代还有很大剩余空间，但没有连续空间来分配对象，最终导致 Full GC。

**优化策略**：业务低峰期（如凌晨四点）显式触发 Full GC（`System.gc()`）。CMS 触发 Full GC 会退化为 Serial Old（标记-整理算法），可以整理堆内存消除碎片，降低高峰期 Full GC 概率。

**优化效果**：高峰期基本没有出现 Full GC。

**总结**：CMS 追求响应时间（标记-清除快），但引入碎片问题。结合业务有高低峰期，在低峰期整理碎片，保证高峰期正常运行。

---

#### 案例二：年轻代过小导致频繁 GC

**环境**：ParNew + CMS + JDK8

**问题现象**：YoungGC 每分钟 50 次，每次 25ms；OldGC 几分钟 1 次，每次 200ms。

**原因分析**：为追求 YGC 较快完成（YGC 全程 STW），年轻代设置较小。但年轻代小导致 YGC 频繁，连锁反应导致大量对象较早晋升到老年代，Old GC 也频繁。随着业务增长，原来的年轻代大小已不合适。

**优化策略**：扩大年轻代内存到原来的 3 倍。

**优化效果**：单次 YGC 耗时增加 5%，YGC 频率降低 60%，Old GC 频率降为几小时 1 次。

**解释**：年轻代扩大 3 倍不代表 YGC 耗时变为 3 倍。YGC 是复制算法，耗时在于复制阶段，大部分对象被回收了，所以单次 YGC 耗时只是轻微增加。

---

#### 案例三：G1 暂停时间目标过小

**环境**：G1 + JDK8

**问题现象**：YoungGC 频繁。

**原因分析**：`-XX:MaxGCPauseMillis` 暂停时间目标参数设置较小，导致 JVM 自动调整降低年轻代的 Region 数量。

**优化策略**（二选一）：
1. 调大 `-XX:MaxGCPauseMillis` 暂停时间目标参数
2. 将年轻代 Region 大小设置为固定值

### CMS 中的 Full GC 和 Old GC

- **Old GC**：仅针对老年代的垃圾回收
- **Full GC**：涉及整个堆内存（包括年轻代、老年代、元空间）
- CMS 中 Full GC 可能在 Old GC 回收失败时触发
- 元空间不在堆中，但元空间内存不足也会触发 Full GC（`OutOfMemoryError: Metaspace` 的垃圾回收依赖 Full GC）
