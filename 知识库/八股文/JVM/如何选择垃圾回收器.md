# 如何选择垃圾回收器

## 问题
JVM 有哪些垃圾回收器？各自有什么特点？如何根据场景选择合适的垃圾回收器？JDK 默认使用哪个？

## 答案

### 一、垃圾回收器概览

JVM 提供了多种垃圾回收器，适用于不同的应用场景。按代际可以分为：

```
新生代回收器              老年代回收器              整堆回收器
├── Serial               ├── Serial Old           ├── G1
├── Parallel Scavenge    ├── CMS                  ├── ZGC（JDK 11+）
└── ParNew               └── Parallel Old         └── Shenandoah（JDK 12+）
```

按发展时间线：

```
Serial → Parallel → CMS → G1 → ZGC
  │         │        │     │     │
  串行      并行     并发   分区   超低延迟
```

回收器的组合使用关系：

| 新生代 | 老年代 | 说明 |
|--------|--------|------|
| Serial | Serial Old | 单线程，简单高效 |
| ParNew | CMS | 低延迟 |
| Parallel Scavenge | Serial Old | JDK 8 默认 |
| Parallel Scavenge | Parallel Old | 吞吐量优先 |
| G1 | G1 | 分 Region，JDK 9+ 默认 |
| ZGC | ZGC | 超低延迟 |

---

### 二、各回收器详细介绍

#### 2.1 Serial 收集器

| 特性 | 说明 |
|------|------|
| **代际** | 新生代 |
| **算法** | 复制算法 |
| **线程** | 单线程 |
| **STW** | 需要，GC 时暂停所有应用线程 |
| **适用场景** | 客户端应用、小堆（几十 MB） |
| **优点** | 简单高效，没有线程交互开销 |
| **缺点** | STW 时间长 |

Serial 是最基础、最古老的收集器。单线程意味着 GC 时只使用一个 CPU 或一个收集线程，且必须暂停所有工作线程（STW）。

```bash
# 使用 Serial + Serial Old
-XX:+UseSerialGC
```

#### 2.2 Serial Old 收集器

| 特性 | 说明 |
|------|------|
| **代际** | 老年代 |
| **算法** | 标记-整理（Mark-Compact） |
| **线程** | 单线程 |
| **适用场景** | 客户端模式、CMS 的后备方案 |

#### 2.3 ParNew 收集器

| 特性 | 说明 |
|------|------|
| **代际** | 新生代 |
| **算法** | 复制算法 |
| **线程** | 多线程 |
| **STW** | 需要 |
| **适用场景** | 与 CMS 搭配使用 |
| **优点** | 多线程并行 GC，减少 STW 时间 |
| **缺点** | 只能与 CMS 搭配（JDK 9 之前） |

ParNew 本质上是 Serial 的多线程版本，使用多个 GC 线程并行回收新生代。

```bash
# 使用 ParNew + CMS
-XX:+UseParNewGC -XX:+UseConcMarkSweepGC
```

#### 2.4 Parallel Scavenge 收集器

| 特性 | 说明 |
|------|------|
| **代际** | 新生代 |
| **算法** | 复制算法 |
| **线程** | 多线程 |
| **STW** | 需要 |
| **目标** | **吞吐量最大化**（而非停顿时间最小化） |
| **适用场景** | 后台计算、批处理 |

**吞吐量** = 用户代码运行时间 / (用户代码运行时间 + GC 时间)

Parallel Scavenge 与 ParNew 的区别：
- Parallel Scavenge 关注吞吐量，ParNew 关注停顿时间。
- Parallel Scavenge 支持自适应调节策略（`-XX:+UseAdaptiveSizePolicy`）。

```bash
# 设置吞吐量目标（默认 99，即 99% 的时间用于用户代码）
-XX:GCTimeRatio=99

# 设置最大停顿时间目标（默认不设置）
-XX:MaxGCPauseMillis=<ms>

# 开启自适应调节（自动调整新生代大小、Eden/Survivor 比例等）
-XX:+UseAdaptiveSizePolicy
```

#### 2.5 Parallel Old 收集器

| 特性 | 说明 |
|------|------|
| **代际** | 老年代 |
| **算法** | 标记-整理（Mark-Compact） |
| **线程** | 多线程 |
| **适用场景** | 与 Parallel Scavenge 搭配，JDK 8 默认组合 |

```bash
# 使用 Parallel Scavenge + Parallel Old（JDK 8 默认）
-XX:+UseParallelGC
```

#### 2.6 CMS 收集器（Concurrent Mark Sweep）

| 特性 | 说明 |
|------|------|
| **代际** | 老年代 |
| **算法** | **标记-清除**（Mark-Sweep） |
| **线程** | 并发（GC 线程与用户线程同时工作） |
| **目标** | **最短停顿时间** |
| **适用场景** | 交互式应用、Web 服务 |
| **缺点** | 内存碎片、CPU 敏感、浮动垃圾、并发失败 |

**CMS 的四个阶段**：

```
阶段1：初始标记（Initial Mark）     — STW，标记 GC Roots 直接关联的对象（快）
阶段2：并发标记（Concurrent Mark）   — 与用户线程并发，遍历引用链（慢但不停顿）
阶段3：重新标记（Remark）           — STW，修正并发标记期间变动的对象（较快）
阶段4：并发清除（Concurrent Sweep）  — 与用户线程并发，清除垃圾（慢但不停顿）
```

| 阶段 | STW | 说明 |
|------|-----|------|
| 初始标记 | 是 | 速度很快，只标记 GC Root 直接关联的对象 |
| 并发标记 | 否 | 耗时最长，与用户线程并发执行 |
| 重新标记 | 是 | 修正并发标记期间的变动（增量更新） |
| 并发清除 | 否 | 清除已标记的垃圾对象 |

**CMS 的缺点**：

| 缺点 | 说明 |
|------|------|
| **内存碎片** | 使用标记-清除算法，会产生内存碎片。可通过 `-XX:+UseCMSCompactAtFullCollection` 在 Full GC 时压缩 |
| **CPU 敏感** | 并发阶段会占用 CPU 资源，降低吞吐量。默认 GC 线程数 = (CPU 核心数 + 3) / 4 |
| **浮动垃圾** | 并发清除阶段用户线程产生的新垃圾无法在本次 GC 中清除，需要下次 GC 回收 |
| **并发失败** | 如果并发阶段用户线程产生的新对象太快，老年代空间不足，会退化为 Serial Old 进行 Full GC |

```bash
# 使用 CMS
-XX:+UseConcMarkSweepGC

# 设置 CMS 触发阈值（老年代使用率达到多少触发 CMS GC，默认 92%，建议 75%）
-XX:CMSInitiatingOccupancyFraction=75

# 开启 CMS 碎片整理
-XX:+UseCMSCompactAtFullCollection
-XX:CMSFullGCsBeforeCompaction=0  # 每次 Full GC 都压缩
```

#### 2.7 G1 收集器（Garbage First）

| 特性 | 说明 |
|------|------|
| **代际** | 整堆（不分新生代/老年代物理隔离） |
| **算法** | 整体标记-整理，Region 间复制 |
| **线程** | 并发 + 并行 |
| **目标** | **可控的停顿时间** + 高吞吐量 |
| **适用场景** | 大堆（6GB+）、需要可控停顿时间 |
| **JDK 默认** | JDK 9+ 默认回收器 |

**G1 的核心设计**：

G1 将堆划分为多个大小相等的 **Region**（默认 2048 个），每个 Region 可以是 Eden、Survivor、Old 或 Humongous（存储大对象）。

```
┌──────┬──────┬──────┬──────┬──────┬──────┐
│ Eden │ Old  │Surv. │ Old  │ Eden │ Hum. │
├──────┼──────┼──────┼──────┼──────┼──────┤
│ Old  │ Eden │ Old  │Surv. │ Old  │ Eden │
├──────┼──────┼──────┼──────┼──────┼──────┤
│ Eden │ Old  │ Eden │ Old  │ Eden │ Old  │
└──────┴──────┴──────┴──────┴──────┴──────┘
```

**G1 的回收过程（Mixed GC）**：

```
阶段1：初始标记（Initial Mark）          — STW，标记 GC Roots 直接关联的对象
阶段2：并发标记（Concurrent Mark）        — 与用户线程并发，遍历引用链
阶段3：最终标记（Final Mark）            — STW，处理并发标记遗留的 SATB 记录
阶段4：筛选回收（Cleanup/Evacuation）    — STW，选择回收价值最高的 Region，复制存活对象
```

**G1 的特点**：

| 特点 | 说明 |
|------|------|
| **可预测停顿** | 通过 `-XX:MaxGCPauseMillis` 设置目标停顿时间，G1 会优先回收垃圾最多的 Region |
| **Region 化** | 不需要整堆回收，可以按 Region 粒度回收 |
| **Humongous 对象** | 大于 Region 50% 的对象存放在 Humongous Region |
| **SATB** | 使用原始快照（Snapshot At The Beginning）解决并发标记的漏标问题 |
| **无内存碎片** | 复制算法在 Region 间移动对象，不会产生碎片 |
| **优先回收价值高的 Region** | 优先回收垃圾最多的 Region（Garbage First 的由来） |

```bash
# 使用 G1（JDK 9+ 默认）
-XX:+UseG1GC

# 设置目标停顿时间（默认 200ms）
-XX:MaxGCPauseMillis=200

# 设置 Region 大小（1-32MB，必须是 2 的幂）
-XX:G1HeapRegionSize=16m

# 触发并发标记的堆使用率阈值
-XX:InitiatingHeapOccupancyPercent=45

# 新生代占比范围
-XX:G1NewSizePercent=5
-XX:G1MaxNewSizePercent=60
```

#### 2.8 ZGC 收集器（Z Garbage Collector）

| 特性 | 说明 |
|------|------|
| **代际** | 整堆 |
| **算法** | 染色指针 + 读屏障 + 内存多重映射 |
| **线程** | 并发（几乎所有阶段都并发） |
| **目标** | **超低停顿时间**（< 10ms，通常 < 1ms） |
| **适用场景** | 超大堆（TB 级别）、超低延迟要求 |
| **JDK 版本** | JDK 11 实验性引入，JDK 15 正式可用 |

**ZGC 的核心技术**：

| 技术 | 说明 |
|------|------|
| **染色指针（Colored Pointer）** | 在 64 位指针中借用几个 bit 来存储 GC 状态信息（Marked0/Marked1/Remapped/Finalizable） |
| **读屏障（Load Barrier）** | 在读取对象引用时检查指针的 GC 状态，必要时修正引用 |
| **内存多重映射** | 将同一块物理内存映射到多个虚拟地址，支持染色指针 |

**ZGC 的工作流程**：

```
阶段1：初始标记（Initial Mark）       — STW（< 1ms）
阶段2：并发标记（Concurrent Mark）     — 并发
阶段3：并发预备重分配（Concurrent Prepare for Relocate） — 并发
阶段4：并发重分配（Concurrent Relocate） — 并发
阶段5：并发重映射（Concurrent Remap）    — 并发
```

**ZGC 的优势**：
- 停顿时间极短（< 1ms），不随堆大小增长
- 支持 TB 级堆内存
- 吞吐量损失较小（约 15%）

```bash
# 使用 ZGC（JDK 15+）
-XX:+UseZGC

# JDK 21+ 分代 ZGC
-XX:+UseZGC -XX:+ZGenerational

# 主动回收间隔（秒）
-XX:ZCollectionInterval=5
```

#### 2.9 Shenandoah 收集器

| 特性 | 说明 |
|------|------|
| **代际** | 整堆 |
| **算法** | 转发指针 + 读屏障 |
| **目标** | 超低停顿时间（与 ZGC 类似） |
| **JDK 版本** | JDK 12+（OpenJDK 特有，Oracle JDK 不包含） |
| **与 ZGC 的区别** | Shenandoah 更注重低延迟，ZGC 更注重低延迟 + 吞吐量的平衡 |

### 三、回收器对比总结

| 对比项 | Serial | Parallel | CMS | G1 | ZGC |
|--------|--------|----------|-----|-----|-----|
| **算法** | 复制/标记-整理 | 复制/标记-整理 | 标记-清除 | 复制+标记-整理 | 着色指针+读屏障 |
| **线程** | 单线程 | 多线程并行 | 并发 | 并发+并行 | 并发 |
| **目标** | 简单高效 | 吞吐量优先 | 低延迟 | 兼顾吞吐和延迟 | 超低延迟 |
| **停顿时间** | 长 | 中等 | 短 | 可控 | 极短（< 1ms） |
| **堆大小** | 小堆 | 中等 | 中等 | 大堆（4GB+） | 超大堆（TB级） |
| **内存碎片** | 无（复制/整理） | 无（复制/整理） | 有（标记-清除） | 无（复制） | 无 |
| **JDK 版本** | JDK 1.3 | JDK 1.4 | JDK 1.5 | JDK 7u4 | JDK 11（实验）/15 |

### 四、如何选择垃圾回收器

#### 4.1 根据堆大小选择

| 堆大小 | 推荐回收器 | 原因 |
|--------|-----------|------|
| < 100MB | Serial | 单线程足够，开销小 |
| 100MB ~ 4GB | Parallel | 并行回收效率高 |
| 4GB ~ 16GB | G1 | Region 机制，可预测停顿 |
| > 16GB | G1 或 ZGC | 大堆需要并发收集 |
| > 100GB | ZGC | 停顿时间不随堆增长 |

#### 4.2 根据应用场景选择

| 应用场景 | 推荐回收器 | 原因 |
|----------|-----------|------|
| **小型应用 / 客户端** | Serial + Serial Old | 简单高效，资源受限 |
| **后台计算 / 批处理** | Parallel Scavenge + Parallel Old | 吞吐量优先 |
| **Web 服务 / 微服务（JDK 8）** | ParNew + CMS | 低延迟 |
| **Web 服务 / 微服务（JDK 9+）** | G1 | 兼顾吞吐和延迟 |
| **实时交易系统** | ZGC 或 G1 | 低延迟要求 |
| **大数据平台** | G1 或 ZGC | 大堆 + 低延迟 |
| **超大堆 / 超低延迟** | ZGC | STW < 1ms |

#### 4.3 根据关注指标选择

| 关注指标 | 推荐回收器 | 说明 |
|----------|-----------|------|
| **吞吐量** | Parallel | 用户代码执行时间占比最高 |
| **响应时间** | G1 / ZGC | 单次停顿时间短 |
| **堆内存大小** | G1 / ZGC | 支持大堆 |
| **简单性** | Serial | 最简单，调优参数少 |

### 五、JDK 默认垃圾回收器变迁

| JDK 版本 | 默认回收器 | 说明 |
|----------|-----------|------|
| **JDK 7** | Parallel Scavenge + Serial Old | 吞吐量优先 |
| **JDK 8** | Parallel Scavenge + Parallel Old | 吞吐量优先 |
| **JDK 9** | G1 | 低延迟 + 平衡 |
| **JDK 11** | G1 | ZGC 实验性引入 |
| **JDK 15** | G1 | ZGC 正式可用 |
| **JDK 21** | G1（ZGC 作为备选） | ZGC 分代模式成熟 |

```bash
# 查看当前 JDK 默认使用的回收器
java -XX:+PrintCommandLineFlags -version

# JDK 8 输出：
# -XX:+UseParallelGC

# JDK 11 输出：
# -XX:+UseG1GC
```

### 六、常用调优参数

```bash
# 1. 选择回收器
-XX:+UseSerialGC          # Serial + Serial Old
-XX:+UseParallelGC        # Parallel Scavenge + Parallel Old（JDK 8 默认）
-XX:+UseConcMarkSweepGC   # ParNew + CMS
-XX:+UseG1GC              # G1（JDK 9+ 默认）
-XX:+UseZGC               # ZGC（JDK 15+）

# 2. 通用参数
-Xms<size>                 # 初始堆大小
-Xmx<size>                 # 最大堆大小
-Xmn<size>                 # 新生代大小
-XX:SurvivorRatio=<ratio>  # Eden/Survivor 比例
-XX:MaxGCPauseMillis=<ms>  # 目标最大停顿时间
-XX:GCTimeRatio=<ratio>    # 吞吐量目标

# 3. 打印 GC 日志
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/path/to/gc.log

# JDK 9+ 统一日志框架：
-Xlog:gc*:file=gc.log:time,uptime,level,tags

# 4. GC 日志分析工具
# GCEasy: https://gceasy.io
# GCViewer: 开源 GC 日志分析工具
```

### 七、如何判断是否需要更换回收器

**1. 监控 GC 停顿时间**：
```bash
# 如果 GC 停顿时间超过业务容忍范围（如 > 200ms），考虑换 G1 或 ZGC
```

**2. 监控 GC 频率**：
```bash
# 如果 Minor GC 频率过高（如每秒 > 1 次），考虑增大新生代
# 如果 Full GC 频率过高（如每天 > 1 次），考虑换 G1 或排查内存泄漏
```

**3. 监控吞吐量**：
```bash
# 如果 GC 时间占比过高（如 > 5%），考虑换 Parallel 或增大堆
```

### 八、实战建议

1. **先看 JDK 版本**：JDK 8 用 Parallel 或 CMS，JDK 9+ 默认 G1 通常不需要更换。
2. **看应用类型**：交互式应用优先考虑延迟，批处理优先考虑吞吐量。
3. **看堆大小**：小堆用 Parallel，大堆用 G1，超大堆用 ZGC。
4. **先监控再调优**：通过 GC 日志和监控工具分析 GC 行为，再针对性调优。
5. **不要过度调优**：JVM 的默认值在大部分场景下已经足够好，只有在明确观察到 GC 问题时才需要调优。
