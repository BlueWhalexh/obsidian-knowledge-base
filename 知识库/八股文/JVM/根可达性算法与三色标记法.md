# 根可达性算法与三色标记法

## 问题
如何判断对象是否是垃圾？什么是三色标记法？并发标记的漏标问题如何解决？多标/浮动垃圾如何解决？

## 答案

### 一、根可达性算法

使用根可达性算法判断对象是否是垃圾。

**判断方式**：以 GC Root 为起点，沿着引用链进行查找。能找到的对象即不可回收（存活），不在 GC Root 引用链上的对象即可视为垃圾。

#### 1.1 四类 GC Root

| 类型 | 说明 |
|------|------|
| **系统类的实例对象** | 核心类库中存活的类对象（如 ClassLoader 加载的类） |
| **本地方法调用相关的实例对象** | JNI 引用的对象（native 方法中引用的对象） |
| **当前活动线程相关的实例对象** | 各线程调用栈中引用的对象（局部变量、参数等） |
| **被加锁的实例对象** | 被 synchronized 锁持有的对象 |

#### 1.2 GC Root 的本质

GC Root 本质上是一组**必须存活**的引用起点。从这些起点出发，通过引用链可达的对象就是存活对象，不可达的就是垃圾。

```
GC Root 1 ──→ Object A ──→ Object B ──→ Object C（存活）
                                          
GC Root 2 ──→ Object D（存活）

Object E ──→ Object F（垃圾，没有 GC Root 可达）
```

### 二、三色标记法

#### 2.1 背景与定义

根可达性算法是理论基础，定义了"如何判断对象存活"的规则。三色标记法是实现手段，是实现上述规则的**高效遍历算法**。

正常标记过程需要 STW（对象引用关系不变才能标记正确），但 STW 时间可能很长。三色标记法允许并发标记（用户线程同时工作），减少停顿时间。

#### 2.2 三种颜色的含义

| 颜色 | 含义 |
|------|------|
| **白色** | 该对象没有被标记过（最终仍为白色的对象将被回收） |
| **灰色** | 该对象已经被标记过，但该对象引用的其他对象还没标记完 |
| **黑色** | 该对象已经被标记过，并且它的全部引用对象也都标记完了 |

#### 2.3 标记流程

```
初始标记(STW) → 并发标记 → 重新标记(STW) → 并发清除
```

1. **初始标记（STW）**：遍历所有 GC Root 根对象，将根对象和直接引用的对象标记为灰色。不扫描整个堆，时间很短。

2. **并发标记（不需要 STW）**：从灰色对象开始遍历，将灰色对象引用的所有白色对象标记为灰色，然后将该灰色对象标记为黑色。不断重复，直到没有更多灰色对象。过程中应用程序线程可能修改对象图，需使用写屏障技术跟踪引用变更。

3. **重新标记（STW）**：处理并发阶段变更的引用关系。修正并发标记期间因用户线程修改引用而导致的标记不一致。

4. **并发清除**：回收所有仍为白色的对象（垃圾）。

#### 2.4 标记过程示例

```
初始状态：
所有对象都是白色

初始标记（STW）：
GC Root 直接引用的对象标记为灰色

并发标记：
灰色对象 A → A 引用的对象 B 标记为灰色，A 变为黑色
灰色对象 B → B 引用的对象 C 标记为灰色，B 变为黑色
...
直到没有灰色对象

重新标记（STW）：
修正并发期间引用变更导致的标记错误

并发清除：
所有白色对象（不可达）被回收
```

### 三、漏标问题（对象消失问题）

#### 3.1 什么是漏标

漏标是指在并发标记过程中，一个**原本应该被标记为存活的对象**（应该被标记为黑色或灰色），因为用户线程修改了引用关系，导致该对象最终被错误地标记为白色，从而被错误地回收。

#### 3.2 漏标的产生条件

**必须同时满足两个条件**才会发生漏标：

1. **条件一**：黑色对象新增了到白色对象的引用（黑色对象已完成扫描，不会再次被处理）
2. **条件二**：灰色对象到白色对象的所有路径被切断（白色对象不再被任何灰色对象引用，无法通过灰色对象被标记）

**单独一个条件发生不会引发问题**：
- 只有条件一：黑色对象引用了白色对象，但灰色对象仍然引用了该白色对象。灰色对象处理时会标记该白色对象。
- 只有条件二：灰色对象断开了对白色对象的引用，但没有黑色对象引用该白色对象。白色对象本来就是不可达的，应该被回收。

**本质**：需要同时满足两个条件。解决方案的核心思想是**记录下其中一个条件的变化，然后在重新标记阶段重新判断**。

#### 3.3 漏标的示例

```
初始状态（并发标记前）：
GC Root ──→ A(灰色) ──→ B(白色)
           └──────────→ C(白色)

并发标记过程中，用户线程做了两件事：
1. A.b = null      （A 断开了对 B 的引用，满足条件二）
2. D.color = B     （黑色对象 D 新增了对 B 的引用，满足条件一）

此时：
- A 从灰色变为黑色（标记完成）
- B 没有被任何灰色对象引用（条件二满足）
- B 被黑色对象 D 引用（条件一满足，但 D 不会再被扫描）
- 结果：B 没有被标记，最终被错误回收！
```

#### 3.4 漏标的解决方案

解决漏标问题的核心思路是**破坏两个条件中的任意一个**：

| 回收器 | 方案 | 破坏的条件 | 原理 |
|--------|------|-----------|------|
| **CMS** | **增量更新**（Incremental Update） | 条件一 | 当黑色对象新增指向白色对象的引用时，记录该引用关系。重新标记阶段根据记录重新扫描这些黑色对象 |
| **G1** | **原始快照**（SATB, Snapshot At The Beginning） | 条件二 | 当灰色对象删除指向白色对象的引用时，记录被删除的引用关系。重新标记阶段根据记录将这些白色对象重新标记为存活 |

---

### 四、CMS 增量更新（Incremental Update）详解

#### 4.1 原理

增量更新针对的是**条件一**（黑色 → 白色的新引用）。

当并发标记阶段，一个黑色对象新增了指向白色对象的引用时：
- 写屏障（Write Barrier）检测到这个变化。
- 将这个**新增的引用关系记录**到一个集合（Card Table / Mod Union Table）中。
- 在重新标记阶段，重新扫描这些黑色对象，检查它们是否引用了白色对象。
- 如果有，将白色对象标记为灰色（然后标记其引用链）。

#### 4.2 执行流程

```
并发标记阶段：
黑色对象 D.d = 白色对象 B  // 写屏障检测到
    │
    ▼
记录：D 引用了 B（加入脏卡表）
    │
    ▼
重新标记阶段（STW）：
    │
    ▼
扫描脏卡表，重新标记 D 的引用
    │
    ▼
将 B 标记为灰色 → 继续标记 B 的引用链
```

#### 4.3 写屏障（Write Barrier）

写屏障类似于 AOP 中的环绕通知，在引用赋值操作前后加入额外的处理逻辑：

```java
// 伪代码：写屏障
void oop_field_store(oop* field, oop new_value) {
    // 写屏障前置操作
    pre_write_barrier(field, new_value);
    
    // 实际的赋值操作
    *field = new_value;
    
    // 写屏障后置操作
    post_write_barrier(field, new_value);
}
```

CMS 使用的是**写后屏障**（Post-Write Barrier），在引用赋值之后记录变化。

### 五、G1 原始快照（SATB）详解

#### 5.1 原理

SATB 针对的是**条件二**（灰色 → 白色的引用被删除）。

原始快照的含义是：在并发标记开始时，为对象图拍一个"快照"。在并发标记过程中，如果引用关系发生了变化（引用被删除），就按照快照中的状态来处理。

具体来说：当灰色对象删除了对白色对象的引用时：
- 写屏障检测到这个变化。
- 将**被删除的引用关系**记录到 SATB 缓冲队列中。
- 在最终标记阶段（STW），处理 SATB 缓冲队列中的引用。
- 对于被删除引用的白色对象，将其**视为存活**（标记为灰色或黑色）。

#### 5.2 执行流程

```
并发标记阶段：
灰色对象 A.b = null  // 写屏障检测到，A 断开了对 B 的引用
    │
    ▼
记录：A 之前引用了 B（B 被记录到 SATB 缓冲队列）
    │
    ▼
最终标记阶段（STW）：
    │
    ▼
处理 SATB 缓冲队列
    │
    ▼
将 B 视为存活对象（标记为灰色/黑色）
```

#### 5.3 SATB 的特点

- **保守策略**：SATB 可能会多保留一些对象（即把一些本该被回收的对象标记为存活），但不会漏标。
- **多保留的对象**：这些对象会在**下一次 GC** 中被正确回收，成为浮动垃圾。
- **写屏障类型**：G1 使用的是**写前屏障**（Pre-Write Barrier），在引用被修改之前记录旧值。

### 六、增量更新 vs SATB 对比

| 维度 | 增量更新（CMS） | SATB（G1） |
|------|----------------|------------|
| **针对条件** | 条件一：黑色→白色的新引用 | 条件二：灰色→白色引用被删除 |
| **记录内容** | 记录新增的引用（新值） | 记录删除的引用（旧值） |
| **重新标记工作量** | 较大（需要重新扫描黑色对象的引用链） | 较小（只需要处理 SATB 缫冲队列中的对象） |
| **写屏障类型** | 写后屏障 | 写前屏障 |
| **浮动垃圾** | 较少 | 较多（保守策略，多标记存活） |
| **漏标风险** | 无 | 无 |
| **重新标记耗时** | 通常较长 | 通常较短 |

**为什么 G1 选择 SATB 而非增量更新？**

G1 使用 SATB 的主要原因是 SATB 的重新标记阶段处理更快。G1 的 Region 化设计使得它需要更高效的并发标记方案，SATB 只需要处理 SATB 缓冲队列中的少量对象，而增量更新需要重新扫描所有脏卡表中的对象引用链。

### 七、多标问题（浮动垃圾）

#### 7.1 什么是多标

多标是指在并发标记过程中，一个**原本应该被回收的对象**（应该标记为白色），因为引用关系的变化，被错误地标记为存活（黑色或灰色）。

**典型场景**：

```
并发标记阶段：
1. 灰色对象 A 引用白色对象 B（B 被标记为灰色/黑色）
2. 用户线程删除了 A 对 B 的引用（A.b = null）
3. 但 B 已经被标记为存活
4. B 实际上已经是垃圾了，但本次 GC 不会回收它
```

#### 7.2 浮动垃圾的产生原因

- 并发标记阶段，用户线程在不断执行，对象引用关系可能发生变化。
- 三色标记法在并发环境下无法精确追踪所有变化。
- 增量更新和 SATB 都可能导致多标（保守策略，宁可多标不可漏标）。

#### 7.3 浮动垃圾的解决方案

**不需要特别解决。** 多标的对象会在**下一次 GC** 中被回收掉，即成为浮动垃圾。

这是并发 GC 的固有代价：
- 漏标会导致**正在使用的对象被错误回收**（严重 bug），必须解决。
- 多标只会导致**本该回收的对象暂时不被回收**（少回收一些垃圾），下次 GC 即可回收，影响不大。

#### 7.4 浮动垃圾的影响

| 影响 | 说明 |
|------|------|
| **内存占用** | 浮动垃圾暂时占用内存，直到下次 GC |
| **触发条件** | 如果浮动垃圾过多，可能导致老年代空间不足，提前触发 GC |
| **CMS 的特殊问题** | CMS 在并发清除阶段产生的浮动垃圾只能等下次 GC 回收。如果浮动垃圾导致老年代空间不足，会退化为 Serial Old 进行 Full GC |

#### 7.5 CMS 的并发失败（Concurrent Mode Failure）

CMS 的一个特殊问题是**并发失败**：在并发标记和并发清除阶段，用户线程还在运行，会产生新的对象。如果老年代空间不足以容纳这些新对象，就会发生并发失败。

**并发失败的后果**：
- JVM 被迫启用 Serial Old 收集器进行老年代的回收（单线程 + STW）。
- STW 时间很长，性能急剧下降。

**预防措施**：
- 设置合适的 CMS 触发阈值：`-XX:CMSInitiatingOccupancyFraction=75`（默认 92%，建议调低到 75%）。
- 留足够的老年代空间容纳浮动垃圾和并发阶段新产生的对象。

### 八、写屏障详解

写屏障是实现增量更新和 SATB 的关键底层技术。

#### 8.1 什么是写屏障

写屏障不是内存屏障（Memory Barrier / Memory Fence），而是 JVM 在引用赋值操作前后插入的一段代码，用于追踪引用关系的变化。

#### 8.2 写前屏障 vs 写后屏障

| 类型 | 时机 | 使用者 | 作用 |
|------|------|--------|------|
| **写前屏障** | 引用赋值**之前** | G1（SATB） | 记录被覆盖的旧引用值 |
| **写后屏障** | 引用赋值**之后** | CMS（增量更新） | 记录新设置的引用值 |

#### 8.3 写屏障的性能影响

- 写屏障会增加每次引用赋值的开销。
- 通过**卡表（Card Table）**和**脏卡（Dirty Card）**机制来减少记录的粒度（按内存页/Region 粒度记录，而非每个引用）。
- 通常额外开销在 5-15% 左右。

### 九、总结

| 问题 | 原因 | 解决方案 | 代价 |
|------|------|----------|------|
| **漏标（对象消失）** | 黑色→白色新引用 + 灰色→白色引用断开 | CMS: 增量更新 / G1: SATB | 重新标记阶段需要 STW 处理 |
| **多标（浮动垃圾）** | 并发期间引用关系变化 | 不解决，下次 GC 回收 | 暂时占用内存 |
| **并发失败（CMS）** | 老年代空间不足 | 降低触发阈值 / Serial Old 兜底 | STW 时间长 |

**核心原则**：
- 漏标必须解决（会导致正在使用的对象被回收，是严重 bug）。
- 多标可以容忍（只是少回收一些垃圾，下次 GC 自然回收）。
- 增量更新和 SATB 都是保守策略，以多标为代价来保证不漏标。
