# 类的生命周期

## 问题
类的生命周期有哪些阶段？每个阶段分别做了什么？什么时候会触发类的初始化？

## 答案

### 一、类的生命周期概述

一个类从被加载到 JVM 内存中，到从内存中卸载，完整的生命周期包含 **7 个阶段**：

```
加载（Loading）
    │
    ▼
链接（Linking）
    ├── 验证（Verification）
    ├── 准备（Preparation）
    └── 解析（Resolution）
    │
    ▼
初始化（Initialization）
    │
    ▼
使用（Using）
    │
    ▼
卸载（Unloading）
```

**注意**：
- 加载、验证、准备、初始化、卸载这 5 个阶段的**开始顺序**是确定的
- 但**执行过程**是交叉进行的（如在加载阶段就可能开始验证阶段的部分工作）
- **解析阶段**不一定在初始化之前，某些情况下会在初始化之后（如 invokedynamic 指令）

---

### 二、加载（Loading）

加载阶段是"类加载"过程的第一步，JVM 需要完成三件事：

**1. 通过类的全限定名获取该类的二进制字节流**

字节流的来源可以是：
- `.class` 文件（本地文件系统）
- JAR/WAR 包中的 `.class` 文件
- 网络（Applet）
- 运行时动态生成（动态代理，`Proxy.newProxyInstance()`）
- 数据库中存储的二进制数据
- 其他文件生成（JSP 编译后的 `.class`）

**2. 将字节流代表的静态存储结构转化为方法区的运行时数据结构**

- 字节码信息存储在**方法区**（JDK 8+ 称为 Metaspace）
- 包括：类的全限定名、父类名、实现的接口、字段信息、方法信息、常量池等

**3. 在堆内存中生成一个 `java.lang.Class` 对象**

- 作为方法区中该类数据的**访问入口**
- 所有对类的反射操作都通过这个 Class 对象

```java
// 获取 Class 对象的三种方式
Class<?> clazz1 = Class.forName("com.example.User"); // 方式1：全限定名
Class<User> clazz2 = User.class;                      // 方式2：类名.class
Class<?> clazz3 = new User().getClass();              // 方式3：对象.getClass()
```

#### 2.1 类加载器（ClassLoader）

| 类加载器 | 加载范围 | 说明 |
|----------|----------|------|
| **Bootstrap ClassLoader**（启动类加载器） | `JAVA_HOME/lib`（rt.jar 等） | C++ 实现，不是 Java 类 |
| **Extension ClassLoader**（扩展类加载器） | `JAVA_HOME/lib/ext` | Java 实现 |
| **Application ClassLoader**（应用程序类加载器） | `classpath` | 加载应用代码，最常用 |
| **自定义 ClassLoader** | 自定义路径 | 继承 ClassLoader |

#### 2.2 双亲委派模型

当一个类加载器收到类加载请求时：

1. 先委派给父类加载器去加载。
2. 父类加载器继续委派给它的父类加载器。
3. 如果父类加载器无法加载（抛出 ClassNotFoundException），子类加载器才自己尝试加载。

```
自定义 ClassLoader
    │ 委派
    ▼
Application ClassLoader
    │ 委派
    ▼
Extension ClassLoader
    │ 委派
    ▼
Bootstrap ClassLoader（尝试加载）
    │ 失败
    ▼
Extension ClassLoader（尝试加载）
    │ 失败
    ▼
Application ClassLoader（尝试加载）
    │ 失败
    ▼
自定义 ClassLoader（尝试加载）
```

**为什么需要双亲委派**：
- **安全性**：防止用户自定义的类冒充核心类（如 java.lang.String）。
- **唯一性**：保证同一个类只被加载一次。

---

### 三、验证（Verification）

验证是连接阶段的第一步，确保字节码文件符合 JVM 规范，不会危害 JVM 安全。

**验证的四个子阶段**：

| 验证类型 | 验证内容 | 示例 |
|----------|---------|------|
| **文件格式验证** | 验证 .class 文件的结构是否符合规范 | 魔数是否为 `0xCAFEBABE`、版本号是否合法、常量池类型是否正确 |
| **元数据验证** | 验证类的元数据信息是否符合 Java 语言规范 | 是否有父类（除 Object 外）、是否继承了 final 类、非抽象类是否实现了所有抽象方法 |
| **字节码验证** | 验证方法体的字节码指令是否合法 | 操作数栈数据类型是否匹配、跳转指令是否跳到合法位置、方法调用参数是否正确 |
| **符号引用验证** | 验证符号引用是否能找到对应的类/字段/方法 | 引用的类是否存在、访问权限是否足够、字段和方法是否存在 |

**为什么需要验证？**
- .class 文件不一定来自 Java 编译器，可以通过任何途径生成
- 不合法的字节码可能导致 JVM 崩溃或安全漏洞
- 验证是 JVM 的**第一道安全防线**

---

### 四、准备（Preparation）

准备阶段为**类变量**（被 `static` 修饰的变量）分配内存并设置**默认初始值**。

**注意**：
- 这里只分配**类变量**（static），不包括**实例变量**（实例变量在对象创建时分配）
- 设置的是**默认初始值**（零值），不是代码中定义的初始值

```java
public class MyClass {
    // 准备阶段：value 分配内存，设置默认值 0（不是 123）
    // 初始化阶段（clinit）：value = 123
    public static int value = 123;
    
    // 准备阶段：常量，直接赋值为 123（final 在编译期就确定了）
    public static final int CONSTANT = 123;
    
    // 准备阶段：str 分配内存，设置默认值 null
    public static String str = "hello";
}
```

**各类型的默认值**：

| 数据类型 | 默认值 |
|----------|--------|
| byte / short / int / char | 0 |
| long | 0L |
| float | 0.0f |
| double | 0.0d |
| boolean | false |
| 引用类型 | null |

**特殊情况**：`static final` 修饰的常量在准备阶段就直接赋值，因为编译器已经确定了值（ConstantValue 属性）。

---

### 五、解析（Resolution）

解析阶段将**常量池中的符号引用**替换为**直接引用**。

**符号引用 vs 直接引用**：

| 类型 | 说明 | 示例 |
|------|------|------|
| **符号引用** | 用一组符号来描述引用的目标（类名、方法名、字段名） | `java/lang/String`、`toString:()Ljava/lang/String;` |
| **直接引用** | 直接指向目标的指针、相对偏移量或句柄 | 内存地址 `0x7f3a2b1c` |

**解析的内容**：

1. **类或接口的解析**：将符号引用 `com.example.User` 解析为方法区中 User 类的直接引用
2. **字段解析**：将符号引用 `User.name:Ljava/lang/String;` 解析为字段在内存中的偏移量
3. **方法解析**：将符号引用 `User.getName:()Ljava/lang/String;` 解析为方法在内存中的入口地址
4. **接口方法解析**：类似方法解析，但针对接口方法

---

### 六、初始化（Initialization）

初始化是类加载过程的最后一个阶段，执行类构造器 `<clinit>()` 方法。

#### 6.1 `<clinit>()` 方法

- 由编译器自动收集类中所有**类变量的赋值动作**和**静态代码块**中的语句合并产生
- 收集顺序与源代码中的顺序一致
- JVM 保证 `<clinit>()` 方法在多线程环境中被正确加锁同步（只会执行一次）

```java
public class MyClass {
    // 准备阶段：value = 0
    // 初始化阶段：执行 <clinit>()，value = 123
    public static int value = 123;
    
    // 静态代码块也会被收集到 <clinit>() 中
    static {
        System.out.println("静态代码块执行");
        value = 456;
    }
    
    // final 常量在准备阶段就赋值了，不参与 <clinit>()
    public static final int CONSTANT = 123;
}
```

#### 6.2 `<clinit>()` 的特点

| 特点 | 说明 |
|------|------|
| **线程安全** | JVM 会保证 `<clinit>()` 方法在多线程环境中被正确地加锁同步。如果多个线程同时初始化一个类，只有一个线程会执行 `<clinit>()`，其他线程阻塞等待 |
| **父类优先** | JVM 保证子类的 `<clinit>()` 执行前，父类的 `<clinit>()` 已经执行完毕。因此 Object 类的 `<clinit>()` 是第一个执行的 |
| **非必需** | 如果类中没有静态变量赋值和静态代码块，编译器可以不生成 `<clinit>()` 方法 |
| **接口特殊** | 接口也有 `<clinit>()` 方法，但接口的 `<clinit>()` 不需要先执行父接口的。只有当接口中定义的变量被使用时才会初始化接口 |

#### 6.3 `<init>()` 方法

除了 `<clinit>()`，还有一个 `<init>()` 方法：
- `<init>()` 是**实例构造器**，在创建对象（new）时调用。
- `<clinit>()` 是**类构造器**，在类初始化时调用。

---

### 七、初始化的触发条件

#### 7.1 会触发初始化的场景（主动引用）

| 场景 | 示例 | 说明 |
|------|------|------|
| **使用 new 创建对象** | `new MyClass()` | 触发类的初始化 |
| **读取/设置类的静态字段** | `MyClass.value`（非 final） | 触发类的初始化 |
| **调用类的静态方法** | `MyClass.doSomething()` | 触发类的初始化 |
| **反射调用** | `Class.forName("com.example.MyClass")` | 触发类的初始化 |
| **初始化子类时父类未初始化** | 子类初始化前先初始化父类 | 父类先初始化 |
| **包含 main() 方法的主类** | JVM 启动时 | 最先初始化 |

#### 7.2 不会触发初始化的场景（被动引用）

| 场景 | 示例 | 说明 |
|------|------|------|
| **子类引用父类的静态字段** | `SubClass.parentValue` | 只初始化父类，不初始化子类 |
| **通过数组定义引用类** | `MyClass[] arr = new MyClass[10]` | 不会初始化 MyClass |
| **引用类的常量** | `MyClass.FINAL_VALUE`（static final） | 常量在编译阶段已放入常量池，不触发初始化 |

```java
class Parent {
    static int value = 123;
    static { System.out.println("Parent 初始化"); }
}

class Child extends Parent {
    static { System.out.println("Child 初始化"); }
}

// 情况1：子类引用父类的静态字段
System.out.println(Child.value);
// 输出：Parent 初始化
// 说明：只初始化了 Parent，没有初始化 Child

// 情况2：定义类的数组
Child[] children = new Child[10];
// 无输出，说明 Child 没有被初始化

// 情况3：引用常量
class Const {
    static final int VALUE = 123;
    static { System.out.println("Const 初始化"); }
}
System.out.println(Const.VALUE);
// 无输出，常量在编译期已确定
```

---

### 八、使用（Using）

类初始化完成后，就可以正常使用了：
- 创建对象（new）
- 调用静态方法
- 读写静态字段
- 反射操作
- 子类继承

---

### 九、卸载（Unloading）

一个类被卸载需要同时满足以下三个条件：

| 条件 | 说明 |
|------|------|
| 该类所有的实例都已被 GC 回收 | 堆中不存在该类的任何实例 |
| 加载该类的 ClassLoader 已被 GC 回收 | ClassLoader 对象不可达 |
| 该类对应的 Class 对象没有在任何地方被引用 | Class 对象不可达 |

**注意**：
- 由**启动类加载器**（Bootstrap ClassLoader）加载的类**永远不会被卸载**
- 在实际应用中，类卸载的情况**非常少见**
- 使用 OSGI、热部署等技术时可能会涉及类卸载

```bash
# 查看类加载和卸载信息
-XX:+TraceClassLoading
-XX:+TraceClassUnloading
```

---

### 十、类的生命周期 vs 类加载

**类加载**（Class Loading）通常指类生命周期的**前 5 个阶段**：加载、验证、准备、解析、初始化。

| 概念 | 包含阶段 | 说明 |
|------|---------|------|
| **类加载**（狭义） | 加载、验证、准备、解析、初始化 | 将 .class 文件加载到 JVM 并完成初始化 |
| **类的生命周期** | 加载、验证、准备、解析、初始化、使用、卸载 | 类在 JVM 中的完整一生 |

**类加载的时机**：
- JVM 规范没有规定何时进行"加载"，由各 JVM 实现自行决定
- 但严格规定了"初始化"必须在以下情况下立即执行（即上面提到的 6 种主动引用）

---

### 十一、总结

| 阶段 | 主要工作 | 关键点 |
|------|----------|--------|
| **加载** | 获取字节流、存入方法区、生成 Class 对象 | 类加载器 + 双亲委派 |
| **验证** | 文件格式/元数据/字节码/符号引用验证 | 保证安全性 |
| **准备** | 为类变量分配内存并赋默认值 | final 常量直接赋值 |
| **解析** | 符号引用→直接引用 | 可能在初始化之后 |
| **初始化** | 执行 `<clinit>()` 方法 | 6 种触发条件、3 种不触发条件 |
| **使用** | 正常使用类 | 创建对象、调用方法 |
| **卸载** | GC 回收类的所有实例和 ClassLoader | 自定义 ClassLoader 加载的类 |
