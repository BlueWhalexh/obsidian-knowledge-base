# BIO NIO AIO 对比

## 问题
BIO、NIO、AIO 有什么区别？同步与异步、阻塞与非阻塞的区别是什么？

## 答案

### 同步与异步如何理解

**同步（Synchronous）**：调用方发出请求后，必须**等待**被调用方处理完成并返回结果，才能继续执行后续操作。调用方主动等待结果。

**异步（Asynchronous）**：调用方发出请求后，**不需要等待**被调用方处理完成，可以立即去做其他事情。被调用方处理完成后通过回调、通知等方式告知调用方。

**经典比喻：打电话 vs 发短信**

| 场景 | 同步（打电话） | 异步（发短信） |
|------|--------------|--------------|
| 发起动作 | 拨号 | 编辑短信并发送 |
| 等待方式 | 拿着电话等对方接听 | 发完就去做别的事 |
| 结果获取 | 实时听到对方说话 | 手机收到回复短信 |
| 期间能否做其他事 | 不能，必须等通话结束 | 可以，不阻塞 |
| 可靠性 | 高，实时反馈 | 需要额外机制保证（如重发） |

**从 CPU 角度理解**：
- **同步**：CPU 发起 I/O 请求后，必须等 I/O 完成才能继续执行（CPU 可能阻塞等待）
- **异步**：CPU 发起 I/O 请求后立即返回，I/O 由 DMA/硬件完成，完成后通过中断通知 CPU

### 阻塞与非阻塞如何理解

**阻塞（Blocking）**：调用方在等待结果的过程中，当前线程被**挂起**（阻塞），不占用 CPU 资源，直到操作完成。

**非阻塞（Non-Blocking）**：调用方在等待结果的过程中，当前线程**不会被挂起**，可以继续执行其他操作，需要通过轮询等方式检查操作是否完成。

**从 CPU 消耗角度理解**：

| 角度 | 阻塞 | 非阻塞 |
|------|------|--------|
| CPU 状态 | 线程挂起，CPU 去调度其他线程 | 线程运行中，CPU 在忙等或做其他事 |
| CPU 利用率 | 等待期间 CPU 不浪费在该线程上 | 忙等时 CPU 空转浪费，但也可能做有用功 |
| 线程状态 | WAITING / BLOCKED | RUNNABLE |
| 响应速度 | 操作完成后立即响应 | 需要轮询才能知道完成 |

**关键区别**：
- **阻塞/非阻塞**关注的是：调用方在等待结果时**线程的状态**
- **同步/异步**关注的是：调用方和被调用方之间的**消息通信机制**

### 同步不一定阻塞，异步也不一定非阻塞

这是一个常见的误区。两组概念是**正交的**，可以自由组合：

| 组合 | 说明 | 示例 |
|------|------|------|
| **同步阻塞** | 调用方等待结果，期间线程挂起 | BIO 的 read() |
| **同步非阻塞** | 调用方等待结果，期间线程不挂起（轮询） | NIO 中 Channel 配合 Selector |
| **异步阻塞** | 调用方发了请求就返回，但线程还是阻塞了（罕见） | 调用了异步 API 但又调了 get() 等结果 |
| **异步非阻塞** | 调用方发了请求就返回，线程也不阻塞 | AIO |

**同步不阻塞的例子**：
- NIO 的 Channel 设置为非阻塞模式后，`read()` 调用是**同步的**（调用方需要处理返回值），但**非阻塞的**（如果没有数据立即返回 0，不会挂起线程）
- 调用方需要不断轮询检查数据是否就绪

**异步阻塞的例子**：
- 使用 `CompletableFuture` 发起异步请求后，调用 `future.get()` 会阻塞当前线程等待结果
- 虽然任务本身是异步执行的，但调用方仍然被阻塞了

### BIO（Blocking I/O）

BIO 是传统的 `java.io` 包，基于流模型实现。

**特点**：
- **同步阻塞**：读写操作时，线程会一直阻塞直到操作完成
- 调用是可靠的线性顺序

**优点**：代码简单、直观

**缺点**：
- I/O 效率低
- 扩展性差
- 容易成为应用性能瓶颈

**典型模型**：一个连接一个线程

```
客户端连接 → 服务端分配线程 → 线程阻塞等待读写 → 读写完成释放线程
```

**代码示例**：
```java
// BIO 服务端
ServerSocket serverSocket = new ServerSocket(8080);
while (true) {
    Socket socket = serverSocket.accept(); // 阻塞等待连接
    new Thread(() -> {
        InputStream in = socket.getInputStream();
        byte[] buf = new byte[1024];
        int len = in.read(buf); // 阻塞等待数据
        // 处理数据...
    }).start();
}
```

**适用场景**：连接数少、每个连接的 I/O 操作频繁且持续

### NIO（Non-Blocking I/O）

NIO 是 Java 1.4 引入的 `java.nio` 包。

**特点**：
- **同步非阻塞**
- 提供了 Channel、Selector、Buffer 等新的抽象
- 构建**多路复用**的同步非阻塞 I/O 程序
- 更接近操作系统底层的高性能数据操作方式

**三大核心组件**：

| 组件 | 说明 |
|---|---|
| **Channel** | 双向通道，既可读也可写，类似流但更灵活 |
| **Buffer** | 缓冲区，数据读写的中转站 |
| **Selector** | 多路复用器，一个线程监听多个 Channel 的事件 |

**工作原理**：
1. Selector 注册多个 Channel 的事件（连接就绪、可读、可写）
2. 一个线程通过 Selector 轮询所有注册的 Channel
3. 当某个 Channel 有事件就绪时，才处理该 Channel

**为什么 NIO 是同步的？**
- NIO 的 read/write 操作仍然是**同步的**：应用程序需要自己负责数据的读写操作
- 只是通过 Selector 实现了**非阻塞**：没有数据就绪时不阻塞，而是返回 0
- 数据从内核缓冲区到用户缓冲区的拷贝，仍然是应用程序主动完成的

**优点**：
- 一个线程处理多个连接，减少线程开销
- 适合连接数多但每个连接 I/O 操作不频繁的场景

**典型模型**：一个线程处理多个连接（多路复用）

```
多个客户端连接 → 注册到 Selector → 单线程轮询 → 处理就绪的 Channel
```

**适用场景**：连接数多、每个连接的 I/O 操作不频繁（如聊天服务器）

### AIO（Asynchronous I/O）

AIO 是 Java 1.7 引入的 `java.nio` 包的升级版本（NIO 2.0）。

**特点**：
- **异步非阻塞**
- 基于**事件和回调**机制实现
- 应用操作后直接返回，不阻塞
- 后台处理完成后，操作系统通知相应线程进行后续操作

**为什么 AIO 是异步的？**
- 应用程序发起 I/O 请求后**立即返回**，不需要等待
- 数据的读写操作完全由**操作系统完成**（包括从内核缓冲区到用户缓冲区的拷贝）
- 操作完成后通过**回调函数**通知应用程序
- 应用程序全程不需要参与 I/O 操作

**工作原理**：
1. 发起 I/O 请求后立即返回
2. 操作系统在后台完成 I/O 操作
3. 操作完成后通过回调函数通知应用

**典型模型**：真正的异步 I/O

```
客户端发起请求 → 立即返回 → 操作系统后台处理 → 完成后回调通知
```

**适用场景**：连接数多且每个连接的 I/O 操作频繁

### 三者对比

| 对比项 | BIO | NIO | AIO |
|---|---|---|---|
| I/O 模型 | 同步阻塞 | 同步非阻塞 | 异步非阻塞 |
| 引入版本 | JDK 1.0 | JDK 1.4 | JDK 1.7 |
| 线程模型 | 一个连接一个线程 | 一个线程处理多个连接 | 事件回调驱动 |
| 编程复杂度 | 简单 | 较复杂 | 复杂 |
| 吞吐量 | 低 | 高 | 最高 |
| 适用场景 | 连接数少、I/O 操作频繁 | 连接数多、I/O 操作不频繁 | 连接数多、I/O 操作频繁 |
| 数据拷贝 | 应用程序负责 | 应用程序负责 | 操作系统负责 |

### 选型建议

- **BIO**：适用于连接数目比较小且固定的架构，对服务器资源要求不高
- **NIO**：适用于连接数目多且连接比较短的架构，如聊天服务器、弹幕系统
- **AIO**：适用于连接数目多且连接比较长的架构，如相册服务器
