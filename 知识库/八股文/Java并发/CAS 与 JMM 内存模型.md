# CAS 与 JMM 内存模型

## 问题
什么是 CAS？什么是 JMM？乐观锁和悲观锁有什么区别？ABA 问题是什么？

## 答案

### CAS 原理

CAS（Compare And Swap，比较再交换）：在更新前检查数据是否被修改，若未被修改则更新，否则自旋重试。

**具体流程**：
1. 主存中有一个共享变量，线程先把变量值拷贝到工作内存中进行计算，得到新值
2. 线程带着旧值和新值再次找到主存中的共享变量
3. **比较**旧值和当前主存中的值是否相同：
   - 若相同：说明期间没有其他线程修改过，线程安全，写入新值
   - 若不同：说明期间有其他线程修改过，自旋重试（重新读取、计算、比较）

**底层实现**：CAS 的核心是 `Unsafe` 类，它提供硬件级别的原子操作。调用链路为：Atomic 包类 → Unsafe 方法 → C 代码 → 汇编 → CPU 原子指令 `cmpxchg`，直接操作内存。

**应用场景**：
- AQS 中大量使用 CAS 实现基本操作
- JUC 包下的原子类（如 AtomicInteger）使用 CAS + volatile 保证线程安全

**优缺点**：
- 优点：不加锁，无锁并发保证线程安全
- 缺点：自旋占用 CPU

### ABA 问题

CAS 只检查值是否变化，不检查是否经历过修改。如果一个值从 A 变为 B 再变回 A，CAS 会认为值没有变化，但实际上已经被修改过。

**解决方案**：
- **版本号机制**：为数据添加 version 字段，每次更新递增版本号。更新时检查版本号是否匹配
- **AtomicStampedReference**：Java 提供的带版本戳的原子引用类

```sql
-- 版本号机制示例
UPDATE table SET value = new_value, version = version + 1
WHERE id = 1 AND version = current_version;
```

### JMM（Java 内存模型）

JMM 是一套规范/规则，目的是在多线程读写共享数据时保证**可见性、有序性和原子性**。

**核心概念**：

| 概念 | 说明 |
|------|------|
| **主内存** | 线程共享的内存区域，存储所有共享变量，是多线程数据通信的桥梁 |
| **工作内存** | 每个线程独占的内存区域，存储局部变量和对主内存共享变量的拷贝副本 |

**工作模式**：线程对共享变量的操作不直接在主内存中进行，而是先将共享变量从主内存拷贝到自己的工作内存中，对副本进行操作，操作完成后再将最新值刷新回主内存。

**底层硬件基础**：MESI 协议（Modified Exclusive Shared Or Invalid）保证各处理器缓存中共享变量副本的一致性。当 CPU 写共享变量时，会发出信号通知其他 CPU 将该变量的缓存行置为无效，其他 CPU 读取时会从内存重新加载。

### happens-before 原则

happens-before 是 JMM 中判断数据是否存在竞争、是否线程安全的重要依据。如果操作 A happens-before 操作 B，则 A 的结果对 B 可见。主要规则包括：
- 程序顺序规则：同一线程中，前面的操作 happens-before 后面的操作
- 锁规则：解锁 happens-before 加锁
- volatile 规则：写 happens-before 读
- 传递性规则：如果 A hb B，B hb C，则 A hb C

### 乐观锁 vs 悲观锁

| 维度 | 乐观锁 | 悲观锁 |
|------|--------|--------|
| 态度 | 认为并发冲突概率低 | 认为并发冲突概率高 |
| 实现方式 | CAS、版本号法 | synchronized、ReentrantLock |
| 适用场景 | 读多写少 | 写多读少 |
| 数据一致性 | 可能因冲突导致重试 | 强一致性保证 |
| 并发性能 | 高 | 低（串行化操作） |
