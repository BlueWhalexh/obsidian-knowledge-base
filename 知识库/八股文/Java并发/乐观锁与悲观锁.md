# 乐观锁与悲观锁

## 问题
什么是乐观锁？什么是悲观锁？各自的应用场景是什么？数据库如何实现乐观锁？

## 答案

### 一、锁的分类概览

锁可以从多个维度进行分类，其中按照**并发控制策略**可以分为乐观锁和悲观锁：

```
锁
├── 按并发控制策略
│   ├── 乐观锁（Optimistic Locking）
│   └── 悲观锁（Pessimistic Locking）
├── 按获取方式
│   ├── 公平锁 / 非公平锁
│   └── 可重入锁 / 不可重入锁
├── 按互斥程度
│   ├── 独占锁（排他锁）
│   └── 共享锁
└── 按实现方式
    ├── synchronized
    └── ReentrantLock / AQS
```

---

### 二、乐观锁

#### 2.1 定义

乐观锁（Optimistic Locking）是一种**不加锁**的并发控制策略。它假设多个线程在访问共享数据时**不会发生冲突**，因此不加锁，而是在**提交修改时**才去检查数据是否被其他线程修改过。如果没有冲突则提交成功，如果有冲突则重试或放弃。

#### 2.2 实现方式

乐观锁主要通过以下两种机制实现：

**1. CAS（Compare And Swap）**

CAS 是乐观锁的核心实现方式，是一条 CPU 原子指令。包含三个操作数：
- **V**（Value）：要更新的内存地址中的值
- **E**（Expected）：期望的旧值
- **N**（New）：要设置的新值

执行逻辑：只有当 V == E 时，才将 V 更新为 N，否则不做任何操作。整个操作是**原子的**。

```java
// Java 中 CAS 的使用（AtomicInteger 为例）
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();  // 内部使用 CAS 实现原子自增
```

**2. 版本号机制（Version Number）**

在数据表中增加一个版本号字段（version），每次更新时检查版本号是否一致：
- 读取数据时获取当前版本号。
- 更新时，WHERE 条件中加上版本号的检查。
- 如果版本号一致则更新成功并将版本号 +1。
- 如果版本号不一致，说明数据被其他事务修改过，更新失败。

#### 2.3 乐观锁的特点

| 特点 | 说明 |
|------|------|
| **不加锁** | 不会阻塞线程，没有死锁风险 |
| **读多写少** | 适合读操作远多于写操作的场景，冲突概率低 |
| **冲突时重试** | 当检测到冲突时，通常采用重试策略（自旋） |
| **CAS 的 ABA 问题** | 值从 A 变为 B 再变回 A，CAS 无法感知变化（可通过版本号解决） |
| **性能** | 无竞争时性能好；高竞争时大量重试，性能下降 |

---

### 三、悲观锁

#### 3.1 定义

悲观锁（Pessimistic Locking）是一种**加锁**的并发控制策略。它假设多个线程在访问共享数据时**一定会发生冲突**，因此在访问数据之前先加锁，确保在自己操作期间不会有其他线程修改数据。

#### 3.2 实现方式

**1. Java 中的悲观锁**

- **synchronized**：JVM 层面的内置锁，通过 Monitor 机制实现。详见 [[悲观锁的实现机制]]。
- **ReentrantLock**：API 层面的显式锁，基于 AQS 实现。

**2. 数据库中的悲观锁**

- **SELECT ... FOR UPDATE**：行级排他锁，锁定查询到的行，其他事务不能修改也不能加排他锁（但可以加共享锁读取）。
- **SELECT ... LOCK IN SHARE MODE**：行级共享锁，其他事务可以加共享锁读取但不能加排他锁修改。

```sql
-- 悲观锁示例：银行转账
BEGIN;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;  -- 加行级排他锁
-- 其他事务无法修改 id=1 的记录，直到当前事务提交
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;  -- 释放锁
```

#### 3.3 悲观锁的特点

| 特点 | 说明 |
|------|------|
| **加锁** | 访问前先加锁，阻塞其他线程 |
| **写多读少** | 适合写操作频繁的场景 |
| **安全性高** | 数据不会被意外修改 |
| **可能死锁** | 多个锁交叉持有可能导致死锁 |
| **性能** | 锁的获取和释放有开销，高并发时性能较差 |

---

### 四、乐观锁 vs 悲观锁 对比

| 维度 | 乐观锁 | 悲观锁 |
|------|--------|--------|
| **核心思想** | 假设不会发生冲突，提交时检查 | 假设一定会发生冲突，先加锁 |
| **是否加锁** | 不加锁 | 加锁 |
| **实现方式** | CAS / 版本号 | synchronized / ReentrantLock / FOR UPDATE |
| **适用场景** | 读多写少，冲突概率低 | 写多读少，冲突概率高 |
| **并发性能** | 无冲突时性能好 | 竞争激烈时更安全 |
| **死锁风险** | 无死锁 | 有死锁风险 |
| **阻塞** | 不阻塞，冲突时重试 | 阻塞等待 |
| **Java 实现** | Atomic* 类、LongAdder | synchronized、ReentrantLock |
| **数据库实现** | 版本号 / 时间戳 | SELECT FOR UPDATE |

### 五、应用场景

#### 5.1 乐观锁的典型应用场景

| 场景 | 说明 |
|------|------|
| **计数器 / 序列号生成** | AtomicInteger、AtomicLong，CAS 实现 |
| **缓存更新** | 读多写少的缓存，更新时用 CAS 检查 |
| **版本控制** | 数据库乐观锁（版本号机制） |
| **CAS 链表/栈** | 无锁数据结构（ConcurrentLinkedQueue 等） |
| **配置更新** | 低频修改的配置信息 |

#### 5.2 悲观锁的典型应用场景

| 场景 | 说明 |
|------|------|
| **银行转账** | 余额变更必须串行执行 |
| **库存扣减** | 防止超卖，必须加锁 |
| **订单状态变更** | 状态机流转需要互斥 |
| **写密集型操作** | 频繁写入，冲突概率高 |
| **数据一致性要求极高** | 不允许任何并发冲突 |

### 六、数据库如何实现乐观锁

#### 6.1 版本号（Version）方式

最常用的数据库乐观锁实现方式。

```sql
-- 表结构
CREATE TABLE product (
    id BIGINT PRIMARY KEY,
    name VARCHAR(100),
    stock INT,
    version INT DEFAULT 0  -- 版本号字段
);

-- 读取数据（获取版本号）
SELECT id, name, stock, version 
FROM product WHERE id = 1;
-- 假设返回 version = 5

-- 更新数据（带版本号检查）
UPDATE product 
SET stock = stock - 1, version = version + 1 
WHERE id = 1 AND version = 5;
-- 如果 version 被其他事务修改过（不等于 5），则更新影响行数为 0，表示更新失败

-- Java 代码层面的处理
int affected = jdbcTemplate.update(
    "UPDATE product SET stock = stock - 1, version = version + 1 WHERE id = ? AND version = ?",
    productId, currentVersion
);
if (affected == 0) {
    // 更新失败，版本号冲突，可以选择重试或抛异常
    throw new OptimisticLockException("数据已被修改，请重试");
}
```

#### 6.2 时间戳（Timestamp）方式

```sql
-- 表结构
CREATE TABLE product (
    id BIGINT PRIMARY KEY,
    name VARCHAR(100),
    stock INT,
    update_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP  -- 时间戳字段
);

-- 读取
SELECT * FROM product WHERE id = 1;
-- 假设返回 update_time = '2026-04-30 10:00:00'

-- 更新（带时间戳检查）
UPDATE product 
SET stock = stock - 1, update_time = NOW() 
WHERE id = 1 AND update_time = '2026-04-30 10:00:00';
```

**注意**：时间戳方式存在精度问题。如果两次更新在同一时间戳精度内发生，可能无法检测到冲突。因此**版本号方式更推荐**。

#### 6.3 乐观锁的重试策略

```java
public boolean deductStock(Long productId, int quantity, int maxRetry) {
    for (int i = 0; i < maxRetry; i++) {
        // 1. 读取当前数据和版本号
        Product product = productDao.selectById(productId);
        if (product.getStock() < quantity) {
            return false;  // 库存不足
        }
        // 2. CAS 更新（带版本号检查）
        int affected = productDao.updateWithVersion(
            productId, quantity, product.getVersion()
        );
        if (affected > 0) {
            return true;  // 扣减成功
        }
        // 3. 更新失败，重试
    }
    return false;  // 超过最大重试次数
}
```

### 七、CAS 的 ABA 问题及解决

#### 7.1 ABA 问题

CAS 检查的是"值是否和预期相同"，但如果值从 A 变为 B 再变回 A，CAS 无法感知中间的变化：

```
线程1：读取 V = A
线程2：V 从 A 改为 B
线程2：V 从 B 改回 A
线程1：CAS(V, A, newValue) → 成功（但 V 已经被修改过了）
```

#### 7.2 解决方案

| 方案 | 说明 |
|------|------|
| **AtomicStampedReference** | 引入版本号（stamp），每次修改都增加 stamp |
| **AtomicMarkableReference** | 引入 boolean 标记，标记是否被修改过 |
| **数据库版本号** | 每次更新 version + 1 |

```java
// AtomicStampedReference 解决 ABA 问题
AtomicStampedReference<Integer> ref = new AtomicStampedReference<>(100, 0);
int stamp = ref.getStamp();  // 获取当前版本号
ref.compareAndSet(100, 200, stamp, stamp + 1);  // 带版本号的 CAS
```
