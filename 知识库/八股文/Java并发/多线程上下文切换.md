# 多线程上下文切换

## 问题
什么是上下文切换？什么时候会发生上下文切换？如何减少上下文切换？

## 答案

### 一、什么是上下文切换

上下文切换（Context Switch）是指 CPU 从一个线程（或进程）切换到另一个线程（或进程）执行的过程。

在多线程环境中，CPU 并不是真正地"同时"执行多个线程，而是在多个线程之间**快速轮转**，每个线程执行一小段时间（称为**时间片**，Time Slice），通过快速切换制造出"同时执行"的假象。

每次上下文切换时，CPU 需要完成以下工作：

1. **保存当前线程的上下文**：
   - 程序计数器（PC）：记录当前执行到哪条指令。
   - 寄存器状态：保存当前线程使用的各种寄存器的值。
   - 栈指针：记录当前线程的栈位置。
   - 线程状态信息。

2. **加载新线程的上下文**：
   - 恢复新线程的程序计数器。
   - 恢复新线程的寄存器状态。
   - 恢复新线程的栈指针。

3. **跳转到新线程上次执行的位置继续执行**。

**开销**：
- **直接开销**：保存和恢复寄存器、切换页表（TLB 失效）、内核调度开销。通常一次上下文切换需要几微秒。
- **间接开销**：CPU 缓存失效（L1/L2/L3 Cache 中的数据可能对新线程无用），导致缓存命中率下降。这是更大的性能损失。

### 二、什么时候会发生上下文切换

#### 2.1 时间片用完（抢占式调度）

最常见的情况。操作系统为每个线程分配一个时间片（通常 10-100ms），时间片用完后，操作系统会强制切换到其他等待的线程。

```
线程A [====执行====] → 时间片用完 → 线程B [====执行====] → 时间片用完 → 线程A
```

这属于**被动切换**，线程本身没有主动让出 CPU。

#### 2.2 I/O 阻塞

当线程执行 I/O 操作（如读写文件、网络请求、数据库查询）时，线程需要等待 I/O 完成。此时操作系统会将该线程切换为阻塞状态，并调度其他可运行的线程。

```
线程A [==执行==] → 发起IO → [等待IO] → IO完成 → [继续执行]
                              ↓ 切换
                     线程B    [====执行====]
```

I/O 阻塞导致的上下文切换在 Web 应用中非常普遍，因为每个请求都会涉及大量的 I/O 操作。

#### 2.3 主动让出 CPU

线程可以主动让出 CPU 的使用权，触发上下文切换：

| 方式 | 说明 |
|------|------|
| `Thread.yield()` | 线程主动让出 CPU 时间片，让调度器重新选择线程（可能选到自己） |
| `Thread.sleep(ms)` | 线程休眠指定时间，让出 CPU |
| `Object.wait()` | 线程在 synchronized 块中等待，释放锁并让出 CPU |
| `LockSupport.park()` | 线程挂起，让出 CPU |
| `Thread.join()` | 等待其他线程执行完毕 |
| `ReentrantLock.lock()` | 获取不到锁时阻塞，让出 CPU |

#### 2.4 高优先级线程抢占

当一个高优先级的线程变为可运行状态时，操作系统可能会中断当前低优先级的线程，切换到高优先级线程执行。这在实时操作系统中更常见。

#### 2.5 垃圾回收（GC）

JVM 的垃圾回收器在执行时通常需要 **STW**（Stop The World），即暂停所有应用线程。虽然 GC 引起的不是严格意义上的"上下文切换"（是 JVM 层面的暂停），但效果类似——所有线程被暂停，GC 线程执行，完成后恢复应用线程。

不同 GC 的 STW 时间差异很大：
- **Serial GC**：单线程 GC，STW 时间长。
- **Parallel GC**：多线程 GC，STW 时间较短。
- **CMS/G1**：尽可能并发执行，STW 时间短。
- **ZGC**：几乎无 STW。

#### 2.6 线程创建和销毁

线程的创建和销毁也会涉及上下文切换，虽然不是直接的上下文切换，但涉及操作系统内核的资源分配和回收。

### 三、如何减少上下文切换

#### 3.1 无锁编程（CAS）

多线程竞争锁时会引起上下文切换。使用**无锁的 CAS 算法**可以避免锁的竞争，从而减少上下文切换。

```java
// 使用 AtomicInteger 而非 synchronized
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();  // CAS 操作，无锁

// 而非
synchronized (lock) {
    count++;  // 有锁竞争，可能引起上下文切换
}
```

适用场景：计数器、序列号生成、标志位更新等简单操作。

**注意**：CAS 在高竞争场景下会大量自旋重试，反而浪费 CPU。此时可以考虑 LongAdder（分散热点）。

#### 3.2 使用最少线程

线程数量过多会导致频繁的上下文切换。应该根据实际任务类型来合理设置线程数：

| 任务类型 | 推荐线程数 | 原因 |
|----------|-----------|------|
| **CPU 密集型** | CPU 核心数 + 1 | 避免过多线程竞争 CPU |
| **IO 密集型** | CPU 核心数 * 2 | 或根据 IO 等待比例计算 |
| **混合型** | 根据实际压测调整 | 先压测再确定最优线程数 |

IO 密集型的更精确公式：
```
线程数 = CPU 核心数 * (1 + IO 耗时 / CPU 耗时)
```

**线程池的合理配置**：

```java
// CPU 密集型
ThreadPoolExecutor cpuPool = new ThreadPoolExecutor(
    Runtime.getRuntime().availableProcessors() + 1,
    Runtime.getRuntime().availableProcessors() + 1,
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000)
);

// IO 密集型
int ioThreads = Runtime.getRuntime().availableProcessors() * 2;
ThreadPoolExecutor ioPool = new ThreadPoolExecutor(
    ioThreads, ioThreads,
    60L, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(1000)
);
```

#### 3.3 协程（Coroutine）

协程是一种**用户态**的轻量级线程，切换时不涉及内核态，切换开销远小于线程切换。

| 维度 | 线程 | 协程 |
|------|------|------|
| 调度方式 | 操作系统内核调度 | 用户态调度 |
| 切换开销 | 大（内核态切换） | 极小（用户态切换） |
| 栈大小 | 通常 1MB | 通常几 KB |
| 数量限制 | 通常数千个 | 可以创建数十万个 |

Java 中可以使用虚拟线程（JDK 21+，Project Loom）来实现类似协程的效果：

```java
// JDK 21 虚拟线程
Thread.startVirtualThread(() -> {
    // 虚拟线程的执行体
    // IO 阻塞时会自动让出底层平台线程
    doSomeIOWork();
});
```

虚拟线程在遇到阻塞操作时会自动让出底层的平台线程（载体线程），避免了传统线程的上下文切换开销。

#### 3.4 合理的 CPU 调度策略

| 策略 | 说明 |
|------|------|
| **CPU 亲和性（Affinity）** | 将线程绑定到特定 CPU 核心，减少缓存失效 |
| **减少线程优先级调整** | 避免频繁的优先级反转导致的切换 |
| **NUMA 感知调度** | 在 NUMA 架构中，将线程调度到内存访问最近的 CPU |

#### 3.5 避免不必要的线程阻塞和唤醒

| 方法 | 说明 |
|------|------|
| **减少 synchronized 的使用** | 使用 CAS 或分段锁减少锁竞争 |
| **使用 Lock 替代 wait/notify** | Condition 可以精确唤醒，避免虚假唤醒 |
| **批量处理** | 将多次小操作合并为一次大操作 |
| **异步非阻塞** | 使用 CompletableFuture、响应式编程减少阻塞等待 |

### 四、上下文切换的监控和诊断

#### 4.1 Linux 命令

```bash
# 查看系统整体上下文切换情况
vmstat 1
# 输出中的 cs 列即为每秒上下文切换次数

# 查看特定进程的上下文切换
pidstat -w -p <pid> 1
# cswch/s: 自愿上下文切换（IO 等导致）
# nvcswch/s: 非自愿上下文切换（时间片用完）

# 查看线程级别的上下文切换
pidstat -w -t -p <pid> 1
```

#### 4.2 Java 工具

```bash
# jstack 查看线程状态
jstack <pid>

# Java Flight Recorder（JFR）记录上下文切换事件
jcmd <pid> JFR.start duration=60s filename=recording.jfr
```

#### 4.3 上下文切换多少算多？

- **正常范围**：每秒几千到几万次。
- **需关注**：每秒超过 10 万次。
- **需优化**：每秒超过 100 万次。

但这取决于具体场景和硬件，需要结合 CPU 使用率一起分析。如果 CPU 使用率低但上下文切换频繁，说明大量线程在等待；如果 CPU 使用率高且上下文切换频繁，说明线程过多。

### 五、总结

| 优化方向 | 具体措施 | 效果 |
|----------|----------|------|
| **减少锁竞争** | CAS / 无锁数据结构 | 减少线程阻塞和唤醒 |
| **减少线程数** | 合理配置线程池 | 减少调度切换频率 |
| **使用协程** | 虚拟线程 / 用户态调度 | 降低切换开销 |
| **异步化** | CompletableFuture / 响应式 | 减少阻塞等待 |
| **批量处理** | 合并小操作 | 减少切换次数 |
