# 悲观锁的实现机制

## 问题
悲观锁有哪些实现方式？synchronized 的原理实现是什么？锁升级过程是怎样的？ReentrantLock 和 synchronized 有什么区别？

## 答案

### 一、悲观锁的实现方式总览

| 实现方式 | 机制 | 特点 |
|----------|------|------|
| **synchronized** | JVM层面，Monitor机制，锁升级 | 自动释放，非公平 |
| **ReentrantLock** | AQS框架，CAS+队列 | 需手动unlock，支持公平/非公平、可中断 |
| **数据库悲观锁** | `SELECT ... FOR UPDATE` | 行级锁，事务提交释放 |

---

### 二、synchronized 的原理实现

#### 2.1 synchronized 的三种使用方式

| 使用方式 | 锁对象 | 示例 |
|----------|--------|------|
| 修饰实例方法 | 当前实例对象 `this` | `public synchronized void doSomething()` |
| 修饰静态方法 | 当前类的 Class 对象 | `public static synchronized void doSomething()` |
| 修饰代码块 | 括号中指定的对象 | `synchronized(obj) { ... }` |

#### 2.2 字节码层面的实现

synchronized 在字节码层面通过 **monitorenter** 和 **monitorexit** 两条指令来实现：

- **monitorenter**：插入到同步代码块的开始位置。执行时会尝试获取对象的 Monitor 所有权。
  - 如果 Monitor 的进入计数器（recursion count）为 0，则获取成功，计数器置为 1，当前线程成为 Monitor 的 owner。
  - 如果当前线程已经持有该 Monitor，计数器加 1（可重入机制）。
  - 如果其他线程已经持有该 Monitor，当前线程进入阻塞状态，直到 Monitor 的计数器为 0。

- **monitorexit**：插入到同步代码块的结束位置和异常位置（保证异常时也能释放锁）。执行时将 Monitor 的计数器减 1，如果计数器为 0，则释放 Monitor，其他被阻塞的线程可以尝试获取。

```java
// 源码
synchronized (obj) {
    // 同步代码块
}

// 字节码（简化）
monitorenter obj
// ... 同步代码块的字节码 ...
monitorexit obj
// 异常处理中也会执行 monitorexit（保证锁释放）
```

#### 2.3 Monitor 机制

每个 Java 对象头中都关联了一个 **Monitor**（监视器锁/管程）。Monitor 是由 C++ 实现的，主要包含以下组成部分：

| 组件 | 作用 |
|------|------|
| **Owner** | 记录当前持有锁的线程，初始为 null |
| **EntryList** | 阻塞等待锁的线程队列（BLOCKED 状态） |
| **WaitSet** | 调用了 wait() 方法等待的线程集合（WAITING 状态） |
| **recursion** | 重入计数器 |

**工作流程**：
1. 线程进入 synchronized 代码块时，JVM 检查 Monitor 的 Owner 是否为空。
2. 如果为空，当前线程成为 Owner，recursion 置为 1。
3. 如果 Owner 是当前线程，recursion 加 1（可重入）。
4. 如果 Owner 是其他线程，当前线程进入 EntryList 阻塞等待。
5. 当 Owner 线程执行完毕离开同步块，recursion 减为 0，释放 Monitor，从 EntryList 中唤醒一个线程。
6. 当 Owner 线程调用 wait()，进入 WaitSet 等待，释放 Monitor。

#### 2.4 Java 对象头与 Mark Word

在 HotSpot 虚拟机中，对象在内存中的布局分为三部分：对象头（Header）、实例数据（Instance Data）和对齐填充（Padding）。

**对象头**又分为两部分：
- **Mark Word**（标记字段）：存储对象的哈希码、GC 分代年龄、锁状态标志、线程持有的锁、偏向线程 ID 等。
- **Klass Pointer**（类型指针）：指向对象的类元数据，JVM 通过此指针确定对象是哪个类的实例。

**Mark Word 的结构**（以 64 位 JVM 为例，共 64 bit）：

| 锁状态 | 内容 | 标志位 |
|--------|------|--------|
| 无锁 | hashcode(31) + age(4) + biased_lock(1) | 01 |
| 偏向锁 | thread_id(54) + epoch(2) + age(4) + biased_lock(1) | 01 |
| 轻量级锁 | 指向栈中锁记录的指针(62) | 00 |
| 重量级锁 | 指向 Monitor 对象的指针(62) | 10 |
| GC 标记 | 空 | 11 |

### 三、synchronized 锁升级过程

JDK 1.6 引入了锁升级机制来减少锁的开销，锁状态可以从低到高依次升级：**无锁 → 偏向锁 → 轻量级锁 → 重量级锁**。升级是**不可逆**的（只能升级不能降级，但偏向锁可以被撤销）。

#### 3.1 无锁状态

- Mark Word 标志位为 `01`，biased_lock 为 `0`。
- 对象没有被任何线程锁定，hashcode 存储在 Mark Word 中。
- 如果开启了偏向锁（默认开启），新建对象在刚创建后的几秒内处于"匿名偏向"状态（biased_lock=1，thread_id=0），等待第一个获取锁的线程来"偏向"它。

#### 3.2 偏向锁（Biased Locking）

**适用场景**：只有同一个线程反复进入同步块，不存在多线程竞争。

**获取过程**：
1. 线程第一次进入 synchronized 代码块时，JVM 使用 CAS 将线程 ID 记录到对象头的 Mark Word 中。
2. 后续该线程再次进入同步块时，只需检查 Mark Word 中的线程 ID 是否是自己的，如果是则直接进入，无需任何 CAS 操作或阻塞。
3. 这大大减少了无竞争情况下的锁获取开销。

**撤销过程**：
1. 当有另一个线程尝试获取偏向锁时，偏向锁会被撤销。
2. 撤销需要等到**全局安全点**（Safe Point，即没有字节码正在执行的时刻）。
3. 检查原来持有偏向锁的线程是否还存活：
   - 如果已不存活，则将对象头恢复为无锁状态。
   - 如果仍存活，则将偏向锁升级为轻量级锁（原来持有偏向锁的线程继续执行）。

**注意**：JDK 15 开始默认禁用偏向锁（JEP 374），因为现代应用中多线程竞争普遍，偏向锁的撤销开销反而成为负担。JDK 18 彻底移除了偏向锁。

#### 3.3 轻量级锁（Lightweight Locking）

**适用场景**：多个线程交替进入同步块（没有同时竞争），或偏向锁被撤销后的状态。

**加锁过程**：
1. JVM 在当前线程的栈帧中创建一个**锁记录**（Lock Record）空间。
2. 将对象头的 Mark Word 复制到锁记录中（称为 **Displaced Mark Word**）。
3. 线程使用 CAS 尝试将对象头的 Mark Word 替换为指向锁记录的指针：
   - 如果成功，当前线程获得轻量级锁。
   - 如果失败，说明存在竞争，进入自旋等待。

**自旋等待**：
- 获取失败的线程不会立即阻塞，而是通过**自旋**（空循环）不断尝试获取锁。
- JDK 1.6 引入了**自适应自旋**：根据前一次在同一个锁上的自旋时间和锁的持有者状态来动态调整自旋次数。
- 如果自旋超过一定次数仍未获取到锁，则膨胀为重量级锁。

**解锁过程**：
1. 使用 CAS 将 Displaced Mark Word 替换回对象头。
2. 如果替换成功，同步完成。
3. 如果替换失败（说明已经膨胀为重量级锁），则在释放锁的同时唤醒被阻塞的线程。

#### 3.4 重量级锁（Heavyweight Locking）

**适用场景**：多个线程同时竞争同一把锁。

**实现**：
1. 对象头的 Mark Word 被替换为指向 **Monitor 对象**的指针。
2. 未获取到锁的线程会被操作系统挂起（从用户态切换到内核态），进入 EntryList 阻塞等待。
3. 当持有锁的线程释放锁后，操作系统负责唤醒一个阻塞的线程。

**开销**：
- 涉及用户态和内核态的切换，开销较大。
- 线程阻塞和唤醒需要操作系统介入，成本远高于自旋。

#### 3.5 锁升级完整流程

```
new Object（匿名偏向 / 无锁）
    │
    ▼
第一个线程获取锁 ──→ 偏向锁（CAS写入线程ID）
    │
    ▼
另一个线程尝试获取锁 ──→ 偏向锁撤销 ──→ 轻量级锁（自旋CAS）
    │
    ▼
自旋失败（竞争激烈）──→ 重量级锁（Monitor，线程阻塞）
```

### 四、ReentrantLock

#### 4.1 ReentrantLock 是什么

ReentrantLock 是 `java.util.concurrent.locks` 包下的一个类，实现了 `Lock` 接口。它基于 **AQS**（AbstractQueuedSynchronizer，抽象队列同步器）实现，提供了与 synchronized 类似的互斥锁功能，但功能更加丰富。

ReentrantLock 支持：
- **可重入**：同一线程可以多次获取同一把锁。
- **公平锁 / 非公平锁**：可以选择是否按照等待顺序来获取锁。
- **可中断获取**：等待锁的过程中可以响应中断。
- **超时获取**：在指定时间内尝试获取锁。
- **多条件变量**：通过 Condition 实现精确的线程等待/通知。

#### 4.2 公平锁与非公平锁

```java
// 非公平锁（默认）
ReentrantLock lock = new ReentrantLock();         // 默认非公平
ReentrantLock lock = new ReentrantLock(false);     // 显式非公平

// 公平锁
ReentrantLock lock = new ReentrantLock(true);      // 公平锁
```

- **非公平锁**：新来的线程会直接尝试 CAS 抢锁，不管队列中是否有等待的线程。如果失败再进入队列排队。吞吐量较高。
- **公平锁**：新来的线程会先检查队列中是否有等待的线程，如果有则乖乖排队。保证 FIFO 顺序，但吞吐量较低。

#### 4.3 synchronized vs ReentrantLock 详细对比

| 特性 | synchronized | ReentrantLock |
|------|--------------|---------------|
| **实现方式** | JVM 层面（Monitor，字节码指令） | API 层面（AQS，CAS + CLH 队列） |
| **释放方式** | 自动释放（退出同步块或异常时） | 手动释放（必须在 finally 中调用 unlock()） |
| **可中断** | 不支持，线程只能被动等待锁 | 支持 `lockInterruptibly()`，可响应中断 |
| **超时获取** | 不支持 | 支持 `tryLock(timeout, unit)` |
| **公平锁** | 只支持非公平 | 支持公平和非公平两种模式 |
| **Condition** | 只有一个等待队列（wait/notify） | 支持多个 Condition，可精确唤醒 |
| **锁绑定** | 一次只能绑定一个条件 | 一个锁可以绑定多个 Condition |
| **性能** | JDK6+ 优化后与 ReentrantLock 接近 | 在高竞争场景下稍优于 synchronized |
| **可重入** | 支持 | 支持 |
| **底层实现** | 字节码层面的 monitorenter/monitorexit | 基于 AQS 的 tryAcquire/tryRelease |
| **锁信息存储** | 对象头 Mark Word | AQS 的 volatile state + Node 队列 |
| **适用建议** | 简单同步场景优先选择（代码简洁、JVM 优化好） | 需要高级功能（公平、中断、多条件、超时）时选择 |

#### 4.4 如何选择

- **优先使用 synchronized**：在只需要简单的互斥同步时，synchronized 是最佳选择。JVM 对其做了大量优化（锁升级、锁消除、锁粗化），且不需要手动释放锁，代码更简洁安全。
- **使用 ReentrantLock**：当需要以下高级功能时：
  - 需要公平锁保证线程获取锁的顺序
  - 需要可中断的锁获取（避免死锁）
  - 需要超时获取锁（`tryLock`）
  - 需要多个条件队列精确唤醒线程
  - 需要非块结构的加锁（在不同的方法中加锁和解锁）

### 五、数据库悲观锁

```sql
-- 在事务中使用 FOR UPDATE 锁定行
BEGIN;
SELECT * FROM accounts WHERE id = 1 FOR UPDATE;
-- 期间其他事务无法修改 id=1 的行
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;
```

特点：
- 行级锁，锁定查询到的行。
- 事务提交或回滚后自动释放。
- 适合写多读少的场景。
- 可能导致死锁，需要合理设计加锁顺序。
