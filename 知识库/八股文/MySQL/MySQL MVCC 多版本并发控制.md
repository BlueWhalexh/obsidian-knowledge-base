# MySQL MVCC 多版本并发控制

## 问题
什么是 MVCC？ReadView 机制是什么？RC 和 RR 隔离级别下 ReadView 的生成时机有什么区别？

## 事务基础

### 事务四大特性（ACID）

事务是数据库操作的最小逻辑单元，具有以下四个特性：

| 特性 | 含义 | 实现原理 |
|------|------|----------|
| **原子性（Atomicity）** | 事务中的操作要么全部成功，要么全部失败回滚 | **undo log**：记录数据修改前的逆向操作，回滚时执行 undo log 恢复数据 |
| **一致性（Consistency）** | 事务前后数据库从一个一致性状态转换到另一个一致性状态 | 由其他三个特性共同保证：原子性 + 隔离性 + 持久性 |
| **隔离性（Isolation）** | 多个并发事务之间互不干扰 | **锁 + MVCC**：通过锁机制和多版本并发控制实现事务隔离 |
| **持久性（Durability）** | 事务提交后，对数据的修改是永久性的 | **redo log**：采用 WAL（Write-Ahead Logging）机制，先写日志再写磁盘，保证宕机后数据不丢失 |

**四大特性的关系**：
- 持久性（redo log）和原子性（undo log）是 InnoDB 存储引擎层面的实现
- 隔离性是数据库层面的实现（锁 + MVCC）
- 一致性是应用层面的目标，由其他三个特性共同保证

### 并发事务带来的问题

当多个事务并发执行时，如果没有合适的隔离机制，会产生以下问题：

#### 1. 脏读（Dirty Read）

**定义**：一个事务读取到了另一个事务**尚未提交**的数据。

**示例**：
```
时间线 | 事务A                     | 事务B
-------|--------------------------|------------------------
T1     | BEGIN;                   |
T2     | UPDATE account SET       |
       | balance = 200 WHERE id=1 |
T3     |                          | BEGIN;
T4     |                          | SELECT balance FROM account WHERE id=1;
       |                          | -- 读到 200（事务A未提交的数据）
T5     | ROLLBACK;                | -- 事务A回滚，balance恢复为100
T6     |                          | -- 事务B读到的200是"脏数据"
```

**危害**：事务B基于脏数据做了业务判断，可能导致严重错误。例如：银行转账场景中，读到了对方尚未提交的转账金额。

#### 2. 不可重复读（Non-Repeatable Read）

**定义**：同一个事务内，多次读取同一行数据，结果不一致。原因是其他事务在两次读之间**修改并提交**了该数据。

**示例**：
```
时间线 | 事务A                     | 事务B
-------|--------------------------|------------------------
T1     | BEGIN;                   |
T2     | SELECT balance FROM      |
       | account WHERE id=1;      |
       | -- 读到 100              |
T3     |                          | BEGIN;
T4     |                          | UPDATE account SET balance=200 WHERE id=1;
T5     |                          | COMMIT;
T6     | SELECT balance FROM      |
       | account WHERE id=1;      |
       | -- 读到 200（前后不一致）  |
T7     | COMMIT;                  |
```

**危害**：同一事务内两次读取结果不同，破坏了事务的一致性视图。例如：先读取库存判断够不够，再读取时发现已被别人改了。

#### 3. 幻读（Phantom Read）

**定义**：同一个事务内，多次执行相同的范围查询，返回的**行数**不一致。原因是其他事务在两次读之间**插入或删除**了满足条件的行。

**示例**：
```
时间线 | 事务A                          | 事务B
-------|-------------------------------|------------------------
T1     | BEGIN;                        |
T2     | SELECT * FROM orders          |
       | WHERE amount > 100;           |
       | -- 返回 3 条记录               |
T3     |                               | BEGIN;
T4     |                               | INSERT INTO orders (amount) VALUES (200);
T5     |                               | COMMIT;
T6     | SELECT * FROM orders          |
       | WHERE amount > 100;           |
       | -- 返回 4 条记录（多了"幻影行"）|
T7     | COMMIT;                       |
```

**危害**：范围查询结果集发生变化，可能导致统计错误或业务逻辑异常。例如：统计高价值订单数量，前后不一致。

**不可重复读 vs 幻读的区别**：
- 不可重复读：侧重于**同一行数据**的值被修改（UPDATE）
- 幻读：侧重于**行数**发生变化（INSERT/DELETE）

### 事务隔离级别

SQL 标准定义了四种隔离级别，从低到高依次解决上述并发问题：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 说明 |
|---------|------|-----------|------|------|
| **读未提交（Read Uncommitted）** | 可能 | 可能 | 可能 | 最低隔离级别，可以读到未提交的数据 |
| **读已提交（Read Committed, RC）** | 解决 | 可能 | 可能 | 只能读到已提交的数据，Oracle 默认级别 |
| **可重复读（Repeatable Read, RR）** | 解决 | 解决 | 可能* | 保证同一事务内多次读取结果一致，MySQL 默认级别 |
| **串行化（Serializable）** | 解决 | 解决 | 解决 | 最高隔离级别，完全串行执行，并发性能最差 |

> *注：MySQL 的 RR 级别通过 **MVCC + Next-Key Lock** 在很大程度上解决了幻读问题，但并非完全消除。在某些特殊场景下（如先快照读再当前读），仍可能出现幻读。

**各隔离级别的实现原理**：
- **Read Uncommitted**：不加锁，直接读最新数据
- **Read Committed**：每次 SELECT 生成新的 ReadView（MVCC）
- **Repeatable Read**：事务中第一次 SELECT 生成 ReadView，后续复用（MVCC）
- **Serializable**：读加共享锁，写加排他锁，完全串行

> MySQL 默认隔离级别是 **Repeatable Read（RR）**，而非 SQL 标准中的 Read Committed。通过 MVCC 机制，MySQL 在 RR 级别下既能保证数据一致性，又能提供良好的并发性能。

## 答案

### 快照读
快照读是指读取的是当前行数据之前的历史版本，而非最新数据。快照读基于某个时间点的查询结果，不考虑同时运行的其他事务所做的更改。

在 MySQL 中，只有 **READ COMMITTED** 和 **REPEATABLE READ** 这两种隔离级别才会使用快照读。

### 版本链
InnoDB 中每行记录在更新时，会生成一条 undo log，这些 undo log 通过 `DB_ROLL_PTR` 指针串联成一个版本链。版本链的头节点是当前记录的最新值，每个节点记录了该版本的事务ID（`DB_TRX_ID`）。

### ReadView 机制
ReadView（读视图）是 MVCC 的核心，包含以下关键字段：
- **m_ids**：创建 ReadView 时，当前系统中所有活跃（未提交）的事务ID列表
- **min_trx_id**：活跃事务中最小的事务ID
- **max_trx_id**：系统应分配给下一个事务的ID（当前最大事务ID + 1）
- **creator_trx_id**：创建该 ReadView 的事务ID

**可见性判断规则**：
1. 如果记录的 `DB_TRX_ID` < `min_trx_id`，说明该版本在 ReadView 创建前已提交，**可见**
2. 如果记录的 `DB_TRX_ID` >= `max_trx_id`，说明该版本在 ReadView 创建后才产生，**不可见**
3. 如果 `min_trx_id` <= `DB_TRX_ID` < `max_trx_id`，则判断是否在 `m_ids` 中：
   - 在 `m_ids` 中：说明该版本由未提交事务产生，**不可见**
   - 不在 `m_ids` 中：说明该版本已提交，**可见**

若不可见，则沿版本链向前查找，直到找到可见版本或链尾。

### RC 与 RR 的核心区别：ReadView 生成时机

| 隔离级别 | ReadView 生成时机 | 效果 |
|---------|-----------------|------|
| **RC（读已提交）** | 每次 SELECT 都创建一个新的 ReadView | 每次查询能看到其他事务已提交的最新数据，存在不可重复读 |
| **RR（可重复读）** | 事务中第一次 SELECT 时创建 ReadView，后续查询复用 | 同一事务内多次查询结果一致，实现可重复读 |

**RR 如何实现可重复读**：由于 ReadView 只在首次查询创建，整个事务期间复用同一个 ReadView，所以即使其他事务提交了新的修改，当前事务看到的仍是首次查询时的数据快照。
