# MySQL 三种日志

## 问题
MySQL 的 binlog、redo log、undo log 分别是什么？有什么作用？

## 答案

### 三种日志概览

| 日志类型 | 层级 | 类型 | 作用 |
|---------|------|------|------|
| **redo log** | InnoDB 引擎层 | 物理日志（记录页修改） | 崩溃恢复（持久性） |
| **undo log** | InnoDB 引擎层 | 逻辑日志（记录逆操作） | 事务回滚（原子性）+ MVCC |
| **binlog** | Server 层 | 逻辑日志（SQL/行变更） | 主从复制、数据备份 |

### binlog（归档日志）

**binlog** 是 MySQL Server 层实现的日志，所有存储引擎都可以使用。

**用途**：主从复制、数据备份、增量恢复。

#### BinLog 的三种格式

**1. Statement 格式**
- **记录内容**：记录每条修改数据的 SQL 语句原文
- **优点**：日志文件小，不需要记录每一行的变化，节约 IO 资源
- **缺点**：某些函数（如 `NOW()`、`UUID()`、`RAND()`）在主从复制时会产生不一致的结果。例如 `UPDATE user SET age=age+1` 在主库执行和在从库执行结果可能不同
- **适用场景**：SQL 语句不涉及不确定函数的场景

**2. Row 格式**
- **记录内容**：记录每行数据修改前后的具体值（而不是 SQL 语句）
- **优点**：数据准确性高，不会出现主从不一致的问题。能够清晰记录每一行数据的变化细节
- **缺点**：日志文件大，尤其是批量更新（`UPDATE users SET status=1`）时，会产生大量日志
- **适用场景**：对数据一致性要求高的生产环境，**推荐使用**

**3. Mixed 格式**
- **记录内容**：MySQL 自动判断，普通 SQL 用 Statement 格式，涉及不确定函数时自动切换为 Row 格式
- **优点**：兼顾了 Statement 的日志量小和 Row 的数据准确性
- **缺点**：在某些特殊情况下仍可能出问题，不如 Row 格式可靠
- **适用场景**：无法确定使用哪种格式时的折中方案

> **生产建议**：推荐使用 **Row 格式**，虽然日志文件较大，但数据一致性最有保障。

### redo log（重做日志）

**redo log** 是 InnoDB 引擎特有的日志，属于物理日志，记录的是"某个数据页上做了什么修改"。

**用途**：宕机恢复，保证事务的持久性（Durability）。

#### 为什么 Redo Log 使用 WAL 技术

**WAL（Write-Ahead Logging）** 即"先写日志，再写磁盘"。

**核心问题**：InnoDB 以页（Page，16KB）为单位管理数据。如果每次事务提交都直接将修改的脏页刷回磁盘，会产生大量的**随机 IO**，性能极差。

**WAL 的解决方案**：
1. 事务提交时，先将修改写入 redo log（顺序 IO，速度快）
2. 脏页的刷盘操作由后台线程异步完成
3. 即使宕机，也可以通过 redo log 恢复未刷盘的数据

**随机 IO vs 顺序 IO**：
- **随机 IO**：写入磁盘的不同位置，磁头需要频繁寻道，速度慢（约 100 IOPS）
- **顺序 IO**：写入连续的磁盘位置，磁头不需要移动，速度快（约 500+ MB/s）
- redo log 采用追加写入的方式，将随机 IO 转化为顺序 IO，极大提升了写入性能

**redo log 的结构**：
- redo log 是固定大小的文件组，采用循环写入方式
- 由 `ib_logfile0` 和 `ib_logfile1` 等文件组成（默认各 48MB）
- 包含两个指针：`write pos`（当前写入位置）和 `checkpoint`（已刷盘位置）
- 当 write pos 追上 checkpoint 时，需要停下来先刷盘再继续

**redo log 的两阶段提交**：
1. **Prepare 阶段**：将 redo log 写入磁盘，并标记为 prepare 状态
2. **Commit 阶段**：将 binlog 写入磁盘，然后将 redo log 标记为 commit 状态

两阶段提交保证了 redo log 和 binlog 的一致性。如果在 prepare 阶段后崩溃，恢复时会检查 binlog 是否完整：binlog 完整则提交事务，不完整则回滚事务。

### undo log（回滚日志）

**undo log** 是 InnoDB 引擎实现的逻辑日志，记录的是数据修改前的逆向操作。

**用途**：
1. **事务回滚**：保证事务的原子性（Atomicity）。事务失败时，执行 undo log 中的逆向操作恢复数据
2. **MVCC**：通过 undo log 版本链实现多版本并发控制。每条 undo log 记录了修改前的数据版本和事务 ID，串联成版本链，配合 ReadView 实现非锁定读

**undo log 的类型**：
- **INSERT 操作**：对应的 undo log 只在事务回滚时使用，事务提交后可立即删除
- **UPDATE/DELETE 操作**：对应的 undo log 还可能被其他事务用于 MVCC 读取，不能立即删除，由 purge 线程负责清理
