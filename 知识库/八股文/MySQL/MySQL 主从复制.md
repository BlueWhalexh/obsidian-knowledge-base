# MySQL 主从复制

## 问题
MySQL 主从复制的原理是什么？有哪些复制方式？

## 答案

### 主从复制原理
MySQL 主从复制是将主数据库的 DDL 和 DML 操作通过二进制日志（binlog）传到从库，从库对这些日志重新执行（重做），从而保持主从数据同步。

### 复制流程

1. **从服务器**在开启主从复制后，创建两个线程：**I/O 线程**和 **SQL 线程**
2. 从服务器的 I/O 线程尝试与主服务器建立连接，主服务器创建 **binlog dump 线程**与之交互
3. 从服务器的 I/O 线程告知主服务器的 dump 线程自己要从什么位置开始接收 binlog
4. 主服务器在更新过程中将更改记录保存到自己的 binlog 中
5. dump 线程检测到 binlog 变化后，从指定位置读取内容，发送给从服务器的 I/O 线程（拉模式）
6. 从服务器的 I/O 线程将接收到的内容保存在**中继日志（relay log）**中
7. 从服务器的 **SQL 线程**不断读取中继日志中的内容，解析为具体操作并写入自己的数据表中

### 复制方式

#### 异步复制（默认）
- 主库执行完事务后，**立即返回客户端**，不关心从库是否完成同步
- **优点**：性能好，延迟低
- **缺点**：主库故障时，可能丢失尚未同步到从库的数据

#### 全同步复制
- 主库执行完事务后，**等待所有从库完成数据复制**后才返回客户端
- **优点**：数据安全性高
- **缺点**：性能差，从库越多延迟越高

#### 半同步复制
- 主库执行完事务后，**等待其中一个从库完成接收事件**后，再返回客户端
- **优点**：兼顾性能和数据安全
- **缺点**：比异步复制稍慢，但比全同步快很多

### 实践优化
- **无磁盘复制**：配置 `repl-diskless-sync yes`，避免全量同步时的磁盘IO
- **控制单节点内存**：减少全量同步时的IO开销
- **提高 repl_baklog 大小**：减少全量同步概率，尽快恢复断连的从节点
- **主-从-从架构**：当从节点过多时，让部分从节点从其他从节点同步，减轻主节点压力
