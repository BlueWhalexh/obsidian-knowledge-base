# MySQL 联合索引最左前缀原则

## 问题
对于 `SELECT * FROM table WHERE a > 0 AND b = 0` 这样的查询，应该如何设计索引？为什么 MySQL 使用 B+ 树作为索引结构？

## 答案

### 为什么使用 B+ 树作为索引

MySQL 的 InnoDB 存储引擎使用 B+ 树作为索引的底层数据结构。选择 B+ 树而非其他数据结构，是经过深思熟虑的设计决策。

#### 为什么不使用哈希表？

- **不支持范围查询**：哈希表只能精确匹配（`=`、`IN`），无法处理 `>`、`<`、`BETWEEN`、`LIKE` 等范围查询
- **不支持排序**：哈希表的存储是无序的，无法利用索引完成 `ORDER BY` 操作
- **哈希冲突**：当数据量大时，哈希冲突会影响查询性能
- **适用场景**：MySQL 的 Memory 引擎支持 Hash 索引，适合等值查询的临时表场景

#### 为什么不使用二叉搜索树 / 红黑树？

- **树的高度过高**：二叉树每个节点最多只有两个子节点，存储大量数据时树的高度会非常大（例如 100 万条数据，理想情况下树高约 20 层）
- **IO 次数过多**：树每高一层，就需要多一次磁盘 IO。数据库的数据存储在磁盘上，过高的树会导致查询时频繁磁盘 IO，性能极差
- **红黑树虽然能自平衡**：但仍然是二叉树，高度问题无法根本解决

#### 为什么不使用 B 树？

B 树虽然比二叉树好，但 B+ 树在数据库场景下更具优势（详见下文 B 树 vs B+ 树对比）。

#### B+ 树的优势

1. **矮胖结构**：每个节点可以存放多个键值（InnoDB 默认一个节点 16KB），3 层 B+ 树就能存储约 2000 万条数据，查询只需 3 次磁盘 IO
2. **叶子节点链表**：叶子节点通过双向链表连接，天然支持范围查询和顺序遍历
3. **非叶子节点只存键**：非叶子节点不存储数据，只存储索引键，使得每个节点能容纳更多键值，进一步降低树高
4. **查询性能稳定**：所有数据都在叶子节点，每次查询的 IO 次数相同（都是树的高度），性能可预测

### B 树和 B+ 树的区别

| 特性 | B 树 | B+ 树 |
|------|------|-------|
| **数据存储位置** | 所有节点都存储数据 | 只有叶子节点存储数据，非叶子节点只存索引键 |
| **叶子节点链表** | 叶子节点之间没有指针连接 | 叶子节点通过双向链表连接 |
| **范围查询** | 需要中序遍历整棵树 | 通过叶子节点链表直接遍历，效率高 |
| **查询性能** | 不稳定（可能在任意层找到数据） | 稳定（必须到叶子节点才能找到数据） |
| **节点空间利用率** | 非叶子节点也存数据，能存的键更少 | 非叶子节点不存数据，能存更多键，树更矮 |

**核心区别详解**：

1. **非叶子节点不存数据**：B+ 树的非叶子节点只存储索引键和子节点指针，不存储实际数据行。这意味着同样大小的节点（16KB），B+ 树能存放更多的索引键，使树更加"矮胖"，减少磁盘 IO 次数。

2. **叶子节点链表**：B+ 树的所有叶子节点通过双向链表相连。这使得范围查询非常高效——找到起始叶子节点后，直接沿链表顺序遍历即可，无需回到父节点再遍历。例如 `WHERE a BETWEEN 10 AND 100`，只需定位到 a=10 的叶子节点，然后沿链表遍历到 a=100。

3. **查询性能稳定**：B 树的查询可能在任意层找到数据（最好 O(1)，最差 O(logN)），而 B+ 树的查询必须到达叶子节点（稳定 O(logN)）。对于数据库这种需要性能可预测的场景，B+ 树更合适。

### 联合索引最左前缀原则

#### 什么是联合索引

联合索引是指在多个列上建立的索引，例如 `INDEX idx_a_b_c (a, b, c)`。联合索引的 B+ 树按照索引列的顺序依次排序：先按 a 排序，a 相同时按 b 排序，b 相同时按 c 排序。

#### 最左前缀原则

联合索引遵循最左前缀原则：查询条件必须从索引的最左列开始，并且不跳过中间列，才能使用索引。

**以 `INDEX idx_a_b_c (a, b, c)` 为例**：

| 查询条件 | 是否使用索引 | 原因 |
|---------|------------|------|
| `WHERE a = 1` | 使用（a） | 匹配最左列 |
| `WHERE a = 1 AND b = 2` | 使用（a, b） | 从最左列连续匹配 |
| `WHERE a = 1 AND b = 2 AND c = 3` | 使用（a, b, c） | 完全匹配 |
| `WHERE b = 2` | 不使用 | 跳过了最左列 a |
| `WHERE b = 2 AND c = 3` | 不使用 | 跳过了最左列 a |
| `WHERE a = 1 AND c = 3` | 部分使用（a） | a 可以用索引，c 被跳过无法使用 |
| `WHERE a > 1 AND b = 2` | 部分使用（a） | a 是范围查询，导致 b 无法使用索引 |

#### 为什么范围查询会导致后续索引失效？

联合索引 `(a, b, c)` 的 B+ 树先按 a 排序，a 相同时再按 b 排序，b 相同时再按 c 排序。当 a 使用范围查询（`>`、`<`、`BETWEEN`）时，a 的值不再是一个固定值，而是一个范围。在这个范围内，b 的值是无序的（因为 b 的有序性建立在 a 相同的基础上），因此无法利用 b 的索引进行快速查找。

### `SELECT * FROM table WHERE a > 0 AND b = 0` 索引怎么加？

这种情况要加联合索引 `(b, a)`。原因是最左前缀原则：b 是等值查询，可以用索引；a 是范围查询，会导致后续索引失效，所以 a 放后面。如果只查部分列，可以考虑覆盖索引，把查询列都放进索引里，避免回表。

如果 a 和 b 都是范围查询，那就只能建单列索引让优化器自己选，或者改写 SQL 拆成多个查询再合并。
