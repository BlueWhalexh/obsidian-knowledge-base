# RR 与 RC 隔离级别区别

## 问题
RR（可重复读）和 RC（读已提交）隔离级别有什么区别？为什么互联网公司普遍选择 RC？

## 答案

### 一、ReadView 生成时机差异

| 隔离级别 | ReadView 生成时机 | 效果 |
|---------|-----------------|------|
| **RC** | 每次 SELECT 创建新的 ReadView | 能看到其他事务已提交的最新数据，存在不可重复读 |
| **RR** | 事务中第一次 SELECT 创建，后续复用 | 同一事务内结果一致，实现可重复读 |

这是两者最本质的区别。RC 每次查询都会更新 ReadView 中的活跃事务列表，所以能看到其他事务刚提交的数据；而 RR 始终使用首次查询的 ReadView 快照。

### 二、锁机制差异

MySQL 有三种锁类型：
- **Record Lock（记录锁）**：锁住索引记录
- **Gap Lock（间隙锁）**：锁住索引记录之间的间隙
- **Next-Key Lock（临键锁）**：Record Lock + Gap Lock，左开右闭区间

| 隔离级别 | 锁机制 |
|---------|--------|
| **RC** | 只使用 Record Lock |
| **RR** | Record Lock + Gap Lock + Next-Key Lock |

RR 级别下为了防止幻读，引入了 Gap Lock 和 Next-Key Lock，锁的粒度更大。

### 三、binlog 格式支持差异

| 隔离级别 | 支持的 binlog 格式 |
|---------|-------------------|
| **RC** | 只支持 **ROW** 格式 |
| **RR** | 支持 **STATEMENT、ROW、MIXED** 三种格式 |

### 四、为什么互联网公司选择 RC

#### 1. 减少锁粒度，提升并发
RC 只使用 Record Lock，不加 Gap Lock 和 Next-Key Lock。锁粒度更小，**并发度更高**。
- 例如双十一订单创建峰值 58.3 万笔/秒，对并发要求极高
- 更小的锁粒度意味着更少的锁冲突

#### 2. 减少死锁概率
RR 级别下 Gap Lock 和 Next-Key Lock 范围更大，多个事务更容易互相等待对方释放锁，导致死锁。
- RC 没有间隙锁，死锁场景大幅减少
- 减少死锁排查和处理的人力成本

#### 3. RC 足够满足业务需求
- 大多数互联网业务不需要解决不可重复读问题
- 业务层可以通过乐观锁、版本号等手段保证数据一致性
- RC 隔离级别已经能防止脏读
