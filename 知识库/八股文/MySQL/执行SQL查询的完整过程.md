# 执行 SQL 查询的完整过程

## 问题
一条 SQL 查询语句在 MySQL 中的执行过程是怎样的？UPDATE 语句的执行过程有什么不同？

## 答案

### MySQL 的整体架构

MySQL 可以分为 **Server 层** 和 **存储引擎层** 两大部分：

- **Server 层**：包括连接器、查询缓存、分析器、优化器、执行器等，涵盖 MySQL 的大多数核心服务功能，以及所有的内置函数（如日期、时间、数学和加密函数等），所有跨存储引擎的功能都在这一层实现（如存储过程、触发器、视图等）
- **存储引擎层**：负责数据的存储和提取，支持 InnoDB、MyISAM、Memory 等多个存储引擎。MySQL 5.5.5 版本后默认使用 InnoDB

### SELECT 语句的执行过程

以 `SELECT * FROM user WHERE id = 1` 为例：

#### 第一步：连接器

**职责**：负责与客户端建立连接、获取权限、维持和管理连接。

**执行流程**：
1. 客户端通过 TCP 三次握手与 MySQL 服务器建立连接
2. 连接器验证用户名和密码是否正确
3. 验证通过后，连接器查询权限表获取该用户的权限
4. 后续该连接的权限判断都基于此时读取到的权限（即使管理员修改了权限，也不影响已建立的连接）

**连接类型**：
- **长连接**：连接成功后，客户端持续使用同一连接。推荐使用长连接，减少建立连接的开销
- **短连接**：每次执行完很少的几次查询就断开连接，下次查询再重新建立

**长连接的问题**：
- MySQL 的临时内存是在连接对象中管理的，长连接累积可能导致内存占用过大
- 解决方案：定期断开长连接（`mysql_reset_connection`）；或者在每次执行较大操作后重新连接

#### 第二步：查询缓存（MySQL 8.0 已移除）

**职责**：将执行过的查询及其结果以 key-value 形式缓存（key 是查询语句，value 是查询结果）。

**执行流程**：
1. 收到 SELECT 语句后，先检查查询缓存
2. 如果命中缓存，直接返回缓存结果（不执行后续步骤）
3. 如果未命中缓存，继续执行后续步骤，执行完成后将结果存入缓存

**为什么 MySQL 8.0 移除了查询缓存**：
- **命中率极低**：只要表有更新，该表的所有查询缓存都会被清空。对于写操作频繁的表，缓存命中率接近于零
- **维护开销大**：缓存的检查和失效需要额外开销
- **适用场景有限**：只适用于读多写少且数据几乎不变化的静态表

#### 第三步：分析器

**职责**：对 SQL 语句进行解析，包括词法分析和语法分析。

**执行流程**：

**1. 词法分析**
- 识别 SQL 语句中的各个关键字和标识符
- 例如：`SELECT * FROM user WHERE id = 1`
  - `SELECT` → 查询关键字
  - `*` → 所有列
  - `FROM` → 来源关键字
  - `user` → 表名
  - `WHERE` → 条件关键字
  - `id` → 列名
  - `=` → 比较运算符
  - `1` → 常量值

**2. 语法分析**
- 根据语法规则判断 SQL 语句是否合法
- 如果语法错误，返回 `You have an error in your SQL syntax` 错误
- 例如：`SELECT * FORM user` 会报错，因为 `FORM` 不是合法关键字

#### 第四步：优化器

**职责**：在表有多个索引时，决定使用哪个索引；或者在多表关联（JOIN）时，决定各表的连接顺序。

**执行流程**：
1. 分析 SQL 语句的查询条件
2. 根据统计信息（如索引基数、数据分布等）评估各种执行计划的代价
3. 选择代价最小的执行计划

**优化器的优化示例**：
```sql
-- 原始查询
SELECT * FROM t1 JOIN t2 ON t1.id = t2.id WHERE t1.name = 'Alice' AND t2.age = 25

-- 优化器可能做的优化：
-- 1. 选择先查 t1 还是 t2（小表驱动大表）
-- 2. 选择使用哪个索引（t1 的 name 索引还是主键索引）
-- 3. 是否可以使用索引覆盖避免回表
```

#### 第五步：执行器

**职责**：根据优化器生成的执行计划，调用存储引擎的接口执行查询。

**执行流程**：
1. 执行前先检查当前用户对该表是否有查询权限
2. 根据执行计划调用存储引擎接口获取数据
3. 将查询结果返回给客户端

**执行示例**（`SELECT * FROM user WHERE id = 1`）：
1. 检查对 `user` 表的 SELECT 权限
2. 调用 InnoDB 引擎接口，通过主键索引查找 id=1 的记录
3. 将找到的记录返回给客户端

### UPDATE 语句的执行过程

以 `UPDATE user SET age = 26 WHERE id = 1` 为例，UPDATE 语句的执行过程比 SELECT 复杂，因为它涉及到日志系统：

#### 整体流程

1. **连接器**：建立连接，验证权限（同 SELECT）
2. **分析器**：词法分析和语法分析（同 SELECT）
3. **优化器**：选择执行计划（同 SELECT）
4. **执行器**：调用存储引擎执行更新

#### InnoDB 引擎的更新流程（涉及 redo log 两阶段提交）

执行器调用 InnoDB 引擎执行更新时，InnoDB 会执行以下操作：

**Step 1：读取数据**
- 从磁盘（或 Buffer Pool）中读取 id=1 这一行的数据
- 如果数据页在 Buffer Pool 中，直接返回
- 如果不在 Buffer Pool 中，从磁盘读入 Buffer Pool，然后返回

**Step 2：更新数据**
- 将 age 从 25 改为 26
- 先将修改写入 **undo log**（用于事务回滚和 MVCC）
- 然后更新 Buffer Pool 中的数据页（此时数据页变为"脏页"）
- 将修改写入 **redo log** 的缓冲区（redo log buffer）

**Step 3：redo log 写入磁盘（Prepare 阶段）**
- 将 redo log buffer 中的内容写入磁盘的 redo log 文件
- 此时 redo log 处于 **prepare** 状态
- 这一步完成后，即使崩溃也能恢复

**Step 4：binlog 写入磁盘**
- 将更新操作写入 binlog 缓冲区
- 将 binlog 缓冲区的内容写入磁盘的 binlog 文件

**Step 5：redo log 标记提交（Commit 阶段）**
- 将 redo log 标记为 **commit** 状态
- 更新完成

#### 为什么需要两阶段提交？

redo log 和 binlog 是两个独立的日志，如果不用两阶段提交，可能会出现不一致：

**场景一：先写 redo log，再写 binlog**
- redo log 写入成功，binlog 写入前崩溃
- 恢复后：redo log 有记录，但 binlog 没有记录
- 从库通过 binlog 同步数据时，会丢失这次更新

**场景二：先写 binlog，再写 redo log**
- binlog 写入成功，redo log 写入前崩溃
- 恢复后：binlog 有记录，但 redo log 没有记录（主库数据丢失）
- 从库通过 binlog 同步了这次更新，但主库没有，主从不一致

**两阶段提交的解决方案**：
- Prepare 阶段写 redo log，Commit 阶段写 binlog 后再标记 redo log 为 commit
- 崩溃恢复时检查 redo log 的状态：
  - 如果是 prepare 状态，检查 binlog 是否完整
  - binlog 完整 → 提交事务
  - binlog 不完整 → 回滚事务

### 各组件的职责总结

| 组件 | 职责 | 关键点 |
|------|------|--------|
| **连接器** | 建立连接、验证身份、获取权限 | 长连接 vs 短连接；权限在连接时确定 |
| **查询缓存** | 缓存查询结果（8.0 已移除） | 表有更新则缓存失效；命中率低 |
| **分析器** | 词法分析、语法分析 | 识别关键字和标识符；检查 SQL 语法 |
| **优化器** | 选择执行计划 | 索引选择；JOIN 顺序优化 |
| **执行器** | 调用存储引擎执行 | 检查表权限；调用引擎接口 |
