# 索引覆盖与回表

## 问题
什么是聚集索引和二级索引？什么是回表？如何通过覆盖索引避免回表？

## 答案

### 聚集索引（聚簇索引）

**定义**：聚集索引是指索引的叶子节点直接存储了完整的数据行。也就是说，数据和索引是放在一起的。

**特点**：
- InnoDB 的主键索引就是聚集索引
- 一张表只能有一个聚集索引（因为数据行只能按一种方式物理排序）
- 聚集索引的叶子节点存储的是**完整的数据行**
- 数据按主键的顺序物理存储

**InnoDB 选择聚集索引的规则**：
1. 如果表定义了主键，则主键就是聚集索引
2. 如果没有主键，则选择第一个非空唯一索引（UNIQUE NOT NULL）作为聚集索引
3. 如果都没有，InnoDB 会自动生成一个隐藏的 6 字节 ROW_ID 作为聚集索引

**聚集索引的 B+ 树结构**：
```
非叶子节点（只存主键值）:
         [10 | 20 | 30]
        /      |       \
叶子节点（存完整数据行）:
[id=1..10, 行数据] <-> [id=11..20, 行数据] <-> [id=21..30, 行数据] <-> ...
```

### 二级索引（辅助索引/非聚簇索引）

**定义**：二级索引是指除聚集索引以外的所有索引。二级索引的叶子节点存储的不是完整数据行，而是**索引列的值 + 主键值**。

**特点**：
- 一张表可以有多个二级索引
- 二级索引的叶子节点存储的是：**索引列的值 + 对应行的主键值**
- 通过二级索引查找数据时，通常需要"回表"到聚集索引获取完整数据

**二级索引的 B+ 树结构**（以 `INDEX idx_name (name)` 为例）：
```
非叶子节点（只存索引值）:
         [Alice | Bob | Charlie]
        /        |         \
叶子节点（存索引值 + 主键值）:
[name=Alice, id=5] <-> [name=Bob, id=12] <-> [name=Charlie, id=20] <-> ...
```

### 回表是什么

**定义**：回表是指通过二级索引找到主键值后，再回到聚集索引（主键索引）中查找完整数据行的过程。

**回表的完整过程**：

假设有一张 `user` 表，有主键 `id` 和二级索引 `idx_name (name)`：

执行 `SELECT * FROM user WHERE name = 'Alice'`：

1. **第一步：在二级索引中查找**
   - 在 `idx_name` 的 B+ 树中查找 `name = 'Alice'`
   - 找到叶子节点，获取到对应的主键值 `id = 5`

2. **第二步：回表到聚集索引**
   - 拿到主键值 `id = 5`
   - 回到聚集索引（主键索引）的 B+ 树中查找 `id = 5`
   - 在聚集索引的叶子节点中找到完整的数据行

3. **第三步：返回结果**
   - 将完整的数据行返回给客户端

**回表的性能问题**：
- 回表意味着需要**两次 B+ 树查找**：一次在二级索引，一次在聚集索引
- 每次 B+ 树查找都可能涉及磁盘 IO（如果数据不在内存中）
- 如果回表的数据量很大（例如范围查询返回大量记录），性能会严重下降
- 回表时可能产生大量**随机 IO**（因为二级索引的主键值不一定是连续的）

### 怎么消除回表：覆盖索引

**定义**：覆盖索引（Covering Index）是指一个索引包含了查询所需的所有列，使得查询只需要扫描索引而不需要回表到聚集索引。

**核心思想**：将查询需要的所有列都放到索引中，这样二级索引的叶子节点就包含了所有需要的数据，无需回表。

**示例**：

假设经常执行以下查询：
```sql
SELECT id, name, age FROM user WHERE name = 'Alice'
```

如果只创建 `INDEX idx_name (name)`，则需要回表获取 `age` 列。

但如果创建联合索引 `INDEX idx_name_age (name, age)`：
- 二级索引的叶子节点存储的是：`name + age + id`（联合索引的列 + 主键）
- 查询时只需要在二级索引中查找，就能获取所有需要的列（name、age、id），无需回表

**EXPLAIN 中的体现**：
- 使用覆盖索引时，Extra 列会显示 `Using index`
- 这表示查询只需要扫描索引，不需要回表

### 覆盖索引的使用场景

#### 1. 高频查询优化

对于频繁执行的查询，可以通过建立覆盖索引来避免回表，显著提升查询性能。

**示例**：电商系统中，根据订单号查询订单状态的高频接口
```sql
-- 高频查询
SELECT order_id, status, amount FROM orders WHERE user_id = 123

-- 创建覆盖索引
INDEX idx_user_id_status (user_id, status, amount)
```

#### 2. 统计查询优化

统计类查询通常只需要部分列，非常适合使用覆盖索引。

**示例**：
```sql
-- 统计用户订单数
SELECT COUNT(*) FROM orders WHERE user_id = 123

-- 只需要在索引中查找，无需回表
INDEX idx_user_id (user_id)
```

#### 3. 分页查询优化

分页查询如果只需要部分列，可以通过覆盖索引优化。

**示例**：
```sql
-- 分页查询
SELECT id, title, status FROM articles WHERE category_id = 1 ORDER BY id LIMIT 10000, 10

-- 覆盖索引
INDEX idx_category_id_id (category_id, id, title, status)
```

#### 4. 避免 SELECT *

`SELECT *` 会查询所有列，很难通过覆盖索引避免回表。最佳实践是只查询需要的列：
```sql
-- 不推荐：SELECT * 几乎无法避免回表
SELECT * FROM user WHERE name = 'Alice'

-- 推荐：只查询需要的列，配合覆盖索引
SELECT id, name, age FROM user WHERE name = 'Alice'
```

### 回表与索引下推（ICP）

**索引下推（Index Condition Pushdown, ICP）** 是 MySQL 5.6 引入的优化，可以在回表前先对索引中的列进行过滤，减少回表次数。

**示例**：
```sql
SELECT * FROM user WHERE name LIKE '张%' AND age = 25
```

有联合索引 `INDEX idx_name_age (name, age)`：

- **没有 ICP**：在索引中找到所有 `name LIKE '张%'` 的记录，逐个回表获取完整数据，再过滤 `age = 25`
- **有 ICP**：在索引中找到所有 `name LIKE '张%'` 的记录后，先在索引中过滤 `age = 25`，只对满足条件的记录回表

ICP 减少了回表次数，在范围查询场景下效果显著。
