# Redis 内存淘汰策略

## 问题
Redis 有哪几种内存淘汰策略？各适用于什么场景？Redis 如何删除过期键？

## 答案

### 为什么要淘汰内存？

Redis 是基于内存的数据库，内存是有限的。当内存使用达到上限（`maxmemory`）时，Redis 需要决定如何处理新的写入请求。内存淘汰策略就是用来决定**淘汰哪些数据**来释放内存的。

### Redis 的 8 种内存淘汰策略

Redis 4.0 之后提供了 8 种内存淘汰策略，可以分为两大类：**针对所有 key 的策略** 和 **针对设置了过期时间的 key 的策略**。

#### 不淘汰（noeviction）

**策略**：当内存不足时，不淘汰任何数据，新的写入操作会返回错误（OOM）。

**适用场景**：
- 数据不能丢失的场景
- 读多写少的场景
- 需要保证数据完整性的场景

**缺点**：写入操作会失败，可能导致业务异常。

#### 针对所有 key 的淘汰策略

**1. allkeys-lru（最常用）**

**策略**：从所有 key 中，使用 **LRU（Least Recently Used，最近最少使用）** 算法淘汰最久没有被访问的 key。

**适用场景**：
- **最通用的策略**，适合大多数场景
- 没有明显的冷热数据区分
- 不同 key 的访问频率差异较大

**原理**：Redis 的 LRU 是近似 LRU，不是严格的 LRU。Redis 随机采样 N 个 key（默认 5 个，可通过 `maxmemory-samples` 配置），从中淘汰最久没有被访问的 key。这样做的好处是不需要维护严格的 LRU 链表，节省内存和 CPU。

**2. allkeys-random**

**策略**：从所有 key 中**随机淘汰**。

**适用场景**：
- 各个 key 的访问频率大致相同
- 没有明显的冷热数据区分

**缺点**：可能淘汰掉热点数据，导致缓存命中率下降。

**3. allkeys-lfu（Redis 4.0+）**

**策略**：从所有 key 中，使用 **LFU（Least Frequently Used，最不经常使用）** 算法淘汰访问频率最低的 key。

**适用场景**：
- 有明显的冷热数据区分
- 某些 key 访问频率很高，某些 key 几乎不被访问
- 需要保留热点数据

**LFU vs LRU 的区别**：
- **LRU**：淘汰最久没有被访问的 key。问题：如果一个 key 最近被访问了一次（但之前一直没被访问），它不会被淘汰，即使它不常用
- **LFU**：淘汰访问频率最低的 key。通过统计每个 key 的访问频率，更准确地区分冷热数据

**LFU 的实现**：
- 每个 key 维护一个访问计数器（logarithmic counter）
- 计数器随时间衰减（decay），避免历史热点数据一直不被淘汰
- 访问频率高的 key 计数器增长慢，访问频率低的 key 计数器增长快

#### 针对设置了过期时间的 key 的淘汰策略

**1. volatile-lru**

**策略**：从**设置了过期时间的 key** 中，使用 LRU 算法淘汰最久没有被访问的 key。

**适用场景**：
- 有明确的过期策略
- 希望只淘汰有过期时间的 key，保护没有设置过期时间的 key

**2. volatile-random**

**策略**：从**设置了过期时间的 key** 中**随机淘汰**。

**适用场景**：
- 有明确的过期策略
- 各个 key 的访问频率大致相同

**3. volatile-ttl**

**策略**：从**设置了过期时间的 key** 中，淘汰**剩余存活时间（TTL）最短**的 key。

**适用场景**：
- 有明确的过期策略
- 希望优先淘汰即将过期的 key

**4. volatile-lfu（Redis 4.0+）**

**策略**：从**设置了过期时间的 key** 中，使用 LFU 算法淘汰访问频率最低的 key。

**适用场景**：
- 有明确的过期策略
- 需要保留访问频率高的 key

### 策略选择建议

| 场景 | 推荐策略 | 原因 |
|------|---------|------|
| **大多数场景** | `allkeys-lru` | 通用性最好，能保留最近使用的数据 |
| **有明显冷热数据** | `allkeys-lfu` | 更准确地区分冷热数据 |
| **需要保护部分 key** | `volatile-lru` | 只淘汰有过期时间的 key |
| **数据不能丢失** | `noeviction` | 不淘汰数据，写入失败时业务自行处理 |
| **有明确过期策略** | `volatile-ttl` | 优先淘汰即将过期的 key |

### Redis 过期键删除策略

Redis 的 key 可以设置过期时间（TTL），过期后需要删除。Redis 使用 **惰性删除 + 定期删除** 两种策略结合来删除过期键。

#### 1. 惰性删除（Lazy Expiration）

**原理**：不主动删除过期键，而是在客户端访问某个 key 时，检查该 key 是否已过期。如果过期则删除并返回空。

**实现**：
```python
def get(key):
    # 检查 key 是否过期
    if is_expired(key):
        delete(key)
        return None
    return key.value
```

**优点**：只在访问时才检查，不浪费 CPU 资源
**缺点**：如果过期 key 一直没有被访问，会一直占用内存（内存泄漏）

#### 2. 定期删除（Periodic Expiration）

**原理**：Redis 定期（默认每秒 10 次）随机抽取一部分设置了过期时间的 key，检查是否过期并删除。

**实现**：
```
每秒执行 10 次（可通过 hz 配置）：
1. 随机抽取 20 个设置了过期时间的 key
2. 删除其中已过期的 key
3. 如果过期的 key 超过 25%（5 个），重复步骤 1
4. 单次执行时间不超过 25ms，避免阻塞主线程
```

**优点**：通过限制执行时间和频率，不会过度消耗 CPU
**缺点**：可能有些过期 key 没有被及时删除

#### 为什么需要两种策略结合？

- **惰性删除**：保证了访问到的 key 一定是有效的，但可能导致过期 key 堆积
- **定期删除**：定期清理过期 key，减少内存占用，但可能有漏网之鱼
- **内存淘汰策略**：作为最后的兜底，当内存不足时强制淘汰

三者配合使用：
1. 定期删除：定期清理大部分过期 key
2. 惰性删除：确保访问到的 key 一定有效
3. 内存淘汰策略：当内存不足时，淘汰剩余的过期 key 或其他 key

### 配置示例

```bash
# 设置最大内存
maxmemory 4gb

# 设置内存淘汰策略
maxmemory-policy allkeys-lru

# 设置 LRU 采样数量（默认 5）
maxmemory-samples 10

# 设置定期删除的频率（默认 10，即每秒 10 次）
hz 10
```

### 内存淘汰的执行流程

```
客户端执行写入命令
       ↓
检查内存是否超过 maxmemory
       ↓
未超过 → 正常写入
   ↓
超过
   ↓
执行淘汰策略
   ↓
淘汰成功 → 写入新数据
   ↓
淘汰失败（noeviction 或无 key 可淘汰）→ 返回 OOM 错误
```
