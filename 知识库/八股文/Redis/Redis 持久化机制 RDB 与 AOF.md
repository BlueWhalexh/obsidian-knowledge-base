# Redis 持久化机制 RDB 与 AOF

## 问题
Redis 的持久化机制有哪些？RDB 和 AOF 有什么区别？什么是混合持久化？

## 答案

### RDB（Redis Database Backup）

**定义**：把内存中的所有数据记录到磁盘文件中（数据快照），Redis 故障重启后读取快照文件恢复数据。

**触发方式**：
- 配置自动触发：`save 900 1`（900秒内至少1个key修改则触发 bgsave）
- 手动触发：`save` 命令（阻塞主进程）、`bgsave` 命令（fork 子进程，非阻塞）

**bgsave 底层原理**：
1. 主进程 fork 出子进程（复制页表，此过程阻塞主进程）
2. 子进程与主进程共享内存数据，使用 **Copy-On-Write（写时复制）** 技术
3. 子进程读取共享内存数据，写入 RDB 文件
4. 主进程执行写操作时，先拷贝一份数据，在拷贝数据上写入，避免冲突

**优点**：
- 宕机后恢复速度快（直接加载文件）
- 文件体积小（二进制压缩格式）

**缺点**：
- 数据安全性差：两次 RDB 之间的写入数据有丢失风险
- fork 子进程和写出 RDB 文件比较耗时

### AOF（Append Only File）

**定义**：Redis 处理的每一个写命令都会记录在 AOF 文件中，可看作命令日志文件。

**刷盘策略**：

| 策略 | 行为 | 数据安全 | 性能 |
|------|------|---------|------|
| **always** | 每次写操作后同步刷盘 | 最高，不丢数据 | 最差 |
| **everysec** | 每秒将缓冲区写入 AOF | 丢失最多1秒数据 | 较好（推荐） |
| **no** | 由操作系统决定何时刷盘 | 不确定 | 最好 |

**AOF 重写**：
- AOF 文件会记录对同一个 key 的多次操作，只有最后一次有意义
- 通过 `bgrewriteaof` 触发重写，用最少的命令达到相同效果
- 可配置触发阈值：文件增长百分比、文件最小体积

**优点**：数据安全性高（everysec 策略最多丢失1秒数据）

**缺点**：
- 宕机恢复慢（需要依次执行所有命令）
- 文件体积大（需要 AOF 重写压缩）

### 混合持久化（Redis 4.0+）

Redis 4.0 推出 RDB-AOF 混合持久化，以 AOF 为基础：

**重写时的文件结构**：
```
[RDB 数据块] + [AOF 增量命令流]
```

1. AOF 文件重写时，先将当前内存数据以 **RDB 格式**写入文件开头
2. 后续增量数据以 **AOF 格式**追加到文件末尾

**优点**：
- 结合 RDB 恢复快的优点（开头是 RDB 格式，加载快）
- 结合 AOF 数据安全的优点（增量部分不会丢失）

**缺点**：AOF 文件格式变化导致不兼容旧版本

### 对比总结

| 特性 | RDB | AOF |
|------|-----|-----|
| 数据安全 | 较差（间隔性快照） | 好（最多丢1秒） |
| 恢复速度 | 快 | 慢 |
| 文件体积 | 小 | 大 |
| 默认状态 | 默认开启 | 需手动开启 |
