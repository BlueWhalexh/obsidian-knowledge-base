# Redis 数据类型与底层实现

## 问题
Redis 有哪些数据类型？各自的底层实现是什么？ZSet 为什么用跳表而不是红黑树？

## 答案

### 五种基本数据类型

| 类型 | 用途 | 底层实现 |
|------|------|---------|
| **String** | 计数器、分布式锁、缓存对象 | int（纯数字）/ SDS 简单动态字符串 |
| **Hash** | 缓存对象（如用户信息） | ZipList / listpack（元素少时）→ Dict 哈希表（元素多时） |
| **List** | 消息列表、朋友圈评论列表 | QuickList（双向链表，每个节点是一个压缩列表） |
| **Set** | 共同关注、去重 | IntSet 整数集合（全整数且元素 < 512）→ Dict 哈希表 |
| **ZSet** | 排行榜、延迟队列 | ZipList / listpack（元素 < 128 且每个 < 64字节）→ **SkipList + Dict** |

### ZSet 底层实现详解

当元素较多时，ZSet 使用 **SkipList（跳表）+ Dict（哈希表）** 双结构存储，同一份数据存两份：

**为什么需要两份数据？**
- **Dict**：用于 O(1) 的键值查询（根据元素查分数）
- **SkipList**：用于范围查询（获取指定分数区间的元素、获取排名区间的元素）

**SkipList 跳表特性**：
- 多层链表结构，元素按 score 升序排列
- 每个节点包含多个指针，跨度不同
- 范围查询时利用高层指针快速跳过大量无关节点，定位范围起始点
- 通过牺牲少量内存空间（多层指针），换取高效的范围查找性能

### 其他底层数据结构

**Dict 哈希表**：
- 底层维护两张哈希表（数组 + 链表）
- ht[0] 存当前数据，ht[1] 在 rehash 时使用
- 扩容/收缩时进行渐进式 rehash

**QuickList**：
- 双向链表，每个节点是一个 ZipList/listpack
- 结合了压缩列表的内存紧凑和双向链表的灵活操作

**ZipList 压缩列表**：
- 连续内存空间的双端链表
- 存储格式紧凑，节省空间
- 缺点：元素过多时分配效率低，存在连锁更新问题
- Redis 7.0 引入 listpack 取代 ZipList，解决了连锁更新问题
