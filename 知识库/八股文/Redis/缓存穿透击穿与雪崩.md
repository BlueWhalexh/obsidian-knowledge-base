# 缓存穿透、击穿与雪崩

## 问题
什么是缓存穿透、缓存击穿、缓存雪崩？各自的原因和解决方案是什么？

## 答案

### 缓存穿透

**定义**：请求的数据在 Redis 缓存和数据库中都**不存在**，缓存永远不会生效，所有请求直接打到数据库。

**典型场景**：恶意攻击、爬虫大量请求不存在的 ID。

**解决方案**：

#### 1. 缓存空值
- 将数据库查不到的结果也缓存到 Redis，设置较短的 TTL
- 优点：实现简单
- 缺点：占用额外内存，可能造成短期数据不一致

#### 2. 布隆过滤器
- **原理**：在插入数据时，通过 n 个哈希函数对 key 计算，在二进制位图上标记特征位
- **判断**：用相同的 n 个哈希函数计算，若所有位置都为 1，则**可能存在**；若有任何位置为 0，则**一定不存在**
- 在缓存前加一层布隆过滤器，快速判断数据是否存在
- 优点：内存占用小，判断速度快
- 缺点：存在误判（可通过调整参数降低概率），不支持删除

### 缓存击穿

**定义**：某个**热点 key** 在缓存中过期的瞬间，大量并发请求同时到达数据库，导致数据库瞬时压力过大。

**典型场景**：秒杀商品、热门新闻等热点数据过期。

**解决方案**：

#### 1. 互斥锁
- 只允许一个线程查询数据库并重建缓存，其他线程阻塞等待
- 实现：线程获取 Redis 缓存为空时，先加分布式锁
  - 加锁成功：从数据库加载数据，写回缓存，释放锁
  - 加锁失败：阻塞等待，锁释放后 double check 获取缓存数据
- 优点：保证数据一致性
- 缺点：其他线程需要等待，影响响应时间

#### 2. 逻辑过期
- 缓存 key **不设置 TTL**，在 value 中额外存储过期时间字段
- 请求命中缓存后检查逻辑过期时间：
  - 未过期：直接返回数据
  - 已过期：开启异步线程获取锁重建缓存，本线程先返回旧数据
- 优点：不会阻塞请求，响应快
- 缺点：短暂时间内返回的是旧数据

### 缓存雪崩

**定义**：大量 key **同时过期**或**缓存服务宕机**，大量请求瞬间打到数据库，导致数据库压力过大甚至崩溃。

**解决方案**：

#### 1. 均匀设置过期时间
- 给过期时间加上随机值，避免大量 key 同时过期
- 例如：`TTL = base_ttl + random(0, 300)`

#### 2. Redis 集群提高可用性
- 使用 Redis Cluster 或哨兵模式，避免单点故障
- 主从复制 + 自动故障转移

#### 3. 多级缓存
- 在 Redis 和数据库之间增加本地缓存（如 Caffeine）
- 即使 Redis 故障，本地缓存仍能抗住部分请求

### 三者对比

| 问题 | 核心原因 | 关键特征 |
|------|---------|---------|
| **穿透** | 数据本身不存在 | 查询的数据缓存和DB都没有 |
| **击穿** | 热点 key 过期 | 单个热点 key 大量并发 |
| **雪崩** | 大量 key 同时过期或缓存宕机 | 大面积缓存失效 |
