# 循环依赖与三级缓存

## 问题
什么是Spring循环依赖？Spring是如何通过三级缓存解决循环依赖的？为什么需要三级缓存？什么情况下无法解决循环依赖？

## 答案

### 什么是循环依赖

循环依赖是指两个或多个Bean之间相互依赖，形成循环引用。例如ServiceA依赖ServiceB，ServiceB又依赖ServiceA。如果不处理会导致应用启动失败。

```java
@Service
public class ServiceA {
    @Autowired
    private ServiceB serviceB;
}

@Service
public class ServiceB {
    @Autowired
    private ServiceA serviceA;
}
```

### 三级缓存机制

Spring通过三级缓存机制提前暴露未完全初始化的Bean实例，解决单例模式下的循环依赖问题。

| 缓存级别 | 名称 | 存储内容 |
|---------|------|---------|
| 一级缓存 | singletonObjects | 完全初始化后的成品Bean，可直接使用 |
| 二级缓存 | earlySingletonObjects | 实例化但未完成初始化的半成品Bean早期引用 |
| 三级缓存 | singletonFactories | Bean的创建工厂ObjectFactory |

### 解决原理

Spring中Bean的创建分为两步：
1. **实例化**：调用构造函数创建对象并分配内存空间
2. **初始化**：给对象的属性进行赋值（`populateBean`）和调用初始化方法（`initializeBean`）

三级缓存的核心思想是：**将实例化和初始化分开处理，在实例化完成后、初始化完成前，提前暴露Bean的引用**。

**三级缓存的源码定义**（`DefaultSingletonBeanRegistry`）：

```java
/** 一级缓存：存放完全初始化好的Bean（成品） */
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

/** 二级缓存：存放提前暴露的Bean（半成品），可能是原始对象也可能是代理对象 */
private final Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>(16);

/** 三级缓存：存放Bean的ObjectFactory（工厂） */
private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);
```

### 三级缓存解决循环依赖的详细步骤

**以A依赖B、B依赖A为例，详细追踪每个缓存的状态变化**：

#### 阶段一：创建Bean A

1. **开始创建A**：`getBean("a")` → `doGetBean("a")` → 发现一级缓存中没有A
2. **标记A正在创建**：将A标记为"正在创建中"（`singletonsCurrentlyInCreation.add("a")`）
3. **实例化A**：调用A的构造函数，创建A的原始对象（此时A的`serviceB`属性为null）
4. **将A的ObjectFactory放入三级缓存**：`addSingletonFactory("a", () -> getEarlyBeanReference("a", mbd, a))`。注意这里放的是一个Lambda工厂，而非A的对象本身。如果A需要AOP代理，这个工厂会在被调用时返回代理对象
5. **填充A的属性**：`populateBean("a")` → 发现需要注入B → 触发`getBean("b")`

此时缓存状态：
```
一级缓存：{}
二级缓存：{}
三级缓存：{a → ObjectFactory<A>}
```

#### 阶段二：创建Bean B

6. **开始创建B**：`getBean("b")` → `doGetBean("b")` → 发现一级缓存中没有B
7. **标记B正在创建**：将B标记为"正在创建中"
8. **实例化B**：调用B的构造函数，创建B的原始对象
9. **将B的ObjectFactory放入三级缓存**：`addSingletonFactory("b", () -> getEarlyBeanReference("b", mbd, b))`
10. **填充B的属性**：`populateBean("b")` → 发现需要注入A → 触发`getSingleton("a")`

此时缓存状态：
```
一级缓存：{}
二级缓存：{}
三级缓存：{a → ObjectFactory<A>, b → ObjectFactory<B>}
```

#### 阶段三：从三级缓存获取A的早期引用

11. **`getSingleton("a")`查找A**：
    - 先查一级缓存 → 不存在
    - 再查二级缓存 → 不存在
    - 最后查三级缓存 → 找到A的ObjectFactory
12. **调用A的ObjectFactory.getObject()**：执行`getEarlyBeanReference("a", mbd, a)`
    - 如果A配置了AOP代理（如`@Transactional`），此处会提前创建A的代理对象
    - 如果A没有AOP代理，直接返回A的原始对象
13. **将A的早期引用从三级缓存移至二级缓存**：`earlySingletonObjects.put("a", earlyRef)`，`singletonFactories.remove("a")`
14. **将A的早期引用注入到B中**：B的`serviceA`属性被赋值

此时缓存状态：
```
一级缓存：{}
二级缓存：{a → A的早期引用（可能是代理对象）}
三级缓存：{b → ObjectFactory<B>}
```

#### 阶段四：B完成创建

15. **B完成属性填充**：B的`serviceA`已注入
16. **B执行初始化**：调用`@PostConstruct`、`InitializingBean.afterPropertiesSet()`等
17. **B的AOP后置处理**：如果B需要AOP代理，在`BeanPostProcessor.postProcessAfterInitialization()`中创建B的代理对象
18. **B创建完成**：将B放入一级缓存，清除二、三级缓存中的B记录
19. **B返回给A的属性注入**

此时缓存状态：
```
一级缓存：{b → B（成品）}
二级缓存：{a → A的早期引用}
三级缓存：{}
```

#### 阶段五：A完成创建

20. **A获取到B实例**：B已创建完成，从一级缓存获取
21. **A完成属性填充**：A的`serviceB`已注入
22. **A执行初始化**：调用`@PostConstruct`等
23. **A的AOP后置处理**：检查A是否已经被提前代理过
    - 如果在阶段12中已经创建了代理（二级缓存中有A的早期引用），则直接使用该代理，不再重复创建
    - 如果未被提前代理，则在此处创建AOP代理
24. **A创建完成**：将A放入一级缓存，清除二、三级缓存中的A记录

最终缓存状态：
```
一级缓存：{a → A（成品）, b → B（成品）}
二级缓存：{}
三级缓存：{}
```

**完整流程图**：
```
创建A → 实例化A → A放入三级缓存 → 填充A属性(需要B)
  → 创建B → 实例化B → B放入三级缓存 → 填充B属性(需要A)
    → 从三级缓存获取A的ObjectFactory → 执行getEarlyBeanReference()
    → A移至二级缓存 → 将A注入B → B初始化完成 → B放入一级缓存
  → A获取到B → A初始化完成 → A放入一级缓存
```

### 为什么需要三级缓存而不是二级

#### 核心原因：解决AOP代理对象的提前暴露问题

**如果没有AOP，二级缓存就够了**：实例化后直接将原始对象放入二级缓存，需要时从二级缓存取即可。

**但存在AOP时，问题变复杂了**：

Spring AOP代理通常是在Bean初始化完成后的`BeanPostProcessor.postProcessAfterInitialization()`中创建的（即`AbstractAutoProxyCreator`）。但循环依赖导致Bean在初始化完成前就需要暴露引用给其他Bean使用。

**如果只有二级缓存**，有两种选择都有问题：

- **方案A：在二级缓存中直接放原始对象**：那么B注入的是A的原始对象，而不是A的代理对象。后续A完成初始化后生成了代理对象，但B持有的仍然是原始对象，导致同一个Bean在不同地方引用不一致。
- **方案B：在实例化时就创建AOP代理**：违背了Spring的设计原则——AOP代理应该在初始化后创建。而且有些场景下（如`@Transactional`）代理的创建依赖于初始化过程中的信息。

**三级缓存的优雅解决方案**：

三级缓存存的不是对象本身，而是`ObjectFactory`（一个Lambda工厂）。这个工厂的`getEarlyBeanReference()`方法在被调用时会判断：如果该Bean配置了需要提前暴露的AOP代理（`SmartInstantiationAwareBeanPostProcessor.getEarlyBeanReference()`），则返回代理对象；否则返回原始对象。

```java
// DefaultSingletonBeanRegistry.addSingletonFactory()
protected void addSingletonFactory(String beanName, ObjectFactory<?> singletonFactory) {
    if (!this.singletonObjects.containsKey(beanName)) {
        this.singletonFactories.put(beanName, singletonFactory);
        this.earlySingletonObjects.remove(beanName);
    }
}

// getEarlyBeanReference 的核心逻辑
protected Object getEarlyBeanReference(String beanName, RootBeanDefinition mbd, Object bean) {
    Object exposedObject = bean;
    // 遍历所有SmartInstantiationAwareBeanPostProcessor
    // 如果有AOP相关的BeanPostProcessor，会在此处提前创建代理
    for (SmartInstantiationAwareBeanPostProcessor bp : getBeanPostProcessors(SmartInstantiationAwareBeanPostProcessor.class)) {
        exposedObject = bp.getEarlyBeanReference(exposedObject, beanName);
    }
    return exposedObject;
}
```

**三级缓存的作用总结**：
- **一级缓存（singletonObjects）**：存放完全初始化好的成品Bean，供使用
- **二级缓存（earlySingletonObjects）**：存放提前暴露的半成品Bean（可能是原始对象也可能是代理对象），用于解决循环依赖
- **三级缓存（singletonFactories）**：存放ObjectFactory，延迟决定返回原始对象还是代理对象，只有在真正发生循环依赖时才会调用工厂创建早期引用。如果一个Bean没有发生循环依赖，其ObjectFactory永远不会被调用，代理对象仍然在正常初始化阶段创建，保证了设计的一致性

### 构造器注入为什么不能解决循环依赖

**根本原因**：构造器注入要求在对象创建（实例化）阶段就完成依赖注入，而setter注入可以先创建对象再注入依赖。

```java
// 构造器注入：创建A时必须先有B，创建B时必须先有A → 死循环
@Service
public class ServiceA {
    private final ServiceB serviceB;
    public ServiceA(ServiceB serviceB) {  // 实例化A时就需要B
        this.serviceB = serviceB;
    }
}

// Setter注入：可以先创建A（此时B为null），再创建B，再将B注入A
@Service
public class ServiceA {
    private ServiceB serviceB;
    @Autowired
    public void setServiceB(ServiceB serviceB) {  // 可以先创建A，后注入B
        this.serviceB = serviceB;
    }
}
```

**详细解释**：

1. Spring创建A时，调用A的构造函数，发现需要参数B
2. 此时A的实例化都还没完成，A的对象都还没有创建出来（没有半成品对象）
3. A没有放入任何缓存（三级缓存是在实例化之后才放入的）
4. 所以无法像setter注入那样，先把半成品的A暴露出去
5. Spring尝试去创建B，但创建B又需要A，而A又没有创建出来
6. 形成无限递归，最终抛出`BeanCurrentlyInCreationException`

**解决方案**：
- **使用`@Lazy`注解**：延迟注入，注入的不是真正的依赖对象，而是一个代理对象，只有在真正使用时才去创建

```java
@Service
public class ServiceA {
    private final ServiceB serviceB;
    public ServiceA(@Lazy ServiceB serviceB) {  // 注入的是代理，不会立即创建B
        this.serviceB = serviceB;
    }
}
```

- **改用Setter注入**：将构造器注入改为Setter注入，利用三级缓存机制解决
- **重新设计**：将共同依赖抽取到第三个类中，打破循环

### 无法解决循环依赖的情况

1. **互相依赖的Bean必须是单例的Bean**：非单例Bean（prototype）每次获取都会创建新实例，无法放入缓存
2. **依赖注入的方式不能都是构造函数注入**：如上所述，构造函数注入在实例化阶段就需要依赖，此时没有半成品对象可用
3. **`@Async`注解导致的代理问题**：如果循环依赖的Bean中有一个使用了`@Async`，其代理创建时机与普通AOP不同，三级缓存可能无法正确处理
4. **Spring Boot 2.6+默认禁止循环依赖**：`spring.main.allow-circular-references=false`，需要手动开启或通过`@Lazy`解决
