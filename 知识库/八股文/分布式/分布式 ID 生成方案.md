# 分布式 ID 生成方案

## 问题
分布式系统中如何生成全局唯一ID？各方案的优缺点是什么？雪花算法的时钟回拨问题如何解决？

## 答案

### 分布式ID的定义

分布式ID即分布式系统下的唯一标识，解决分库分表后单机自增ID的局限性，满足高并发、高可用场景下对唯一标识的需求。

### 分布式ID应具备的特性

1. **全局唯一性**：ID必须全网唯一（基本要求）
2. **趋势递增**：ID整体按时间递增，提升数据库写入性能（避免页分裂）
3. **高性能高可用**：生成速度快（低延迟、高QPS），系统可用性达99.999%
4. **信息安全**：避免连续可预测，防止恶意爬取
5. **长度适中**：通常64位或更短，便于存储和查询

### 方案一：UUID

标准格式：xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx（32个字符）

基于当前时间、计数器、机器MAC地址等数据计算生成。

| 优点 | 缺点 |
|------|------|
| 生成性能高，不依赖网络 | 无含义，排查调试困难 |
| 本地生成 | 非递增，写入数据库可能导致页分裂 |
| | 长度较长（36字符） |

### 方案二：数据库自增

利用数据库的自增ID特性。

| 优点 | 缺点 |
|------|------|
| 简单易用 | 存在性能瓶颈（每次需访问数据库） |
| 保证有序 | 数据库单点故障风险 |
| | 分库分表后无法保证全局唯一 |

### 方案三：Redis自增

使用Redis的`incr`或`incrby`命令实现原子性自增。

| 优点 | 缺点 |
|------|------|
| 基于内存，性能高 | Redis可能存在数据丢失 |
| ID自增有序 | 自增ID容易被猜测（可加前缀或时间戳） |
| | 依赖Redis服务 |

### 方案四：雪花算法（Snowflake）

分布式ID生成算法，生成全局唯一且递增的ID。

**核心思想**：将64位ID划分成多个部分：

```
| 1bit符号位 | 41bit时间戳 | 10bit机器标识 | 12bit序列号 |
```

| 部分 | 位数 | 说明 |
|------|------|------|
| 符号位 | 1bit | 始终为0 |
| 时间戳 | 41bit | 精确到毫秒，支持约69年 |
| 机器标识 | 10bit | 区分不同机器，最多1024台服务器 |
| 序列号 | 12bit | 单台机器每毫秒最多生成4096个ID |

**优点**：
- 高性能高吞吐：单台服务每秒可生成百万级ID
- 不依赖网络，本地生成
- 单机部署时ID严格递增

**集群部署时**：同一毫秒内不同机器生成的ID不能保证全局递增（因机器标识无大小关系）。

### 雪花算法的时钟回拨问题

雪花算法使用服务器时间戳作为ID的一部分，如果系统时钟回拨（网络时间校准或人工设置），可能导致ID重复。

**解决方案**：

**方案一：阻塞等待**
- 发现时钟回拨时，直接抛异常或阻塞线程，直到系统追平最后一次生成ID的时间
- 适用场景：能容忍短暂服务中断

**方案二：改造ID构造**
- 将10位机器码拆分为3位时钟序列 + 7位机器码
- 影响：最多部署128个节点（原1024个）
- 当发生时钟回拨时，递增时钟序列，3位可解决8次时钟回拨
- 若并发量不大，可减少序列号位数，增加时钟序列位数

**机器标识分配**：可通过ZooKeeper、etcd等系统动态分配唯一ID，节点启动时注册并获取，宕机后释放。

### 方案选择建议

| 场景 | 推荐方案 |
|------|---------|
| 简单场景，性能要求不高 | 数据库自增 |
| 需要高性能、趋势递增 | 雪花算法 |
| 需要全局唯一，不要求有序 | UUID |
| 已有Redis基础设施 | Redis自增 |
