# 4C8G 机器各项系统指标正常范围

## 问题
4C8G 的机器，各项系统指标什么范围算是正常？

## 面试考察点
- 系统监控能力：对 CPU、内存、磁盘、网络等关键指标的掌握程度
- 性能分析：指标异常时的诊断思路与关联分析能力
- 优化意识：资源利用的平衡，识别瓶颈并给出调优建议
- 实战经验：真实运维场景中的指标判断经验

## 解题思路
> 核心在于理解各指标的含义及其健康阈值，关键是要能关联多个指标进行综合判断

4C8G 配置下，CPU 核心数为 4，Load Average 长期不应超过 4；内存 8GB 需保留至少 20%-30% 空闲避免频繁 swap；磁盘 IO 等待时间 SSD 应低于 10ms、HDD 低于 30ms；网络带宽利用率不超过 70%-80%。不同业务类型（Web 服务、数据库、计算服务）的指标特征差异很大，需要结合场景判断。

GC 指标同样重要：每次 GC 时长不应超过 200ms，每分钟 GC 次数超过 2 次可能需要调整堆大小或选择不同垃圾回收器。关联指标分析是关键：CPU 高但 IO 低说明计算密集型，IO 等待高说明存储瓶颈，内存高加 swap 使用可能是内存泄漏。

## 具体方案
### 各指标健康范围
- **CPU 使用率**：用户态(us) < 70%，内核态(sy) < 15%，IO 等待(wa) < 20%
- **Load Average**：1 分钟 < 核数 x 0.7（即 2.8），15 分钟 < 核数（即 4）
- **内存**：used < 90%（即 < 7.2GB），available 始终 > 500MB，swap used < 500MB
- **磁盘 IO**：%util < 70%（SATA）/ < 90%（SSD），await < 10ms（SSD）/ < 30ms（HDD）
- **网络**：带宽利用率 < 70%，包错误率 < 0.1%，TCP 重传 < 1%
- **GC**：每次时长 < 200ms，每分钟次数 < 2 次

### 监控工具
- CPU：`vmstat 1`、`top`/`htop`
- 内存：`free -m`、`swapon --show`
- 磁盘：`iostat -x 1`、`iotop`
- 网络：`sar -n DEV 1`、`iftop`
- GC：`-Xloggc` JVM 参数、JVisualVM、Prometheus + Grafana

### 异常关联分析
- CPU 高 + IO 低：计算密集型任务
- IO 等待高：存储瓶颈（考虑换 SSD 或调整电梯算法）
- 内存高 + swap 使用：内存泄漏可能（用 MAT 分析堆转储）
- 网络高 + CPU 低：可能遭受攻击

## 要点总结
- Load Average 不应长期超过 CPU 核心数，短期不超过核数 x 0.7
- 内存必须预留 20%-30% 空闲，swap 使用超过 500MB 是预警信号
- 磁盘 IO 和网络带宽都应控制在 70% 以下留有余量
- GC 每次超过 200ms 或每分钟超过 2 次需要关注调优
- 关联多个指标综合判断，单一指标异常不能得出结论
