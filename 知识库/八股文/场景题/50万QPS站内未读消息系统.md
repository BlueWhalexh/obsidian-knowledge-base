# 50万QPS站内未读消息系统

## 问题

如何设计支撑50万QPS的站内未读消息系统？系统通知和信息流未读数如何实现？

## 面试考察点

- 系统通知未读数设计（共享列表+偏移量）
- 信息流未读数设计（博文数快照）
- 全量用户打点的高效方案
- 缓存在高并发场景中的核心作用

## 解题思路

未读消息分两类：**系统通知**（全量用户共享）和**信息流**（基于关注关系），方案完全不同。

## 具体方案

### 1. 系统通知未读数

**错误方案**：逐个用户递增未读计数
- 1亿用户 * 1ms = 10万秒（超过1天）
- 大量不活跃用户浪费存储

**正确方案**：共享列表 + 用户偏移量
- 系统通知存储在一个共享列表中
- 每个用户记录**最近看过的通知ID**
- 未读数 = 最新通知ID - 用户最近看过的ID
- 用户查看后更新偏移量为最新ID

### 2. 全量打点（红点提醒）
- 记录用户**最近点击红点的时间戳**
- 记录全局**最新打点时间戳**
- 用户时间戳 < 全局时间戳 -> 显示红点

### 3. 信息流未读数（50万QPS）

**难点**：
- 每个人关注的人不同，无法共享存储
- 请求量极大（客户端轮询，用户不操作也会请求）
- 大V发博延迟问题

**方案**：博文数快照
1. 通用计数器记录每个用户的**博文总数**
2. Redis缓存用户**所有关注人的博文数快照**
3. 未读数 = 关注人当前博文总数之和 - 快照中博文总数之和
4. 用户查看后刷新快照

**性能**：全内存操作，微博仅用**16台服务器**支撑50万QPS

## 要点总结

1. 系统通知用共享列表+偏移量，避免操作全量用户
2. 信息流未读用博文数快照，全内存操作高性能
3. 缓存是提升性能的神器，16台服务器支撑50万QPS
4. 未读数允许一定误差，可牺牲准确性换取性能
