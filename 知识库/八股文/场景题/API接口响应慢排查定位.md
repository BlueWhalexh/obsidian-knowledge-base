# API接口响应慢排查定位

## 问题

线上API接口响应慢，如何快速排查和定位问题？

## 面试考察点

- 慢查询日志分析
- 可观测性三支柱（Metrics/Logs/Traces）
- 链路追踪定位瓶颈
- CPU/内存/死锁排查

## 解题思路

接口响应慢分为**个别接口慢**和**所有接口慢**两种情况，排查方向不同。

## 具体方案

### 1. 个别接口响应慢

**链路追踪（Skywalking）**
- 通过traceId串联完整调用链路
- 查看每个环节耗时：DB查询、Redis访问、RPC调用
- 快速定位是哪个环节慢

**链路打印日志**
- 在关键节点打印耗时日志
- 逐段排查

**死锁排查**
- `jstack pid` 查看线程栈
- `jconsole` 图形化检测死锁
- 关注 `waiting to lock` 和 `locked` 关键字

### 2. 所有接口响应慢

排查顺序：
1. **CPU占用过高** -> `top` -> `top -Hp pid` -> `jstack`
2. **内存占用过高** -> `jmap -dump` -> MAT分析
3. **磁盘问题** -> `df -h` / `iostat` / `vmstat`
4. **网络故障** -> `dstat` 查看网络流量
5. **数据库问题** -> 慢查询日志、索引失效、死锁
6. **GC问题** -> `jstat -gcutil pid` 查看GC频率

### 3. 压测监控核心指标
- **RT（响应时间）**：互联网 < 500ms
- **TPS/QPS**：系统处理能力
- **错误率**：< 0.6%
- **CPU/Memory**：资源使用率
- **数据库**：连接数、缓存命中率

### 4. 慢查询日志
```sql
SET GLOBAL slow_query_log='ON';
SET GLOBAL long_query_time=2;
```
- 记录超过2秒的SQL
- 配合EXPLAIN分析执行计划

## 要点总结

1. 个别接口慢用链路追踪定位，所有接口慢排查系统资源
2. Skywalking traceId是串联调用链的利器
3. top + jstack + jmap三板斧定位CPU/线程/内存问题
4. 慢查询日志是排查DB瓶颈的基础手段
5. 压测关注RT/TPS/错误率/资源使用率四大指标
