# BitMap一亿整数存在性判断

## 问题
在100M内存下存储一亿个整数（范围1到2亿），如何快速判断给定整数是否存在？

## 面试考察点
- 数据结构能力：对紧凑型数据结构（BitMap、BloomFilter）的掌握
- 空间优化：严格内存限制下的精确计算能力
- 算法思维：时间/空间复杂度的权衡取舍
- 工程实现：方案在生产环境的可行性

## 解题思路
> 核心在于用最少的内存表达"存在/不存在"这个二值状态

每个整数只需要1bit来标记是否存在。整数范围是1到2亿，即2亿个可能值，需要2亿bit = 200,000,000 / 8 = 25,000,000字节，约23.8MB，远低于100MB限制。这就是BitMap（位图）方案的由来。

如果内存更紧张或允许一定误判率，可以考虑BloomFilter（布隆过滤器）。它通过多个哈希函数将元素映到位数组，空间更省但存在假阳性。需要注意，BitMap不支持删除和计数，BloomFilter不支持删除且有误判。

## 具体方案
### 方案一：BitMap（推荐）
- 原理：每个整数对应1bit，用1表示存在、0表示不存在
- 空间：23.8MB（精确，无误判）
- 查询：O(1) 时间复杂度
- Java实现：`BitSet` 或直接 `byte[]` 数组
- 局限：仅适用于整数范围已知且连续的场景

### 方案二：BloomFilter（备选）
- 原理：多个哈希函数 + 位数组，允许假阳性
- 参数：n=1亿，p=0.1时约57MB；p=0.01时约114MB（超出限制）
- Java实现：Guava `BloomFilter` 或 Redis `BF.ADD/BF.EXISTS`
- 局限：存在误判，不支持删除

### 方案三：Roaring Bitmap
- 适合稀疏数据分布，压缩效率更高
- 支持集合运算（交集、并集），适合广告去重等场景

## 要点总结
- 100MB内存存一亿整数，BitMap仅需约24MB，空间富余
- BitMap是O(1)查询，100%准确；BloomFilter省空间但有误判
- 生产环境可考虑分段位图减少连续内存分配，或热/冷数据分级存储
- 若需支持删除，可用计数布隆过滤器（Counting BloomFilter）
- 广告点击去重系统实战：5亿用户ID范围0-10亿，位图仅119MB，查询200万QPS
