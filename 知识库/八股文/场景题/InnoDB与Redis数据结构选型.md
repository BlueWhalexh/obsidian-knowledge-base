# InnoDB与Redis数据结构选型

## 问题
InnoDB为什么不用跳表，Redis为什么不用B+树？

## 面试考察点
- 存储引擎理解：对不同数据库底层存储的理解深度
- 数据结构应用：对数据结构适用场景的把握
- 系统设计思维：技术选型的权衡分析能力
- 性能优化意识：对不同场景性能需求的理解

## 解题思路
> 核心思路：从存储介质差异（磁盘vs内存）出发分析数据结构选型

根本原因是存储介质不同。InnoDB基于磁盘存储，磁盘IO是主要瓶颈，需要最小化随机IO次数。Redis基于内存存储，内存访问无寻址代价，指针操作成本低。

B+树是多叉树，每个节点16KB数据页，扇出很高，三层即可存储2000万数据（3次磁盘IO）。跳表是链表结构，一个节点一条数据，存储2000万数据需要约24层（24次磁盘IO）。因此InnoDB选B+树。而Redis在内存中运行，不关心层数，跳表实现更简单、写入无需页分裂和旋转平衡开销，因此Redis选跳表。

## 具体方案
### 方案一：InnoDB选择B+树的原因
- 磁盘友好：节点大小=磁盘页大小（16KB），顺序预读优化
- 范围查询优势：叶子节点形成链表，范围查询只需定位首尾页
- 写入优化：仅需修改受影响页，页分裂成本可控
- 性能对比：存储2000万数据，B+树需3次IO，跳表需24次IO

### 方案二：Redis选择跳表的原因
- 内存友好：动态层级适应内存访问模式，无寻址代价
- 写入优势：插入只需修改相邻节点指针，无页分裂和旋转平衡开销
- 实现简单：代码量减少40%，调试难度显著降低
- 并发控制：无锁化实现更容易
- Redis ZSET底层：skiplist + hash，跳表支持范围查询，hash支持精确查询

## 要点总结
- 选型本质：InnoDB面向磁盘优化选IO效率最高的结构，Redis面向内存优化选实现简单且并发友好的结构
- B+树3层存2000万（3次IO），跳表24层存2000万（24次IO）
- 跳表写入性能优于B+树：无页分裂、无旋转平衡开销
- Redis ZSET结合跳表+hash，用空间换时间
- 实测：Redis改用B+树后ZSET写入QPS从15万降至9万
