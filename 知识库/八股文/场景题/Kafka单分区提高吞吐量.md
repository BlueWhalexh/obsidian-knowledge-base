# Kafka单分区提高吞吐量

## 问题
Kafka单分区单消费者实例，如何提高吞吐量？

## 面试考察点
- Kafka原理理解：消费者工作机制的掌握
- 性能优化能力：系统性提升吞吐量的思路
- 工程实践能力：实际调优经验
- 技术深度：对底层机制的深入理解

## 解题思路
> 核心思考方向：单分区单消费者意味着无法通过增加分区/消费者来并行，必须优化"消费处理链路"

约束条件明确了：不能加分区、不能加消费者实例。那就在现有条件下优化整个链路。

思考路径：消费流程是 poll -> process -> commit。瓶颈可能在哪里？1. 网络IO（拉取太频繁或太少） 2. 处理逻辑（单条处理慢） 3. 提交方式（同步提交阻塞）。逐个优化。

## 具体方案

### 方案一：参数调优（网络IO优化）
- 原理：增大单次拉取的数据量和消息数，减少网络往返
- 步骤：1. 增大fetch.min.bytes（默认1B->64KB） 2. 调整fetch.max.wait.ms（默认500ms->1s） 3. 增大max.partition.fetch.bytes（默认1MB->8MB） 4. 增加max.poll.records（默认500->5000）
- 优点：改动小，效果明显
- 缺点：可能增加单次处理延迟

### 方案二：处理逻辑优化
- 原理：批处理+异步流水线，减少单条处理开销
- 步骤：1. 批量写入数据库（合并SQL） 2. CompletableFuture异步处理 3. 内存队列缓冲+定时批量处理 4. 异步提交offset（commitAsync）
- 优点：大幅提升处理效率（批处理可提升8倍）
- 缺点：需要处理好消息失败的重试逻辑

### 方案三：本地并行化（突破性方案）
- 原理：单消费者内对消息进行二次哈希分区，多线程并行处理
- 步骤：1. poll拉取消息 2. 按key哈希分到本地多个队列 3. 多线程并行处理各队列 4. 处理完成后异步提交
- 优点：突破单线程限制，充分利用多核CPU
- 缺点：顺序性需要额外保证，实现复杂

### 监控与调优
- 关键指标：records-lag-max（消费延迟）、records-consumed-rate（吞吐）、avg-process-time-per-batch（批处理效率）
- 动态策略：根据负载调整poll间隔，背压机制防OOM

## 要点总结
- 瓶颈分析：网络IO -> 处理逻辑 -> 提交方式，逐一排查
- 参数调优是第一步：增大fetch.min.bytes和max.poll.records
- 批处理是最大提升点（可提升8倍），异步流水线次之（可提升3倍）
- 本地并行化是突破单线程限制的思路，但要注意顺序性
- 异步提交offset避免阻塞消费线程
- 生产者端也可优化：batch.size、linger.ms、compression.type
