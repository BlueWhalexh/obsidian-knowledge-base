# LBS交友系统地理空间邻近算法

## 问题

如何在LBS交友系统中高效查询附近的用户？有哪些地理空间算法可选？

## 面试考察点

- 地理空间索引算法对比
- GeoHash原理与精度
- Redis GEO数据结构
- 动态网格/四叉树方案

## 解题思路

核心问题是**如何高效检索某个坐标附近的点**，从暴力SQL到空间索引逐步优化。

## 具体方案

### 方案一：SQL邻近算法（不推荐）
- 直接用经纬度范围查询：`WHERE lat BETWEEN x1 AND x2 AND lng BETWEEN y1 AND y2`
- 问题：不精确（矩形而非圆形）、无法利用索引、性能差

### 方案二：地理网格
- 将地图划分为固定大小的网格
- 每个用户归属一个网格，查询时只搜索相邻网格
- 简单但网格大小固定，稀疏/密集区域体验不一致

### 方案三：动态网格 / 四叉树
- **四叉树**：将区域递归四分
- **阈值500用户分裂**：当一个网格内用户超过500时，分裂为4个子网格
- 自适应密度，稀疏区域大网格，密集区域小网格

### 方案四：GeoHash算法（最终选择）
- 将二维坐标编码为**一维字符串**（Base32编码）
- **前缀匹配**：相近的地理位置有相同的GeoHash前缀
- **Redis使用52bit精度**：误差约0.6米
- Redis GEO底层使用GeoHash + sorted set

```
GEOADD users 116.40 39.90 "user1"
GEORADIUS users 116.40 39.90 5 km ASC COUNT 20
```

## 要点总结

1. SQL暴力查询不可取，需要空间索引
2. 四叉树按密度自适应分裂，阈值500用户是经验值
3. GeoHash将二维降为一维，前缀匹配实现邻近查询
4. Redis GEO基于GeoHash，52bit精度0.6米，生产环境首选
5. 最终选择GeoHash方案，兼顾精度、性能和实现复杂度
