# MQ消息堆积排查与解决

## 问题
线上MQ消息队列出现消息堆积，导致CPU飙升、系统响应变慢，如何排查和解决？

## 面试考察点
- 对MQ三大阶段（生产、存储、消费）的理解，定位瓶颈在消费阶段
- 区分突发性堆积和缓慢持续增长堆积的不同处理策略
- 消息堆积对系统资源（CPU、网络IO）的连锁影响分析

## 排查思路
> 消息堆积出现在消费阶段。先判断堆积类型（突发 vs 缓慢增长），再从消费者数量、消费逻辑、下游依赖逐层排查。

## 具体排查步骤
1. **确认堆积情况**：通过MQ管理控制台查看队列深度、消费速率、生产速率
2. **检查消费者数量**：当前消费者节点数是否足够，消费速率是否低于生产速率
3. **分析消费逻辑**：消费方法中是否有慢操作（HTTP调用、数据库慢查询、锁等待）
4. **检查下游依赖**：消费者调用的外部服务是否正常（如库存系统是否超时/崩溃）
5. **查看CPU和GC**：消息堆积会导致MQ频繁存储/检索/传递，CPU飙升；重试机制也会增加额外开销

## 常见原因与解决方案
| 原因 | 解决方案 |
|------|---------|
| 消费者数量不足 | 增加消费者节点，提升消费并行度 |
| 消费逻辑中有慢调用 | 优化消费逻辑，将慢操作异步化或移除 |
| 下游服务故障导致消费阻塞 | 临时将消息快速消费转存到数据库，待下游恢复后重新投递 |
| 消息重试风暴 | 将失败消息转入死信队列，避免频繁重试 |
| MQ配置不合理 | 调整内存、连接数、堆积阈值等参数 |

## 要点总结
- 增加消费者不一定能解决问题，如果瓶颈在下游服务，增加消费者反而加剧压力
- 紧急处理：新建消费者快速消费堆积消息转存到DB，等下游恢复后再重新投递
- 预防措施：促销活动前对全链路做压测，找出瓶颈提前优化
- 消息堆积导致CPU飙升的原因：频繁存储/检索、重试开销、过多连接和网络IO
