# MySQL B+树与MongoDB B树对比

## 问题
为什么MySQL用B+树，MongoDB用B树？

## 面试考察点
- 存储引擎底层原理：B树和B+树数据结构的深入理解
- 数据库设计哲学：不同数据库的设计目标和技术选型依据
- 工程实践思维：理论知识与实际系统设计的联系
- 技术对比分析能力：客观分析不同方案的优缺点

## 解题思路
> 核心思考方向：从两种数据库的"使用场景差异"出发，反推数据结构选择的原因

不要直接对比B树和B+树的结构差异，而是先想清楚MySQL和MongoDB各自的典型使用场景，再理解为什么各选各的。

MySQL是关系型数据库，核心场景是范围查询、多表JOIN、排序操作；MongoDB是文档数据库，核心场景是单文档随机访问。场景不同，最优的数据结构自然不同。同时要考虑磁盘IO特性——索引存在磁盘上，树的高度等于IO次数。

## 具体方案

### 方案一：理解B+树（MySQL的选择）
- 原理：非叶子节点只存key不存数据，叶子节点存数据并通过双向链表连接
- 优势：1. 非叶子节点不存数据，每个节点可容纳更多key，树更矮，IO更少 2. 叶子节点链表天然支持范围查询 3. 查询时间复杂度固定为O(log n)
- 适用场景：范围查询、全表扫描、排序操作、多表关联

### 方案二：理解B树（MongoDB的选择）
- 原理：所有节点都存储key和数据，搜索可在非叶子节点结束
- 优势：1. 单点查询可能O(1)就找到（数据在非叶子节点） 2. 所有节点都有数据字段，单文档访问更快 3. 更适合处理变长文档数据
- 适用场景：单文档随机访问、聚合查询、嵌套文档结构

### 对比总结
- 查询模式：B+树适合范围查询（叶子链表遍历），B树适合点查询（任意节点可返回）
- 磁盘IO：B+树分支因子更高，同样数据量树更矮
- 设计目标：MySQL追求遍历和范围查询效率，MongoDB追求单文档访问速度

## 要点总结
- B+树非叶子节点不存数据 -> 节点能存更多key -> 树更矮 -> IO更少
- B+树叶子节点双向链表 -> 范围查询只需遍历叶子层
- B树所有节点都存数据 -> 单点查询可能更快（非叶子节点就命中）
- 选择哪种数据结构取决于数据库的核心使用场景
- 面试官可能追问：为什么Redis用跳表？——内存场景看重简单性和并发性能
