# MySQL热点数据更新问题

## 问题
MySQL热点数据更新会带来哪些问题？

## 面试考察点
- 并发控制理解：对锁机制和事务隔离的掌握程度
- 性能瓶颈分析：识别和解决性能问题的能力
- 高可用意识：对系统稳定性的考量
- 实战经验：实际处理热点更新场景的经验

## 解题思路
> 核心思路：热点行更新的本质是lock-update-unlock串行化，导致三大后果

MySQL的更新模式是"lock-update-unlock"，大量写请求并发访问少数数据行时，行锁强制串行执行，高并发下热点行处理能力仅约200 TPS。且写热点无法通过增加副本解决（不同于读热点）。

以库存扣减为例：select预检查 -> begin事务 -> insert交易流水 -> update库存 -> select结果 -> commit。瓶颈在于update操作的行锁串行排队。

## 具体方案
### 方案一：短期缓解
- 应用层排队：Redis List削峰（rightPush入队）
- 批处理合并：将多次stock-1合并为stock-100一次更新
- 优点：实现简单，快速见效
- 缺点：实时性差，用户体验受影响

### 方案二：中期优化 - 分桶打散
- 原理：将单个热点行拆分为多个逻辑分片
- 实现：UPDATE inventory_shards SET stock=stock-1 WHERE product_id=1001 AND shard_id=#{threadId % 10}
- 缓存中间结果：先更新Redis incr，再异步持久化到MySQL
- 优点：显著提升并发能力
- 缺点：增加业务复杂度

### 方案三：长期架构
- 分布式计数器：使用Cassandra等擅长高频写的数据库
- 事件溯源模式：INSERT INTO inventory_events (product_id, change_amount) VALUES (1001, -1)
- Redis+Lua原子扣减 + 异步批量同步到MySQL

### 热点三大后果
- 非热点数据处理性能下降：热点行事务占用线程池资源不释放
- 热点行TPS归零、RT飙升：行锁竞争 + 死锁检测成本
- 连接池耗尽雪崩：连接不能及时释放，Server端压力进一步增加

## 要点总结
- 热点行更新本质：lock-update-unlock串行化，高并发仅约200 TPS
- 写热点与读热点不同：无法通过增加副本解决
- 三大后果：非热点受影响、热点TPS归零、连接池耗尽雪崩
- 分桶打散是核心优化思路：将单行拆为多行分散锁竞争
- 实战方案：库存拆10个分片 + Redis+Lua原子扣减 + 异步同步MySQL，峰值QPS从200提升到50000
