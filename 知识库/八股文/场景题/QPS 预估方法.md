# QPS 预估方法

## 一、如何预估接口上线后的 QPS

### 1.1 二八法则（最常用）

**核心思想**：80% 的请求集中在 20% 的时间段内。

**计算公式**：

```
总 PV（日均页面访问量）× 80% ÷ (每天秒数 × 20%) = 峰值 QPS
```

**举例**：某系统日均 PV 为 1000 万

```
峰值 QPS = 10,000,000 × 0.8 ÷ (86,400 × 0.2)
         = 8,000,000 ÷ 17,280
         ≈ 463 QPS
```

**细化版本**：实际使用时还需考虑以下系数

- **业务高峰系数**：电商类系统在大促期间可能是日常的 5~10 倍
- **读写比**：读多写少的系统，写接口 QPS 远低于读接口
- **接口粒度**：一个页面可能调用多个接口，需分别估算

### 1.2 历史数据推算

**适用场景**：已有线上系统，新接口与老接口有业务关联。

**方法**：

1. 从监控系统（Prometheus/Grafana/ELK）获取现有接口的 QPS 数据
2. 根据新接口与老接口的业务比例推算
3. 考虑新功能上线后的用户增长预期

**示例**：

```
现有下单接口 QPS = 200
新接口"订单确认"预计与下单接口 1:1 调用
→ 新接口预估 QPS ≈ 200
```

### 1.3 压测数据反推

**适用场景**：系统已部署到测试/预发布环境。

**方法**：

1. 使用压测工具（JMeter/Locust/wrk）逐步加压
2. 观察系统指标：RT、错误率、CPU、内存、GC
3. 找到系统的性能拐点（错误率突增或 RT 急剧上升的临界点）
4. 该拐点的 QPS 即为系统承载上限

**压测关键指标**：

| 指标 | 含义 | 关注阈值 |
|------|------|----------|
| QPS | 每秒查询数 | 核心指标 |
| RT（P99） | 99% 请求的响应时间 | < 200ms |
| 错误率 | 失败请求占比 | < 0.1% |
| CPU 使用率 | 服务器 CPU 占用 | < 80% |
| GC 停顿 | Full GC 频率 | 尽量为 0 |

## 二、如何预估/回答接口本身能抗住多少 QPS

### 2.1 单机 QPS 估算

**方法一：理论计算**

```
单机 QPS = 1000ms ÷ 单次请求平均耗时(ms) × 线程数
```

**示例**：

- Tomcat 默认线程数 200
- 接口平均 RT = 50ms
- 单机 QPS = 1000 ÷ 50 × 200 = 4000 QPS

**方法二：压测实测**

这是最准确的方法。直接对单台机器进行压测，逐步增加并发数，找到性能拐点。

```
步骤：
1. 准备一台与线上配置相同的服务器
2. 使用 JMeter 设置阶梯式加压：50 → 100 → 200 → 500 → 1000 并发
3. 每个阶梯持续运行 5 分钟
4. 记录每个阶梯的 QPS、RT、错误率
5. 找到 RT 陡增或错误率突增的拐点
```

**方法三：经验公式（快速估算）**

| 接口类型 | 典型 RT | 典型单机 QPS |
|----------|---------|-------------|
| 纯内存操作（Redis/本地缓存） | 1~5ms | 20,000~100,000 |
| 简单 DB 查询（走索引） | 5~20ms | 5,000~20,000 |
| 复杂业务逻辑 + DB | 20~100ms | 1,000~5,000 |
| 调用外部服务 | 50~500ms | 200~2,000 |
| 大量计算/文件处理 | 100ms+ | 200~1,000 |

### 2.2 集群总 QPS

```
集群总 QPS = 单机 QPS × 机器数 × 线性系数
```

**线性系数**：实际中不会完全线性扩展，通常取 0.7~0.9，受以下因素影响：

- 负载均衡是否均匀
- 是否有共享资源（数据库、Redis）成为瓶颈
- 会话粘滞（Session Sticky）导致负载不均
- 网络带宽限制

**示例**：

```
单机压测 QPS = 3000
部署 10 台机器
线性系数取 0.8
集群 QPS = 3000 × 10 × 0.8 = 24,000 QPS
```

### 2.3 面试回答模板

**问：你们这个接口能抗多少 QPS？**

> 我们这个接口经过压测，单机 QPS 大约在 XXX 左右（说明机器配置），线上部署了 N 台机器，所以整体能抗住大约 XXX 的 QPS。我们的日常峰值大概在 XXX，大促期间预估峰值在 XXX，所以留了 N 倍的余量。

**回答要点**：

1. **有数据支撑**：说出具体的压测数据，而非"应该没问题"
2. **说明机器配置**：4C8G 还是 8C16G，让面试官有参考基准
3. **说明业务场景**：日常多少，峰值多少，是否有大促
4. **说明余量设计**：通常预留 2~3 倍余量
5. **说明瓶颈在哪**：CPU 密集还是 IO 密集，瓶颈在 DB 还是 Redis

## 三、QPS 评估后的架构设计

### 3.1 根据 QPS 量级选择架构

| QPS 量级 | 架构方案 |
|----------|---------|
| < 100 | 单机即可，无需缓存 |
| 100~1,000 | 加 Redis 缓存 + MySQL 读写分离 |
| 1,000~10,000 | 多机部署 + Redis 集群 + MQ 削峰 |
| 10,000~100,000 | 分布式架构 + 分库分表 + 多级缓存 |
| > 100,000 | CDN + 限流降级 + 异步化 + 单元化部署 |

### 3.2 容量规划公式

```
所需机器数 = 预估峰值 QPS ÷ (单机 QPS × 线性系数) × 冗余系数
```

**冗余系数**：通常取 1.5~3，考虑故障容错和突发流量。

## 四、常见追问

### Q：二八法则估算出来的 QPS 和实际差距大吗？

> 在电商、社交等流量波动明显的场景下差距可能较大。二八法则是基于均匀分布的假设，实际流量可能呈现脉冲式分布（如秒杀场景下几秒内涌入大量请求）。所以二八法则只适合做初步估算，最终还是要靠压测验证。

### Q：压测数据和线上真实数据会差多少？

> 通常压测数据会比线上高 10%~20%，因为压测环境数据量较小、缓存命中率较高、没有真实的网络抖动等因素。所以线上实际承载能力要按压测数据打 8 折来估算。

### Q：如何动态监控线上 QPS 是否接近瓶颈？

> 1. 设置 Grafana 监控大盘，实时展示 QPS、RT、错误率
> 2. 设置告警规则：QPS 超过阈值 80% 时告警
> 3. 关注系统水位指标：CPU > 70%、GC 频率升高、连接池使用率 > 80%
> 4. 定期进行全链路压测，验证系统容量是否满足业务增长
