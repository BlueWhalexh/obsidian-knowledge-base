# RPC 请求客户端超时但服务端不超时

## 问题
一次 RPC 请求，客户端显示超时，但是服务端不超时，可能是什么原因？

## 面试考察点
- 网络诊断能力：对 RPC 调用全链路（序列化 -> 网络 -> 处理 -> 返回）的理解
- 全链路思维：客户端到服务端完整路径的分层分析
- 调优经验：实际处理 RPC 超时问题的排查经验
- 技术广度：Dubbo/gRPC 超时机制、连接池、负载均衡等中间件知识

## 解题思路
> 超时是客户端视角的概念，核心排查"请求是否到达服务端"以及"响应是否成功返回客户端"

RPC 调用全流程：客户端序列化 -> 网络传输 -> 服务端处理 -> 响应返回 -> 客户端反序列化。超时可能发生在任何环节。首先要确认服务端是否收到请求（查日志或 tcpdump 抓包），如果没收到则问题在网络层；如果收到了且处理完成，问题在响应返回路径。

排查遵循分层原则：先网络层（ping/traceroute/tcpdump 检查 RTT、丢包、重传） -> 再应用层（Arthas 追踪调用链耗时） -> 最后中间件（线程池状态、连接池配置）。七大常见原因各有特征：配置过短、网络丢包、异步处理、响应未返回、连接池问题、路由错误、防火墙拦截。

## 具体方案
### 七大可能原因
- **客户端超时配置过短**：客户端设 500ms，服务端实际需 600ms，对比两端配置
- **网络延迟或丢包**：请求/响应在传输中被延迟，用 ping/traceroute/tcpdump 排查
- **服务端异步处理**：收到请求后提交线程池但未同步返回，检查 @Async 注解
- **响应未正确返回**：响应数据过大、线程池满、序列化耗时过长
- **客户端连接池问题**：连接池满导致请求排队，或连接泄漏导致无法建立新连接
- **负载均衡路由错误**：请求被路由到异常节点，检查健康检查和心跳检测
- **防火墙/中间件拦截**：响应包被丢弃，用 tcpdump 对比两端数据包

### 排查步骤
1. 确认服务端是否收到请求（服务端日志 + tcpdump 抓包）
2. 检查服务端处理耗时（记录开始和结束时间）
3. 检查网络延迟（ping/traceroute/telnet 端口连通性）
4. 分析客户端连接池状态（最大连接数、是否有泄漏）
5. 检查中间件和防火墙（Nginx/Envoy 日志、防火墙规则）
6. 分析响应数据大小和 jstack 线程状态

### 解决方案
- 网络问题：改用专线/VPC 内网，启用 TCP 快速重传
- 配置对齐：客户端超时与服务端默认超时保持一致
- 序列化优化：大对象改用 Protobuf 替代 JSON
- 动态超时：根据历史 P99 响应时间自动调整（timeout = avgResponseTime x 3）

## 要点总结
- 超时是客户端视角，服务端可能完全正常，问题可能在网络或配置
- 排查顺序：确认请求到达 -> 检查处理耗时 -> 网络诊断 -> 中间件检查
- 最常见原因是客户端超时配置过短和网络延迟/丢包
- 连接池满和线程池满是最容易忽略的隐性原因
- 生产环境建议动态超时（基于 P99）+ 熔断降级兜底
