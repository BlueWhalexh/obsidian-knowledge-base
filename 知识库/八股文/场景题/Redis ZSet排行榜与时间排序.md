# Redis ZSet排行榜与时间排序

## 问题
Redis的ZSet实现排行榜，分数相同按时间顺序排序，怎么做？

## 面试考察点
- Redis高级特性应用：能否灵活运用ZSET解决实际问题
- 数据编码设计：复合排序条件的处理能力
- 性能优化意识：保证功能的同时考虑性能影响
- 并发安全意识：原子操作的保证

## 解题思路
> 核心思考方向：ZSet的score是64位浮点数，将分数和时间"编码"到一个score中

ZSet只支持按score排序，要实现"分数相同按时间排序"，就需要把两个维度的信息编码到一个score里。关键问题是：如何编码才能保证分数优先级高于时间？如何保证并发安全？

思考路径：score是double类型 -> 分数放高位，时间放低位 -> 需要保证原子操作（Lua脚本）。

## 具体方案

### 方案一：分数+时间戳拼接
- 原理：score = 分数 + (活动结束时间 - 当前时间)的固定位数拼接
- 步骤：1. 计算时间差 = 结束时间 - 当前时间 2. 补零到固定位数 3. 拼接为 score = 分数 + 时间差 4. 提取分数时截取前N位
- 优点：思路简单直观
- 缺点：拼接方式精度受限，需Lua脚本保证原子性

### 方案二：位运算方案（推荐）
- 原理：score = 分数 << 27 + (结束时间 - 当前时间)，左移保证分数在高位
- 步骤：1. 分数左移27位（留出时间戳空间） 2. 加上时间戳差值 3. 提取真实分数时右移27位 4. 用Lua脚本保证原子操作
- 优点：位运算高效，精度可控
- 缺点：需要理解位运算

### 方案三：雪花算法思想
- 原理：借鉴雪花算法，1位保留+22位分数+41位时间戳
- 步骤：1. 分数放在高位 2. 时间戳放在低位 3. 分数越大值越大，同分数则时间越小（越早）值越大
- 优点：结构清晰，扩展性好
- 缺点：22位分数范围有限（2^21-1），可通过压缩时间戳位数解决

### Java实现参考
- 原理：score = 原始分数 + (1 - 时间戳/最大时间戳)
- 关键：时间戳标准化为小数部分，分数为整数部分，同分时时间越大标准化值越大，1-normalizedTime越小，排名越靠后

## 要点总结
- 核心思想：将多维度排序编码到ZSet的单个score中
- 分数必须在高位，时间在低位，保证分数优先排序
- 并发安全是重点：必须用Lua脚本保证读取-计算-写入的原子性
- score是64位double，注意精度问题（分数超过2^52时可能丢失精度）
- 可用evalsha缓存Lua脚本减少网络开销
