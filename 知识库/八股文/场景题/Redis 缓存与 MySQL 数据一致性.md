# Redis 缓存与 MySQL 数据一致性

## 问题
如何保证Redis缓存与MySQL数据的一致性？先删缓存还是先更新数据库？各种方案的优缺点是什么？

## 答案

### 缓存一致性问题

在数据更新时，如何保证MySQL数据和Redis缓存的数据一致性。

### 方案一：先更新DB，再删除缓存（推荐）

**流程**：更新MySQL数据 → 同步删除Redis缓存（缓存设置TTL兜底）

**优点**：简单实现，同步删除成功时数据不一致时间极短

**并发场景分析**：
```
读请求：缓存未命中，读取数据库值为20
写请求：更新数据库值为21，删除缓存
读请求：将20写回缓存  ← 数据不一致
```

但此情况概率极小：因为更新MySQL耗时 >> 写入Redis耗时，正常流程下读请求不应在写请求之后才写回缓存。

**为什么不推荐先删缓存再更新DB？**
```
写请求：删除缓存
读请求：缓存未命中，读取数据库旧值20，写入缓存
写请求：更新数据库值为21
```
因为更新MySQL耗时远大于删除Redis耗时，容易在"删除缓存"和"更新DB"之间有并发读请求读到旧值。

### 方案二：延迟双删

**流程**：
1. 先删除缓存
2. 更新数据库
3. 延迟一段时间后，再次删除缓存

**第一次先删除缓存的原因**：避免"先更新DB再删缓存"导致的一段数据不一致时间。

**第二次延迟删除的原因**：解决读写并发导致的数据不一致。延迟时间需等待并发读请求完成写入缓存，经验值500ms-1s。

**延迟双删的问题**：
1. 延迟时间难以精确设定
2. 如果未发生并发读写，第二次删除可能删掉正确的缓存值
3. 第二次删除不一定成功，仍需引入消息队列重试

### 方案三：消息队列重试

**流程**：
1. 更新MySQL数据，同步删除缓存
2. 如果删除失败，发送MQ消息
3. 消费者监听消息，删除缓存
4. 失败触发重试机制，超过最大次数则告警

**缺点**：对代码侵入性强，引入MQ增加系统复杂度。

### 方案四：Canal监听BinLog + 消息队列重试

**流程**：
1. 更新MySQL数据，同步删除缓存
2. Canal模拟MySQL主从复制协议，获取BinLog（数据库更新记录）
3. 下游服务订阅Canal数据，发送MQ删除缓存消息
4. 消费者监听消息删除缓存，有重试机制和告警

**优点**：解除代码侵入问题
**缺点**：引入组件多，系统复杂度高

### 为什么是删除缓存而不是更新？

1. 删除更简单耗时更短；更新若涉及复杂数据聚合则耗时长
2. 写写并发场景下更新缓存可能导致数据不一致

### 删除失败的兜底方案

无论选择哪种方案，都需要为缓存设置TTL作为兜底，到期自动删除缓存。

### 方案总结与推荐

| 方案 | 优点 | 缺点 |
|------|------|------|
| 先更新DB再删缓存 + MQ重试 + TTL | 同步删除保证不一致窗口短，重试机制保证 | 引入MQ |
| Canal + MQ重试 | 无代码侵入 | 组件多，复杂度高 |
| 延迟双删 | 无额外组件 | 延迟时间难确定 |

**推荐方案**：先更新数据库，再删除缓存，若删除失败使用消息队列重试 + TTL兜底。同步删除保证数据不一致时间段很短，删除失败重试机制保证，TTL兜底。

**重要结论**：以上方案都不能保证强一致性，只保证最终一致性。

### 最佳实践

1. 为所有缓存设置TTL作为兜底
2. 定期对比缓存和数据库的数据是否一致
3. 发现多次删除失败时进行数据一致性校验，强行修改缓存数据
