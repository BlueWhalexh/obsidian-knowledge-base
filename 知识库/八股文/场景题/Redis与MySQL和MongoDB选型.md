# Redis与MySQL和MongoDB选型

## 问题
Redis、MySQL和MongoDB的区别是什么，各自适用场景呢？

## 面试考察点
- 三种数据库的核心特性差异：数据模型、一致性、扩展性
- 技术选型原则：根据业务需求匹配数据库特性
- 混合架构设计：MySQL + Redis + MongoDB组合使用

## 解题思路
> 核心思路：没有万能数据库，需根据数据模型、一致性要求、性能需求、扩展性要求选择合适的数据库，实际项目中常组合使用。

Redis是键值型内存数据库，微秒级响应，适合缓存/计数器/分布式锁；MySQL是关系型数据库，强一致性ACID事务，适合金融交易/订单管理；MongoDB是文档型NoSQL，天然水平扩展，适合日志/内容管理/IoT数据。三者常结合使用构建高性能系统。

选型关键维度：数据模型（键值/关系/文档）、一致性（最终/强一致）、扩展性（垂直/水平）、查询能力（简单键值/复杂SQL/灵活查询）、存储成本（内存/磁盘）。

## 具体方案
### 方案一：按业务场景选型
- Redis适用：高并发缓存、实时计数器、分布式锁、消息队列
- MySQL适用：强一致性事务、复杂JOIN查询、中小规模结构化数据
- MongoDB适用：半结构化数据、灵活Schema、水平扩展、高并发写入

### 方案二：混合架构（推荐）
- 原理：MySQL存核心数据（用户/订单），Redis做缓存（库存/会话），MongoDB存日志/行为数据
- 流程：用户下单 -> MySQL记录订单 -> Redis缓存库存原子扣减 -> MongoDB存浏览日志
- 优点：QPS从1万提升至10万，存储成本降低60%

## 要点总结
- Redis：微秒级响应，单线程IO多路复用，适合缓存/限流/实时计数，不适合海量存储和复杂查询
- MySQL：强一致性ACID，支持复杂SQL和JOIN，适合金融/订单，不适合高并发写入和非结构化数据
- MongoDB：灵活Schema、水平扩展、高并发写入，适合日志/IoT/内容管理，不适合强事务场景
- 实际项目推荐混合架构：Redis + MySQL + MongoDB，按数据特性分层存储
- 用Redis存冷数据是反模式（内存成本过高），用MySQL处理海量日志也是反模式
