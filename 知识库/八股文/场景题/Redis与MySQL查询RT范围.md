# Redis与MySQL查询RT范围

## 问题
Redis和MySQL的一次普通查询，RT在什么范围内是合理的？

## 面试考察点
- 数据库性能基准的理解（什么算正常）
- 生产环境性能敏感度（如何定义快慢）
- 不同场景的性能预期管理
- 问题排查经验（如何判断性能劣化）

## 解题思路
> 核心思路：RT合理范围需分场景讨论，从理论基准和生产实际两个维度来看，建立分层的性能标准。

Redis查询RT：简单命令（GET/SET）0.1-1ms，复杂命令（Pipeline/Lua）1-10ms，大Value读写10-100ms。影响因素：网络延迟、Value大小、Pipeline批量、Lua脚本复杂度、持久化策略。

MySQL查询RT：主键/唯一索引查询1-5ms，覆盖索引5-20ms，全表扫描/未命中索引50-500ms，JOIN查询10-100ms，聚合函数10-500ms。影响因素：索引使用、锁竞争、磁盘IO、连接池配置、SQL优化。

## 具体方案
### Redis RT标准
- <1ms：理想状态（缓存/计数器/热点数据）
- 1-10ms：正常波动（网络或大Value影响）
- >10ms：需排查慢命令/大Value/Lua阻塞/网络抖动

### MySQL RT标准
- <5ms：理想状态（高频核心查询）
- 5-50ms：正常波动（索引扫描或小表JOIN）
- 50-100ms：需优化（检查索引/filesort/大表扫描）
- >100ms：严重性能问题（全表扫描/锁竞争/磁盘瓶颈）

### 异常预警线
- Redis P99 > 10ms：检查大key/热点key/网络
- MySQL简单查询P90 > 30ms：检查索引/锁竞争
- 同一业务RT突增50%以上需介入排查

## 要点总结
- Redis简单查询理想RT < 1ms，生产环境P99应 <= 5ms
- MySQL主键查询理想RT < 5ms，生产环境P99应 <= 20ms
- Redis > 10ms需排查：大Key、AOF重写fork阻塞、连接池耗尽
- MySQL > 100ms需排查：全表扫描、锁等待、缓冲池命中率低
- 避免绝对化表述，区分冷热数据场景，考虑批量操作对平均RT的影响
