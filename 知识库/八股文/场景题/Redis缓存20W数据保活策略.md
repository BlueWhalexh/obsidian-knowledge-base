# Redis缓存20W数据保活策略

## 问题
MySQL 2000W数据，Redis只存20W，如何保证Redis中的数据是热点数据？

## 面试考察点
- 缓存策略理解：对缓存淘汰和更新机制的掌握
- 系统设计能力：多组件协同工作的设计能力
- 性能优化意识：识别和优化热点数据的能力
- 数据一致性：保证缓存与数据库一致的方案

## 解题思路
> 核心思路：热点识别 + LFU淘汰 + 动态更新 + 分层缓存

关键在于三点：如何识别热点数据、如何淘汰冷数据、如何保持数据一致性。Redis提供8种淘汰策略，LFU（最不经常使用）比LRU更适合热点场景，因为它基于访问频率而非最近访问时间。

热点识别可以通过实时统计（慢查询日志分析）、业务标记（注解标记热点接口）、动态评分模型（热度分数=访问频率x0.6+业务权重x0.3+时间衰减x0.1）三种方式结合。

## 具体方案
### 方案一：LFU淘汰策略
- 配置：maxmemory-policy allkeys-lfu，lfu-log-factor 10，lfu-decay-time 60
- 原理：自动淘汰访问频率最低的数据，热点数据自然留存
- 优点：无需额外开发，Redis原生支持
- 命中率对比：LRU约85%，LFU约92%

### 方案二：热点识别系统
- 实时统计层：分析MySQL慢查询日志，按query_count排序取Top 20万
- 业务标记层：@HotSpotData注解标记热点业务接口
- 动态评分模型：热度分数=访问频率x0.6+业务权重x0.3+时间衰减x0.1
- 冷启动：从MySQL分析日志获取热点ID，批量pipeline加载到Redis

### 方案三：分层缓存架构
- 请求流转：客户端 -> 本地二级缓存 -> Redis热点缓存 -> MySQL
- 未命中回写：Redis未命中时从MySQL查询并回写Redis
- 主动推送：数据库变更时同步更新Redis
- 被动淘汰：监控淘汰日志，分析淘汰原因

## 要点总结
- LFU比LRU更适合热点场景：基于访问频率而非最近访问时间
- 热点识别三板斧：实时统计 + 业务标记 + 动态评分模型
- 冷启动方案：分析日志获取热点ID，pipeline批量加载，限流控制每秒1000条
- 监控验证：keyspace_hits/(keyspace_hits+keyspace_misses)计算命中率
- 实战效果：Redis命中率从70%提升到98%，MySQL负载减少30%
