# RocketMQ顺序消费性能优化

## 问题

RocketMQ原生顺序消费机制性能差，如何优化？如何实现关联顺序性？

## 面试考察点

- 原设计三次加锁机制的瓶颈
- 关联顺序性概念（同Key串行，不同Key并发）
- 按Key hashCode路由到任务队列
- 并发度优化

## 解题思路

原生RocketMQ顺序消费是**全局顺序**（一个队列一个消费线程），并发度极低。优化方向是**关联顺序性**：只有相关消息才需要顺序，无关消息可以并发。

## 具体方案

### 原设计的三次加锁（性能瓶颈）
1. **队列锁**：锁定MessageQueue，防止多消费者同时消费同一队列
2. **MessageQueue锁**：本地锁，控制单个队列的消费
3. **ProcessQueue锁**：处理队列的消费锁

问题：一个队列只能串行消费，并发度 = 队列数，严重受限。

### 优化方案：关联顺序性

**核心思想**：同Key消息串行，不同Key消息并发

**实现方式**：
1. 消费者从队列拉取消息
2. 按消息Key的**hashCode % 线程数**路由到不同的**任务队列**
3. 同Key消息路由到同一任务队列，保证顺序
4. 不同Key消息路由到不同任务队列，并发消费

```
消息Key -> hashCode -> % 线程数 -> 对应任务队列
订单A   -> 123      -> % 4 = 3  -> 队列3
订单B   -> 456      -> % 4 = 0  -> 队列0
订单A'  -> 123      -> % 4 = 3  -> 队列3（与订单A串行）
```

### 并发度提升
- 原方案：并发度 = 队列数（通常4-8个）
- 优化后：并发度 = 队列数 × 线程数（可达数十倍）

## 要点总结

1. 原生三次加锁导致并发度严重受限
2. 关联顺序性是核心优化思路：同Key串行，不同Key并发
3. hashCode % 线程数路由是实现方案，简单高效
4. 并发度可提升数十倍，显著改善消费性能
