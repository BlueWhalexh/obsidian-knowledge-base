# Top10搜索关键词统计

## 问题
如何从1TB的搜索日志中找出搜索量最高的10个关键词？

## 面试考察点
- 大数据处理能力：如何高效处理远超内存的数据集
- TopK问题的解决方案：哈希统计+堆排序
- MapReduce分布式计算范式的理解
- 近似算法（Count-Min Sketch）的了解

## 解题思路
> 核心思路：分块处理+哈希映射确保相同关键词到同一文件+局部词频统计+全局归并找Top10。

分块处理：将1TB数据分割成1000个小文件（每块约1GB）。使用哈希函数hash(keyword) % 1000将关键词分配到对应文件，确保相同关键词在同一文件中。对每个小文件用HashMap统计词频，再用最小堆维护全局Top10。

全局归并：遍历所有小文件的词频结果，使用大小为10的最小堆。堆未满时直接入堆，堆已满时若新词频>堆顶则替换并调整。时间复杂度O(N log K)，K=10。

## 具体方案
### 方案一：分块+哈希+最小堆（精确方案）
- 步骤：分1000块 -> hash分配 -> HashMap统计词频 -> 最小堆取Top10
- 优点：100%精确，无需外部框架
- 适用：单机32GB内存处理1TB数据

### 方案二：分布式处理（Spark/Hadoop）
- Map阶段：统计每个分片词频
- Shuffle阶段：按关键词聚合
- Reduce阶段：合并词频输出Top10
- 适用：PB级数据、需快速处理

### 方案三：Count-Min Sketch（近似方案）
- 原理：多个哈希函数映射到二维数组，统计时取最小值
- 优点：内存占用极小（10MB），支持实时统计
- 缺点：约3%计数误差
- 适用：允许误差的高频更新场景

## 要点总结
- 核心方案：哈希分块 + HashMap统计 + 最小堆取TopK，时间复杂度O(N log K)
- 哈希分块保证相同关键词到同一文件，避免跨文件重复统计
- 最小堆维护TopK：堆未满直接入堆，堆满后比堆顶大则替换
- 近似方案Count-Min Sketch可将内存降至10MB但有~3%误差
- 若需求升级为实时Top10，可用Flink滑动窗口机制，每5分钟更新榜单
