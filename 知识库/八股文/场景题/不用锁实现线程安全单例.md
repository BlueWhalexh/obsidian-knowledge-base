# 不用锁实现线程安全单例

## 问题
不使用synchronized和Lock如何设计一个线程安全的单例？

## 面试考察点
- 并发编程深度：Java内存模型和并发机制的掌握
- 设计模式理解：单例模式的各种实现方式
- JVM知识：类加载机制的理解
- 创新思维：突破常规解决方案的能力

## 解题思路
> 核心思考方向：不显式用锁，但可以借助JVM的类加载机制保证线程安全

面试官说"不用synchronized和Lock"，但底层ClassLoader.loadClass方法本身就用了synchronized。所以关键是理解：我们不直接用锁，而是借助JVM层面的线程安全保证。

思考路径：1. 能否在类加载时就完成初始化？（饿汉式） 2. 能否延迟加载但借助类加载机制？（静态内部类） 3. 有没有更安全的方式？（枚举） 4. 能否展示并发知识？（CAS）

## 具体方案

### 方案一：饿汉式（静态初始化）
- 原理：利用静态字段只初始化一次的特性
- 实现：`private static final Singleton INSTANCE = new Singleton();` 直接返回INSTANCE
- 优点：最简单、线程安全
- 缺点：非懒加载，类加载时就初始化

### 方案二：静态内部类（推荐）
- 原理：Holder类只有在getInstance()调用时才加载，JVM类加载时加锁保证线程安全
- 实现：`private static class Holder { static final Singleton INSTANCE = new Singleton(); }` 返回Holder.INSTANCE
- 优点：懒加载+无锁+高效+实现简单
- 缺点：无法防止反射攻击

### 方案三：枚举单例（最安全）
- 原理：枚举实例创建由JVM保证线程安全，且天然防止反射和反序列化破坏
- 实现：`public enum Singleton { INSTANCE; }`
- 优点：防反射攻击+防序列化破坏+代码简洁
- 缺点：不够直观，实际使用较少

### 方案四：CAS实现（展示并发知识）
- 原理：AtomicReference + compareAndSet，无锁乐观竞争
- 实现：for循环中get()检查 -> new Singleton() -> compareAndSet(null, instance)
- 优点：展示CAS知识，真正无锁
- 缺点：可能创建多余实例，实际项目不推荐

## 要点总结
- 静态内部类是最佳推荐：懒加载+无锁+JVM保证线程安全
- 枚举是最安全方案：防反射+防序列化，Effective Java推荐
- 本质都是借助ClassLoader的线程安全机制（loadClass方法内部有synchronized）
- CAS方案可作为加分项展示并发知识，但不推荐生产使用
- 面试时先说方案，再分析原理，最后给出推荐和理由
