# 与外部服务交互防拖垮

## 问题

系统需要调用外部合作方的接口，如何防止外部服务不可用时拖垮自己的系统？

## 问题分析

外部服务不可用时会导致：

1. **线程阻塞**：调用超时时间内线程被占用，线程池耗尽
2. **响应变慢**：大量请求等待外部服务响应，用户体验差
3. **级联故障**：上游服务超时导致调用方也超时，故障传播
4. **资源耗尽**：连接池、内存等资源被占满

---

## 方案一：超时控制

### 核心原则

**必须设置合理的超时时间**，永远不要使用默认的无限超时。

### 超时时间设置

```java
@Configuration
public class ExternalServiceConfig {
    
    @Bean
    public RestTemplate externalRestTemplate() {
        // 连接超时：建立TCP连接的超时时间
        RequestConfig requestConfig = RequestConfig.custom()
            .setConnectTimeout(3000)      // 3秒
            .setSocketTimeout(5000)       // 读取超时5秒
            .setConnectionRequestTimeout(2000) // 从连接池获取连接超时2秒
            .build();
        
        CloseableHttpClient httpClient = HttpClientBuilder.create()
            .setDefaultRequestConfig(requestConfig)
            .build();
        
        return new RestTemplate(new HttpComponentsClientHttpRequestFactory(httpClient));
    }
}
```

### 超时时间如何确定

```
超时时间 = 外部服务 P99 响应时间 × 1.5 ~ 2 倍
```

- 先看外部服务的 SLA 文档或历史监控数据
- 通常连接超时 3 秒，读取超时 5~10 秒
- 不同接口设置不同超时（非核心接口更短）

### 超时后的处理

```java
try {
    Response result = externalService.call(request);
    return result;
} catch (TimeoutException e) {
    // 记录日志
    log.warn("External service timeout: {}", e.getMessage());
    // 返回降级结果
    return getDefaultValue();
}
```

## 方案二：熔断降级

### 什么是熔断

当下游服务的错误率超过阈值时，自动"断开"调用，直接返回降级结果，避免持续调用失败的服务消耗资源。

### 熔断器状态机

```
        失败率超过阈值
关闭(Closed) ──────→ 打开(Open) ──────→ 半开(Half-Open)
    ↑                                        │
    │        探测成功                         │
    └─────────────────────────────────────────┘
              探测失败 → 回到 Open
```

- **关闭（Closed）**：正常调用，统计失败率
- **打开（Open）**：直接返回降级结果，不调用外部服务
- **半开（Half-Open）**：允许少量请求通过，试探外部服务是否恢复

### Sentinel 实现（推荐）

```java
// 1. 定义资源
@SentinelResource(
    value = "externalService",
    blockHandler = "handleBlock",
    fallback = "handleFallback"
)
public String callExternalService(String param) {
    return externalService.call(param);
}

// 2. 限流降级处理
public String handleBlock(String param, BlockException ex) {
    return "服务繁忙，请稍后重试";
}

// 3. 熔断降级处理
public String handleFallback(String param, Throwable ex) {
    return getDefaultValue();
}

// 4. 配置熔断规则
@PostConstruct
public void initDegradeRule() {
    DegradeRule rule = new DegradeRule("externalService")
        .setGrade(CircuitBreakerStrategy.ERROR_RATIO.getType())
        .setCount(0.5)         // 错误率阈值 50%
        .setTimeWindow(10)     // 熔断持续时间 10 秒
        .setMinRequestAmount(10) // 最少请求数
        .setStatIntervalMs(10000); // 统计窗口 10 秒
    DegradeRuleManager.loadRules(Collections.singletonList(rule));
}
```

### Hystrix 实现

```java
@HystrixCommand(
    fallbackMethod = "fallback",
    commandProperties = {
        // 10 秒内请求数 > 20 且错误率 > 50% 时熔断
        @HystrixProperty(name = "circuitBreaker.requestVolumeThreshold", 
                         value = "20"),
        @HystrixProperty(name = "circuitBreaker.errorThresholdPercentage", 
                         value = "50"),
        @HystrixProperty(name = "circuitBreaker.sleepWindowInMilliseconds", 
                         value = "10000"),
        // 超时时间
        @HystrixProperty(name = "execution.isolation.thread.timeoutInMilliseconds", 
                         value = "5000")
    }
)
public String callExternalService(String param) {
    return externalService.call(param);
}

public String fallback(String param) {
    return getDefaultValue();
}
```

### Sentinel vs Hystrix

| 特性 | Sentinel | Hystrix |
|------|----------|---------|
| 熔断策略 | 慢调用比例/异常比例/异常数 | 错误率/错误数 |
| 流控 | QPS/并发数线程数 | 线程池/信号量 |
| 实时监控 | Dashboard 实时监控 | 依赖 Turbine |
| 规则配置 | 动态推送 | 配置文件 |
| 社区活跃 | 阿里持续维护 | 已停止维护 |

**结论**：新项目用 Sentinel。

## 方案三：异步化

### 核心思想

将同步调用改为异步调用，不阻塞主线程。

### 使用 CompletableFuture

```java
// 同步调用（阻塞）
public OrderDTO getOrderDetail(long orderId) {
    OrderDTO order = orderService.getById(orderId); // 10ms
    UserDTO user = externalUserService.getById(order.getUserId()); // 200ms（外部服务）
    ProductDTO product = productService.getById(order.getProductId()); // 10ms
    return merge(order, user, product);
}
// 总耗时：10 + 200 + 10 = 220ms

// 异步调用（并行）
public OrderDTO getOrderDetailAsync(long orderId) {
    OrderDTO order = orderService.getById(orderId); // 10ms
    
    CompletableFuture<UserDTO> userFuture = CompletableFuture
        .supplyAsync(() -> externalUserService.getById(order.getUserId()));
    
    CompletableFuture<ProductDTO> productFuture = CompletableFuture
        .supplyAsync(() -> productService.getById(order.getProductId()));
    
    // 等待所有结果返回
    CompletableFuture.allOf(userFuture, productFuture).join();
    
    return merge(order, userFuture.join(), productFuture.join());
}
// 总耗时：10 + max(200, 10) = 210ms（外部服务耗时被并行掩盖）
```

### 使用 MQ 异步解耦

```
同步方式：订单服务 → 调用外部风控服务 → 创建订单
异步方式：订单服务 → 发送 MQ 消息 → 风控服务消费消息 → 回调通知
```

**适用场景**：不需要实时返回结果的操作。

## 方案四：缓存兜底

### 核心思想

对外部服务的返回结果进行缓存。当外部服务不可用时，使用缓存数据作为兜底。

```java
public UserDTO getUserWithCache(long userId) {
    String cacheKey = "external:user:" + userId;
    
    // 1. 先查缓存
    UserDTO cached = cache.get(cacheKey);
    if (cached != null) return cached;
    
    try {
        // 2. 调用外部服务
        UserDTO user = externalService.getUser(userId);
        // 3. 写入缓存（设置合理过期时间）
        cache.put(cacheKey, user, 5, TimeUnit.MINUTES);
        return user;
    } catch (Exception e) {
        // 4. 外部服务异常时，返回缓存的旧数据
        UserDTO staleData = cache.get(cacheKey);
        if (staleData != null) {
            log.warn("External service failed, returning stale data", e);
            return staleData;
        }
        // 5. 连缓存都没有，返回默认值
        return getDefaultUser();
    }
}
```

## 方案五：隔离

### 线程池隔离

为不同的外部服务分配独立的线程池，防止一个服务拖垮所有服务。

```java
// 外部服务 A 专用线程池
ThreadPoolExecutor serviceAPool = new ThreadPoolExecutor(
    10, 20, 60, TimeUnit.SECONDS, new LinkedBlockingQueue<>(100));

// 外部服务 B 专用线程池
ThreadPoolExecutor serviceBPool = new ThreadPoolExecutor(
    10, 20, 60, TimeUnit.SECONDS, new LinkedBlockingQueue<>(100));

// 调用外部服务 A
CompletableFuture.supplyAsync(() -> externalServiceA.call(), serviceAPool);

// 调用外部服务 B
CompletableFuture.supplyAsync(() -> externalServiceB.call(), serviceBPool);
```

### 信号量隔离

限制对外部服务的并发调用数。

```java
private final Semaphore semaphore = new Semaphore(10); // 最多 10 个并发

public String callExternal(String param) {
    if (!semaphore.tryAcquire(100, TimeUnit.MILLISECONDS)) {
        return getDefaultValue(); // 获取不到信号量，直接降级
    }
    try {
        return externalService.call(param);
    } finally {
        semaphore.release();
    }
}
```

## 综合防护方案

```
                    ┌─ 超时控制（第一道防线）
                    │
外部服务调用 ───────┼─ 熔断降级（第二道防线）
                    │
                    ├─ 异步化（减少阻塞）
                    │
                    ├─ 缓存兜底（第三道防线）
                    │
                    └─ 线程池隔离（防止扩散）
```

**完整的调用流程**：

```java
@SentinelResource(value = "externalCall", fallback = "fallback")
public Result callExternalWithProtection(Request request) {
    // 1. 信号量限流
    if (!semaphore.tryAcquire()) {
        return Result.fail("too many requests");
    }
    
    try {
        // 2. 从缓存获取
        Result cached = cache.get(buildCacheKey(request));
        if (cached != null) return cached;
        
        // 3. 异步调用 + 超时控制
        CompletableFuture<Result> future = CompletableFuture
            .supplyAsync(() -> externalService.call(request), pool)
            .orTimeout(5, TimeUnit.SECONDS); // 5秒超时
        
        Result result = future.join();
        
        // 4. 更新缓存
        cache.put(buildCacheKey(request), result, 5, TimeUnit.MINUTES);
        
        return result;
    } finally {
        semaphore.release();
    }
}

// 降级方法
public Result fallback(Request request, Throwable e) {
    // 尝试返回缓存数据
    Result cached = cache.get(buildCacheKey(request));
    if (cached != null) return cached;
    // 返回默认值
    return Result.success(getDefaultValue());
}
```

## 常见追问

### Q：如何判断外部服务是否可靠？

> 1. 看 SLA 承诺（99.9% 还是 99.99%）
> 2. 查看历史监控数据（错误率、P99 延迟）
> 3. 进行混沌工程测试（模拟外部服务不可用）
> 4. 询问对方的容灾方案（是否有主备切换）

### Q：熔断恢复后如何避免流量突增？

> 1. 使用半开状态逐步放量
> 2. 设置恢复速率（如每秒增加 10% 的流量）
> 3. 使用预热模式（Sentinel 支持 warm up）

### Q：外部服务的接口文档和实际不一致怎么办？

> 1. 所有外部调用都要做防御性编程（空指针检查、类型转换异常处理）
> 2. 使用契约测试（Contract Testing）验证接口规范
> 3. 与合作方建立接口变更通知机制

## 场景题补充：第三方接口超时处理

面试常问："第三方接口不稳定经常超时，如何保证不影响自己接口？"这道题考察系统容错能力和韧性设计。

**核心目标**：故障隔离（第三方异常不影响自身）、容错降级（失败也能提供基础服务）、性能保障（防止拖慢自身响应）。

**六大防护策略**：

1. **超时控制**：HTTP客户端（Feign/RestTemplate）设置connectTimeout和readTimeout；RPC框架（Dubbo/gRPC）配置超时参数。超时时间 = 第三方P99响应时间 x 1.5~2倍。

2. **重试机制**：指数退避重试（Exponential Backoff），仅对幂等接口重试（GET/查询），非幂等接口（支付/写操作）禁止重试。

3. **熔断降级**：Sentinel/Hystrix/Resilience4j实现。失败率超阈值自动熔断，返回默认值或缓存数据。

4. **异步化+消息队列**：非关键路径调用异步化，用CompletableFuture或MQ解耦。

5. **缓存策略**：本地缓存（Caffeine）+ 分布式缓存（Redis），减少对第三方直接依赖。第三方不可用时返回缓存兜底数据。

6. **限流与线程池隔离**：为不同第三方服务分配独立线程池，防止一个服务故障耗尽所有线程。

**推荐组合**：线程池隔离 + 熔断降级 + 缓存策略 + 监控告警。实测效果：核心接口成功率从75%提升至99.95%，第三方全量故障时缓存兜底保障80%请求可处理。
