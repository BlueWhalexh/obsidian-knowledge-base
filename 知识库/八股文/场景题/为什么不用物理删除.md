# 为什么不用物理删除

## 问题
为什么很多公司数据库不允许物理删除(delete)数据？

## 面试考察点
- 数据安全与恢复能力：误删数据的恢复机制
- 审计与合规意识：GDPR、SOX等法规要求
- 架构思维全面性：关联完整性、统计分析、灾备恢复
- 技术实现：逻辑删除替代方案的可行性

## 解题思路
> 物理删除不可逆，逻辑删除（is_deleted标记）保留数据可恢复性，满足审计合规和业务连续性需求

禁止物理删除主要基于六大考量：一是数据安全，物理删除不可逆，误操作导致关键数据永久丢失，逻辑删除可通过update恢复；二是审计合规，金融/医疗等行业法规要求保留数据访问和修改记录；三是关联完整性，物理删除主表数据会导致子表外键约束冲突；四是性能优化，频繁DELETE导致表空间碎片化，索引效率下降；五是历史数据价值，支持数据挖掘分析和业务复盘；六是灾备恢复，逻辑删除保留数据完整性便于迁移和回滚。

## 具体方案
### 方案一：逻辑删除标准实现
- **步骤**：添加is_deleted字段 -> 所有查询自动过滤（MyBatis拦截器） -> 定期归档历史数据
- 优点：数据可恢复、满足审计、保持关联完整性
- 缺点：表数据量持续增长，需定期归档
- 实现：ALTER TABLE users ADD is_deleted TINYINT DEFAULT 0

### 方案二：必须物理删除的例外
- 敏感数据脱敏后归档：INSERT INTO archive SELECT * FROM table WHERE is_deleted=1; DELETE FROM table WHERE is_deleted=1 AND created_at < '2020-01-01'
- 性能敏感的分区表历史数据清理

## 要点总结
- 物理删除不可逆，误删数据恢复成本极高
- 逻辑删除满足审计合规（GDPR要求保留5年以上、银行交易记录10年以上）
- 避免外键约束破坏，保持数据关联完整性
- 频繁DELETE导致表碎片化，逻辑删除+定期归档是更优方案
- 敏感数据脱敏后可物理删除，需建立数据生命周期管理机制
