# 用了主键索引反而查询很慢

## 问题
用了主键索引反而查询很慢，可能是什么原因？

## 面试考察点
- 索引原理理解：对 B+ 树索引结构和聚簇索引的深入掌握
- 问题诊断：EXPLAIN 分析、执行计划解读的系统化能力
- 调优经验：实际解决慢查询的实践经验
- 技术广度：覆盖索引、回表、ICP、虚拟列等优化手段

## 解题思路
> 使用索引不等于高效查询，关键看"扫描行数"而非"是否用了索引"

核心认知：EXPLAIN 显示使用了 PRIMARY 索引，不代表查询高效。InnoDB 是索引组织表，所有查询在逻辑上都用了索引。`select * from t where id > 0` 虽然 EXPLAIN 显示 PRIMARY，实际是全表扫描。真正有意义的是扫描行数，查询优化的过程就是减少扫描行数的过程。

慢查询的四大根因：全表扫描、全索引扫描、索引过滤性不好、频繁回表的开销。例如 14 亿人的表按 age BETWEEN 10 AND 15 查询，虽然有索引但满足条件有 1 亿行，扫描量依然巨大。回表问题在 `like '张%'` 场景更严重，8000 万次回表是性能杀手。

## 具体方案
### 六大可能原因及解决
- **回表开销过大**：用覆盖索引（只查索引列）或延迟关联（先子查询取 id 再 JOIN）
- **索引碎片严重**：碎片率 > 30% 需要 `OPTIMIZE TABLE` 或 `ALTER TABLE ENGINE=InnoDB`
- **隐式类型转换**：id 为 varchar 但用 int 查询，导致索引失效
- **长事务影响**：MVCC 导致需要回溯 undo log，检查 `innodb_trx` 中超过 60s 的事务
- **硬件瓶颈**：IO 等待过高，用 `iostat -x` 检查 %util 和 await
- **优化器误判**：`SET optimizer_trace="enabled=on"` 跟踪索引选择原因

### 关键优化手段
- **覆盖索引**：`SELECT id, name` 代替 `SELECT *`，避免回表
- **索引下推 (ICP)**：MySQL 5.6+ 在联合索引遍历中下推条件，减少回表次数（8000 万次降到 100 万次）
- **虚拟列**：MySQL 5.7+ 用 `GENERATED ALWAYS AS (left(name,1))` 创建紧凑索引
- **强制索引**：`FORCE INDEX(PRIMARY)` 绕过优化器误判

## 要点总结
- 使用索引和是否进入慢查询没有必然联系，核心看扫描行数
- 全索引扫描 `select a from t` 虽然用了索引但扫描整棵索引树，和全表扫描无本质区别
- 索引过滤性（区分度）要足够好，age 字段在 14 亿人中过滤性很差
- 回表是最大性能杀手，优先用覆盖索引或 ICP 优化
- UUID 主键导致页分裂和碎片，推荐用雪花 ID + 月分区表
