# 全链路压测平台设计

## 问题

如何设计一个生产环境的全链路压测平台？如何保证压测不影响线上用户？

## 面试考察点

- 压测流量构造与拷贝
- 数据隔离（影子库/影子表）
- 流量染色与标记
- 压测实施策略（逐步加压）

## 解题思路

全链路压测核心挑战是**流量隔离**和**风险控制**。

## 具体方案

### 1. 压测数据产生
- **流量拷贝**：使用GoReplay等工具拷贝线上流量
- 存储到**流量数据工厂**（HBase/MongoDB/S3）
- 流量清洗：过滤无效请求
- 压测时从工厂获取数据，切分多份下发到多个压测节点

### 2. 流量染色
- HTTP请求头增加 `is_stress_test` 标记
- 染色标记在整条链路中传递（ThreadLocal + Filter）

### 3. 数据隔离

**下行流量（读请求）**
- 不能压测的服务做Mock处理
- 用户行为日志不记录（避免污染业务报表）
- 推荐服务做Mock（避免压测流量消耗推荐商品）

**上行流量（写请求）**
- **MySQL影子库**：同一实例不同Schema，表结构和数据相同
- **Redis影子key**：增加统一前缀
- **ES影子索引**：单独索引表

### 4. 压测实施策略
1. 设定目标QPS（如20万/秒）
2. **逐步加压**：每次增加1万QPS
3. 每次加压后稳定运行一段时间，观察性能
4. 性能达到阈值时减少流量，排查瓶颈
5. **流量监控组件**：预设CPU 60-70%、RT < 1s、慢请求 < 1%
6. 超过阈值自动减少流量并报警

### 5. 压测平台核心模块
- 流量构造和产生模块
- 压测数据隔离模块
- 系统健康度检查和流量干预模块

## 要点总结

1. GoReplay拷贝线上流量，比模拟流量更真实
2. 流量染色是隔离的基础，全链路传递标记
3. 影子库/影子key/影子索引实现读写完全隔离
4. 逐步加压 + 自动监控熔断，保障压测安全
5. 压测应周期性执行，系统迭代可能引入新的瓶颈
