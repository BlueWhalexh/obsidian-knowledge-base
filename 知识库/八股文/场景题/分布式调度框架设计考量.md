# 分布式调度框架设计考量

## 问题

如何设计分布式调度框架？如何保证分布式事务一致性和任务可靠性？

## 面试考察点

- 本地消息表保证分布式事务一致性
- ElasticJob流式任务（DataflowJob接口）
- 分片机制
- 重试与防阻塞策略

## 解题思路

分布式调度框架核心挑战是**任务可靠性**和**分布式事务一致性**。

## 具体方案

### 1. 本地消息表 + 定时任务
- 业务操作和消息写入在**同一本地事务**中
- 定时任务扫描本地消息表，发送未确认的消息
- 保证**分布式事务一致性**（最终一致性）
- 消费成功后标记消息为已消费

### 2. ElasticJob流式任务
- **DataflowJob接口**：支持流式处理
- `fetchData()`：从数据源拉取一批数据
- `processData()`：处理数据
- 循环执行直到无数据可拉取
- 适合大数据量的批处理场景

### 3. 分片机制
- 将任务拆分为多个分片
- 多台机器各领一个或多个分片并行处理
- 分片数可配置，灵活调整并行度
- 机器上下线时自动重分配分片

### 4. 重试与防阻塞
- 任务失败后自动重试，支持配置重试次数
- **next_select_time**：设置下次执行时间，避免失败后立即重试阻塞
- 重试间隔递增（指数退避）

## 要点总结

1. 本地消息表是保证分布式事务一致性的经典方案
2. ElasticJob DataflowJob支持流式处理，适合大批量数据
3. 分片机制实现任务并行，灵活调整并行度
4. 重试+指数退避避免频繁重试打垮下游系统
