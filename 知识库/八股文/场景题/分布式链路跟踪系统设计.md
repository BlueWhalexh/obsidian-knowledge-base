# 分布式链路跟踪系统设计

## 问题

如何设计分布式链路跟踪系统？如何在微服务架构下追踪请求链路？

## 面试考察点

- OpenTelemetry标准（Metrics+Tracing+Logging融合）
- TraceID + RPCID/SpanID链路标识
- ELK + 链路跟踪定制化方案
- 日志格式标准与SDK埋点

## 解题思路

分布式链路跟踪核心是**生成并传递TraceID**，将同一请求的所有日志串联起来。

## 具体方案

### 1. OpenTelemetry标准
- **Metrics**：监控指标（QPS、RT、错误率）
- **Tracing**：链路追踪（请求在各服务间的调用路径）
- **Logging**：日志记录
- 三者融合，统一数据格式

### 2. TraceID单次请求标识
- 请求入口生成全局唯一UUID作为TraceID
- 所有依赖资源请求（MySQL/Redis/RPC）都携带TraceID
- 收集所有相同TraceID的日志，按时间/RPCID串联

### 3. RPCID vs SpanID

**RPCID（层级计数器）**
- 格式：1.1 -> 1.2 -> 1.2.1 -> 1.2.2
- 前缀标识层级，计数器标识同级顺序
- 优点：可记录请求方日志分析被调用方；日志丢失仍可大致定位层级

**SpanID（链式记录）**
- 每个span记录自己的ParentSpanID
- 通过ParentID关联上下游关系
- 优点：结构清晰，业界标准

### 4. ELK + 定制化方案
- **Filebeat**：日志收集
- **Kafka + Logstash**：日志传输
- **Elasticsearch**：日志存储+检索
- **Kibana**：可视化分析
- 自定义日志格式标准 + SDK埋点即可实现完整链路跟踪

### 5. 日志格式
- JSON格式，字段类型固定
- 必填：name, trace_id, rpc_id, timestamp, duration, module
- 选填：uid, server_ip, action, param, response

## 要点总结

1. TraceID是链路跟踪的核心，请求入口生成，全链路传递
2. RPCID层级计数器和SpanID链式记录是两种串联方式
3. ELK提供完整的日志收集/传输/存储/分析能力
4. 自定义日志格式+SDK埋点即可实现定制化链路跟踪
5. 核心系统记录全量日志，周边业务可采样
