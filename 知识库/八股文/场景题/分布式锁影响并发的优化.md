# 分布式锁影响并发的优化

## 问题
加分布式锁之后影响并发了怎么办？

## 面试考察点
- 锁竞争根本原因：锁粒度过大、热点数据集中、锁持有时间过长
- 性能瓶颈量化：QPS下降幅度、延迟增加、锁等待队列
- 优化策略选型：锁粒度细化、无锁化设计、异步化补偿
- 故障容错：锁续期失败、网络分区下锁误释放、雪崩效应

## 解题思路
> 分布式锁确实降低并发性能，但通过精细控制锁粒度、缩短持有时间、选择合适锁类型可有效缓解

锁竞争导致并发下降的根本原因有三：全局锁导致所有请求串行化、热点数据高频访问冲突、事务处理时间超出合理范围。优化遵循四条原则：能不用锁尽量不用、能用乐观锁不用悲观锁、能用细粒度锁不用全局锁、能缩短时间绝不长持有。

## 具体方案
### 方案一：锁粒度细化 + 分段锁
- **原理**：将全局锁拆分为多个分段锁，减少锁冲突
- **步骤**：按商品ID hash分段（如10段），每个段独立加锁
- 优点：并发能力提升5-10倍
- 缺点：实现复杂度上升，数据聚合成本增加

### 方案二：缩短锁持有时间
- **原理**：仅锁住核心计算逻辑，前置查询和后置操作放在锁外
- **步骤**：前置查询 -> lock.lock() -> 核心计算 -> lock.unlock() -> 后置操作
- 原则：锁内代码行数不超过10行

### 方案三：无锁化设计（乐观锁兜底）
- **原理**：库存分段 + 本地缓存前置校验 + 数据库乐观锁最终兜底
- 电商大促案例：库存拆分为100逻辑段，峰值QPS从500提升到15000

## 要点总结
- 锁粒度细化是最直接有效的优化手段（分段锁提升5-10倍）
- 缩短锁持有时间：只锁核心逻辑，非关键操作移到锁外
- 读多写少场景用读写锁分离，写多场景考虑乐观锁
- 架构级优化：本地锁降级（先本地锁再分布式锁）、异步化处理
- 监控锁等待时间，超过阈值自动降级为乐观锁模式
