# 分库分表

## 一、什么是分库分表

**分库**：将一个数据库拆分为多个数据库，每个数据库部署在不同的服务器上。

**分表**：将一张大表拆分为多张小表，可以是同一个数据库内的多张表（分表），也可以分布在不同数据库中（分库分表）。

**垂直拆分**：按业务模块拆分（如用户库、订单库、商品库）。

**水平拆分**：按某个字段（分片键）将数据分散到多张结构相同的表中（如 order_0, order_1, order_2）。

---

## 二、什么时候需要分库分表

### 2.1 分表时机

| 指标 | 阈值 | 说明 |
|------|------|------|
| 单表行数 | > 500 万行 | 查询性能开始明显下降 |
| 单表数据量 | > 2GB | B+ 树层数增加，IO 次数增多 |
| 单表索引大小 | > 内存大小 | 索引无法全部放入内存 |

### 2.2 分库时机

| 指标 | 阈值 | 说明 |
|------|------|------|
| 单库连接数 | > 最大连接数 | 应用无法获取连接 |
| 单库 QPS | > 5000 | 数据库 CPU 打满 |
| 单库数据量 | > 1000 万行 | 多表联合查询性能差 |

### 2.3 如何预估分库分表数量

```
分表数量 = 预估总数据量 ÷ 单表推荐数据量（500万）
分库数量 = 分表数量 ÷ 单库推荐表数量（200~500）
```

**示例**：

```
预估 3 年订单量：5 亿
分表数量 = 500,000,000 ÷ 5,000,000 = 100 张表
单库存 50 张表 → 需要 2 个库
建议取 2 的幂次 → 分 128 张表、4 个库（每个库 32 张表）
```

## 三、分片键选择

分片键（Sharding Key）是分库分表的核心，决定了数据如何分布。

### 3.1 选择原则

1. **高频查询字段**：绝大多数查询都带有该字段作为查询条件
2. **数据分布均匀**：该字段的值足够分散，不会导致数据倾斜
3. **不可变**：分片键的值不应该被修改

### 3.2 常见分片键示例

| 业务 | 分片键 | 理由 |
|------|--------|------|
| 订单表 | 用户 ID | 大部分查询是"查我的订单" |
| 用户表 | 用户 ID | 天然唯一且均匀 |
| 消息表 | 会话 ID | 查询基本按会话维度 |
| 日志表 | 时间 | 按天/月分表 |

### 3.3 分片键选择错误的后果

- 数据倾斜：某些分片数据量远大于其他分片
- 跨分片查询：大量请求需要扫描所有分片
- 热点问题：某个分片成为性能瓶颈

## 四、分片算法

### 4.1 范围分片（Range Sharding）

按分片键的范围划分。

```
order_id < 1000000      → order_0
1000000 ≤ order_id < 2000000 → order_1
2000000 ≤ order_id < 3000000 → order_2
...
```

**优点**：范围查询友好，扩容方便（新增分片即可）。

**缺点**：数据分布不均匀，新数据集中在最新分片（热点）。

### 4.2 哈希分片（Hash Sharding）

对分片键取哈希值，再对分片数取模。

```java
int shardIndex = userId.hashCode() % 8; // 0~7
String tableName = "order_" + shardIndex;
```

**优点**：数据分布均匀。

**缺点**：
1. 扩容时需要迁移大量数据（取模基数变化）
2. 范围查询需要扫描所有分片

### 4.3 一致性哈希分片

**核心思想**：将哈希值空间组织成一个虚拟的环（0 ~ 2^32-1），节点和数据都映射到环上。

```
步骤：
1. 将 0 ~ 2^32-1 的哈希空间组成一个环
2. 将每个节点（数据库/表）通过哈希函数映射到环上
3. 将每条数据通过哈希函数映射到环上
4. 数据沿顺时针方向找到的第一个节点，就是该数据所属的节点
```

**虚拟节点**：

为了解决数据倾斜问题，每个物理节点对应多个虚拟节点。

```
物理节点 A → 虚拟节点 A1, A2, A3, ..., A150
物理节点 B → 虚拟节点 B1, B2, B3, ..., B150
物理节点 C → 虚拟节点 C1, C2, C3, ..., C150
```

虚拟节点数量越多，数据分布越均匀。通常每个物理节点 100~200 个虚拟节点。

**代码实现**：

```java
public class ConsistentHash {
    private final TreeMap<Long, String> ring = new TreeMap<>();
    private final int virtualNodeCount = 150;
    
    public void addNode(String node) {
        for (int i = 0; i < virtualNodeCount; i++) {
            long hash = hash(node + "#" + i);
            ring.put(hash, node);
        }
    }
    
    public void removeNode(String node) {
        for (int i = 0; i < virtualNodeCount; i++) {
            long hash = hash(node + "#" + i);
            ring.remove(hash);
        }
    }
    
    public String getNode(String key) {
        long hash = hash(key);
        // 找到第一个 >= hash 的节点
        Map.Entry<Long, String> entry = ring.ceilingEntry(hash);
        if (entry == null) {
            entry = ring.firstEntry(); // 环形：回到起点
        }
        return entry.getValue();
    }
    
    private long hash(String key) {
        // 使用 MurmurHash 或 FNV 等哈希函数
        return Hashing.murmur3_128().hashUnencodedChars(key).asLong();
    }
}
```

**优点**：

1. 新增/删除节点只影响相邻节点的数据，迁移量最小
2. 数据分布均匀（虚拟节点足够多时）

**缺点**：

1. 实现复杂
2. 范围查询仍然需要扫描多个分片

### 4.4 算法对比

| 算法 | 数据均匀性 | 扩容成本 | 范围查询 | 实现复杂度 |
|------|-----------|---------|---------|-----------|
| 范围分片 | 差 | 低 | 好 | 低 |
| 哈希分片 | 好 | 高（全量迁移） | 差 | 低 |
| 一致性哈希 | 好 | 低（最小迁移） | 差 | 高 |

## 五、非分片键查询怎么办

### 5.1 问题

分表后，只有带分片键的查询才能快速定位到具体分片。非分片键查询需要扫描所有分片（全表扫描）。

**示例**：订单表按 `user_id` 分片
- `SELECT * FROM order WHERE user_id = 123` → 快速定位
- `SELECT * FROM order WHERE order_id = 456` → 需要扫描所有分片

### 5.2 方案一：基因法

**核心思想**：在分片键中嵌入其他查询字段的信息。

**实现**：将 user_id 的低位（如后 4 位）嵌入到 order_id 中。

```
生成 order_id 时：
order_id = snowflake_id | (user_id % 16)

查询时：
SELECT * FROM order 
WHERE order_id = 12345678 
AND user_id IN (
    -- 取 order_id 低 4 位，反推出可能的 user_id 范围
    SELECT user_id FROM user WHERE user_id % 16 = 12345678 % 16
)
```

**简化理解**：order_id 的最后几位就是 user_id 的分片信息，用 order_id 查询时可以直接定位到分片。

### 5.3 方案二：映射表

维护一张映射表，记录 order_id → user_id 的关系。

```sql
CREATE TABLE order_user_mapping (
    order_id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    INDEX idx_user_id (user_id)
);
```

查询流程：
1. 先查映射表获取 user_id
2. 用 user_id 定位分片查询订单

**缺点**：映射表本身也会成为瓶颈，可以放在 Redis 中加速。

### 5.4 方案三：搜索引擎

使用 Elasticsearch 建立索引，支持任意字段查询。

```
同步流程：
MySQL Binlog → Canal → Kafka → Elasticsearch

查询流程：
1. 复杂查询 → ES 查出主键列表
2. 用主键列表 → 回查 MySQL 获取完整数据
```

### 5.5 方案对比

| 方案 | 实时性 | 复杂度 | 适用场景 |
|------|--------|--------|---------|
| 基因法 | 实时 | 中 | ID 可控生成 |
| 映射表 | 实时 | 低 | 查询字段固定 |
| ES 索引 | 准实时 | 高 | 复杂查询场景 |

## 六、分库分表带来的问题

### 6.1 分布式事务

单库中用 `@Transactional` 就能保证事务。分库后，一个业务操作可能涉及多个数据库，需要分布式事务。

**方案**：

| 方案 | 一致性 | 性能 | 复杂度 |
|------|--------|------|--------|
| 2PC（XA） | 强一致 | 低 | 中 |
| TCC | 最终一致 | 高 | 高 |
| Saga | 最终一致 | 高 | 中 |
| 本地消息表 | 最终一致 | 高 | 低（推荐） |
| Seata AT | 最终一致 | 中 | 低 |

### 6.2 跨库 JOIN

分库后无法直接 JOIN 不同库的表。

**解决方案**：
1. **冗余字段**：将需要关联的字段冗余到主表
2. **应用层组装**：先查 A 表，再用结果查 B 表，应用层合并
3. **宽表**：将多张表合并为一张宽表，同步更新
4. **数据同步到 ES**：通过搜索引擎关联查询

### 6.3 全局唯一 ID

分库分表后，自增 ID 无法保证全局唯一。

**方案**：参考 [[分布式 ID 生成方案]]
- Snowflake 算法（推荐）
- Leaf（美团）
- UUID（不推荐，无序）

### 6.4 数据迁移

从单库单表迁移到分库分表是一个复杂的过程。

**迁移步骤**：

```
1. 双写阶段：新数据同时写入旧库和新库
2. 数据校验：对比新旧库数据一致性
3. 历史数据迁移：将历史数据迁移到新库
4. 读切新库：读流量切到新库
5. 停写旧库：停止写入旧库
6. 下线旧库
```

### 6.5 跨分片分页

`SELECT * FROM order ORDER BY create_time DESC LIMIT 100000, 10` 在分库分表后无法直接执行。

**方案**：
1. **禁止深分页**：只允许查看前 N 页
2. **二次查询法**：每个分片查 `LIMIT 100000, 10`，合并后取 Top 10
3. **游标分页**：使用 `WHERE id < lastId LIMIT 10` 替代 OFFSET

## 七、常用分库分表中间件

| 中间件 | 类型 | 特点 |
|--------|------|------|
| ShardingSphere-JDBC | 客户端分片 | 无需代理，性能好（推荐） |
| ShardingSphere-Proxy | 代理分片 | 对应用透明，有网络开销 |
| MyCat | 代理分片 | 社区活跃度下降 |
| TDDL | 客户端分片 | 阿里内部使用 |

### ShardingSphere 配置示例

```yaml
spring:
  shardingsphere:
    datasource:
      names: ds0, ds1
      ds0:
        url: jdbc:mysql://host1:3306/db0
      ds1:
        url: jdbc:mysql://host2:3306/db1
    rules:
      sharding:
        tables:
          order:
            actual-data-nodes: ds$->{0..1}.order_$->{0..31}
            table-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: order-inline
            key-generate-strategy:
              column: order_id
              key-generator-name: snowflake
        sharding-algorithms:
          order-inline:
            type: INLINE
            props:
              algorithm-expression: order_$->{user_id % 32}
```

## 常见追问

### Q：分库分表后如何做数据统计（如报表）？

> 1. 将数据同步到数据仓库（ClickHouse/Doris），在数仓中做统计
> 2. 使用 Canal 监听 Binlog，实时同步到 ES/数仓
> 3. 避免直接在分库分表上做聚合查询

### Q：分库分表键选错了怎么办？

> 这是一个非常严重的问题，通常需要重新迁移数据。所以分片键的选择一定要慎重，考虑 3~5 年的数据增长。如果真的选错了，只能做数据迁移（参考上面的迁移步骤）。

### Q：不分库分表，还有什么其他方案？

> 1. 读写分离（解决读压力）
> 2. 归档历史数据（将老数据迁移到历史表）
> 3. 冷热分离（热数据在 MySQL，冷数据在 TiDB/ClickHouse）
> 4. 使用 NewSQL 数据库（TiDB/CockroachDB），自动分片
