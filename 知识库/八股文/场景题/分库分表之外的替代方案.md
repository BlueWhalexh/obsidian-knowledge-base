# 分库分表之外的替代方案

## 问题
单表数据量大，只能考虑分库分表吗？有哪些替代方案？

## 面试考察点
- 技术广度：对多种大数据量解决方案的了解
- 架构能力：根据场景选择合适方案的决策力
- 成本意识：方案投入产出比的评估
- 实践经验：是否经历过真实的大数据量优化

## 解题思路
> 分库分表是"核武器"，能不用就不用——它带来跨库事务、分布式ID、查询聚合、运维复杂度等一系列问题

在考虑分库分表之前，应先评估7个"轻量级"方案：索引优化、冷热分离、分区表、读写分离、垂直拆分、硬件升级、NoSQL辅助。大多数场景下，这些方案的组合就能解决问题。

分库分表的真正触发条件：单表超5000万行且持续增长、硬件已到顶配仍卡顿、写入QPS超单机极限。不到这个体量硬上分库分表，等于"小学生穿西装"。

## 具体方案
### 方案一：索引优化 + 硬件升级（首选）
- 用EXPLAIN分析SQL，消除全表扫描和临时表
- 联合索引遵循最左匹配原则
- innodb_buffer_pool_size调到内存70%，换SSD性能翻10倍

### 方案二：冷热数据分离
- 近期数据（热）放主表，历史数据（冷）归档到archive表
- 主表瘦身，查询速度提升；删旧数据用 `TRUNCATE PARTITION` 比DELETE快10倍

### 方案三：分区表
- MySQL按时间/ID范围分区，应用无感知
- 限制：所有分区须同引擎，主键须包含分区键，最大8192分区

### 方案四：读写分离
- 主库写，多个从库轮询读，用ShardingSphere自动分流
- 注意从库延迟，关键操作查主库

### 方案五：垂直拆分 + NoSQL辅助
- 大字段拆到扩展表，主表只留核心字段
- 搜索交给ES，缓存用Redis，日志存MongoDB

## 要点总结
- 分库分表前先试索引优化、冷热分离、分区表、读写分离，能解决80%问题
- 分库分表的代价：跨库事务、分布式ID、查询聚合、运维复杂度飙升
- 决策路径：数据量>1亿 → 有分区键? → 分区表；读多写少? → 读写分离；仍不够? → 冷热分离；仍不够 → 分库分表
- 小公司单库 + 索引优化足够撑到B轮；设计时预留sharding_key字段方便日后扩展
- 终极原则：能用钱解决别玩命，能用简单方案别堆复杂度
