# 分库分表时每个城市人口不一样如何均匀分布

## 问题
分库分表时，每个城市的人口不一样，有的密集，有的稀疏，如何实现均匀分布？

## 面试考察点
- 分片策略的灵活性：如何应对数据分布不均匀（如一线城市数据量是县城的100倍）
- 负载均衡算法：一致性哈希与虚拟节点的应用、动态权重分配
- 动态扩容能力：数据迁移成本控制、分片再平衡机制
- 典型反模式识别：直接按城市分片导致热点问题、固定哈希分片无法适应数据增长

## 解题思路
> 核心问题：直接按城市ID作为分片键会导致数据倾斜，大城市单分片过载，小城市资源浪费

分片策略设计需遵循四个原则：均匀分布（避免单一分片过载）、查询效率（减少跨分片查询）、可扩展性（支持动态调整分片数量）、低维护成本（策略简单易管理）。

关键在于选择合适的分片键和分片算法。不能简单按城市分片，需要引入复合分片键、虚拟节点、权重分配等机制来打散数据，同时兼顾按城市查询的业务需求。

## 具体方案
### 方案一：复合分片键（City + User ID）
- 使用city_id和user_id组合作为分片键
- 哈希计算：hash(city_id, user_id) % shard_count
- 同一城市用户分散到多个分片，缓解热点
- 查询时可通过city_id定位分片，减少跨分片查询

### 方案二：一致性哈希 + 虚拟节点
- 为每个物理分片分配多个虚拟节点（10~20个）
- 数据键哈希后分配到虚拟节点，再映射到物理分片
- 增减分片时仅影响邻近节点，减少数据迁移量
- 结合虚拟节点后分布更均匀，支持动态扩展

### 方案三：按权重分片
- 根据城市人口设置权重，权重高的城市分配更多分片
- 例如：北京（权重4）-> shard0~3，上海（权重3）-> shard4~6
- 适合数据分布已知且相对稳定的场景
- 需人工维护权重，动态调整困难

## 要点总结
- 核心原则：不能直接按城市分片，需用复合分片键或虚拟节点打散数据
- 推荐方案：虚拟节点 + 动态权重组合，兼顾均匀性和查询效率
- 生产效果：分片数据量差异可从500倍降至2倍以内，热点QPS从10万降至2万
- 工具支持：Apache ShardingSphere、Vitess等中间件简化分片管理
