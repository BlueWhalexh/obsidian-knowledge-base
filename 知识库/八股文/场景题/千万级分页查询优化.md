# 千万级分页查询优化

## 问题
MySQL单表一千万条数据怎么做分页查询？

## 面试考察点
- 传统LIMIT offset, size的性能瓶颈：offset越大扫描行数越多
- 覆盖索引与复合索引的使用
- 游标分页（Cursor-based Pagination）与延迟关联（Deferred Join）
- 深度分页导致全表扫描的反模式识别

## 解题思路
> 核心问题：LIMIT 1000000, 10需要扫描100万+10行后丢弃前100万行，随offset增大查询时间线性增长。

传统分页的问题在于offset越大MySQL需要扫描丢弃的数据越多。实测一千万数据表中，LIMIT 100,100耗时0.014s，而LIMIT 9000000,100耗时12.591s。优化方向是避免大offset扫描，利用索引定位数据。

可选方案包括：游标分页（利用上一页最后一条记录的ID定位）、延迟关联（先用覆盖索引查ID再JOIN回表）、分区表（按时间范围隔离数据）。需根据业务是否需要随机跳页来选择策略。

## 具体方案
### 方案一：游标分页（Cursor-based Pagination）
- 原理：利用上一页最后一条记录的ID/时间戳作为起点，避免offset扫描
- 步骤：`SELECT * FROM table WHERE id > last_id ORDER BY id LIMIT 100`
- 优点：查询时间恒定，不随页码增大而退化
- 缺点：不支持随机跳页，只能上下翻页

### 方案二：延迟关联（Deferred Join）
- 原理：先用覆盖索引查出ID，再通过ID回表取完整数据
- 步骤：`SELECT * FROM table a JOIN (SELECT id FROM table LIMIT 9000000, 100) b ON a.id = b.id`
- 优点：覆盖索引扫描速度快，实测从12.5s降至3.4s
- 缺点：需要主键或唯一索引支持

### 方案三：控制每页数据量+覆盖索引
- 原理：减少每页返回条数，配合覆盖索引降低IO
- 优点：实现简单，适合浅层分页场景
- 缺点：无法根本解决深度分页问题

## 要点总结
- LIMIT语句查询时间与起始记录位置成正比，千万级表不适合直接使用大offset
- 游标分页是深度分页的首选方案，但需要业务接受只能顺序翻页
- 延迟关联通过覆盖索引+JOIN的方式减少回表次数，显著提升性能
- 查询时使用覆盖索引，非必要字段不返回，有利于查询速度提升
- 分页时应控制每页数据量，业务允许时优先使用游标分页
