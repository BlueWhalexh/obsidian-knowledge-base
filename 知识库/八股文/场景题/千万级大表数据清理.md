# 千万级大表数据清理

## 问题
MySQL千万级大表如何做数据清理？

## 面试考察点
- 大数据量处理能力：对海量数据操作的实践经验
- 数据库原理理解：对MySQL存储机制的深入掌握
- 业务影响控制：最小化清理操作对生产的影响
- 系统化思维：全链路考虑数据生命周期

## 解题思路
> 核心思路：千万级数据不能直接DELETE，必须分批/分区/重建，控制对线上影响

直接DELETE千万级数据会导致长时间锁表，影响线上服务。核心原则是：小批量操作、控制节奏、可中断恢复。方案选择取决于删除条件和数据量：条件复杂用分批删除，按时间分区用分区裁剪，删除大量数据用表重建。

清理后的优化同样重要：重建索引（ALTER TABLE ENGINE=InnoDB）、更新统计信息（ANALYZE TABLE）、整理碎片（OPTIMIZE TABLE）。

## 具体方案
### 方案一：分批删除（最常用）
- 原理：每次DELETE LIMIT 1000，循环执行直到无数据可删
- 优化参数：innodb_flush_neighbors=0（禁用相邻页刷新）、关闭自适应哈希
- 控制节奏：每次删除后sleep 1秒，业务低峰期执行
- 中断恢复：记录断点位置，支持断点续删
- 工具推荐：pt-archiver控制节奏

### 方案二：分区表快速清理
- 原理：按时间/范围分区，直接DROP PARTITION（秒级完成）
- 实施：ALTER TABLE DROP PARTITION p_2022
- 优点：瞬时完成，对业务无影响
- 缺点：需提前规划分区策略

### 方案三：表重建
- 原理：创建新表 -> 迁移保留数据 -> 原子切换（RENAME TABLE）
- 适用：删除大量数据（如删除80%以上）
- 缺点：需要停机窗口

## 要点总结
- 核心原则：小批量、控节奏、可中断、低峰期
- 分批删除每次1000条，配合sleep控制对线上影响
- 分区表DROP PARTITION是最快方案，但需要提前规划
- 清理后三步走：重建索引 -> 更新统计信息 -> 整理碎片
- 实战效果：8000万记录分批删除耗时3.6小时，系统负载增加不超过15%
