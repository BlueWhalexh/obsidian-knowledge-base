# 千万级数据查询优化

## 问题
MySQL千万级数据量，查询如何做优化？

## 面试考察点
- 全表扫描vs索引扫描的识别（type: ALL vs ref/range）
- 覆盖索引解决回表问题
- 小表驱动大表的优化策略
- 千万级数据IN查询索引失效时的强制索引技巧

## 解题思路
> 核心问题：单表千万级数据在表设计差、不允许拆分子查询的情况下，如何通过SQL优化提升查询性能。

千万级数据对MySQL来说本身不合理，应优先考虑分表分库、创建汇总表。但在不允许拆分的极端情况下，需要通过索引优化来解决问题。优化顺序为：普通索引 -> 覆盖索引 -> 减少数据量 -> 小表驱动大表 -> 强制索引。

实测1500万orders表JOIN 1000个users的统计查询：无索引191s，普通索引反而变慢到460s（回表开销），覆盖索引降至10s，减少数据量降至7s。普通索引变慢的原因是命中索引后仍需通过主键回表获取数据。

## 具体方案
### 方案一：覆盖索引
- 原理：索引包含查询所需所有列，无需回表访问数据行
- 步骤：创建`idx_orders_user_id_total_amount (user_id, total_amount)`复合索引
- 优点：避免回表操作，实测从460s降至10s
- 索引列顺序需遵循MySQL执行顺序：FROM -> ON -> JOIN -> WHERE -> GROUP BY -> SELECT

### 方案二：减少数据量
- 原理：通过WHERE条件提前过滤数据，配合分区表
- 步例：`WHERE user_id > 1033`，配合range type访问
- 优点：查询时间从10s降至7s

### 方案三：强制索引
- 原理：千万级数据IN查询时优化器可能选择错误索引，需FORCE INDEX指定
- 适用：explain显示命中索引但查询仍然很慢的场景

## 要点总结
- 普通索引添加后查询可能更慢，原因是回表机制的开销，需使用覆盖索引
- 覆盖索引的列顺序很重要，需遵循MySQL关键字执行顺序（FROM->ON->WHERE->GROUP BY->SELECT）
- 千万级数据接近时应考虑分表（按用户ID取模），用汇总表代替子查询
- EXPLAIN结果type排序：system > const > eq_ref > ref > range > index > all
- 关联字段冗余到同一张表可避免JOIN，直接单表查询
