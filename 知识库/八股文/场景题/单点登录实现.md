# 单点登录实现

## 问题

如何实现用户在 A 网站登录后，访问 B 网站时自动登录（Single Sign-On, SSO）？

## 一、为什么需要单点登录

一个公司可能有多个子系统：
- 主站（www.example.com）
- 后台管理（admin.example.com）
- 商城（shop.example.com）

用户在主站登录后，希望访问商城时不需要再次登录。

**核心难点**：HTTP 是无状态协议，每个站点的 Cookie 是独立的，A 站的登录状态无法直接传递给 B 站。

---

## 二、方案一：共享 Cookie

### 实现原理

利用 Cookie 的 Domain 属性，让多个子域名共享同一个 Cookie。

```java
// 登录成功后设置 Cookie
Cookie cookie = new Cookie("token", jwtToken);
cookie.setDomain(".example.com"); // 关键：设置父域名
cookie.setPath("/");
cookie.setHttpOnly(true);
response.addCookie(cookie);
```

**效果**：用户在 `www.example.com` 登录后，Cookie 会被发送到 `*.example.com` 的所有子域名。

### 限制

1. **必须同域**：只能用于子域名（如 `*.example.com`），不能跨主域名
2. **安全风险**：一个子站被攻破，所有子站的登录状态都暴露
3. **隐私法规**：某些浏览器限制第三方 Cookie（Safari/Chrome）

### 适用场景

同一公司的多个子域名系统，简单快速的方案。

---

## 三、方案二：CAS 协议（Central Authentication Service）

### 核心思想

设立一个中央认证服务器（CAS Server），所有应用系统（CAS Client）的登录都通过它来完成。

### 流程详解

```
用户访问系统 A（未登录）
    │
    ▼
系统 A 发现未登录，重定向到 CAS Server
URL: https://cas.example.com/login?service=https://a.example.com/callback
    │
    ▼
CAS Server 显示登录页面，用户输入账号密码
    │
    ▼
CAS Server 验证通过，生成 TGT（Ticket Granting Ticket）
设置 CAS Server 的 Cookie（TGT）
生成 ST（Service Ticket），重定向回系统 A
URL: https://a.example.com/callback?ST=ST-xxxxx
    │
    ▼
系统 A 收到 ST，向 CAS Server 验证 ST
CAS Server 验证 ST 有效，返回用户信息
系统 A 建立本地会话，设置自己的 Cookie
    │
    ▼
用户访问系统 B（未登录）
    │
    ▼
系统 B 重定向到 CAS Server
CAS Server 发现已有 TGT Cookie，不需要再次登录
直接生成新的 ST，重定向回系统 B
    │
    ▼
系统 B 验证 ST，建立本地会话
用户无需输入密码，自动登录
```

### 核心代码

```java
// CAS Server 端
@Controller
public class CasServerController {
    
    @GetMapping("/login")
    public String login(@RequestParam String service, 
                        HttpServletRequest request,
                        HttpServletResponse response) {
        // 检查是否已有 TGT
        String tgt = getTgtFromCookie(request);
        if (tgt != null) {
            // 已登录，直接生成 ST 并重定向
            String st = generateServiceTicket(tgt, service);
            return "redirect:" + service + "?ST=" + st;
        }
        // 未登录，显示登录页面
        return "login";
    }
    
    @PostMapping("/login")
    public String doLogin(@RequestParam String username,
                          @RequestParam String password,
                          @RequestParam String service,
                          HttpServletResponse response) {
        // 验证用户名密码
        if (authenticate(username, password)) {
            // 生成 TGT
            String tgt = generateTgt(username);
            // 设置 TGT Cookie
            setTgtCookie(response, tgt);
            // 生成 ST
            String st = generateServiceTicket(tgt, service);
            return "redirect:" + service + "?ST=" + st;
        }
        return "login?error=true";
    }
    
    @GetMapping("/validate")
    @ResponseBody
    public Map<String, Object> validate(@RequestParam String st) {
        // 验证 ST 并返回用户信息
        String username = validateServiceTicket(st);
        Map<String, Object> result = new HashMap<>();
        result.put("valid", username != null);
        result.put("username", username);
        return result;
    }
}

// CAS Client 端（系统 A/B）
@Component
public class CasClientFilter implements Filter {
    
    private String casServerUrl = "https://cas.example.com";
    private String localUrl = "https://a.example.com";
    
    @Override
    public void doFilter(ServletRequest req, ServletResponse res, 
                         FilterChain chain) throws IOException, ServletException {
        HttpServletRequest request = (HttpServletRequest) req;
        HttpServletResponse response = (HttpServletResponse) res;
        
        // 检查本地会话
        if (isLoggedIn(request)) {
            chain.doFilter(req, res);
            return;
        }
        
        // 检查是否有 ST 参数
        String st = request.getParameter("ST");
        if (st != null) {
            // 向 CAS Server 验证 ST
            String username = validateWithCasServer(st);
            if (username != null) {
                // 建立本地会话
                createLocalSession(request, username);
                chain.doFilter(req, res);
                return;
            }
        }
        
        // 重定向到 CAS Server
        String loginUrl = casServerUrl + "/login?service=" + 
            URLEncoder.encode(localUrl, "UTF-8");
        response.sendRedirect(loginUrl);
    }
}
```

### 关键概念

| 概念 | 说明 |
|------|------|
| TGT | Ticket Granting Ticket，用户在 CAS Server 的登录凭证 |
| ST | Service Ticket，一次性凭证，用于应用系统验证用户身份 |
| TGC | Ticket Granting Cookie，CAS Server 设置的 Cookie，存储 TGT |

## 四、方案三：OAuth 2.0

### 核心思想

OAuth 2.0 本质是一种授权协议，但也可用于 SSO。用户通过第三方平台（如微信、Google）的身份来登录你的系统。

### 授权码模式流程

```
用户点击"微信登录"
    │
    ▼
重定向到微信授权页面
URL: https://open.weixin.qq.com/connect/oauth2/authorize
    ?appid=xxx
    &redirect_uri=https://a.example.com/callback
    &response_type=code
    &scope=snsapi_userinfo
    │
    ▼
用户授权后，微信重定向回应用
URL: https://a.example.com/callback?code=AUTH_CODE
    │
    ▼
应用用 code 换取 access_token
POST https://api.weixin.qq.com/sns/oauth2/access_token
    ?appid=xxx&secret=xxx&code=AUTH_CODE
    │
    ▼
用 access_token 获取用户信息
GET https://api.weixin.qq.com/sns/userinfo?access_token=xxx&openid=xxx
    │
    ▼
根据用户信息创建/登录本地账户
```

### Spring Security OAuth2 实现

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/public/**").permitAll()
                .anyRequest().authenticated()
            )
            .oauth2Login(oauth2 -> oauth2
                .loginPage("/login")
                .defaultSuccessUrl("/home")
                .userInfoEndpoint(userInfo -> userInfo
                    .userService(customOAuth2UserService)
                )
            );
        return http.build();
    }
}
```

## 五、方案对比

| 方案 | 跨域 | 复杂度 | 安全性 | 适用场景 |
|------|------|--------|--------|---------|
| 共享 Cookie | 仅同父域 | 低 | 中 | 子域名系统 |
| CAS | 任意域 | 中 | 高 | 企业内部多系统（推荐） |
| OAuth 2.0 | 任意域 | 高 | 高 | 第三方登录/开放平台 |
| JWT + Redis | 任意域 | 低 | 中 | 前后端分离微服务 |

## 六、微服务场景下的 SSO

在微服务架构中，通常使用 **JWT + 网关** 实现 SSO：

```
用户登录 → 网关验证 → 颁发 JWT Token
用户请求 → 网关校验 Token → 转发到具体服务

所有微服务共享同一个 JWT Token，无需每个服务单独登录。
```

```java
// 网关层统一校验
@Component
public class AuthFilter implements GlobalFilter {
    
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, 
                              GatewayFilterChain chain) {
        String token = exchange.getRequest().getHeaders()
            .getFirst("Authorization");
        
        if (token == null || !jwtUtils.validate(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        
        // 将用户信息传递给下游服务
        Claims claims = jwtUtils.parseToken(token);
        ServerHttpRequest request = exchange.getRequest().mutate()
            .header("X-User-Id", claims.getSubject())
            .build();
        
        return chain.filter(exchange.mutate().request(request).build());
    }
}
```

## 常见追问

### Q：CAS 和 OAuth 2.0 的区别？

> 1. CAS 是认证协议，解决"你是谁"的问题
> 2. OAuth 2.0 是授权协议，解决"你能做什么"的问题
> 3. CAS 适用于企业内部系统，OAuth 2.0 适用于第三方登录
> 4. CAS 需要自建认证中心，OAuth 2.0 依赖第三方平台

### Q：Token 被盗怎么办？

> 1. 设置较短的过期时间（如 2 小时）
> 2. 使用 HTTPS 传输
> 3. 将 Token 存在 HttpOnly Cookie 中，防止 XSS 攻击
> 4. 使用 Refresh Token 机制，Access Token 过期后用 Refresh Token 换新
> 5. 服务端维护 Token 黑名单（注销时加入黑名单）

### Q：如何实现登出（注销）？

> 1. CAS：向 CAS Server 发送注销请求，CAS Server 销毁 TGT 并通知所有已登录的应用
> 2. JWT：客户端删除 Token + 服务端将 Token 加入黑名单
> 3. OAuth 2.0：调用第三方平台的注销接口

## 场景题补充：SSO系统设计要点

面试中可能会问："如何设计一个SSO系统？"这道题考察对认证授权体系和分布式会话管理的理解。

**SSO核心组件**：
- 前台站点（需要登录的业务站点）
- SSO站点（登录页面、登出入口）
- SSO服务（登录服务、登录状态校验服务、登出服务）
- 数据库（用户账户信息）
- 缓存（Redis存储登录信息，保证高可用和线性扩展）

**登录状态存储方案**：
- 用户登录成功后生成AuthToken（UUID或UserName+时间戳加密），浏览器存Cookie，App存本地缓存
- 服务端用Redis集群存储登录信息（UserId、UserName），支持负载均衡和水平扩展
- Cookie设置domain为`.test.com`，所有`*.test.com`子站自动携带AuthToken

**跨域SSO实现思路**：
- 登录完成后通过回调方式将AuthToken传递给非主域名站点，各站点自行存入当前域Cookie
- 登出时通过回调调用各站点登出接口，清除各域Cookie中的AuthToken
- 核心是解决Cookie的跨域读写问题

**关键设计决策**：
- AuthToken建议用UUID+时间戳加密生成，服务端解密验证合法性
- 登出需同时清除Redis缓存和客户端Cookie
- 手机App场景需额外考虑Token刷新和本地安全存储
