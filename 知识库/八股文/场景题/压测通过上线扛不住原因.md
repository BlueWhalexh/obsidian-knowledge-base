# 压测通过上线扛不住原因

## 问题
压测600没问题，上线后300就扛不住了，可能是什么原因？

## 面试考察点
- 压测与真实流量的差异（模拟vs生产）
- 系统瓶颈分析能力（数据/依赖/网络/监控开销）
- 环境差异的敏感性（测试环境和生产环境的区别）
- 分布式系统的隐藏问题（缓存失效、锁竞争、线程泄漏）

## 解题思路
> 核心问题：压测环境与生产环境存在关键差异，需从压测场景设计、数据差异、依赖差异、网络环境、监控开销、线程竞争、JVM配置、缓存预热等维度系统排查。

压测场景不合理：只测单一接口，未覆盖真实业务多接口并发调用导致资源竞争。数据差异：压测用少量静态数据，生产数据复杂导致查询变慢。系统依赖未模拟：压测Mock外部服务，生产调用真实Redis/MQ/第三方成为瓶颈。网络差异：压测内网，生产有公网延迟/DNS/网关。

生产额外开销：链路追踪(APM)、日志采集(ELK)、监控(Prometheus)增加CPU和IO消耗。缓存未预热：生产环境缓存为空导致大量穿透到DB。线程竞争：真实流量有热点数据竞争导致锁激烈。JVM差异：生产环境GC参数不同导致频繁STW。

## 具体方案
### 方案一：环境对齐
- 统一JVM参数（-Xms4g -Xmx4g -XX:+UseG1GC）
- 压测时连接真实依赖（数据库/Redis/MQ）
- 使用生产数据集进行压测

### 方案二：多维度压测
- 多接口混合压测，按真实比例分配请求
- 预热缓存，模拟缓存穿透/雪崩场景
- 长时间持续压测（非短时间峰值）

### 方案三：生产监控排查
- SkyWalking/Pinpoint定位慢接口
- jstat -gc查看Full GC频率
- jstack查看线程阻塞和锁竞争
- Redis info stats查看缓存命中率

## 要点总结
- 压测必须覆盖真实业务路径（多接口混合），按真实流量比例分配请求
- 压测数据应使用生产数据集，预热缓存，模拟缓存穿透/失效场景
- 压测时连接真实依赖（DB/Redis/MQ），Mock会掩盖依赖瓶颈
- 生产环境APM/日志/监控有额外开销，压测环境也应开启
- 排查顺序：对比环境配置 -> 链路追踪定位慢接口 -> JVM GC分析 -> 线程状态 -> DB慢查询
