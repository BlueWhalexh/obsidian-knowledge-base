# 同城多活数据同步方案

## 问题

如何实现同城多活架构下的数据同步？如何处理数据冲突？

## 面试考察点

- 核心数据中心方案（主从同步+时间戳对比）
- Otter工具（Canal + binlog + 双向同步）
- 行冲突与字段冲突处理
- 适用距离限制（50-100公里）

## 解题思路

同城多活核心挑战是**数据一致性**和**冲突解决**。

## 具体方案

### 方案一：核心数据中心方案
- 设定一个**核心数据中心**作为主写入点
- 其他数据中心**主从同步**到核心中心
- 读请求就近访问本机房，写请求路由到核心中心
- **客户端时间戳对比**：解决冲突时以时间戳大的为准

### 方案二：Otter双向同步（推荐）
- **Canal**监听MySQL binlog，捕获数据变更
- Otter负责跨机房数据同步
- **双向同步**：两个机房都可以写入，互相同步

#### 冲突处理
- **行冲突**：同一行在两个机房同时修改
  - 解决：以时间戳大的为准（最后写入胜出）
- **字段冲突**：同一行不同字段在两个机房分别修改
  - 解决：**字段级别合并**，保留两个机房各自的修改

### 适用条件
- 同城多活适用于**50-100公里**内的机房间
- 网络延迟 < 3ms，同步延迟可接受
- 超过100公里建议采用异地多活（更复杂的方案）

## 要点总结

1. 核心数据中心方案简单但有单点风险
2. Otter + Canal是成熟的双向同步方案
3. 行冲突用时间戳解决，字段冲突用字段合并
4. 同城多活限50-100公里，超出需考虑异地方案
5. 双向同步需注意循环同步问题（Otter内置处理）
