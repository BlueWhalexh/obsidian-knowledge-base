# 堆外内存泄漏排查

## 问题
应用内存持续增长，但堆内存和元空间都没变化，可能是什么原因？如何排查？

## 面试考察点
- JVM内存模型理解：堆外内存（Direct Memory、Native Memory）的认知
- 排查能力：系统化的内存泄漏定位思路
- 工具使用：熟悉各类诊断工具（jmap、NMT、pmap、async-profiler等）
- 实战经验：是否处理过真实的内存泄漏问题

## 解题思路
> 堆内存和元空间不变但RSS持续增长，大概率是堆外内存泄漏

Java进程的内存 = 堆 + 元空间 + 线程栈 + 直接内存(DirectByteBuffer) + Native内存(Malloc) + 映射文件(MappedByteBuffer) + JNI分配。堆内存稳定说明泄漏不在Java对象层面，需要关注DirectByteBuffer、NIO、JNI调用的第三方库（如Netty、Protobuf、压缩库）。

排查思路分四步：确认泄漏范围 → 定位泄漏类型 → 抓取详细分析 → 修复。先用操作系统工具确认进程RSS增长，再用JVM工具排除堆和元空间，最后用NMT或pmap定位具体是哪种堆外内存。

## 具体方案
### 方案一：NMT (Native Memory Tracking) 分析
- 启动参数加 `-XX:NativeMemoryTracking=detail`
- 命令 `jcmd <pid> VM.native_memory detail` 查看各区域内存分布
- 对比两次快照找到增长的区域（Direct/Mmap/Thread/Code等）
- 优点：JVM内置，无需额外工具
- 缺点：有约5-10%性能开销

### 方案二：系统级工具定位
- `pmap -x <pid>` 查看进程内存映射，找出异常大的匿名映射区域
- `/proc/<pid>/smaps` 详细查看每段内存的RSS/PSS
- 对比发现增长的内存段地址，关联到具体的分配来源

### 方案三：Async-Profiler + Malloc追踪
- 用async-profiler的 `-e malloc` 事件采样Native分配
- 或用 `LD_PRELOAD` 注入malloc追踪库，记录分配调用栈
- 定位到具体的代码位置和第三方库

## 要点总结
- RSS增长但堆不变，首先怀疑DirectByteBuffer（NIO未释放）和第三方Native库
- 排查四步法：top/ps确认RSS → jcmd NMT定位区域 → pmap定位地址 → profiler定位代码
- 常见泄漏源：Netty ByteBuf未release、JNI调用的C库Malloc未Free、MappedByteBuffer未unmap
- DirectByteBuffer由Cleaner回收，Full GC时才触发，需确保GC正常运行
- 生产环境建议开启NMT定期采样，配合Prometheus+Grafana监控堆外内存趋势
