# 外部数据交互注意事项

## 问题
和其他公司做数据交互时，有什么需要注意的？如何保证安全、可靠、可追溯？

## 面试考察点
- 安全意识：数据传输中的安全防护措施
- 可靠性设计：数据一致性、幂等性、重试机制
- 接口设计：协议规范、版本管理、限流防护
- 工程思维：可追溯性、监控告警、异常处理

## 解题思路
> 外部交互的核心风险：数据被窃取、接口被攻击、数据不一致、对方系统拖垮我方

需要从四个维度构建防护体系：**安全**（认证鉴权、加签加密、IP白名单）、**可靠**（幂等设计、重试补偿、对账机制）、**防护**（限流熔断、超时控制、数据校验）、**可追溯**（全链路日志、唯一请求ID、对账报表）。

最关键的原则是：永远不信任外部数据。所有外部输入必须做格式校验、范围校验、签名验证后再使用。

## 具体方案
### 方案一：安全防护层
- 接口鉴权：OAuth2/API Key + Secret，请求签名（HMAC-SHA256）
- 传输安全：HTTPS + 敏感字段加密（AES/RSA）
- IP白名单 + 请求频率限制（令牌桶/滑动窗口）
- 防重放：请求带时间戳+Nonce，服务端校验时效性和唯一性

### 方案二：可靠性保障层
- 幂等设计：请求携带唯一业务ID，服务端去重（Redis/数据库唯一索引）
- 超时重试：指数退避 + 最大重试次数 + 幂等保证
- 对账机制：定时任务比对双方数据，差异数据触发补偿
- 签名验签：所有请求/响应双向签名，防篡改

### 方案三：防护隔离层
- 限流：对外部接口调用做QPS限制，防止对方系统异常拖垮自身
- 熔断：调用失败率超阈值自动熔断，走降级逻辑
- 超时控制：设置合理超时（连接超时、读超时），避免线程池被占满
- 数据校验：所有外部数据入参校验（类型、长度、范围、格式）

### 方案四：可追溯层
- 全链路日志：请求ID贯穿调用链，记录请求/响应原文
- 数据脱敏：日志中敏感字段（身份证、手机号、银行卡）脱敏处理
- 监控告警：接口成功率、响应时间、错误码分布实时监控
- 审计合规：数据交互记录留痕，满足合规审计要求

## 要点总结
- 永远不信任外部数据：所有入参必须校验，所有请求必须验签
- 幂等是外部交互的基石：用唯一业务ID + Redis/数据库去重保证
- 防护三件套：限流（防突发）、熔断（防雪崩）、超时（防阻塞）
- 对账机制不可少：定时比对双方数据，差异触发人工介入或自动补偿
- 数据安全合规：传输加密、存储加密、日志脱敏、访问审计，满足GDPR等法规要求
