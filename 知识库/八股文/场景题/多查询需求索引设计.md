# 一个表有用户和时间两个列三种查询需求怎么建索引

## 问题
一个表有用户和时间两个列，现有3个需求：根据用户查；根据日期查；根据日期和用户查；怎么建立索引？

## 面试考察点
- 联合索引设计原则：最左前缀原则的理解和应用
- 索引覆盖策略：如何用最少的索引覆盖最多的查询场景
- 读写性能权衡：索引数量对INSERT/UPDATE/DELETE的影响
- 查询模式分析：不同查询频率下的索引优化策略

## 解题思路
> 核心问题：三种查询需求（WHERE user=?、WHERE create_time=?、WHERE user=? AND create_time=?）需要合理设计索引

方案一：建立两个复合索引 (user, create_time) 和 (create_time, user)。根据最左前缀原则，WHERE user=?命中第一个索引的前缀，WHERE create_time=?命中第二个索引的前缀，联合查询两个索引均可命中。优点是覆盖所有查询场景，缺点是索引维护成本高。

方案二：一个复合索引 (user, create_time) + 一个单列索引 (create_time)。WHERE user=?和联合查询命中复合索引，WHERE create_time=?命中单列索引。优点是节省空间、写入性能更优。

选择建议：读多写少用方案一，写入频繁用方案二。

## 具体方案
### 方案一：两个复合索引
- CREATE INDEX idx_user_time ON table(user, create_time)
- CREATE INDEX idx_time_user ON table(create_time, user)
- 三个查询均可命中索引，避免全表扫描
- 适合读写比例高、三种查询频率接近的场景

### 方案二：复合索引 + 单列索引
- CREATE INDEX idx_user_time ON table(user, create_time)
- CREATE INDEX idx_time ON table(create_time)
- 节省空间，写入效率更高
- 适合写入频繁、时间查询频率较低的场景

## 要点总结
- 最左前缀原则：复合索引(a,b)可以匹配WHERE a=?但不能直接匹配WHERE b=?
- 推荐方案一（两个复合索引）适用于读多写少、查询模式多样的场景
- 索引顺序很重要：更常用的字段作为复合索引的前导列
- 每个索引都会增加写入成本，需在查询性能和写入性能间权衡
