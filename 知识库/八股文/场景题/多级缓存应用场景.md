# 多级缓存应用场景

## 问题
说一说多级缓存是如何应用的？

## 面试考察点
- 架构设计能力：对缓存体系的理解深度
- 性能优化思维：如何通过缓存提升系统性能
- 技术广度：对不同缓存技术的掌握程度
- 问题预见性：对缓存击穿、雪崩等问题的认知

## 解题思路
> 核心思路：从单层缓存的瓶颈出发，逐层叠加解决带宽、穿透、成本问题

多级缓存的本质是参考CPU缓存层级思想，构建分层缓存体系。主从模式解决基本读写分离，但热点数据带宽和单点穿透是其瓶颈。在主从前面加L1缓存层解决带宽问题，再加本地缓存进一步降低成本。关键挑战是各层数据一致性，本地缓存通常采用短过期时间策略平衡。

请求流转顺序：本地缓存 -> L1缓存 -> 主从缓存 -> 数据库。未命中时回种到上层，通过LRU淘汰保证热点数据留存。L1缓存有分组概念，同一数据轮流落到不同组，实现负载分散。

## 具体方案
### 方案一：L1+主从双层模式
- 原理：在主从缓存前增加L1缓存层，L1有分组概念
- 优点：解决单点带宽瓶颈和穿透问题，L1宕机不会全部打到DB
- 缺点：内存成本较高
- 适用：热点数据带宽压力大的场景

### 方案二：本地缓存+L1+主从多层模式
- 原理：应用服务器本身承担部分缓存，使用Caffeine等本地缓存
- 优点：充分利用应用机器带宽，纳秒级访问，成本低
- 缺点：数据一致性问题（应用机器数量随扩缩容变化）
- 适用：应用层机器部署较多的场景
- 一致性方案：本地缓存采用短过期时间（如5秒）

### 方案三：五级缓存架构（完整方案）
- L1客户端缓存（HTTP缓存头）-> L2 CDN缓存 -> L3 Nginx反向代理缓存 -> L4应用级缓存（Caffeine）-> L5分布式缓存（Redis）
- 一致性保障：主动更新+延迟双删+版本控制
- 异常处理：布隆过滤器防击穿、随机过期时间防雪崩、本地缓存+分片策略防热点Key

## 要点总结
- 多级缓存解决的核心问题：带宽瓶颈、单点穿透、缓存成本
- 本地缓存一致性采用短过期时间折中方案（如5秒过期）
- L1缓存分组设计：同一数据轮流落到不同组，穿透后回种实现多副本
- 缓存异常三板斧：布隆过滤器防击穿、随机过期防雪崩、分片防热点
- 实战效果：电商秒杀峰值QPS从5k提升到50k，数据库负载降低90%
