# 实时订阅推送设计

## 问题

如何设计实时订阅推送系统？如何在1分钟内完成20万条推送？

## 面试考察点

- 传统定时任务的局限性
- Redis SortedSet作为推送队列
- 定时任务集群设计（多机并发/秒级精度/无单点故障）
- 动态配置与负载均衡

## 解题思路

核心问题是**秒级时效性**和**大吞吐量**。传统定时任务无法满足，需要改造为集群模式。

## 具体方案

### 方案对比

**MQ延迟投递（不适用）**
- 不支持精确时间点投递
- 取消订阅需要delete已发出的MQ消息，困难

**传统定时任务（不适用）**
- 单机处理，性能瓶颈
- 频率不能太高（分钟级），时效性差
- 单点故障风险

**定时任务集群（推荐）**

### 架构设计

```
用户订阅 -> Redis SortedSet（score=提醒时间戳）
                    ↓
        多台Server每秒拉取（定时器1s）
                    ↓
          负载均衡分配队列
                    ↓
           MQ推送到消息中心
```

### 核心组件

**Redis SortedSet队列**
- 以提醒时间戳作为score
- `ZRANGEBYSCORE` 获取当前需要推送的记录
- 天然支持按时间排序

**队列分片**
- 按 `user_id % 队列数` hash到不同队列
- 同一用户的多条推送可合并
- 队列数 = 服务器数（动态可配置）

**负载均衡**
- Redis自增key % 队列数，保证不同Server竞争不同队列
- 避免多台机器同时处理同一队列

**动态配置**
- 队列数可动态调整（淘宝Diamond配置中心）
- 每次拉取数量可配置
- 大促加机器时自动调整

### 性能估算
- 10台机器 * 2000条/次 = 20000条/次
- 发MQ异步推送约0.5秒
- **20万条推送约10几秒完成**

## 要点总结

1. Redis SortedSet天然支持按时间排序，是推送队列的理想选择
2. 定时任务集群多机并发，秒级精度，无单点故障
3. 队列数动态可配置，随机器数调整
4. 负载均衡保证无竞争，提升集群效率
5. 10台机器可在十几秒内完成20万条推送
