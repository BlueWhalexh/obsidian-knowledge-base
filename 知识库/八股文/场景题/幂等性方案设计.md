# 幂等性方案设计

## 问题
你是如何设计幂等性方案避免数据重复处理的？

## 面试考察点
- 分布式理解：对重复请求问题的本质认识
- 架构能力：复杂场景下的解决方案设计
- 技术广度：多维度处理方案的掌握
- 实战经验：实际业务中的落地经验

## 解题思路
> 核心思路：理解幂等本质 -> 分析重复原因 -> 分层设计方案

幂等性：f(x) = f(f(x)，一次操作和多次操作产生相同影响。重复请求的原因：网络阻塞/延迟导致超时重试、用户重复点击、MQ/RPC重试机制。

单数据CRUD中，查询天然幂等，基于主键的Delete天然幂等，需要保证幂等的是：新增操作、基于主键的计算式Update（如stock=stock-1）、基于条件查询的Update。多数据并发操作是重点场景（高并发抢红包、下单、支付）。

## 具体方案
### 方案一：基础性方案（按场景选择）
- 全局唯一ID：请求前先根据ID判断是否已执行（UUID/Snowflake/数据库自增）
- 唯一索引/去重表：将唯一标识作为唯一索引，重复插入会抛唯一约束异常
- 插入或更新（upsert）：INSERT ON DUPLICATE KEY UPDATE
- 多版本控制：UPDATE goods SET name=#{newName}, version=#{version} WHERE id=#{id} AND version<#{version}
- 状态机控制：UPDATE goods_order SET status=#{status} WHERE id=#{id} AND status<#{status}

### 方案二：综合性方案 - 一锁二判三更新（支付宝方案）
- 第一步加锁：Redis分布式锁（高性能），可引入锁分段机制提升100倍
- 第二步判断：基于状态机/流水表/唯一性索引判断是否已执行
- 第三步更新：通过幂等判断后执行数据更新，完成后释放锁
- 优点：通用性强，覆盖高并发场景
- 适用：秒杀、支付、订单等关键业务

### 方案三：多级幂等控制（完整方案）
- 轻量级校验层：BloomFilter拦截80%重复请求
- 分布式锁控制层：Redis setIfAbsent处理并发请求
- 持久化记录层：idempotent_records表（request_id唯一索引）最终保障
- 消息队列幂等：Redis setnx记录msgId，24小时过期

## 要点总结
- 幂等性是对写操作的要求：f(x) = f(f(x))
- 重复原因：网络超时重试 + 用户重复操作 + MQ/RPC重试机制
- 一锁二判三更新是支付宝验证的通用方案：Redis分布式锁 + 状态判断 + 数据更新
- 分层设计：BloomFilter前置拦截 -> Redis分布式锁并发控制 -> 数据库唯一索引最终保障
- 实战效果：三级幂等控制抗住618峰值5万笔/秒支付请求，重复支付率为0
