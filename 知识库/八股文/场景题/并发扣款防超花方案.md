# 账户只有十块钱同时两笔订单怎么保证不超花

## 问题
账户里面只有十块钱，同时发来两笔订单一共大于十块钱，怎么保证不超花？

## 面试考察点
- 并发写入风险识别：脏读/不可重复读导致的余额透支
- 悲观锁 vs 乐观锁的适用场景和实现方式
- 事务隔离级别应用：读已提交 vs 可重复读 vs 串行化
- 分布式场景扩展：本地事务 vs 分布式事务（TCC、SAGA）

## 解题思路
> 核心问题：两个请求同时读取余额10元，各自判断余额足够后执行扣款，最终余额变负

竞态条件的本质是"读取-判断-更新"不是原子操作。解决方案的核心是让扣款操作变成原子性的，或者让并发请求串行化执行。

五种方案按场景选择：1）乐观锁（版本号）——适合单体应用、低并发；2）悲观锁（SELECT FOR UPDATE）——适合强一致性要求；3）Redis分布式锁——适合微服务架构；4）消息队列削峰——适合高并发场景；5）预扣款+确认机制——适合电商/金融。

## 具体方案
### 方案一：数据库乐观锁（推荐）
- 账户表添加version字段
- UPDATE account SET balance=balance-8, version=version+1 WHERE user_id=123 AND version=1
- 影响行数为0则说明版本冲突，重试或提示失败
- 无锁设计，适合读多写少场景

### 方案二：悲观锁 SELECT FOR UPDATE
- START TRANSACTION
- SELECT * FROM account WHERE user_id=123 FOR UPDATE（锁定行）
- 判断余额足够则UPDATE，否则失败
- COMMIT
- 强一致性但性能较低，有死锁风险

### 方案三：消息队列顺序处理
- 订单请求入Kafka队列，按账户ID分区保证顺序
- 消费者串行处理同一账户的扣款请求
- 削峰填谷，避免并发问题
- 缺点是用户需等待异步处理结果

## 要点总结
- 核心是将"读取-判断-更新"变为原子操作或串行执行
- 乐观锁适合低冲突场景，悲观锁适合强一致性场景
- 分布式环境推荐Redis分布式锁或消息队列方案
- 预扣款机制（冻结余额）是电商/金融场景的最佳实践
