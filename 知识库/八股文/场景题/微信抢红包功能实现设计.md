# 微信抢红包功能实现设计

## 问题

微信抢红包功能如何实现？如何应对春晚1400万/秒的峰值？

## 面试考察点

- 红包金额分配算法
- 高并发下红包防超领
- 接入服务内嵌摇一摇逻辑
- 过载保护策略

## 解题思路

红包系统核心挑战是**金额分配算法**和**超高并发下的系统稳定性**。

## 具体方案

### 1. 金额分配算法
- 期望收益与先后顺序无关
- 保证每人至少0.01元
- 所有子红包之和 = 总金额
- 算法：随机金额在 [0.01, 剩余平均值*2] 之间
- 最后一人拿剩余金额

### 2. 微信红包特点
- **实时计算**：拆的时候才算金额，不预分配（纯内存计算）
- **CAS更新**：每抢一个红包，CAS更新剩余金额和个数
- **cache计数**：红包个数原子递减，到0表示抢完
- 一个红包一条数据，没有队列

### 3. 春晚架构演进

**V0.1原型（信心10%）**
- 问题：带宽3000PB/s、同时在线3.5亿、请求1000万/秒

**V0.5测试版（信心50%）**
- 解决方案：
  - 静态资源提前下发客户端（闲时下载）
  - 18个接入集群，638台服务器，支持14.6亿在线
  - **摇一摇逻辑内嵌接入服务**（去掉独立摇一摇服务）
  - 接入服务拆分为IO模块+逻辑模块+Agent

**V0.8预览版（信心70%）**
- 全程压测 + 专题Code Review + 内部演练 + 线上预热

**V1.0正式版（信心80%）**
- 春晚结果：110亿次摇动，峰值1400万/秒

### 4. 过载保护
- **前端保护后端**：客户端每5-10秒发一个请求（而非每次摇都发）
- **后端拒绝前端**：实时计算接入负载，返回限速档位
- 实时CPU负载监控，动态调整限速策略

## 要点总结

1. 金额实时计算不预分配，CAS保证并发安全
2. 摇一摇逻辑内嵌接入服务，消除独立服务的网络开销
3. 前端保护后端+后端拒绝前端的过载保护策略
4. 80%信心+20%运气，架构设计无法做到万无一失
