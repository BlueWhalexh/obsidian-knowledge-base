# 扫码登录实现原理

## 问题

如何设计一个扫码登录功能？整体流程分为哪几个阶段？

## 面试考察点

- 多端交互设计能力（PC端、移动端、服务端三方协作）
- 状态机设计（待扫描/已扫描待确认/已确认三阶段）
- 安全性考量（防token劫持、防重放攻击）
- 长轮询/WebSocket等实时通信方案

## 解题思路

扫码登录本质是**借助已登录的移动端来授权PC端登录**，核心流程分三个阶段：

### 阶段一：待扫描

1. PC端请求服务端，获取**唯一二维码ID**，绘制成二维码展示
2. PC端启动定时器，**轮询查询**二维码状态
3. 二维码设置过期时间，超时后失效需重新获取

### 阶段二：已扫描待确认

1. 移动端扫描二维码，获取二维码ID
2. 移动端将**已登录token + 二维码ID**发送给服务端
3. 服务端将token与二维码ID关联，生成**临时token**返回给移动端
4. PC端轮询检测到状态变更，显示"已扫描，请在手机确认"

### 阶段三：已确认

1. 用户在移动端点击确认，携带临时token请求服务端
2. 服务端校验通过后，更新二维码状态，为PC端生成**正式token**
3. PC端轮询获取到正式token，完成登录

## 具体方案

```
关键设计点：
- 二维码ID必须全局唯一（UUID），否则会导致账号错乱
- 临时token有效期短，仅用于确认环节，防止被截获重放
- 确认环节是必须的，防止恶意扫描直接登录
- 确认后推送登录提醒通知到用户App
```

**轮询 vs 长轮询 vs WebSocket**：
- 轮询：实现简单但请求频繁
- 长轮询：减少无效请求，推荐方案
- WebSocket：实时性最好但实现复杂

## 要点总结

1. 三阶段状态机是核心：待扫描 -> 已扫描待确认 -> 已确认
2. 二维码ID唯一性是关键，绑定身份信息后不可重复
3. 必须有确认环节，防token劫持导致的安全问题
4. 确认后应推送登录通知，便于用户发现异常登录
