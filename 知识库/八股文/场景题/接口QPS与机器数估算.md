# 接口QPS与机器数估算

## 问题
一个接口3000QPS，RT为200MS，预估需要几台机器？

## 面试考察点
- 容量规划能力：对系统负载的量化分析能力
- 性能计算：基础性能指标的运算能力
- 工程思维：理论计算与实际部署的结合
- 扩展意识：对冗余和弹性的考量

## 解题思路
> 核心思路：先算单机理论QPS，再用环境因素修正，最后加冗余

基础公式推导：
1. 单线程QPS = 1000ms / RT = 1000 / 200 = 5 QPS
2. 假设Tomcat默认200线程，单机理论QPS = 5 x 200 = 1000 QPS
3. 理论机器数 = 3000 / 1000 = 3台

但这是纯理论值，需要考虑实际环境因素修正。实际预估需要在单机上压测，根据压测结果来估算。一般按理论值的2-3倍来算，即6-9台。

## 具体方案
### 方案一：理论计算法
- 单线程QPS = 1000ms / RT(ms)
- 单机QPS = 单线程QPS x 线程数
- 理论机器数 = 总QPS / 单机QPS
- 本题：5 x 200 = 1000，3000/1000 = 3台

### 方案二：环境修正法（更接近实际）
- CPU利用率修正（目标70%）：1000 x 70% = 700
- GC影响修正（降低10%）：700 x 90% = 630
- 网络损耗修正（5%）：630 x 95% ≈ 600
- 修正后机器数 = 3000 / 600 = 5台
- 加冗余（N+1）：5+1 = 6台
- 峰值缓冲（30%余量）：6 x 1.3 ≈ 8台

### 方案三：完整容量规划
- 压测验证：wrk -t8 -c200 -d60s --latency http://service:8080/api
- 监控指标：CPU利用率<=70%、线程池活跃度<=80%、GC频率<=1次/10s
- 弹性方案：4台常备实例 + 4台弹性伸缩实例
- 机器配置参考：4C8G混合型QPS约1000-1500，8C16G约2000-3000

## 要点总结
- 核心公式：单机QPS = (1000ms/RT) x 线程数
- 本题理论值：3台，实际建议6-9台（2-3倍冗余）
- 环境修正因素：CPU利用率、GC开销、网络损耗、IO等待
- 必须通过压测验证，不能只靠理论计算
- 高可用设计：N+1冗余 + 30%峰值缓冲 + 弹性伸缩
