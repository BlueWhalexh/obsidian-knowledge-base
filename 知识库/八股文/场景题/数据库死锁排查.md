# 数据库死锁排查

## 问题
线上MySQL出现死锁异常，如何排查定位原因并解决？

## 面试考察点
- 死锁日志的获取和分析能力（show engine innodb status）
- binlog日志分析能力，找到死锁事务的完整SQL执行序列
- 对MySQL锁类型（记录锁、间隙锁、Next-key锁、插入意向锁）的理解

## 排查思路
> 错误日志报警 -> 查看堆栈信息 -> 获取死锁日志 -> 分析binlog -> 找出两个死锁事务的SQL -> 根据锁理论分析原因 -> 修改代码

## 具体排查步骤
1. **发现死锁**：线上错误日志报警发现Deadlock异常
2. **查看堆栈信息**：定位被回滚事务对应的业务代码
3. **获取死锁日志**：
   - `show engine innodb status`（仅最近一次）
   - 开启监控：`set GLOBAL innodb_print_all_deadlocks=ON`（记录到错误日志）
4. **分析死锁日志**：关注事务ID、活跃时间、持有锁/等待锁信息、锁类型
5. **获取binlog**：用`mysqlbinlog`工具获取完整事务SQL
6. **锁冲突分析**：根据两个事务的SQL和锁信息，分析死锁原因
7. **修改业务代码**

## 死锁日志关键信息
| 字段 | 含义 |
|------|------|
| TRANSACTION ... ACTIVE N sec | 事务活跃时间 |
| LOCK WAIT | 事务正在等待锁 |
| lock_struct(s) | 事务持有的锁结构数 |
| row lock(s) | 持有的行锁个数 |
| lock_mode X locks rec but not gap | 记录锁 |
| lock_mode X locks gap before rec | 间隙锁 |
| lock_mode X | Next-key锁 |
| lock_mode X locks gap before rec insert intention | 插入意向锁 |

## 要点总结
- 开启`innodb_print_all_deadlocks`可自动记录所有死锁到错误日志，建议生产环境常开
- 死锁日志只显示两个事务，多事务死锁时信息可能不完整
- binlog可获取事务的完整SQL序列，弥补死锁日志信息不足
- 分析死锁需结合锁理论：记录锁保护已有行、间隙锁保护范围、Next-key锁是两者组合
