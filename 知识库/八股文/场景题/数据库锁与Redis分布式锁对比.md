# 数据库锁与Redis分布式锁对比

## 问题
数据库乐观锁和悲观锁以及Redis分布式锁的区别和使用场景？

## 面试考察点
- 各类锁的底层实现原理掌握程度
- 不同业务场景下的技术选型能力
- 锁冲突问题的诊断与解决思路
- 实际应用中的权衡经验

## 解题思路
> 三种锁的核心目标都是保证数据一致性，但实现方式、适用场景差异显著

乐观锁假设冲突少发生，在提交时检查版本号（UPDATE SET version=version+1 WHERE version=N），适合读多写少场景，无锁竞争开销。悲观锁假设冲突频繁，事务开始即加锁（SELECT ... FOR UPDATE），适合写多场景，强一致性保障但性能较低。Redis分布式锁通过SETNX实现跨服务节点互斥，适用于分布式系统但需处理网络依赖和锁续期问题。

## 具体方案
### 方案一：乐观锁
- **原理**：版本号/时间戳校验，冲突时重试
- **示例**：UPDATE inventory SET stock=stock-1, version=version+1 WHERE product_id=1001 AND version=1
- 优点：性能高、无锁竞争、适合高并发低冲突
- 缺点：冲突频繁时需重试，无法保证绝对一致性
- 适用：电商秒杀库存扣减、订单状态变更、缓存更新

### 方案二：悲观锁
- **原理**：SELECT ... FOR UPDATE 行级锁，事务全程持有
- 优点：强一致性、简单直接
- 缺点：性能低、死锁风险、连接池资源占用高
- 适用：银行转账、财务流水、短事务写操作

### 方案三：Redis分布式锁
- **原理**：SETNX + Lua脚本原子释放
- 优点：跨服务协调、高性能（内存操作）、支持集群
- 缺点：网络依赖、需处理锁续期和脑裂
- 适用：分布式任务调度、跨服务协调、分布式限流

## 要点总结
- 乐观锁：应用层实现，无锁竞争，适合读多写少（电商秒杀、订单状态变更）
- 悲观锁：数据库层实现，强一致性，适合写多读少（银行转账、库存锁定）
- Redis分布式锁：分布式中间件实现，适合跨服务场景（任务调度、缓存预热）
- 选型决策：单机优先乐观锁/悲观锁，跨服务才用分布式锁
