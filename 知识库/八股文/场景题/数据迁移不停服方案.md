# 数据迁移不停服方案

## 问题
如何在生产环境不停服情况下进行数据迁移，从原来的16张表迁移到64张表中？

## 面试考察点
- 系统架构能力：能否设计高可用、低风险的迁移方案
- 分布式系统理解：数据一致性、事务处理的深度理解
- 风险控制意识：对迁移过程中问题的预见性和回滚方案
- 技术广度：对Canal、DRS等数据同步工具链的熟悉程度

## 解题思路
> 核心思考方向：在线迁移必须满足三个目标——不停服、数据完整、可回滚

简单的mysqldump或主从同步只支持单库到单库，无法支持16表到64表的分库分表场景。关键是设计一个渐进式方案，让新旧库并行运行，逐步切换流量。

思考路径：先同步数据 -> 双写保证增量一致 -> 校验数据 -> 灰度切读流量 -> 切写流量 -> 清理旧库。每一步都要有回滚方案。

## 具体方案

### 方案一：双写方案
- 原理：新旧库同时写入，通过Canal获取Binlog增量日志按分片逻辑写入新表
- 步骤：1. 用Canal同步历史数据到新表 2. 改造业务代码双写（新库可异步写入） 3. 抽样校验数据一致性 4. 灰度切读流量（10%->50%->100%） 5. 确认无误后改为只写新库
- 优点：全程可回滚，风险最低，通用性强（MySQL/Redis/MQ迁移都适用）
- 缺点：周期长，应用有改造成本

### 方案二：级联同步方案
- 原理：旧库->新库->备库三级级联同步，适合自建机房迁移上云场景
- 步骤：1. 新库配置为旧库从库同步数据 2. 备库配置为新库从库做备份 3. 三库写入一致后切读流量 4. 低峰期暂停写入，切换写流量到新库
- 优点：业务无改造成本，实施简单
- 缺点：切写时需短暂停止写入（需安排在低峰期）

### 缓存迁移补充：改造副本组预热
- 原理：云上部署副本组，写入时异步同步到云上节点，读请求按比例走云上缓存
- 步骤：1. 云上部署mc副本组 2. 写入异步同步到云上 3. 10%读流量走云上缓存 4. 命中率达90%后全量切换
- 优点：自动预热，专线带宽可控
- 缺点：预热期间有跨专线延迟

## 要点总结
- 迁移的核心三要素：在线、完整、可回滚
- 双写方案是最通用的方案，适用于各种存储介质的迁移
- 数据校验是最容易出问题的步骤，务必提前准备好校验工具
- 灰度切换流量是降低风险的关键手段
- 缓存迁移的重点是保持缓存热度，避免穿透到数据库
