# 本地缓存设计

## 问题

如何设计一个本地缓存？需要考虑哪些方面？

## 一、核心设计要素

### 1.1 数据结构选择

| 数据结构 | 特点 | 适用场景 |
|----------|------|---------|
| HashMap | 高速查找，非线程安全 | 单线程环境 |
| ConcurrentHashMap | 线程安全，分段锁 | 多线程环境（首选） |
| LinkedHashMap | 维护插入/访问顺序 | 实现 LRU |
| TreeMap | 有序，O(logN) 查找 | 需要有序遍历 |

**最常用**：`ConcurrentHashMap` 作为底层存储。

### 1.2 缓存接口设计

```java
public interface LocalCache<K, V> {
    V get(K key);
    void put(K key, V value);
    void put(K key, V value, long ttl, TimeUnit unit);
    void remove(K key);
    void clear();
    long size();
}
```

### 1.3 缓存条目设计

```java
public class CacheEntry<V> {
    private V value;
    private long createTime;     // 创建时间
    private long accessTime;     // 最后访问时间
    private long expireTime;     // 过期时间（0=永不过期）
    private int accessCount;     // 访问次数（用于 LFU）
    private long ttl;            // 存活时间
    
    public boolean isExpired() {
        return expireTime > 0 && System.currentTimeMillis() > expireTime;
    }
}
```

## 二、过期策略

### 2.1 TTL（Time To Live）

数据写入后 N 秒过期。

```java
public void put(K key, V value, long ttl, TimeUnit unit) {
    long expireTime = System.currentTimeMillis() + unit.toMillis(ttl);
    cache.put(key, new CacheEntry<>(value, expireTime));
}
```

### 2.2 TTI（Time To Idle）

数据最后一次被访问后 N 秒过期。适合不常访问的数据快速过期。

```java
public V get(K key) {
    CacheEntry<V> entry = cache.get(key);
    if (entry == null) return null;
    
    // 更新最后访问时间
    entry.setAccessTime(System.currentTimeMillis());
    return entry.getValue();
}
```

### 2.3 过期检查方式

| 方式 | 实现 | 特点 |
|------|------|------|
| 惰性删除 | get 时检查是否过期 | 简单，但已过期数据占用内存 |
| 定时扫描 | 后台线程定期清理 | 及时释放内存 |
| 两者结合 | get 时检查 + 定时扫描 | 推荐方案 |

```java
// 定时清理任务
@Scheduled(fixedRate = 60000) // 每分钟执行
public void cleanExpiredEntries() {
    long now = System.currentTimeMillis();
    cache.entrySet().removeIf(entry -> 
        entry.getValue().getExpireTime() > 0 
        && now > entry.getValue().getExpireTime());
}
```

## 三、缓存淘汰策略

当缓存达到最大容量时，需要淘汰旧数据。

### 3.1 LRU（Least Recently Used）— 最近最少使用

**核心思想**：淘汰最久没有被访问的数据。

**实现方式**：`ConcurrentHashMap` + `LinkedList`（或 `LinkedHashMap`）

```java
public class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int maxSize;
    
    public LRUCache(int maxSize) {
        // accessOrder = true 表示按访问顺序排序
        super(maxSize, 0.75f, true);
        this.maxSize = maxSize;
    }
    
    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > maxSize;
    }
}
```

**手写 LRU**（面试常考）：

```java
public class LRUCache<K, V> {
    private final int capacity;
    private final Map<K, Node<K, V>> map;
    private final Node<K, V> head; // 哨兵头
    private final Node<K, V> tail; // 哨兵尾
    
    public LRUCache(int capacity) {
        this.capacity = capacity;
        this.map = new ConcurrentHashMap<>();
        this.head = new Node<>(null, null);
        this.tail = new Node<>(null, null);
        head.next = tail;
        tail.prev = head;
    }
    
    public V get(K key) {
        Node<K, V> node = map.get(key);
        if (node == null) return null;
        
        // 移动到链表头部（标记为最近使用）
        moveToHead(node);
        return node.value;
    }
    
    public void put(K key, V value) {
        Node<K, V> node = map.get(key);
        if (node != null) {
            node.value = value;
            moveToHead(node);
        } else {
            node = new Node<>(key, value);
            map.put(key, node);
            addToHead(node);
            
            if (map.size() > capacity) {
                // 移除链表尾部（最久未使用）
                Node<K, V> removed = removeTail();
                map.remove(removed.key);
            }
        }
    }
    
    private void addToHead(Node<K, V> node) {
        node.prev = head;
        node.next = head.next;
        head.next.prev = node;
        head.next = node;
    }
    
    private void moveToHead(Node<K, V> node) {
        // 先从当前位置移除
        node.prev.next = node.next;
        node.next.prev = node.prev;
        // 添加到头部
        addToHead(node);
    }
    
    private Node<K, V> removeTail() {
        Node<K, V> node = tail.prev;
        node.prev.next = tail;
        tail.prev = node.prev;
        return node;
    }
    
    static class Node<K, V> {
        K key;
        V value;
        Node<K, V> prev;
        Node<K, V> next;
        
        Node(K key, V value) {
            this.key = key;
            this.value = value;
        }
    }
}
```

### 3.2 LFU（Least Frequently Used）— 最不经常使用

**核心思想**：淘汰访问频率最低的数据。

**实现方式**：`HashMap` + 频率桶

```java
public class LFUCache<K, V> {
    private final int capacity;
    private int minFreq; // 当前最小频率
    private Map<K, Node<K, V>> keyMap;      // key → Node
    private Map<Integer, LinkedHashSet<Node<K, V>>> freqMap; // freq → nodes
    
    public LFUCache(int capacity) {
        this.capacity = capacity;
        this.keyMap = new HashMap<>();
        this.freqMap = new HashMap<>();
        this.minFreq = 0;
    }
    
    public V get(K key) {
        Node<K, V> node = keyMap.get(key);
        if (node == null) return null;
        increaseFreq(node);
        return node.value;
    }
    
    public void put(K key, V value) {
        if (capacity == 0) return;
        
        Node<K, V> node = keyMap.get(key);
        if (node != null) {
            node.value = value;
            increaseFreq(node);
        } else {
            if (keyMap.size() >= capacity) {
                // 淘汰最小频率的节点
                LinkedHashSet<Node<K, V>> minFreqSet = 
                    freqMap.get(minFreq);
                Node<K, V> removed = minFreqSet.iterator().next();
                minFreqSet.remove(removed);
                keyMap.remove(removed.key);
            }
            
            node = new Node<>(key, value, 1);
            keyMap.put(key, node);
            freqMap.computeIfAbsent(1, k -> new LinkedHashSet<>()).add(node);
            minFreq = 1;
        }
    }
    
    private void increaseFreq(Node<K, V> node) {
        int freq = node.freq;
        freqMap.get(freq).remove(node);
        if (freqMap.get(freq).isEmpty()) {
            freqMap.remove(freq);
            if (minFreq == freq) minFreq++;
        }
        node.freq++;
        freqMap.computeIfAbsent(node.freq, k -> new LinkedHashSet<>()).add(node);
    }
}
```

### 3.3 策略对比

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| LRU | 实现简单，效果好 | 偶尔访问一次的数据不会被淘汰 | 通用场景（推荐） |
| LFU | 更精确地淘汰冷数据 | 实现复杂，新数据容易被淘汰 | 有明显热点数据 |
| FIFO | 最简单 | 不考虑访问频率 | 极简场景 |
| TTL | 按时间控制 | 无法根据热度淘汰 | 有时效性要求的数据 |

## 四、线程安全

### 4.1 方案对比

| 方案 | 特点 | 性能 |
|------|------|------|
| synchronized | 简单粗暴 | 低（全表加锁） |
| ReentrantReadWriteLock | 读写分离 | 中 |
| ConcurrentHashMap | 分段锁/ CAS | 高（推荐） |
| ConcurrentHashMap + ReadWriteLock | 细粒度控制 | 最高 |

### 4.2 推荐实现

```java
public class ThreadSafeCache<K, V> {
    private final ConcurrentHashMap<K, CacheEntry<V>> cache;
    private final ReadWriteLock evictionLock = new ReentrantReadWriteLock();
    
    public V get(K key) {
        CacheEntry<V> entry = cache.get(key);
        if (entry == null || entry.isExpired()) {
            if (entry != null) cache.remove(key);
            return null;
        }
        entry.setAccessTime(System.currentTimeMillis());
        return entry.getValue();
    }
    
    public void put(K key, V value, long ttl) {
        evictionLock.writeLock().lock();
        try {
            if (cache.size() >= maxSize) {
                evict(); // 淘汰操作需要独占锁
            }
            cache.put(key, new CacheEntry<>(value, ttl));
        } finally {
            evictionLock.writeLock().unlock();
        }
    }
}
```

## 五、生产级缓存框架

### 5.1 Caffeine（推荐）

```java
Cache<String, Object> cache = Caffeine.newBuilder()
    .maximumSize(10_000)           // 最大条目数
    .expireAfterWrite(10, TimeUnit.MINUTES)  // 写入后过期
    .expireAfterAccess(5, TimeUnit.MINUTES)  // 访问后过期
    .recordStats()                 // 统计命中率
    .build();

// 使用
cache.put("key", "value");
Object value = cache.getIfPresent("key");

// 异步加载
cache.get("key", k -> loadFromDB(k));
```

**Caffeine 的优势**：

1. 使用 W-TinyLFU 算法，命中率接近理论最优
2. 高性能：读写接近 O(1)
3. 支持异步刷新
4. Spring Boot 默认缓存实现

### 5.2 Guava Cache

```java
Cache<String, Object> cache = CacheBuilder.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(10, TimeUnit.MINUTES)
    .removalListener(notification -> {
        // 淘汰回调
        log.info("Key {} was removed", notification.getKey());
    })
    .build();
```

### 5.3 框架选择

**直接选择 Caffeine**。理由：

1. Caffeine 是 Guava Cache 作者的升级版
2. 性能更好（W-TinyLFU vs LRU）
3. API 更友好
4. Spring Boot 2.x 默认使用 Caffeine

## 六、多级缓存架构

```
请求 → L1 本地缓存（Caffeine） → L2 分布式缓存（Redis） → 数据库
         ↓ 未命中               ↓ 未命中
      查询 Redis              查询 DB
      回写本地缓存            回写 Redis + 本地缓存
```

**实现**：

```java
@Component
public class MultiLevelCache {
    @Autowired
    private Cache<String, Object> localCache; // Caffeine
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    public Object get(String key) {
        // L1: 本地缓存
        Object value = localCache.getIfPresent(key);
        if (value != null) return value;
        
        // L2: Redis
        value = redisTemplate.opsForValue().get(key);
        if (value != null) {
            localCache.put(key, value); // 回写 L1
            return value;
        }
        
        return null;
    }
}
```

## 常见追问

### Q：本地缓存和分布式缓存怎么选？

> 1. 数据量小、变更频率低、允许短暂不一致 → 本地缓存（如配置信息、字典表）
> 2. 数据量大、需要多实例共享、要求强一致 → 分布式缓存（Redis）
> 3. 高并发场景 → 多级缓存（L1 本地 + L2 Redis）

### Q：缓存一致性如何保证？

> 1. 本地缓存：设置较短的 TTL（如 1~5 分钟），容忍短暂不一致
> 2. 使用消息广播（如 Redis Pub/Sub）通知所有实例清除本地缓存
> 3. 版本号机制：每次更新递增版本号，读取时校验版本

### Q：如何预估本地缓存的大小？

> 1. 先估算单条数据大小（如 1KB）
> 2. 估算热点数据总量（如 10000 条）
> 3. 缓存大小 = 单条大小 × 条目数 = 1KB × 10000 = 10MB
> 4. 设置 JVM 堆内存时预留缓存空间
