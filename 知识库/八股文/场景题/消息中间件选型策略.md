# 消息中间件选型策略

## 问题

如何选择合适的消息中间件？RabbitMQ、RocketMQ、Kafka各适用于什么场景？

## 面试考察点

- 消息队列两大核心场景（异步解耦/削峰填谷）
- 三种MQ的特性对比
- 堆积能力、消息检索、吞吐量的权衡

## 解题思路

消息队列选型需根据**业务场景**、**性能要求**和**运维成本**综合评估。

## 具体方案

### 两大核心场景

**异步解耦**
- 下游系统不需要实时处理
- 生产者和消费者速率不一致
- 系统间解耦，独立演进

**削峰填谷**
- 瞬时高流量涌入，MQ缓冲
- 消费者按自身能力匀速消费
- 保护下游系统不被打垮

### 三种MQ对比

| 特性 | RabbitMQ | RocketMQ | Kafka |
|------|----------|----------|-------|
| 语言 | Erlang | Java | Scala/Java |
| 堆积能力 | 弱 | 强 | 极强 |
| 消息检索 | 不支持 | 支持 | 不支持 |
| 吞吐量 | 万级 | 十万级 | 百万级 |
| 延迟 | 微秒级 | 毫秒级 | 毫秒级 |
| 适用场景 | 小型业务 | 核心业务 | 日志/大数据 |

### 选型建议
- **RabbitMQ**：业务量小、需要低延迟，但堆积能力弱
- **RocketMQ**：核心业务交易，支持消息检索（适合对账/补偿）
- **Kafka**：日志采集、大数据管道，吞吐量最高

## 要点总结

1. 异步解耦和削峰填谷是消息队列的两大核心价值
2. RabbitMQ适合小规模业务，堆积能力是短板
3. RocketMQ消息检索能力适合核心业务（对账/补偿/幂等）
4. Kafka日志和大数据场景首选，吞吐量最高
5. 选型核心：堆积能力 > 消息检索 > 吞吐量 > 延迟
