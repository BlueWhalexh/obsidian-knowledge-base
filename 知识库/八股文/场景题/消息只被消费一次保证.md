# 消息只被消费一次保证

## 问题

如何确保消息只被消费一次？消息丢失和重复消费如何处理？

## 面试考察点

- 消息丢失的三个环节（生产/存储/消费）
- 幂等性概念与实现
- 生产端幂等（Producer Idempotency）
- 消费端幂等（全局ID + 乐观锁）

## 解题思路

完全避免消息重复几乎不可能，核心思路是**保证幂等性**：重复消费的结果与消费一次相同。

## 具体方案

### 1. 消息丢失的三个环节

**生产端丢失**
- 网络抖动导致发送失败
- 方案：超时重试（2-3次）
- 重试可能导致重复 -> 需要消费端幂等

**存储端丢失**
- Kafka异步刷盘，宕机可能丢数据
- 方案：集群部署 + `acks=all`（ISR全部确认）
- 权衡：acks=all性能下降，需根据容忍度选择

**消费端丢失**
- 处理成功但更新消费进度前宕机
- 方案：处理完成后再更新进度（可能重复消费）

### 2. 幂等性保证

**生产端幂等（Kafka 0.11+）**
- 给每个生产者分配唯一ID
- 每条消息有唯一ID
- 服务端存储 `<生产者ID, 最后消息ID>` 映射
- 重复消息自动丢弃

**消费端通用方案：全局唯一ID**
```
if (selectByID(ID)) {
    return; // 已消费，跳过
} else {
    process(message); // 处理消息
    saveID(ID); // 记录已消费
}
```

**消费端业务方案：乐观锁**
```sql
UPDATE user SET amount = amount + 20, version = version + 1 
WHERE userId = 1 AND version = 1;
```
- version不匹配则更新失败，避免重复扣款

### 3. 权衡建议
- 要求严格不丢：acks=all + 同步刷盘（性能差）
- 允许少量丢失：acks=1（推荐）
- 红包等场景：丢失后补发比重复发放代价低

## 要点总结

1. 消息重复不可避免，核心是保证消费幂等性
2. 全局唯一ID + 去重表是通用幂等方案
3. 乐观锁适合金额/库存等业务场景
4. Kafka Producer Idempotency从生产端防重复
5. 根据业务容忍度选择acks级别，平衡可靠性和性能
