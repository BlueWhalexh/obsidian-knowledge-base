# 实现一个登录拉黑功能

## 问题
实现一个登录拉黑功能，实现拉黑用户和把已经登录的用户踢下线。

## 面试考察点
- 用户身份验证与会话管理：JWT/Session的踢下线机制
- 黑名单存储设计：Redis存储方案、TTL过期策略
- 登录拦截实现：Filter/Interceptor的请求拦截
- 分布式环境下的会话一致性：多实例间的状态同步

## 解题思路
> 核心需求：两个功能——登录拉黑（禁止登录）和踢下线（强制已登录用户失效）

登录拉黑功能需要解决两个核心问题：1）在登录时检查用户是否在黑名单中；2）对于已登录用户，需要使其当前会话立即失效。

技术选型上，黑名单使用Redis存储（支持TTL快速查询），会话管理也使用Redis（存储Token或Session ID），登录验证通过Spring Interceptor/Filter实现。关键设计是：拉黑时不仅要将用户加入黑名单，还要删除该用户在Redis中的所有活跃会话，这样后续请求在拦截器中验证时会发现会话已失效。

## 具体方案
### 方案一：Redis黑名单 + 会话管理
- 黑名单存储：SET blacklisted_user:{userId} "true" EX 86400
- 登录检查：登录接口中先查询Redis黑名单，存在则拒绝登录
- 踢下线：删除用户所有活跃会话（user_session:{userId}:{sessionId}）
- 拦截器验证：每次请求检查Token是否在黑名单中、会话是否仍有效

### 方案二：完整流程设计
- 用户登录 -> 检查黑名单 -> 生成Token -> 存储会话到Redis -> 返回Token
- 用户被拉黑 -> 加入黑名单Redis -> 删除所有活跃会话
- 后续请求 -> 拦截器检查黑名单 -> 检查会话有效性 -> 返回未授权

## 要点总结
- 黑名单和会话都用Redis存储，利用其高性能和TTL特性
- 拉黑操作需要同时处理：加入黑名单 + 删除所有活跃会话
- 拦截器是关键：在每个请求前检查黑名单和会话有效性
- 扩展能力：支持批量拉黑（Redis Pipeline）、自动解封（TTL过期）
