# 短URL生成器无冲突设计

## 问题
短URL生成器：百亿短URL怎样做到无冲突？

## 面试考察点
- 分布式唯一ID生成算法（Snowflake、Redis原子自增）
- 哈希冲突控制与概率分析
- 短URL编码策略（进制选择、长度计算）
- 系统水平扩展能力与存储方案

## 解题思路
> 核心思路：对比三种生成方案的冲突风险，选择预生成池化方案彻底消除冲突

三种短URL生成方案对比：
1. 单向散列函数（MD5/SHA256）：截取前6位可能冲突，冲突处理需多次查存储，无法保证性能
2. 自增长自然数：免冲突但可猜测，不满足不可预测需求
3. 预生成短URL池：预先用随机数生成所有短URL，用布隆过滤器去重，离线完成无性能问题

预生成方案的精髓：上线前用随机数预生成144亿条短URL（120亿需求+20%冗余），存储在HDFS中。运行时从池中直接取用，从根源上消除冲突。用户自定义短URL限制长度不等于6位，通过数据库唯一索引防冲突。

## 具体方案
### 方案一：预生成池化（Fuxi采用）
- 原理：随机数生成 + 布隆过滤器去重，离线预生成144亿条短URL
- 存储：HDFS文件存储（144亿 x 6B = 86.4GB），预加载服务器内存缓存
- 分配：预加载服务器从HDFS加载1万条到内存，用链表管理，剩余不足2000时异步补充
- 并发安全：利用HDFS偏移量文件写操作锁实现多服务器互斥
- 优点：彻底无冲突、高性能、不可猜测
- 缺点：需要预热、可能浪费ID资源

### 方案二：Snowflake变体+哈希校验混合
- 原理：雪花算法生成ID -> 转62进制 -> Redis setIfAbsent校验冲突
- 冲突处理：检测到冲突则重新生成（MD5换截取位置）
- 优点：分布式生成、无状态
- 缺点：仍有极小冲突概率（约0.0001%）

### 方案三：编码策略设计
- 6位Base64编码：64^6 ≈ 680亿组合，满足120亿需求
- URL安全编码：将Base64中的"+"改为"-"、"/"改为"_"，避免URL编码冲突
- 不可猜测性：随机数生成而非自增，无法通过一个短URL推算其他

## 要点总结
- 预生成池化是彻底无冲突的核心方案：离线随机生成 + 布隆过滤器去重
- 6位URL安全Base64编码支持680亿组合，满足百亿级需求
- 用户自定义短URL限制长度（不等于6位）+ 数据库唯一索引防冲突
- 存储架构：HDFS存预生成文件 -> 预加载服务器内存 -> Redis缓存热点 -> HBase持久化映射
- 使用302重定向（非301）以支持访问统计
