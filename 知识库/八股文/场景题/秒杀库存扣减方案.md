# 秒杀库存扣减方案

## 问题

秒杀场景下，如何安全高效地扣减库存？如何防止超卖？

## 一、秒杀场景特点

- **瞬时高并发**：几秒内涌入几十万请求
- **读多写少**：大量请求查询库存，少量请求实际下单
- **库存少**：商品数量有限（如 100 件），但请求量可能是库存的 1000 倍
- **要求强一致**：库存不能超卖（扣成负数）

---

## 二、整体架构

```
用户请求 → CDN/前端限流 → Nginx 限流 → 网关限流 → 应用服务 → Redis 预扣减 → MQ → DB 最终扣减
```

**核心思想**：层层拦截，最终只有少量请求到达数据库。

---

## 三、Redis 预扣减 + MQ 异步下单

### 3.1 整体流程

```
1. 秒杀开始前，将库存预热到 Redis
2. 用户请求到达 → Redis Lua 脚本原子扣减库存
3. 扣减成功 → 发送 MQ 消息
4. 消费者消费 MQ → 数据库最终扣减 + 创建订单
5. 扣减失败 → 返回"已售罄"
```

### 3.2 Redis Lua 脚本原子扣减

```java
/**
 * Lua 脚本：原子扣减库存
 * KEYS[1] = 库存 key（如 seckill:stock:1001）
 * KEYS[2] = 已购买用户集合 key（如 seckill:bought:1001）
 * ARGV[1] = 用户 ID
 */
private static final String STOCK_LUA = 
    "local stock = tonumber(redis.call('get', KEYS[1])) " +
    "if stock == nil then return -1 end " +  // 库存不存在
    "if stock <= 0 then return 0 end " +      // 库存不足
    "local bought = redis.call('sismember', KEYS[2], ARGV[1]) " +
    "if bought == 1 then return -2 end " +    // 重复购买
    "redis.call('decr', KEYS[1]) " +          // 扣减库存
    "redis.call('sadd', KEYS[2], ARGV[1]) " + // 记录已购买
    "return 1";                                // 扣减成功

public Long tryDeductStock(long skuId, long userId) {
    DefaultRedisScript<Long> script = new DefaultRedisScript<>();
    script.setScriptText(STOCK_LUA);
    script.setResultType(Long.class);
    
    return redisTemplate.execute(script,
        List.of("seckill:stock:" + skuId, "seckill:bought:" + skuId),
        String.valueOf(userId));
}
```

**为什么用 Lua 脚本？**

Redis 执行 Lua 脚本是原子操作，不会被其他命令打断。如果用 `get` + `decr` 两步操作，在高并发下可能出现：
1. 线程 A 读取库存 = 1
2. 线程 B 读取库存 = 1
3. 线程 A 扣减，库存 = 0
4. 线程 B 扣减，库存 = -1（超卖！）

### 3.3 MQ 异步下单

```java
@Service
public class SeckillService {
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;
    
    @Autowired
    private RocketMQTemplate mqTemplate;
    
    public Result seckill(long skuId, long userId) {
        // 1. Redis 预扣减
        Long result = tryDeductStock(skuId, userId);
        
        if (result == 1) {
            // 2. 扣减成功，发送 MQ 消息异步下单
            SeckillMessage message = new SeckillMessage(skuId, userId);
            mqTemplate.convertAndSend("seckill:order:topic", message);
            return Result.success("排队中，请稍后查看订单");
        } else if (result == 0) {
            return Result.fail("已售罄");
        } else if (result == -2) {
            return Result.fail("请勿重复购买");
        }
        return Result.fail("系统异常");
    }
}

// MQ 消费者
@Component
@RocketMQMessageListener(topic = "seckill:order:topic", 
                          consumerGroup = "seckill-order-group")
public class SeckillOrderConsumer implements RocketMQListener<SeckillMessage> {
    
    @Override
    public void onMessage(SeckillMessage message) {
        // 3. 数据库最终扣减 + 创建订单
        orderService.createSeckillOrder(message.getSkuId(), message.getUserId());
    }
}
```

### 3.4 数据库最终扣减

```java
@Transactional
public void createSeckillOrder(long skuId, long userId) {
    // 使用乐观锁扣减库存
    int affected = stockMapper.decrStock(skuId, 1);
    if (affected == 0) {
        // 库存不足（理论上不会发生，因为 Redis 已扣减）
        throw new RuntimeException("库存不足");
    }
    
    // 创建订单
    Order order = new Order();
    order.setSkuId(skuId);
    order.setUserId(userId);
    order.setStatus(OrderStatus.UNPAID);
    orderMapper.insert(order);
}
```

## 四、超卖问题解决方案

### 4.1 超卖原因

| 原因 | 说明 |
|------|------|
| 并发扣减 | 多个请求同时读取到相同的库存值 |
| 库存回滚异常 | 事务回滚但 Redis 库存已扣减 |
| Redis 与 DB 不一致 | Redis 扣减成功但 DB 写入失败 |

### 4.2 防超卖方案

**方案一：Redis Lua 脚本（推荐）**

原子操作，上面已详细说明。

**方案二：数据库乐观锁**

```sql
UPDATE stock SET quantity = quantity - 1 
WHERE sku_id = 1001 AND quantity > 0;
```

利用 `quantity > 0` 条件防止库存变负数。`affected rows = 0` 时说明库存不足。

**方案三：数据库悲观锁**

```sql
SELECT * FROM stock WHERE sku_id = 1001 FOR UPDATE;
-- 检查库存
UPDATE stock SET quantity = quantity - 1 WHERE sku_id = 1001;
```

**缺点**：性能差，不适合秒杀场景。

**方案四：Redis 原子操作 `DECR`**

```java
Long remaining = redisTemplate.opsForValue().decrement("stock:" + skuId, 1);
if (remaining >= 0) {
    // 扣减成功
} else {
    // 库存不足，回滚
    redisTemplate.opsForValue().increment("stock:" + skuId, 1);
}
```

**注意**：`DECR` 是原子的，但 `DECR` + 判断 + 回滚 不是原子操作。并发场景下仍可能超卖。所以推荐使用 Lua 脚本。

### 4.3 超卖方案对比

| 方案 | 原子性 | 性能 | 适用场景 |
|------|--------|------|---------|
| Redis Lua 脚本 | 是 | 极高 | 秒杀首选 |
| 数据库乐观锁 | 是 | 中 | 低并发通用 |
| 数据库悲观锁 | 是 | 低 | 不适合秒杀 |
| Redis DECR + 回滚 | 否 | 高 | 不推荐 |

## 五、其他优化手段

### 5.1 前端限流

```javascript
// 按钮点击后置灰，防止重复提交
btn.onclick = function() {
    btn.disabled = true;
    btn.textContent = "处理中...";
    fetch('/api/seckill', { method: 'POST' })
        .finally(() => {
            btn.disabled = false;
            btn.textContent = "立即抢购";
        });
};
```

### 5.2 Nginx 限流

```nginx
# 限制每秒 10 个请求
limit_req_zone $binary_remote_addr zone=seckill:10m rate=10r/s;

location /api/seckill {
    limit_req zone=seckill burst=20 nodelay;
    proxy_pass http://backend;
}
```

### 5.3 库存预热

```java
@PostConstruct
public void initStock() {
    // 秒杀开始前，将库存加载到 Redis
    List<SeckillProduct> products = productMapper.getAllSeckillProducts();
    for (SeckillProduct product : products) {
        redisTemplate.opsForValue().set(
            "seckill:stock:" + product.getSkuId(), 
            String.valueOf(product.getStock()));
    }
}
```

### 5.4 内存标记

在应用服务内存中维护一个"售罄标记"，避免大量请求打到 Redis。

```java
private Map<Long, Boolean> soldOutMap = new ConcurrentHashMap<>();

public Result seckill(long skuId, long userId) {
    // 1. 内存标记检查（最快的拦截）
    if (soldOutMap.getOrDefault(skuId, false)) {
        return Result.fail("已售罄");
    }
    
    // 2. Redis 扣减
    Long result = tryDeductStock(skuId, userId);
    if (result == 0) {
        soldOutMap.put(skuId, true); // 标记售罄
        return Result.fail("已售罄");
    }
    
    // ... 后续逻辑
}
```

## 常见追问

### Q：Redis 和 DB 库存不一致怎么办？

> 1. Redis 扣减成功但 DB 写入失败：消费端重试 + 对账机制
> 2. 定时任务对账：对比 Redis 和 DB 的库存，不一致时告警
> 3. 库存回补：订单超时未支付时，需要同时回补 Redis 和 DB 库存

### Q：秒杀商品数量特别多（如 10 万件）怎么办？

> 1. Redis 扣减完全能胜任（单机 10 万 QPS）
> 2. 主要瓶颈在 MQ 消费端和 DB 写入
> 3. 可以分多个队列并行消费
> 4. 批量扣减 DB（一次 SQL 扣减多条）

### Q：如果 Redis 挂了怎么办？

> 参考 [[Redis 挂了怎么办]]。秒杀场景下通常需要 Redis 高可用（哨兵/集群），如果 Redis 完全不可用，可以降级为数据库乐观锁扣减（牺牲性能保可用性）。
