# 秒杀系统设计

## 问题
如何设计一个秒杀系统？如何保证高并发下库存不超卖？

## 答案

### 秒杀场景特点

- 访问请求数量远大于库存数量，只有少部分用户能成功
- 瞬时流量激增，高并发冲击
- 业务流程简单：下订单、减库存

### 设计理念

- **限流**：限制大部分流量，只允许少部分进入后端
- **削峰**：将瞬间高流量变成平稳流量
- **异步处理**：提高系统并发量
- **内存缓存**：将数据和业务逻辑转移到内存，减少数据库IO
- **可扩展**：弹性扩展机器应对流量

### 前端方案

1. **页面静态化**：将活动页面的静态元素全部静态化，通过CDN抗峰值
2. **禁止重复提交**：用户提交后按钮置灰
3. **用户限流**：某一时间段内只允许提交一次请求，可采取IP限流

### 网关层方案

- 限制同一UserID的访问频率，防止恶意攻击
- 令牌桶限流算法控制请求速率

### 服务层方案（核心）

**请求处理流程**：

1. **商品信息初始化**：将秒杀商品信息和库存初始化到Redis缓存
2. **合法性校验**：检查用户是否登录，请求非法直接返回错误码
3. **秒杀状态判断**：内存标识判断秒杀是否已结束，已结束直接返回
4. **Redis预扣减库存**：使用`decr`命令预减库存，若库存量<0则标记秒杀结束

```lua
-- Redis Lua脚本保证原子性
local stock = redis.call('decr', KEYS[1])
if stock < 0 then
    return -1
end
return stock
```

5. **防重复秒杀**：使用Redis布隆过滤器判断用户是否已秒杀过
   - 布隆过滤器判断已秒杀 → 再次查库确认（布隆过滤器有误判率）
   - 布隆过滤器判断未秒杀 → 直接放行（无误判）
6. **异步下单**：发送MQ消息给业务端处理，返回"排队中"给客户端
7. **客户端轮询**：自动查询秒杀结果，直到返回成功或失败

**业务端处理**：
- 再次校验（秒杀是否结束、库存是否充足）
- 将用户和商品ID写入布隆过滤器
- 真正扣减数据库库存（`UPDATE SET stock = stock - 1 WHERE stock > 0`）
- 生成秒杀订单
- 返回秒杀成功

### 数据库层方案

- 使用乐观锁：`UPDATE SET stock = stock - 1 WHERE stock > 0`
- 添加用户ID和商品ID的联合唯一索引，防止重复下单
- 数据库层只承担"能力范围内"的请求

### 秒杀接口安全优化

- **隐藏接口**：点击秒杀按钮时生成加密串存入缓存，再次请求时携带加密串校验
- **频率限制**：拦截器配合自定义注解限制接口访问频率

### 整体架构

```
用户 → CDN（静态资源）→ Nginx（限流）→ 网关（令牌桶限流）
  → 服务层（Redis预扣减 + 布隆过滤器）→ MQ（异步）→ 业务服务（DB扣减）
```

### Redis挂了怎么办？

1. **监控**：基于Redis调用成功率做监控，发现长时间不可用
2. **降级**：降级为调用数据库
3. **限流**：数据库扛不住高并发，需在前置拦截加入限流
4. **高可用**：Redis哨兵、集群模式保证高可用
5. **开关机制**：提前配置开关，故障时一键切换备选方案

## 场景题补充：为什么不用分布式锁实现秒杀

面试常问："秒杀场景为什么不用分布式锁？"这道题考察对高并发场景下锁机制性能瓶颈的理解。

**真实事故案例**：某电商飞天茅台抢购（库存100瓶），使用分布式锁导致超卖P0事故。原因链：用户服务高负载 -> HTTP响应延迟超10s -> 分布式锁过期（10s）被其他线程获取 -> 释放了别人的锁 -> 非原子性库存校验失效 -> 超卖。

**分布式锁不适用的核心原因**：

1. **性能天花板**：每个请求需获取/释放锁，单Redis节点分布式锁QPS很难超过5000，而秒杀场景需要10万+QPS。锁操作增加10-50ms延迟。

2. **锁安全性隐患**：线程A执行时间超过锁过期时间 -> 线程B获取锁 -> 线程A执行完释放了线程B的锁 -> 恶性循环。即使用LUA脚本保证set/del一一映射，也无法保证业务绝对安全。

3. **与秒杀特性冲突**：瞬时并发100万QPS需要200个锁服务节点；锁服务故障导致全站不可用。

**更优方案**：
- **Redis原子操作**（decr）：QPS可达50000+，配合Lua脚本保证原子性
- **本地库存+异步**：ConcurrentHashMap预扣减 + MQ异步提交，QPS可达100000+
- **终极方案**：库存均摊到集群各服务器，网关层按用户ID hash路由，用本地缓存（AtomicInteger）扣减，完全不依赖Redis

> 性能对比实测：分布式锁4200 QPS/50节点 vs Redis原子操作58000 QPS/8节点 vs 混合方案120000 QPS/15节点。
