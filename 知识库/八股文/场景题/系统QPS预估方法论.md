# 如何预估一个系统的 QPS

## 问题
如何预估一个系统的 QPS？

## 面试考察点
- 容量规划能力：从用户量到请求量的量化分析能力
- 性能建模：建立合理计算模型的能力
- 全链路思维：从业务指标到技术指标的转换能力
- 实战经验：实际项目中的预估方法与压测验证

## 解题思路
> QPS 预估的核心是"业务指标转换 + 单机基准测试 + 流量模型 + 安全余量"，漏斗模型逐层衰减

预估分为五个步骤：评估总访问量（问产品/运营） -> 评估平均 QPS（总量除以 4 万秒，即白天时间） -> 评估高峰 QPS（平均的 3-4 倍） -> 压测单机极限 QPS -> 适当冗余度。关键经验公式：QPS = 总请求量 / 时间窗口，或 QPS = 并发数 / 平均响应时间。

已知的经验型参考数据：日志 QPS 6-10 万、RPC QPS 10 万、Redis QPS 8-10 万、MySQL 大致 6k-2 万。预测时分析一次请求有多少次写日志、读写底层资源、RPC 调用，取最低值即为系统最大 QPS 上限。漏斗模型适用于有转化链路的场景，如活动页 -> 商品详情 -> 下单，每层 QPS 逐步递减。

## 具体方案
### 五步预估法
1. **评估总访问量**：从产品/运营获取日活用户数和用户行为数据
2. **评估平均 QPS**：每日总请求量 / 40000 秒（白天时段）
3. **评估高峰 QPS**：平均 QPS x 3-4 倍（根据业务曲线）
4. **压测单机极限**：使用 wrk/JMeter 测出单机最大 QPS 和 P99 延迟
5. **计算集群容量**：所需机器数 = 高峰 QPS / (单机 QPS x 安全系数 0.6-0.7)

### 实际案例
- **社交平台**：DAU 500 万，人均 20 次/天 -> 每日 1 亿请求 -> 高峰 4 小时占 20% -> 高峰 QPS 约 1400 -> 单机 800 QPS -> 初始部署 4 台
- **Feed 系统**：日写 1000 万条，每条 10 人读 -> 写 463 QPS，读 4630 QPS -> 再乘 4 倍余量 -> 写 1852，读 18520

### 漏斗衰减模型
活动页（全部用户） -> 商品详情（约 60%） -> 下单（约 10%），每层 QPS 递减，按最底层设计数据库容量。

## 要点总结
- 一天按 4 万秒估算（仅白天），高峰 QPS 是平均的 3-4 倍
- 集群容量公式：高峰 QPS / (单机 QPS x 0.7)，留 30% CPU 余量
- 已知经验：Redis 8-10 万 QPS、RPC 10 万、MySQL 6k-2 万
- 取请求链路中最低资源的 QPS 作为系统上限
- 压测验证猜测，有经验的人看架构就能预估最大 QPS
