# 系统升级数据平滑迁移

## 问题

系统升级过程中如何实现数据的平滑迁移？如何保证可回滚？

## 面试考察点

- 双写方案（可回滚）
- 级联同步方案（简单易实施）
- 缓存预热策略
- 灰度切换

## 解题思路

数据迁移核心要求：**在线迁移**（不停服）、**数据完整**、**可回滚**。

## 具体方案

### 1. 双写方案（推荐）

**步骤**：
1. 新库配置为旧库从库，同步存量数据（可用Canal获取binlog同步到分库分表）
2. 业务代码改造：写入旧库的同时**异步写入新库**（写入失败记录日志）
3. **数据校验**：抽取部分数据验证一致性
4. **灰度切读流量**：先10% -> 50% -> 100%
5. 观察几天无问题后，**去掉旧库写入**，只写新库

**优势**：随时可回滚到旧库
**劣势**：周期长，有改造成本

### 2. 级联同步方案

**步骤**：
1. 新库配置为旧库从库
2. 备库配置为新库从库（保留在自建机房）
3. 三个库写入一致后，切读流量到新库
4. 低峰期暂停写入，切写流量到新库

**优势**：简单，业务无改造
**劣势**：切写时需短暂停止写入

### 3. 缓存预热

**问题**：空缓存会导致所有请求穿透到DB

**方案一：副本组预热**
- 云上部署副本组，从自建机房主从加载数据
- 缓存命中率达90%后切换

**方案二：改造副本组**
- 写入时异步写入云上缓存节点
- 读取时10%流量走云上缓存（可控的穿透）
- 命中率达标后全量切换

## 要点总结

1. 双写方案可随时回滚，风险最低，是推荐方案
2. 级联同步简单但切写时需停写，安排在低峰期
3. 灰度切换（10%->50%->100%）降低风险
4. 缓存预热是迁移的关键环节，避免DB被穿透打垮
5. 数据校验工具需提前开发和测试
