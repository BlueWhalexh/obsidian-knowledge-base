# 系统架构优化提升并发性能

## 问题

如何通过架构优化来提升系统并发能力？以直播互动场景为例。

## 面试考察点

- 可预估/不可预估流量的不同应对策略
- 客户端互动合并减少请求
- 服务端树形多层汇总架构
- 数据分片与动态扩容

## 解题思路

直播互动是典型的**读多写多**场景，无法简单用缓存解决。需要从架构层面做灵活拆分。

## 具体方案

### 1. 可预估用户量（游戏房间模式）
- 通过创建房间**预分配资源**
- 压测单台服务器承载人数，精确规划集群容量
- 调度服务根据各集群负载分配新用户

### 2. 不可预估用户量（直播场景）

**聊天：信息合并**
- 用户聊天内容放入分布式队列
- 大数据实时分析压缩重复内容（刷屏合并）
- 分发到多个内容分发服务器

**答题：瞬时拉取高峰**
- 客户端加入**随机延迟**再请求，分散峰值
- 题目预加载到客户端
- 失败重试使用**退火算法**

**点赞：客户端合并**
- 用户本地狂点100次 -> 合并为3秒内10次
- 大幅减少服务器请求量

**点赞：服务端树形多层汇总**
- 一层写缓存本地实时汇总
- 周期性汇总到二层
- 最终同步到核心缓存
- 适合一致性要求不高的计数器场景

### 3. 打赏/购物：数据分片
- 按用户ID hash分片
- **虚拟桶**方式：256个桶分配到N台服务器
- **树形热迁移**：扩容时新增从节点同步 -> 切换流量 -> 删除旧数据

### 4. 服务降级
- 分布式队列汇总缓冲
- 牺牲实时性换取可用性
- 客户端暂存点赞数据，服务器空闲时同步

## 要点总结

1. 可预估流量用房间模式精确控制，不可预估需要更灵活的架构
2. 客户端合并互动操作是最高效的减压手段
3. 树形多层汇总适合一致性要求不高的计数场景
4. 虚拟桶+热迁移实现数据分片的动态扩容
