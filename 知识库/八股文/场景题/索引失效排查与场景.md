# 索引失效排查与场景

## 问题
索引失效的问题是如何排查的，有哪些种情况？

## 面试考察点
- SQL优化能力：对索引工作原理的理解深度
- 问题诊断技巧：系统化排查问题的能力
- 实战经验：实际处理索引失效问题的经验
- 知识广度：对各类失效场景的掌握程度

## 解题思路
> 核心思路：先用EXPLAIN诊断，再对照失效场景逐一排查

排查索引失效的核心工具是EXPLAIN语句。重点关注三个字段：type列（ALL表示全表扫描）、key列（显示实际使用的索引）、Extra列（Using where/filesort/temporary等警告）。如果type=ALL、key=null、Extra=Using where，则说明索引失效。

排查时先确认索引是否存在且统计信息准确（SHOW INDEX、ANALYZE TABLE），再对照常见失效场景逐一分析。建立预防体系比事后排查更重要：制定索引使用规范、代码审查检查SQL写法、部署慢查询告警。

## 具体方案
### 方案一：EXPLAIN诊断四步法
- 执行计划分析：EXPLAIN查看type/key/extra
- 索引状态检查：SHOW INDEX确认索引存在、Cardinality准确
- 性能监控：慢查询日志定位问题SQL、performance_schema监控索引使用频率
- 统计信息更新：ANALYZE TABLE更新过期统计信息

### 方案二：八大失效场景及修复
- 隐式类型转换：字符串字段用数字查询 -> 确保类型一致
- 对索引列使用函数/运算：YEAR(create_time)=2023 -> 改为范围查询
- 违反最左前缀原则：复合索引跳过前导列 -> 调整查询条件或索引顺序
- 使用不等于(!=/ <>)：改写为范围查询
- LIKE以通配符开头：使用全文索引或改写为'张%'
- OR条件使用不当：改用UNION ALL或补充索引
- 优化器误判：FORCE INDEX或更新统计信息
- 字符集不匹配：统一字符集

## 要点总结
- 排查工具链：EXPLAIN + 慢查询日志 + performance_schema + SHOW INDEX
- EXPLAIN中type=ALL、key=null是索引失效的典型信号
- 最常见的失效原因：隐式类型转换、函数运算、违反最左前缀
- 建立预防体系比事后排查更重要：规范 + 审查 + 监控 + 告警
- 实战案例：对索引列使用DATE_FORMAT导致全表扫描，优化后QPS从50提升到2000
