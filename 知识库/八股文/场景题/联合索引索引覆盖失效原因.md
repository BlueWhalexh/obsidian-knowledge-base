# a,b联合索引 select b where a=xx无法走索引覆盖

## 问题
a,b的联合索引，select b where a = xx，无法走索引覆盖，什么原因？

## 面试考察点
- 索引覆盖原理：覆盖索引（Covering Index）的定义和工作机制
- 优化器决策逻辑：成本模型、统计信息对执行计划的影响
- 索引选择性：基数（Cardinality）对索引使用的影响
- 执行计划分析：EXPLAIN输出中Extra字段的含义

## 解题思路
> 核心理论：SELECT b FROM table WHERE a=xx中，a和b都在索引(a,b)中，应该走索引覆盖，但实际可能不走

不走索引覆盖的常见原因有：1）优化器选择回表——当a=xx匹配行数很多（如占20%以上），使用索引导致大量随机IO，优化器可能选择全表扫描；2）索引选择性过低——a列基数过低（如性别字段），匹配行数过多，优化器放弃索引；3）查询隐含其他字段——实际用了SELECT *或函数；4）存储引擎限制——MyISAM对覆盖索引支持不如InnoDB；5）包含排序或分组——ORDER BY/GROUP BY可能导致放弃覆盖索引；6）索引字段含NULL值。

排查方法：用EXPLAIN查看执行计划，关注Extra字段是否为"Using index"（覆盖索引）还是"Using index condition"（索引下推但未覆盖）。

## 具体方案
### 方案一：检查并优化执行计划
- EXPLAIN SELECT b FROM table WHERE a=xx，查看Extra字段
- 强制使用索引：SELECT b FROM table USE INDEX(idx_ab) WHERE a=xx
- 更新统计信息：ANALYZE TABLE table

### 方案二：调整索引和查询
- 若a选择性差，考虑添加更多字段到索引：(a, b, c)
- 避免查询中使用函数：WHERE DATE(a)='2023-01-01'改为范围查询
- 确保查询字段和条件字段均在索引中，避免SELECT *

## 要点总结
- EXPLAIN是排查索引问题的第一步，关注Extra字段
- 优化器基于成本决策，匹配行数过多时可能放弃索引覆盖
- ANALYZE TABLE更新统计信息可解决优化器误判
- 避免在索引列上使用函数，会导致索引失效
