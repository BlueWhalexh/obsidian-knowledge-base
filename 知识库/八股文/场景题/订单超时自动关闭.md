# 订单超时自动关闭

## 问题

用户下单后未在规定时间内支付（如30分钟），订单需要自动关闭。如何实现？

## 方案一：定时任务扫描

### 实现原理

定时任务每隔一段时间扫描数据库中"未支付且已超时"的订单，将其状态改为"已关闭"。

### 实现代码

```java
@Scheduled(cron = "0 */1 * * * ?") // 每分钟执行一次
public void closeTimeoutOrders() {
    // 查询未支付且创建时间超过30分钟的订单
    List<Order> timeoutOrders = orderDao.selectList(
        new QueryWrapper<Order>()
            .eq("status", OrderStatus.UNPAID)
            .le("create_time", LocalDateTime.now().minusMinutes(30))
    );
    
    for (Order order : timeoutOrders) {
        order.setStatus(OrderStatus.CLOSED);
        orderDao.updateById(order);
        // 恢复库存
        stockService.incrStock(order.getSkuId(), order.getQuantity());
    }
}
```

### 优化：使用时间轮扫描表

直接扫描订单大表性能差，可以维护一张"扫描表"：

```sql
CREATE TABLE order_timeout_scan (
    id BIGINT PRIMARY KEY,
    order_id BIGINT NOT NULL,
    timeout_time DATETIME NOT NULL,
    status TINYINT DEFAULT 0 COMMENT '0-待扫描 1-已处理',
    INDEX idx_timeout_time (timeout_time, status)
);
```

下单时同时插入一条记录到扫描表，定时任务只需扫描这张小表。

### 优缺点

| 维度 | 说明 |
|------|------|
| 优点 | 实现简单、容易理解、便于排查问题 |
| 缺点 | 有延迟（取决于扫描间隔）、频繁扫描影响 DB 性能 |
| 适用场景 | 订单量不大（日单量 < 10万）、对时效性要求不高 |

## 方案二：延迟队列

### 方案 2.1：RabbitMQ TTL + 死信队列

**原理**：

1. 消息设置 TTL（Time To Live），即消息的过期时间
2. 消息过期后进入死信队列（Dead Letter Exchange）
3. 消费者监听死信队列，收到消息后关闭订单

**架构图**：

```
下单 → 订单交换机(TTL=30min) → 订单队列(无消费者)
                                      ↓ (消息过期)
                                死信交换机 → 死信队列 → 消费者(关闭订单)
```

**代码实现**：

```java
// 1. 配置交换机和队列
@Configuration
public class RabbitMQConfig {
    
    // 普通交换机
    @Bean
    public DirectExchange orderExchange() {
        return new DirectExchange("order.exchange");
    }
    
    // 普通队列（设置TTL和死信交换机）
    @Bean
    public Queue orderQueue() {
        Map<String, Object> args = new HashMap<>();
        args.put("x-message-ttl", 30 * 60 * 1000); // 30分钟
        args.put("x-dead-letter-exchange", "order.dlx.exchange");
        args.put("x-dead-letter-routing-key", "order.close");
        return new Queue("order.queue", true, false, false, args);
    }
    
    // 死信交换机
    @Bean
    public DirectExchange orderDlxExchange() {
        return new DirectExchange("order.dlx.exchange");
    }
    
    // 死信队列
    @Bean
    public Queue orderDlxQueue() {
        return new Queue("order.dlx.queue");
    }
    
    // 绑定关系
    @Bean
    public Binding orderBinding() {
        return BindingBuilder.bind(orderQueue())
            .to(orderExchange()).with("order.create");
    }
    
    @Bean
    public Binding orderDlxBinding() {
        return BindingBuilder.bind(orderDlxQueue())
            .to(orderDlxExchange()).with("order.close");
    }
}

// 2. 下单时发送消息
public void createOrder(OrderDTO dto) {
    // 创建订单（状态：未支付）
    Order order = orderService.create(dto);
    // 发送延迟消息
    rabbitTemplate.convertAndSend("order.exchange", "order.create", 
        order.getId());
}

// 3. 监听死信队列
@RabbitListener(queues = "order.dlx.queue")
public void handleCloseOrder(Long orderId) {
    Order order = orderDao.selectById(orderId);
    if (order != null && order.getStatus() == OrderStatus.UNPAID) {
        order.setStatus(OrderStatus.CLOSED);
        orderDao.updateById(order);
        stockService.incrStock(order.getSkuId(), order.getQuantity());
    }
}
```

**注意**：RabbitMQ 的 TTL 是队列级别或消息级别。队列级别 TTL 更高效（消息到期后直接进死信队列），但一个队列只能有一个 TTL 值。如果不同订单有不同超时时间，需要使用消息级别 TTL（但此时消息不会立即被丢弃，而是在被消费时才检查是否过期，存在"队列头阻塞"问题）。

### 方案 2.2：RocketMQ 延迟消息

**原理**：RocketMQ 原生支持延迟消息，内置了 18 个延迟等级。

```java
// 发送延迟消息（延迟等级 9 = 30分钟）
Message msg = new Message("OrderTopic", "close", 
    orderId.toString().getBytes());
msg.setDelayTimeLevel(9); // RocketMQ 4.x 支持固定等级
producer.send(msg);
```

**RocketMQ 5.x 支持任意时间延迟**：

```java
msg.setDelayTime(30 * 60 * 1000); // 任意延迟时间（毫秒）
```

**优点**：原生支持、无需额外配置死信队列、性能好。

### 方案 2.3：Redis ZSet + 轮询

**原理**：

1. 下单时将订单 ID 和过期时间作为 score 存入 Redis ZSet
2. 定时任务（或独立线程）轮询 ZSet，取出已到期的订单进行关闭

```java
// 下单时
public void createOrder(OrderDTO dto) {
    Order order = orderService.create(dto);
    long timeout = System.currentTimeMillis() + 30 * 60 * 1000;
    redisTemplate.opsForZSet().add("order:timeout", 
        order.getId().toString(), timeout);
}

// 定时轮询（每秒执行）
@Scheduled(fixedRate = 1000)
public void checkTimeoutOrders() {
    long now = System.currentTimeMillis();
    Set<String> orderIds = redisTemplate.opsForZSet()
        .rangeByScore("order:timeout", 0, now);
    
    if (orderIds != null && !orderIds.isEmpty()) {
        for (String orderId : orderIds) {
            // 先移除再处理（原子操作）
            Long removed = redisTemplate.opsForZSet()
                .remove("order:timeout", orderId);
            if (removed != null && removed > 0) {
                closeOrder(Long.parseLong(orderId));
            }
        }
    }
}
```

**优化**：使用 Lua 脚本保证原子性

```lua
-- 取出并删除到期订单
local orders = redis.call('ZRANGEBYSCORE', KEYS[1], 0, ARGV[1])
if #orders > 0 then
    redis.call('ZREMRANGEBYSCORE', KEYS[1], 0, ARGV[1])
end
return orders
```

## 方案三：Redis Key 过期通知

**原理**：利用 Redis 的键空间通知（Keyspace Notification），当 key 过期时发布事件。

```java
// 订单创建时设置 Redis key
redisTemplate.opsForValue().set(
    "order:timeout:" + orderId, "1", 30, TimeUnit.MINUTES);

// 监听过期事件
@Configuration
public class RedisExpiredListener {
    @Bean
    public RedisMessageListenerContainer container(
            RedisConnectionFactory factory) {
        RedisMessageListenerContainer container = 
            new RedisMessageListenerContainer();
        container.setConnectionFactory(factory);
        container.addMessageListener(
            (message, pattern) -> {
                String key = new String(message.getBody());
                String orderId = key.replace("order:timeout:", "");
                closeOrder(Long.parseLong(orderId));
            },
            new PatternTopic("__keyevent@0__:expired")
        );
        return container;
    }
}
```

**为什么不推荐**：

1. **不可靠**：Redis 不保证过期事件一定被发送。如果消息丢失，订单永远不会被关闭
2. **性能问题**：大量 key 过期时会产生大量事件通知
3. **配置依赖**：需要开启 `notify-keyspace-events Ex` 配置
4. **集群不友好**：在 Redis Cluster 中，事件只在当前节点发布，需要监听所有节点

## 各方案优缺点对比

| 方案 | 延迟精度 | 可靠性 | 复杂度 | 性能 | 适用场景 |
|------|---------|--------|--------|------|---------|
| 定时任务扫描 | 低（分钟级） | 高 | 低 | 中（DB压力大） | 小规模系统、对精度要求低 |
| RabbitMQ TTL+死信 | 高（秒级） | 高 | 中 | 高 | 中大规模系统 |
| RocketMQ 延迟消息 | 高（秒级） | 高 | 低 | 高 | 中大规模系统（推荐） |
| Redis ZSet+轮询 | 高（秒级） | 中 | 中 | 高 | 已有 Redis 基础设施 |
| Redis 过期通知 | 高（秒级） | 低 | 低 | 中 | 不推荐用于生产 |

## 生产环境推荐

1. **小规模系统**：定时任务扫描即可，简单可靠
2. **中大规模系统**：RocketMQ 延迟消息（首选）或 RabbitMQ TTL+死信队列
3. **已有 Redis**：Redis ZSet + 轮询（需做好容错处理）
4. **兜底方案**：无论用哪种方案，都建议加一个定时任务作为兜底，处理因消息丢失等原因未被关闭的订单

## 常见追问

### Q：用户支付了但延迟消息已经发出去了怎么办？

> 消费者收到消息后需要先检查订单状态。如果订单已经是"已支付"状态，则直接忽略，不做任何处理。这是幂等性的体现。

### Q：订单关闭后需要恢复库存吗？

> 需要。关闭订单时需要将占用的库存归还。可以通过 MQ 异步处理，避免同步调用库存服务导致性能问题。

### Q：大量订单同时超时导致消息洪峰怎么办？

> 1. 在下单时为每个订单的超时时间加一个随机偏移量（如 ±30秒），打散消息
> 2. 消费者使用批量处理，一次取出多条消息批量关闭
> 3. 消费者使用线程池并发处理

## 场景题补充：为什么不建议用MQ

面试中常被追问："为什么不用MQ延迟队列实现订单超时关闭？"这道题考察对MQ能力边界和业务场景适配的理解。

**核心回答要点**：

1. **可靠性问题**：MQ消息投递存在三个不确定性——可能丢失（Broker宕机）、可能重复（网络重传）、可能乱序（分片场景）。订单关单需要强一致性，一次订单只能关一次，用MQ实现精确控制成本很高。

2. **时效性问题**：订单超时是精准时间控制（如30分钟），MQ延迟队列存在两个缺陷：时间精度依赖轮询（RocketMQ约1s级精度）；堆积时出现"队头阻塞"，导致本该到期的订单被延迟处理。

3. **状态同步问题**：订单有复杂状态流转（待支付->已支付->已发货），MQ触发关单时需要检查支付状态、处理部分退款等，业务逻辑耦合在消费者端很复杂。

4. **事务一致性问题**：订单支付和关单消息同时到达时，无论先处理哪个都需要做幂等，交叉场景用MQ很难优雅处理。

**更好的替代方案**：状态机+定时任务、时间轮算法（Netty HashedWheelTimer）、Redis ZSet+Lua脚本。这些方案能提供毫秒级精确控制、严格状态检查和可视化监控。

> 实际项目AB测试对比：MQ方案超时关单误差+-3秒/异常率0.5%，时间轮方案误差+-50ms/异常率0.01%。
