# 购物车系统设计

## 问题

如何设计购物车系统？登录前后数据如何衔接？存储选MySQL还是Redis？

## 面试考察点

- 暂存购物车 vs 用户购物车
- Cookie vs LocalStorage选型
- MySQL vs Redis存储对比
- 多端数据同步

## 解题思路

购物车系统看似简单，但**登录状态切换**和**多端同步**是核心设计难点。

## 具体方案

### 1. 两种购物车

**暂存购物车（未登录）**
- 保存在客户端（Cookie或LocalStorage）
- 无需服务端存储，无全局标识问题
- **Cookie**：实现简单，服务端可读写，但4KB上限
- **LocalStorage**：容量大，但只能客户端访问
- 小型电商用Cookie，大批量加购用LocalStorage

**用户购物车（已登录）**
- 保存在服务端，多端同步
- 存储方案：MySQL或Redis

### 2. 数据格式
```json
{
  "SKUID": 8888,
  "timestamp": 1578721136,
  "count": 1,
  "selected": true
}
```
- SKUID、数量、加购时间、勾选状态
- 商品价格/介绍从其他系统实时获取

### 3. 登录状态切换
- **未登录 -> 登录**：自动合并暂存购物车到用户购物车，清除暂存
- **登录 -> 关闭浏览器 -> 重新打开**：暂存购物车已清空，显示用户购物车
- **多端登录**：同一用户ID，各端看到同一购物车

### 4. MySQL vs Redis

| 维度 | MySQL | Redis |
|------|-------|-------|
| 性能 | 一般 | 高一个量级 |
| 可靠性 | 高 | 异步刷盘可能丢数据 |
| 查询方式 | 丰富 | 有限 |
| 事务 | 支持 | 有限 |

**推荐MySQL**：考虑需求变化（如统计今日加购总数），MySQL扩展性更好
**选择Redis**：追求极致性能，少量数据丢失可接受

### 5. Redis存储结构
- Key：用户ID
- Hash：field=商品ID, value={timestamp, count, selected}

## 要点总结

1. 两种购物车：暂存（客户端）+ 用户（服务端）
2. 登录时自动合并暂存购物车，清除暂存
3. Cookie适合小型电商，LocalStorage适合大批量场景
4. MySQL推荐（扩展性好），Redis可选（性能高但丢少量数据可接受）
5. 存储设计需权衡：数据可靠性 vs 性能 vs 扩展性
