# 跨库JOIN实现方案

## 问题
在微服务/分库分表架构下，数据分布在不同数据库实例中，如何实现跨库JOIN查询？

## 面试考察点
- 分布式系统理解：数据分布带来的查询挑战
- 解决方案广度：掌握多种跨库查询技术及其适用场景
- 性能优化意识：不同方案的性能代价评估
- 工程实践：实际落地时的一致性和扩展性考量

## 解题思路
> 跨库JOIN的本质问题是：数据不在同一个数据库实例中，无法直接用SQL JOIN

解决方案按数据量和实时性要求分层：小数据量用应用层内存关联；中等规模用数据库中间件透明代理；大规模用数据同步+宽表冗余；实时分析用联邦查询引擎。核心原则是"能不跨库就不跨库"，通过合理的数据归属设计减少跨库查询需求。

需要注意数据一致性问题：跨库查询的结果可能不是同一时刻的快照，尤其在并发写入场景下。

## 具体方案
### 方案一：应用层JOIN（最简单）
- 分别查询各库数据，在Java/Python内存中用HashMap关联
- 优点：实现简单，适用于任意数据源（SQL、NoSQL、API）
- 缺点：数据量大时内存压力大，一致性难保证
- 适用：几千条以内、实时性要求不高的场景

### 方案二：数据库中间件（ShardingSphere/MyCat）
- 中间件虚拟化为逻辑数据库，解析SQL自动路由到各物理库
- 支持跨库JOIN的SQL改写和结果归并
- 优点：应用无感知，兼容标准SQL
- 缺点：性能受限于中间件能力，复杂查询支持有限

### 方案三：数据冗余/宽表同步
- 将需要关联的数据通过Binlog/CDC同步到一个查询库，构建宽表
- 用Canal监听变更 → 消费者组装宽表 → 查询走宽表
- 优点：查询性能最好，无跨库开销
- 缺点：数据有延迟（秒级），需要维护同步链路

### 方案四：应用层聚合 + 缓存
- API Gateway层聚合多个微服务的查询结果
- 常用关联数据缓存到Redis，减少跨库查询频率
- 适用：微服务架构下跨服务查询

## 要点总结
- 优先通过合理的数据归属设计避免跨库JOIN，这是根本解决方案
- 小数据量（千级）用应用层JOIN，简单直接
- 中等规模用ShardingSphere等中间件，透明代理
- 大规模高并发用数据同步+宽表，牺牲实时性换查询性能
- 跨库查询结果不保证同一时刻快照，对一致性要求高的场景需额外处理
