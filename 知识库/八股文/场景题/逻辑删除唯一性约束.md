# 逻辑删除唯一性约束

## 问题
数据库逻辑删除后，唯一约束列插入相同值会报Duplicate entry，但业务上该值必须可重新插入，怎么解决？

## 面试考察点
- 业务设计能力：对逻辑删除场景的深入理解
- 数据库技能：对约束机制和索引的掌握
- 解决方案能力：复杂约束场景下的方案权衡
- 工程思维：线上已运行系统的改造成本考量

## 解题思路
> 问题本质：逻辑删除标记（is_deleted=1）的数据仍参与唯一约束校验

传统唯一索引不区分"已删除"和"未删除"数据，导致删除后无法重新插入相同值。核心思路是让唯一约束"忽略"已删除的数据。

有五种方案，按推荐程度：方案五（新增del_unique_key字段）和方案四（时间戳替代标记位）最适合线上系统；方案一（物理删除）需要业务支持；方案三（Redis判重）引入额外复杂度。

## 具体方案
### 方案一：复合唯一索引 + 删除标记特殊处理
- 原理：将 `is_deleted` 加入唯一索引 `(username, is_deleted)`，删除时将 `is_deleted` 设为该行主键id（保证唯一）
- 优点：全数据库兼容，改动最小
- 缺点：删除标记不再简单（0/1），需改删除逻辑

### 方案二：过滤索引/部分索引
- MySQL 8.0+ 函数索引 / PostgreSQL 部分索引 / SQL Server 过滤索引
- `CREATE UNIQUE INDEX uk ON users(username) WHERE is_deleted = 0`
- 优点：直观清晰
- 缺点：依赖数据库版本

### 方案三：删除时间戳替代标记位
- `del_flag` 字段用时间戳而非0/1，联合唯一索引 `(A, del_flag)`
- 删除时更新时间戳，每次不同，不冲突
- 适合新建业务，线上改造需改删除逻辑

### 方案四：新增 del_unique_key 字段（推荐线上使用）
- 新增字段默认值0，联合唯一索引 `(A, del_unique_key)`
- 删除时将 `del_unique_key` 设为该行主键id
- 优点：对已有业务影响最小，新增字段无需改现有逻辑

## 要点总结
- 逻辑删除破坏唯一约束的根本原因是删除标记值不变，唯一索引"看到"已删除数据
- 线上系统推荐新增字段方案（del_unique_key），改造成本最低
- 新建系统可直接用时间戳方案，逻辑更自然
- 恢复数据时需先校验是否存在活跃记录，避免违反约束
- 生产级方案：数据库复合索引 + 应用层校验 + 触发器兜底，三层防护
