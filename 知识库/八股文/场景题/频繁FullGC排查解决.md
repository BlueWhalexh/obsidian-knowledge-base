# 频繁FullGC排查解决

## 问题
线上JVM出现频繁FullGC，如何排查定位原因并解决？

## 面试考察点
- 对FullGC触发条件的全面理解（System.gc、Minor GC检查、大对象、长期引用）
- Heap Dump分析能力，能从dump文件定位内存问题根因
- JVM参数调优和GC日志分析能力

## 排查思路
> 先排除显式触发（System.gc、jmap），再通过Heap Dump分析是堆太小还是内存泄漏，最后针对具体对象类型定位代码问题。

## 具体排查步骤
1. **开启GC日志和Heap Dump**：配置`-XX:+HeapDumpBeforeFullGC -XX:HeapDumpPath=.`和`-Xloggc:gc.log`
2. **分析GC日志**：观察FullGC频率、老年代使用率、GC前后内存变化
3. **用MAT/jvisualvm分析Dump文件**：查看占用内存最多的对象类型
4. **追踪对象引用链**：找到是什么代码在持有这些对象的引用
5. **定位根因并修复**

## 常见原因与解决方案
| 原因 | 解决方案 |
|------|---------|
| 堆内存设置过小 | 适当增大-Xmx参数 |
| 大对象直接进入老年代 | 优化对象大小，避免大数组/大集合 |
| Druid等框架的SQL监控功能缓存大量SQL | 关闭不必要的监控功能 |
| ThreadLocal未清理 | 使用后及时调用remove() |
| 未分页的大数据量查询 | 添加分页限制 |
| 定时任务中List未清空每次追加 | 每次执行前清空集合 |
| 对象长期持有引用导致无法回收 | 检查缓存、集合是否无限增长 |

## 要点总结
- 分析Dump时用MAT建议勾选"Keep unreachable objects"，否则会过滤掉不可达对象导致分析不准确
- jvisualvm对OQL支持不好，需要OQL查询时建议用MAT
- Heap Dump操作会触发STW，线上建议分布式部署，只在一台机器开启
- 排查闭环：发现 -> 止血 -> 分析 -> 规避
