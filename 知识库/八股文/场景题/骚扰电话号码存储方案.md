# 大量手机号码被标记成骚扰电话如何存储

## 问题
大量的手机号码被标记成骚扰电话，如何存储这些号码？

## 面试考察点
- 数据结构选型能力：布隆过滤器 vs Redis vs 位图的适用场景
- 查询性能与存储效率的平衡：高频查询下的低延迟要求
- 隐私安全设计：避免明文存储手机号
- 扩展性设计：支持千万级数据量和水平扩展

## 解题思路
> 核心需求：高频实时查询（每秒数万次）+ 低存储成本 + 实时更新 + 隐私安全

需要兼顾五个维度：查询性能（毫秒级响应）、存储效率（数千万级号码）、更新能力（实时增删）、隐私安全（防泄露）、扩展性（水平扩展）。

推荐组合方案：布隆过滤器作为第一层快速过滤（内存占用小、查询O(k)），Redis作为第二层精确匹配（毫秒级响应、支持实时更新），MySQL作为持久化存储（完整数据、管理后台）。查询流程：布隆过滤器判断"可能存在" -> Redis精确查询 -> 返回结果。

## 具体方案
### 方案一：布隆过滤器 + Redis + MySQL（推荐）
- 布隆过滤器：快速判断号码是否可能存在（1亿号码约100MB内存，有小概率误判）
- Redis：缓存高频访问的骚扰号码，精确匹配，支持实时更新
- MySQL：持久化存储完整列表，定时同步到布隆过滤器和Redis
- 查询流程：布隆过滤器 -> Redis -> 返回结果

### 方案二：RoaringBitmap压缩位图
- 将手机号转为64位整数，使用RoaringBitmap存储
- 1亿号码仅需约12MB，极致空间效率
- 无误判，支持高效批量查询和交并差集操作
- 适合批量处理场景，不适合高频变更

## 要点总结
- 分层架构是关键：布隆过滤器（快速过滤）+ Redis（精确匹配）+ MySQL（持久化）
- 布隆过滤器空间效率极高但有误判，适合做第一层过滤
- Redis适合高频实时查询和更新，但内存成本较高
- 隐私保护：可对手机号哈希（SHA-256）或加密（AES）后存储
