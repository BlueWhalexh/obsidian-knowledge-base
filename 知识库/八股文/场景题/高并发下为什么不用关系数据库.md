# 高并发下为什么不用关系数据库

## 问题

高并发场景下关系数据库为什么成为瓶颈？有哪些替代方案？

## 面试考察点

- B+Tree存储结构与16KB Page
- LSM Tree写优化原理（RocksDB）
- 列存储适用场景（OLAP）
- HTAP融合趋势

## 解题思路

关系数据库的瓶颈源于其**存储结构**（B+Tree）和**事务机制**。

## 具体方案

### 1. B+Tree的局限
- InnoDB使用B+Tree索引
- 三层B+Tree约支撑**2000万行**数据
- 每次查询需要3次磁盘I/O（三层深度）
- **16KB Page**：单页存储有限，大量数据需要多页
- 高并发下随机I/O成为瓶颈

### 2. LSM Tree（写多读少场景）
- **写入**：先写内存（MemTable），满了刷磁盘（SSTable）
- **读取**：需要合并多个层级的SSTable
- **RocksDB**：典型的LSM Tree实现
- 写入性能极高（顺序写），读取需要多层查找
- 适合**写多读少**的日志/时序数据场景

### 3. 列存储（OLAP场景）
- 行存储：一行数据连续存储（适合OLTP）
- 列存储：同一列数据连续存储（适合OLAP）
- 列存储压缩率高，聚合查询快
- 不适合频繁的单行增删改

### 4. HTAP融合
- 同时支持OLTP和OLAP
- TiDB：分布式HTAP数据库
- 行列混合存储，TP请求走行存，AP请求走列存

## 要点总结

1. B+Tree三层约2000万行，高并发随机I/O是瓶颈
2. LSM Tree牺牲读性能换取写性能，适合写多读少
3. 列存储适合OLAP聚合查询，不适合OLTP
4. HTAP融合是趋势，TiDB是典型代表
5. 选型核心：读写比例决定存储引擎
