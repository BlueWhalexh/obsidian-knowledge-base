# 高并发库存抢购超卖解决

## 问题

高并发场景下如何解决库存超卖问题？有哪些技术方案？

## 面试考察点

- Redis原子操作（decr）+ 补偿
- 令牌库存（list + rpop）
- 拆分多个key减少锁争抢
- 自旋互斥锁
- CAS乐观锁（watch + multi + exec）
- Redis Lua脚本

## 解题思路

超卖本质是**并发写入导致的数据不一致**。解决方案从数据库层面到缓存层面逐步优化。

## 具体方案

### 1. Redis原子操作（decr）
```lua
if redis.call('get', KEYS[1]) > 0 then
    return redis.call('decr', KEYS[1])
else
    return -1
end
```
- decr是原子操作，天然防超卖
- 扣减成功后异步写入数据库
- 扣减失败返回-1，表示库存不足

### 2. 令牌库存（list + rpop）
- 将库存数量转化为令牌（list中的元素）
- 抢购 = rpop一个令牌
- rpop是原子操作，list为空即售罄
- 简单直观

### 3. 拆分多个key减少锁争抢
- 将1000库存拆为10个key，每个100
- 请求随机或轮询选择key扣减
- 分散热点，减少单key争抢

### 4. 自旋互斥锁
- 适用于数据库层面
- 获取锁失败则自旋重试
- 适合库存量小、并发不太高的场景

### 5. CAS乐观锁
```
watch stock
multi
decr stock
exec
```
- watch监控key，被其他客户端修改则事务失败
- 失败后重试

### 6. Redis Lua脚本（推荐）
- 将判断+扣减逻辑放在一个Lua脚本中
- Redis保证Lua脚本原子执行
- 无锁方案，性能最高

## 要点总结

1. Redis Lua脚本是生产环境首选，原子性+高性能
2. 令牌库存方案直观易懂，rpop天然原子
3. 拆分多个key分散热点，减少锁争抢
4. 数据库层面用CAS乐观锁兜底
5. 高并发下应尽量将库存操作前置到Redis
