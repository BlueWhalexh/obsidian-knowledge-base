# 过滤黑名单网址的系统如何实现用到哪些数据结构

## 问题
做一个过滤黑名单网址的系统，你觉得要怎么实现，会用到哪些数据结构？

## 面试考察点
- 数据结构选型能力：哈希表 vs 布隆过滤器 vs Trie树的适用场景
- 匹配精度控制：精确匹配 vs 通配符匹配（*.example.com）
- 动态更新能力：实时增删黑名单的性能影响
- 系统架构权衡：内存占用与查询效率的平衡、单机与分布式架构选择

## 解题思路
> 核心需求：高性能（每秒数万次查询）+ 低内存占用 + 支持通配符匹配 + 实时更新

需要根据不同匹配需求选择不同数据结构的组合。布隆过滤器作为第一层快速判断（内存极低、O(k)查询、有误判），Redis哈希表作为第二层精确匹配（O(1)查询、无误判），Trie树支持通配符匹配（*.example.com），MySQL作为持久化存储。

系统架构：请求 -> 布隆过滤器（可能存在？）-> Redis（精确匹配？）-> Trie树（通配符匹配？）-> 返回结果。这种分层架构既保证了查询性能，又支持了多种匹配模式。

## 具体方案
### 方案一：布隆过滤器 + Redis + Trie树 + MySQL
- 布隆过滤器（第一层）：快速判断URL是否可能存在，内存占用极低（1亿URL约100MB）
- Redis（第二层）：缓存高频访问的黑名单URL，精确匹配，毫秒级响应
- Trie树（第三层）：支持通配符匹配（*.example.com、/admin/*）
- MySQL（持久化）：存储完整黑名单数据，支持管理后台增删改查

### 方案二：AC自动机多模式匹配
- 将多个黑名单URL构建成AC自动机
- 一次扫描即可匹配所有黑名单模式
- 支持增量更新，无需重建整棵树
- 适合大规模多模式串匹配场景

## 要点总结
- 分层架构是核心：布隆过滤器（粗筛）-> Redis（精确）-> Trie（模糊）-> MySQL（持久化）
- 布隆过滤器适合大规模数据的快速过滤，但有误判且不支持删除
- Trie树是通配符匹配的最佳选择，但内存占用较高
- 生产效果：查询延迟从50ms降至0.1ms，内存占用降低85%
