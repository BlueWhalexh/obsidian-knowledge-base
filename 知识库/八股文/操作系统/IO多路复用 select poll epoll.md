# IO多路复用 select poll epoll

## 问题
什么是IO多路复用？select、poll、epoll有什么区别？为什么epoll性能最高？

## 答案

### IO多路复用概念

IO多路复用是一种高效的IO模型，允许单个线程同时监听多个文件描述符（FD），并在某个FD可读、可写或出现异常时得到通知。避免无效等待，充分利用CPU资源。

### select、poll、epoll对比

| 对比项 | select | poll | epoll |
|-------|--------|------|-------|
| **数据结构** | 固定大小数组（fd_set） | 链表 | 红黑树 + 就绪链表 |
| **最大连接数** | 1024（FD_SETSIZE限制） | 理论无限制 | 理论无限制 |
| **FD拷贝** | 每次调用将全部FD从用户空间拷贝到内核空间 | 每次调用将全部FD从用户空间拷贝到内核空间 | 使用mmap共享内存，无需重复拷贝 |
| **就绪检测** | 遍历所有FD找出就绪的 | 遍历所有FD找出就绪的 | 回调机制，只将就绪FD放入就绪链表 |
| **时间复杂度** | O(n) | O(n) | O(1) |
| **触发模式** | 水平触发（LT） | 水平触发（LT） | 支持LT和ET |

### select模式

```c
// 固定大小为1024的位图数组
fd_set read_fds;
FD_SET(fd, &read_fds);
select(max_fd + 1, &read_fds, NULL, NULL, NULL);
```

**特点**：
- 内核空间存储FD的结构是固定大小为1024的数组，最大并发连接数受限
- 当某个FD就绪时，从内核空间将所有FD拷贝到用户空间
- 用户空间需要遍历数组找出就绪的FD

### poll模式

```c
struct pollfd fds[1024];
fds[0].fd = fd;
fds[0].events = POLLIN;
poll(fds, 1024, -1);
```

**特点**：
- 内核空间存储FD的结构是链表，理论上没有最大连接数限制
- 当某个FD就绪时，从内核空间将所有FD拷贝到用户空间
- 用户空间需要遍历链表找出就绪的FD，链表越长遍历耗时越长

### epoll模式

```c
int epfd = epoll_create(1024);
struct epoll_event ev;
ev.events = EPOLLIN;
ev.data.fd = fd;
epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &ev);

struct epoll_event events[1024];
int nfds = epoll_wait(epfd, events, 1024, -1);
// events数组中只包含就绪的FD，直接处理
```

**特点**：
- 内核空间存储FD的结构是红黑树，增删改查性能稳定（O(logn)）
- 当有数据就绪时，通过回调机制将就绪FD加入就绪链表
- 只将就绪的FD从内核空间拷贝到用户空间，减少数据拷贝
- 用户空间无需遍历所有FD，直接处理返回的就绪FD

### 为什么epoll性能最高？

1. **无FD数量限制**：红黑树结构，不受1024限制
2. **减少FD拷贝**：使用mmap共享内存，避免每次调用都拷贝全部FD
3. **避免全量遍历**：回调机制直接将就绪FD放入链表，时间复杂度O(1)
4. **支持边缘触发**：减少epoll_wait调用次数

### LT模式与ET模式

| 模式 | 说明 | 特点 |
|------|------|------|
| **LT（水平触发）** | 只要FD处于就绪状态，每次epoll_wait都会通知 | 默认模式，编程简单，但可能重复通知 |
| **ET（边缘触发）** | 只在FD状态变化时通知一次 | 性能更高，但编程复杂，必须一次性读完数据 |

**ET模式注意事项**：必须配合非阻塞IO使用，且必须循环读取直到返回EAGAIN，否则可能丢失数据。
