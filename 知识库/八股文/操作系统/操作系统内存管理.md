# 操作系统内存管理

## 问题
什么是虚拟内存？页表和 TLB 是什么？段式管理和页式管理有什么区别？

## 答案

### 内存管理概述

操作系统的内存管理负责管理计算机的物理内存，主要目标：
1. **内存分配与回收**：为进程分配和回收内存空间
2. **地址转换**：将程序使用的虚拟地址转换为物理地址
3. **内存保护**：防止进程访问其他进程的内存空间
4. **内存扩充**：通过虚拟内存技术，让程序使用比物理内存更大的地址空间

### 连续内存管理 vs 非连续内存管理

| 对比项 | 连续内存管理 | 非连续内存管理 |
|-------|------------|--------------|
| **定义** | 进程占用一块连续的物理内存 | 进程的内存可以分散在物理内存的不同位置 |
| **优点** | 实现简单，访问速度快 | 灵活，减少内存碎片，支持虚拟内存 |
| **缺点** | 容易产生外部碎片，内存利用率低 | 实现复杂，地址转换需要额外开销 |
| **典型方式** | 固定分区、动态分区 | 段式管理、页式管理、段页式管理 |

**连续内存管理的问题**：
- **外部碎片**：内存中有许多小的空闲分区，但每个都太小，无法满足进程需求
- **内部碎片**：分配给进程的分区比进程实际需要的大，多出的部分被浪费
- **紧凑（Compaction）**：通过移动进程将空闲内存合并，但开销很大

---

### 段式管理（Segmentation）

段式管理按照程序的**逻辑结构**将内存分成若干个段。

**基本概念**：
- **段**：一个逻辑单元，如代码段、数据段、栈段、堆段
- 每个段有自己的段名、段长和基址（起始地址）
- 段的长度可以不同
- 地址表示：`(段号, 段内偏移)`

**地址转换**：
```
逻辑地址 = (段号 S, 段内偏移 D)

段表：
+--------+--------+--------+
| 段号 S | 段基址 | 段长   |
+--------+--------+--------+
| 0      | 1000   | 500    |
| 1      | 2000   | 300    |
| 2      | 5000   | 400    |
+--------+--------+--------+

物理地址 = 段基址 + 段内偏移
         = 1000 + D（如果 S=0）

检查：D < 段长，否则产生段越界中断
```

**优点**：
- 符合程序的逻辑结构，便于共享和保护
- 可以按段设置访问权限（如代码段只读、数据段可读写）
- 便于动态增长（如堆段、栈段）

**缺点**：
- 段的长度不固定，分配和回收复杂
- 容易产生外部碎片
- 段的大小受限于物理内存

---

### 页式管理（Paging）

页式管理将物理内存和逻辑地址空间都分成固定大小的**页**。

**基本概念**：
- **页（Page）**：逻辑地址空间的划分单位
- **页框（Page Frame / 物理块）**：物理内存的划分单位
- 页和页框大小相同（通常 4KB）
- 页的编号从 0 开始：第 0 页、第 1 页、...
- 地址表示：`(页号 P, 页内偏移 W)`

**地址转换**：
```
逻辑地址 = (页号 P, 页内偏移 W)

页表：
+--------+--------+
| 页号 P | 页框号 |
+--------+--------+
| 0      | 3      |
| 1      | 7      |
| 2      | 1      |
+--------+--------+

物理地址 = 页框号 × 页大小 + 页内偏移
         = 3 × 4096 + W（如果 P=0）

页大小 = 4KB = 4096 字节
页号 = 逻辑地址 / 页大小
页内偏移 = 逻辑地址 % 页大小
```

**优点**：
- 没有外部碎片（页的大小固定）
- 内存利用率高
- 便于实现虚拟内存（按需调页）

**缺点**：
- 有内部碎片（最后一个页可能未填满）
- 页表占用额外内存
- 不符合程序的逻辑结构（一个逻辑段可能跨越多个页）

---

### 段页式管理（Segmented Paging）

段页式管理结合了段式和页式的优点。

**基本概念**：
- 先将程序分成若干段（按逻辑结构）
- 再将每个段分成若干页（按固定大小）
- 地址表示：`(段号 S, 页号 P, 页内偏移 W)`

**地址转换**：
```
逻辑地址 = (段号 S, 页号 P, 页内偏移 W)

1. 根据段号 S 查段表，找到该段的页表起始地址
2. 根据页号 P 查页表，找到页框号
3. 物理地址 = 页框号 × 页大小 + 页内偏移 W

段表：
+--------+-----------+
| 段号 S | 页表地址  |
+--------+-----------+
| 0      | 0x1000    |
| 1      | 0x2000    |
+--------+-----------+

页表（段0的页表）：
+--------+--------+
| 页号 P | 页框号 |
+--------+--------+
| 0      | 5      |
| 1      | 2      |
+--------+--------+
```

**优点**：
- 兼有段式和页式的优点
- 便于共享和保护（按段）
- 没有外部碎片（按页）

**缺点**：
- 地址转换需要三次访存（段表 → 页表 → 数据）
- 实现复杂

---

### 虚拟内存（Virtual Memory）

虚拟内存是操作系统提供的一种内存管理技术，让程序使用比物理内存更大的地址空间。

#### 什么是虚拟内存

- 每个进程拥有独立的**虚拟地址空间**（如 32 位系统为 4GB）
- 程序使用**虚拟地址**访问内存，操作系统负责将虚拟地址转换为**物理地址**
- 不是所有虚拟地址都需要映射到物理内存，可以映射到磁盘（Swap 分区）
- 进程间虚拟地址空间相互隔离，互不影响

#### 虚拟内存的好处

1. **扩大可用内存**：程序可以使用比物理内存更大的地址空间
2. **内存隔离**：每个进程有独立的地址空间，互不影响
3. **简化编程**：程序员不需要关心物理内存的分配和管理
4. **共享内存**：多个进程可以映射到同一物理内存（如共享库）
5. **按需加载**：只加载当前需要的页面，减少启动时间和内存占用

#### 虚拟地址到物理地址的转换

```
虚拟地址 ──> 页表查询 ──> 物理地址
              |
              ├── 页在内存中：直接获取物理地址
              └── 页不在内存中：触发缺页中断
                    ├── 从磁盘加载页面到内存
                    ├── 更新页表
                    └── 重新执行指令
```

#### 缺页中断（Page Fault）

当程序访问的虚拟页面不在物理内存中时，触发缺页中断：

1. CPU 访问虚拟地址
2. MMU（内存管理单元）查询页表，发现页面不在内存中（有效位为 0）
3. 触发缺页中断，CPU 切换到内核态
4. 操作系统检查访问是否合法
5. 如果合法：
   - 从磁盘（Swap 分区或文件）加载页面到物理内存
   - 如果物理内存满了，需要选择一个页面换出（页面置换算法）
   - 更新页表
6. 重新执行触发缺页的指令

**页面置换算法**：
- **OPT（最佳置换）**：换出将来最长时间不会被访问的页面（理论最优，无法实现）
- **FIFO（先进先出）**：换出最早进入内存的页面（简单但效果差，有 Belady 异常）
- **LRU（最近最少使用）**：换出最近最长时间未被访问的页面（效果好但实现开销大）
- **Clock（时钟算法）**：LRU 的近似实现，使用访问位（Access Bit）
- **LFU（最不经常使用）**：换出访问次数最少的页面

### 页表（Page Table）

页表是虚拟地址到物理地址的映射表。

#### 单级页表的问题

32 位系统、4KB 页大小：
- 虚拟地址空间 = 4GB
- 页面数 = 4GB / 4KB = 2^20 = 1M 个页面
- 每个页表项 4 字节
- 页表大小 = 1M × 4B = **4MB**
- 每个进程都需要一个页表，100 个进程需要 400MB

#### 多级页表

为了解决页表过大的问题，使用多级页表（如二级页表）：

```
二级页表（32位系统）：
虚拟地址 = [一级页号(10位)] [二级页号(10位)] [页内偏移(12位)]

一级页表（页目录）：1024 个条目
  └── 每个条目指向一个二级页表

二级页表：1024 个条目
  └── 每个条目指向一个物理页框

优势：不需要为没有使用的虚拟地址空间分配二级页表
```

**多级页表的优势**：
- 只有被使用的虚拟地址空间才需要分配页表
- 大大减少了页表占用的内存
- 32 位系统中，二级页表通常只需要几 KB

**多级页表的缺点**：
- 地址转换需要多次访存（每级页表一次）
- 增加了地址转换的延迟

### TLB（Translation Lookaside Buffer）

TLB 是 MMU 中的高速缓存，用于加速地址转换。

#### 为什么需要 TLB

- 每次访问内存都需要查询页表
- 多级页表需要多次访存（如二级页表需要 3 次访存：2 次查页表 + 1 次访问数据）
- TLB 缓存最近使用的页表项，避免每次都查页表

#### TLB 的工作原理

```
CPU 发出虚拟地址
       |
       v
  查询 TLB ──> 命中 ──> 直接获取物理地址（1 个时钟周期）
       |
      未命中
       |
       v
  查询页表 ──> 更新 TLB ──> 获取物理地址（几十到几百个时钟周期）
```

**TLB 命中率**：
- TLB 通常有 64~1024 个条目
- 由于程序的**局部性原理**（时间局部性和空间局部性），TLB 命中率通常在 99% 以上
- TLB 命中率对系统性能影响很大

#### TLB 与上下文切换

- 进程切换时，TLB 中的条目可能对新进程无效
- 解决方案：
  - **全部刷新**：切换时清空 TLB（简单但性能损失大）
  - **ASID（Address Space ID）**：为每个进程分配唯一标识，TLB 条目带上 ASID，不同进程的条目可以共存

---

### 地址转换完整流程

```
CPU 访问虚拟地址
       |
       v
  查询 TLB
       |
  ┌────┴────┐
 命中      未命中
  |          |
  v          v
 直接获取   查询多级页表
 物理地址     |
  |       ┌──┴──┐
  |      找到   未找到（页不在内存）
  |       |        |
  |       v        v
  |    更新TLB   触发缺页中断
  |    获取物理    |
  |    地址       v
  |            从磁盘加载页面
  |            更新页表
  |            更新TLB
  |            获取物理地址
  |              |
  v              v
 访问物理内存中的数据
```
