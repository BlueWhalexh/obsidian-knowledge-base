# HTTP 与 HTTPS 的区别

## 问题
HTTP和HTTPS有什么区别？HTTPS的加密过程是怎样的？对称加密和非对称加密的区别是什么？HTTP各版本之间有什么区别？QUIC协议了解吗？

## 答案

### HTTP与HTTPS的核心区别

| 对比项 | HTTP | HTTPS |
|-------|------|-------|
| **传输方式** | 明文传输 | 加密传输（TLS/SSL） |
| **端口** | 80 | 443 |
| **安全性** | 不安全 | 安全，防窃听、防篡改、防冒充 |
| **证书** | 不需要 | 需要CA证书 |
| **性能** | 较快 | 有一定开销（握手阶段） |

---

### HTTP 1.0 vs HTTP 1.1

HTTP 1.1 是对 HTTP 1.0 的重大改进，主要区别如下：

| 对比项 | HTTP 1.0 | HTTP 1.1 |
|-------|----------|----------|
| **连接方式** | 短连接，每次请求都建立新的TCP连接，用完即关闭 | **持久连接（Keep-Alive）**，默认开启，一个TCP连接可以发送多个请求 |
| **管线化（Pipelining）** | 不支持，请求必须串行（上一个响应返回后才能发下一个） | 支持管线化，客户端可以连续发送多个请求，不必等前一个响应返回（但服务端仍按序响应，存在队头阻塞） |
| **缓存处理** | 使用 `Expires`（绝对过期时间）和 `If-Modified-Since`（协商缓存） | 新增 `Cache-Control`（相对过期时间，优先级更高）、`ETag`/`If-None-Match`（更精确的协商缓存） |
| **Host 头** | 不需要，每个IP只对应一个域名 | **必须携带 Host 头**，支持同一IP部署多个虚拟主机（同一服务器托管多个域名） |
| **带宽优化** | 不支持断点续传，每次传输完整资源 | 支持 `Range` 头部实现断点续传（状态码 206 Partial Content） |
| **错误处理** | 错误状态码较少 | 新增了更多状态码，如 409 Conflict、410 Gone 等 |

**持久连接详解**：
- HTTP 1.0 默认是短连接，每次请求都要经历 TCP 三次握手和四次挥手，开销很大
- HTTP 1.1 默认启用 `Connection: Keep-Alive`，TCP 连接在请求完成后不会立即关闭，后续请求复用同一连接
- 持久连接可以通过 `Connection: Close` 显式关闭
- 持久连接有超时时间，超过空闲时间后会自动关闭

**管线化详解**：
- HTTP 1.1 引入管线化，允许客户端在收到前一个响应之前就发送下一个请求
- 但服务端必须按照请求的顺序返回响应（FIFO），如果第一个请求处理很慢，后面的请求即使处理完了也得等着，这就是**队头阻塞（Head-of-Line Blocking）**
- 实际上由于队头阻塞和实现复杂性，大多数浏览器默认关闭了管线化
- HTTP 2.0 通过多路复用彻底解决了这个问题

---

### HTTP 1.1 vs HTTP 2.0

HTTP 2.0（2015年发布）是一次重大升级，核心目标是提升性能。

| 对比项 | HTTP 1.1 | HTTP 2.0 |
|-------|----------|----------|
| **传输格式** | 文本格式（可读性强但解析效率低） | **二进制帧（Binary Frame）**，更紧凑高效 |
| **多路复用** | 不支持，同一连接上请求串行，存在队头阻塞 | **同一TCP连接上可并行传输多个请求和响应**，彻底解决应用层队头阻塞 |
| **头部压缩** | 每次请求都携带完整的 Header，重复传输浪费带宽 | 使用 **HPACK 算法**压缩 Header，维护 Header 字典表，只传输差异部分 |
| **服务端推送** | 不支持，只能客户端主动请求 | **服务端可以主动推送资源**给客户端（如推送 CSS、JS 文件） |
| **流（Stream）** | 没有流的概念 | 每个请求/响应对应一个流，流之间独立，有唯一 ID |

**多路复用详解**：
- HTTP 1.1 中，浏览器对同一域名的并发连接数有限制（通常 6 个），超过就得排队
- HTTP 2.0 中，所有请求和响应都可以在一个 TCP 连接上并行传输
- 数据被拆分成一个个**帧（Frame）**，每个帧都有所属流的标识，不同流的帧可以交错发送，到达后重新组装
- 这就是"多路复用"——多条逻辑通道共享一个物理连接

**头部压缩详解（HPACK）**：
- HTTP 1.1 中每个请求都携带大量重复的 Header（如 Cookie、User-Agent），浪费带宽
- HTTP 2.0 使用 HPACK 算法：
  - 维护一个**静态表**（包含常见头部字段和值，如 `:method: GET`）
  - 维护一个**动态表**（记录本次连接中出现过的头部字段）
  - 使用**霍夫曼编码**压缩头部值
  - 对于已出现过的头部字段，只发送索引号，不发送完整内容

**服务端推送详解**：
- 客户端请求 `index.html`，服务端可以主动推送 `style.css` 和 `script.js`
- 推送的资源在客户端有缓存时可以拒绝
- 实际使用中，服务端推送的效果并不总是理想的，Chrome 已在 HTTP/3 中移除了对服务端推送的支持

**HTTP 2.0 的局限**：
- 虽然解决了应用层的队头阻塞，但 TCP 层的队头阻塞仍然存在
- 如果一个 TCP 包丢失，整个连接上的所有流都会被阻塞，即使其他流的数据已经到达
- 这也是 HTTP 3.0 要放弃 TCP 转用 UDP 的原因

---

### HTTP 2.0 vs HTTP 3.0

HTTP 3.0（基于 QUIC 协议）是一次根本性的架构变革。

| 对比项 | HTTP 2.0 | HTTP 3.0 |
|-------|----------|----------|
| **传输层协议** | TCP | **QUIC（基于 UDP）** |
| **队头阻塞** | TCP 层仍存在队头阻塞 | **彻底解决**，流之间完全独立，一个流丢包不影响其他流 |
| **连接建立** | TCP 三次握手 + TLS 握手 = 2~3 RTT | **0-RTT 或 1-RTT**（首次 1-RTT，后续 0-RTT） |
| **连接迁移** | 基于四元组（IP+端口），网络切换时连接断开 | 使用 **64 位 Connection ID** 标识连接，网络切换时连接不断开 |
| **加密** | TLS 可选 | **TLS 1.3 内置**，所有流量都加密 |
| **流量控制** | 基于 TCP 的流控 | 基于 QUIC 的流控，更灵活 |

**为什么 HTTP 3.0 要放弃 TCP**：
1. **TCP 的队头阻塞无法解决**：TCP 是可靠的字节流协议，必须保证数据有序交付。一个包丢失会阻塞整个连接上的所有数据
2. **TCP 握手延迟高**：TCP 三次握手 + TLS 握手至少需要 2~3 个 RTT
3. **TCP 连接与四元组绑定**：手机从 WiFi 切换到 4G 时，IP 地址变化导致连接断开，需要重新建立连接
4. **TCP 协议在内核中实现**：升级困难，部署新特性需要更新操作系统内核

---

### QUIC 协议详解

QUIC（Quick UDP Internet Connections）是 Google 开发的基于 UDP 的传输层协议，是 HTTP 3.0 的基础。

**核心特性**：

1. **基于 UDP**：
   - UDP 本身是不可靠的、无连接的传输协议
   - QUIC 在 UDP 之上实现了可靠传输（重传、拥塞控制、流量控制等原本 TCP 的功能）
   - 选择 UDP 的原因是：TCP 在内核中实现，改进困难；UDP 简单灵活，可以在用户态自由实现

2. **TLS 1.3 内置**：
   - QUIC 将 TLS 握手集成到连接建立过程中
   - 不像 TCP + TLS 那样需要先建立 TCP 连接再进行 TLS 握手
   - 首次连接只需 1-RTT（TCP + TLS 1.3 需要 2-RTT）
   - 会话恢复时支持 0-RTT（可以连握手带数据一起发送）

3. **多路复用，无队头阻塞**：
   - QUIC 中的流（Stream）是独立的，每个流有自己的序号和重传机制
   - 流 A 的丢包不会影响流 B 的数据交付
   - 彻底解决了 TCP 层的队头阻塞问题

4. **连接迁移**：
   - TCP 连接由四元组（源IP、源端口、目的IP、目的端口）标识
   - 网络切换时（如 WiFi → 4G），IP 地址变化，连接断开
   - QUIC 使用 64 位 Connection ID 标识连接，与网络地址无关
   - 网络切换时，只需将新地址与已有 Connection ID 关联，连接无缝迁移

5. **前向纠错（FEC）**：
   - QUIC 支持前向纠错，发送数据包时附带冗余信息
   - 少量丢包时可以通过冗余信息直接恢复，不需要重传
   - 适合实时性要求高的场景（如视频通话、游戏）

**QUIC 的 0-RTT 连接过程**：
```
首次连接（1-RTT）：
客户端                          服务端
  | ---- Client Hello + 支持的加密套件 ---> |
  | <--- Server Hello + 证书 + 选择的密钥 --- |
  | ---- 数据请求（可以随握手一起发送） ---> |

后续连接（0-RTT）：
客户端                          服务端
  | ---- 数据请求 + 缓存的密钥 ---> |  （无需等待握手，直接发送数据）
  | <--- 数据响应 ---------------- |
```

### HTTPS加密过程

HTTPS = HTTP + TLS/SSL，通过加密保证数据传输的安全性。加密过程分为以下 6 步：

**第1步：客户端发起请求（Client Hello）**
- 客户端向服务器发送请求，包含：
  - 支持的 TLS 版本列表（如 TLS 1.2、TLS 1.3）
  - 支持的加密算法列表（密码套件，如 RSA、ECDHE）
  - 客户端生成的随机数 **Client Random**

**第2步：服务器响应（Server Hello）**
- 服务器返回：
  - 选定的 TLS 版本和加密算法
  - 服务器生成的随机数 **Server Random**
  - 服务器的**数字证书**（包含服务器公钥、域名、CA 签名等）

**第3步：客户端验证证书**
- 客户端验证服务器证书的合法性：
  - 证书是否由受信任的 CA（证书颁发机构）签发
  - 证书是否在有效期内
  - 证书中的域名是否与访问的域名一致
  - 证书是否被吊销（通过 CRL 或 OCSP 检查）
- 验证通过后，从证书中提取服务器公钥

**第4步：密钥交换（Pre-Master Secret）**
- 客户端生成一个随机数 **Pre-Master Secret**
- 使用服务器的**公钥**通过**非对称加密**对 Pre-Master Secret 进行加密
- 将加密后的 Pre-Master Secret 发送给服务器
- 服务器使用**私钥**解密，获取 Pre-Master Secret
- （TLS 1.3 使用 ECDHE 密钥交换，双方各自生成临时密钥对，交换公钥后计算出共享密钥）

**第5步：生成会话密钥（Session Key）**
- 客户端和服务器各自使用三个随机数：
  - Client Random
  - Server Random
  - Pre-Master Secret
- 通过相同的算法生成**会话密钥（Session Key）**
- Session Key 是对称加密密钥，双方计算结果一致

**第6步：加密通信**
- 双方使用 Session Key 进行**对称加密**通信
- 后续所有数据传输都使用对称加密，速度快、开销小
- 会话结束后，Session Key 被丢弃，下次通信重新协商

```
客户端                                              服务端
  | ---- Client Hello (TLS版本, 加密算法, Client Random) --> |
  | <-- Server Hello (选定算法, Server Random, 数字证书) ---- |
  |        [客户端验证证书，提取公钥]                          |
  | ---- Pre-Master Secret (用公钥加密) ------------------> |
  |        [双方计算 Session Key]                            |
  | <========== 对称加密通信（使用 Session Key）==============> |
```

### 为什么HTTPS要同时使用两种加密算法？

HTTPS 结合了非对称加密和对称加密的优势，原因是：

**对称加密**：
- 加密和解密使用**相同的密钥**
- 优点：**速度快**，适合大量数据的加密传输
- 缺点：密钥需要在网络上传输，如果直接传输密钥，密钥可能被截获
- 典型算法：AES、DES、3DES

**非对称加密**：
- 使用**公钥加密、私钥解密**（或私钥签名、公钥验证）
- 优点：**安全性高**，公钥可以公开分发，私钥始终保留在服务器
- 缺点：**速度慢**，比对称加密慢 100~1000 倍，不适合大量数据加密
- 典型算法：RSA、ECC、DSA

**为什么不能只用一种**：
1. **只用对称加密**：密钥无法安全传输。如果直接在网络上发送密钥，攻击者可以截获密钥，后续所有通信都可以被解密
2. **只用非对称加密**：虽然安全，但速度太慢，会严重影响通信性能。每个字节都需要进行复杂的数学运算
3. **两者结合**：用非对称加密安全地交换对称密钥（少量数据，速度可接受），然后用对称密钥加密后续通信数据（大量数据，速度快）

**总结**：非对称加密是"密钥的安全快递员"，对称加密是"数据的高速传输通道"。HTTPS 先用非对称加密安全地交换密钥，再用对称加密高效地传输数据。

### CA证书

CA（Certificate Authority）是数字证书颁发机构。服务器向CA申请证书，证书中包含服务器公钥和域名等信息。客户端通过验证CA证书来确认服务器身份的真实性，防止中间人攻击。

**证书链**：浏览器信任根CA → 根CA签发中间CA证书 → 中间CA签发服务器证书。验证时从服务器证书向上追溯到根CA。
