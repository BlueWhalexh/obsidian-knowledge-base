# TCP 三次握手与四次挥手

## 问题
请详细描述TCP三次握手和四次挥手的过程，为什么握手是三次而不是两次？为什么挥手是四次？TCP如何保证可靠性？什么是TCP粘包和拆包？

## 答案

### TCP三次握手

TCP是面向连接的协议，使用前必须先建立连接。初始时客户端和服务端都处于Close状态，服务端主动监听端口处于Listen状态。

```
客户端                              服务端
  |  ---- SYN (client_isn) ----->   |   SYN_Sent
  |  <-- SYN+ACK (server_isn) ----  |   SYN_Received
  |  ---- ACK ----->                 |   Established
  |                                  |   Established
```

**过程**：

1. **第一次握手**：客户端随机初始化序列号`client_isn`，将SYN标志位置为1，发送SYN报文，之后客户端处于`SYN_Sent`状态
2. **第二次握手**：服务端收到SYN报文，随机初始化自己的序列号`server_isn`，将应答号置为`client_isn + 1`，SYN和ACK标志位置为1，回复SYN+ACK报文，之后服务端处于`SYN_Received`状态
3. **第三次握手**：客户端收到SYN+ACK报文，将应答号置为`server_isn + 1`，ACK标志位置为1，回复ACK报文，之后客户端处于`Established`状态。服务端收到ACK后也进入`Established`状态

**注意**：第三次握手可以携带数据，前两次握手不能携带数据。

---

### 为什么是三次握手而不是两次？

1. **防止建立历史连接**：客户端发送的 SYN 报文可能在网络中滞留。如果是两次握手，服务端收到滞留在网络中的无效历史 SYN 报文后，会直接分配资源建立连接并回复 SYN+ACK，但客户端并不会理会这个连接，导致服务端白白浪费资源。三次握手时，客户端收到服务端对历史 SYN 的响应后，会发送 RST 报文拒绝连接，避免资源浪费。

2. **同步双方初始序列号**：TCP 需要双方各自确认对方的序列号。两次握手时，服务端无法确认它发送的序列号（server_isn）客户端是否已成功接收。三次握手中，客户端的第三次 ACK 明确告知服务端"我已经收到了你的序列号"，确保双方的序列号可靠同步。

3. **避免重复连接的初始化**：如果是两次握手，失效的连接请求到达服务端后，服务端会立即建立连接并开始发送数据，但客户端并未请求该连接，会造成混乱。

### TCP四次挥手

```
客户端                              服务端
  |  ---- FIN ----->                 |   FIN_Wait_1
  |  <-- ACK ----                    |   Close_Wait
  |                                  |   (发送完数据后)
  |  <-- FIN ----                    |   Last_ACK
  |  ---- ACK ----->                 |   TIME_WAIT → Close
  |                                  |   Close
```

**过程**：

1. **第一次挥手**：客户端将FIN标志位置为1，发送FIN报文表示关闭连接，之后客户端处于`FIN_Wait_1`状态
2. **第二次挥手**：服务端收到FIN报文，回复ACK报文，之后服务端处于`Close_Wait`状态。客户端收到ACK后处于`FIN_Wait_2`状态
3. **第三次挥手**：服务端发送完剩余数据后，发送FIN报文给客户端，之后服务端进入`Last_ACK`状态
4. **第四次挥手**：客户端收到FIN报文，回复ACK报文，之后客户端进入`TIME_WAIT`状态（等待2MSL后关闭）。服务端收到ACK后关闭连接

---

### 为什么挥手需要四次？

TCP 是**全双工**通信，数据可以在两个方向上同时传输。关闭连接时，每个方向都需要单独关闭：

- **第一次挥手**：客户端告诉服务端"我没有数据要发了"（客户端→服务端方向关闭）
- **第二次挥手**：服务端确认收到"我知道你没有数据要发了"
- **第三次挥手**：服务端告诉客户端"我也没有数据要发了"（服务端→客户端方向关闭）
- **第四次挥手**：客户端确认收到"我知道你也没有数据要发了"

**为什么 ACK 和 FIN 不能合并**：
- 服务端收到客户端的 FIN 后，可能还有数据没有发送完
- 所以服务端先回复 ACK 告诉客户端"我知道你要关了"，但自己这边继续发送剩余数据
- 等数据发完后，再发送 FIN 告诉客户端"我这边也发完了"
- 因此 ACK 和 FIN 通常是分开发送的，形成四次挥手

**特殊情况**：如果服务端收到 FIN 时恰好没有数据要发送，可以将 ACK 和 FIN 合并成一次发送（三次挥手），但这种情况较少见。

---

### 为什么有TIME_WAIT状态？为什么是2MSL？

**TIME_WAIT 的作用**：

1. **确保最后的 ACK 能到达服务端**：如果客户端发送的最后一个 ACK 丢失，服务端会重发 FIN。TIME_WAIT 状态下客户端可以重新发送 ACK，确保服务端正常关闭连接。

2. **防止旧连接的数据干扰新连接**：经过 2MSL 时间后，本次连接的所有报文都会在网络中消失。如果立即建立新连接，旧连接中滞留的延迟报文可能被新连接误收，导致数据混乱。

**为什么是 2MSL（两倍的最大报文生存时间）**：
- MSL（Maximum Segment Lifetime）是报文在网络中的最大生存时间
- 客户端发送 ACK 后，如果 ACK 丢失，服务端会在 1MSL 后重发 FIN
- FIN 到达客户端最多需要 1MSL
- 所以等待 2MSL = ACK 丢失的检测时间（1MSL）+ FIN 到达的时间（1MSL）
- 这样可以确保最后一个 ACK 和可能重传的 FIN 都有足够时间在网络中传输

---

### TIME_WAIT过多怎么处理？

在高并发场景下，大量 TIME_WAIT 状态的连接会占用系统资源（端口、内存）。

**TIME_WAIT 过多的危害**：
- 占用大量端口资源（默认端口范围 32768~61000）
- 占用内存（每个 TCP 连接约占 300 字节）
- 导致新连接无法建立

**解决方案**：

1. **调整 `tcp_max_tw_buckets` 参数**：
   - 系统允许的最大 TIME_WAIT 连接数，超过后新的 TIME_WAIT 连接会被直接关闭
   - `sysctl -w net.ipv4.tcp_max_tw_buckets=5000`
   - 缺点：可能导致连接异常关闭

2. **开启 `tcp_tw_reuse`（TIME_WAIT 复用）**：
   - 允许将 TIME_WAIT 状态的端口用于新的出站连接
   - `sysctl -w net.ipv4.tcp_tw_reuse=1`
   - 前提：需要开启时间戳（`tcp_timestamps=1`），通过时间戳判断旧连接的报文
   - 只对客户端（主动发起连接方）有效

3. **开启 `tcp_tw_recycle`（快速回收，已废弃）**：
   - Linux 4.12 版本后已移除此选项
   - 在 NAT 环境下会导致问题：同一内网的不同主机时间戳不一致，导致合法连接被拒绝

4. **使用长连接**：
   - 通过 HTTP Keep-Alive 或 TCP Keep-Alive 复用连接
   - 减少连接建立和关闭的频率

5. **使用连接池**：
   - 应用层维护连接池，避免频繁创建和销毁连接

### TCP可靠性保证

TCP 通过多种机制确保数据传输的可靠性：

**1. 序列号（Sequence Number）**
- 每个字节的数据都有唯一的序列号
- 接收方通过序列号对数据进行排序，确保数据有序
- 可以检测数据是否丢失或重复

**2. 确认应答（ACK）**
- 接收方收到数据后，返回 ACK 确认，告诉发送方"我已经收到了哪些数据"
- ACK 中的确认号表示"这个序号之前的数据我都收到了"
- 累积确认：确认号 N 表示序号 N 之前的所有数据都已收到

**3. 超时重传**
- 发送方发送数据后启动定时器
- 如果在超时时间内没有收到 ACK，认为数据丢失，重新发送
- 超时时间（RTO）根据网络延迟动态计算：`RTO = RTT + 4 * RTT_Dev`（RTT 是往返时间，RTT_Dev 是偏差）

**4. 滑动窗口**
- 发送方维护一个发送窗口，窗口内的数据可以连续发送而无需等待 ACK
- 窗口大小由接收方告知（通过 TCP 头部的 Window 字段）
- 滑动窗口提高了传输效率，避免了每个数据包都等待 ACK

**5. 流量控制**
- 接收方通过 TCP 头部的**窗口大小字段（Window Size）**告知发送方自己的接收缓冲区剩余空间
- 发送方的发送速率不能超过接收方的处理能力
- 如果接收方缓冲区满了，窗口大小为 0，发送方暂停发送
- 发送方会定期发送**窗口探测报文**，询问接收方窗口是否恢复

**6. 拥塞控制**
- 防止网络拥塞导致大量丢包
- 发送方维护一个**拥塞窗口（cwnd）**，实际发送窗口 = min(拥塞窗口, 接收方窗口)
- 拥塞控制算法：慢启动、拥塞避免、快重传、快恢复（详见下文）

---

### 拥塞控制详解

TCP 拥塞控制的目的是避免网络拥塞，通过动态调整发送速率来适应网络状况。

**核心变量**：
- `cwnd`（Congestion Window）：拥塞窗口大小，由发送方维护
- `ssthresh`（Slow Start Threshold）：慢启动阈值，初始值通常为 65535 字节
- 实际发送窗口 = `min(cwnd, rwnd)`（rwnd 是接收方窗口）

**1. 慢启动（Slow Start）**
- 连接建立初期，cwnd 从 1 MSS（最大报文段大小）开始
- 每收到一个 ACK，cwnd 增加 1 MSS（指数增长：1 → 2 → 4 → 8 → 16...）
- 当 cwnd >= ssthresh 时，进入拥塞避免阶段
- 目的：快速探测网络可用带宽

**2. 拥塞避免（Congestion Avoidance）**
- cwnd 增长方式变为线性增长（每个 RTT 增加 1 MSS）
- 当检测到丢包（超时或收到 3 个重复 ACK）时：
  - 超时：ssthresh = cwnd / 2，cwnd = 1 MSS，重新进入慢启动
  - 3 个重复 ACK：进入快重传

**3. 快重传（Fast Retransmit）**
- 当发送方连续收到 3 个重复的 ACK 时，立即重传丢失的报文（不等超时）
- 3 个重复 ACK 意味着网络可能只是轻微拥塞，不需要大幅降低发送速率

**4. 快恢复（Fast Recovery）**
- 快重传后进入快恢复阶段：
  - ssthresh = cwnd / 2
  - cwnd = ssthresh + 3 MSS（加上 3 个重复 ACK 表示有 3 个数据包已离开网络）
  - 每收到一个重复 ACK，cwnd 增加 1 MSS
  - 收到新的 ACK 后，cwnd = ssthresh，进入拥塞避免阶段
- 目的：避免像超时那样将 cwnd 降到 1 MSS，快速恢复传输

**拥塞控制状态机**：
```
慢启动 ──(cwnd >= ssthresh)──> 拥塞避免
   ^                              |
   |                              |
   └──(超时丢包)──────────────────┘
                    |
                    ├──(3个重复ACK)──> 快重传 ──> 快恢复 ──> 拥塞避免
```

### TCP粘包和拆包

**什么是粘包和拆包**：

TCP 是**面向字节流**的传输协议，没有消息边界的概念。发送方多次发送的数据可能被合并成一个包发送（粘包），也可能被拆分成多个包发送（拆包）。

```
粘包示例：
发送方：[消息A][消息B][消息C]
接收方可能收到：[消息A消息B][消息C]  （多条消息粘在一起）

拆包示例：
发送方：[大数据包]
接收方可能收到：[数据包前半部分][数据包后半部分]  （一条消息被拆开）
```

**产生原因**：
1. **Nagle 算法**：TCP 为了提高效率，将多个小数据包合并成一个大包发送
2. **接收方缓冲区**：应用层读取数据时，可能一次读取了多条消息或只读取了部分消息
3. **MSS/MTU 限制**：数据包大小超过 MSS（最大报文段大小）时会被拆分

**解决方案**：

1. **固定长度**：
   - 每条消息固定长度，不足部分用填充字符补齐
   - 接收方每次读取固定长度的数据
   - 缺点：浪费带宽，不灵活

2. **分隔符**：
   - 在每条消息末尾添加特殊分隔符（如 `\n`、`\r\n`）
   - 接收方通过分隔符来划分消息边界
   - 缺点：消息内容不能包含分隔符字符，需要转义

3. **长度字段（最常用）**：
   - 在消息头部添加长度字段，表示消息体的长度
   - 接收方先读取长度字段，再根据长度读取对应长度的消息体
   - 这是大多数应用协议的做法（如 HTTP 的 Content-Length、自定义协议的 TLV 格式）

```
长度字段方案：
+--------+--------+
| 长度(4字节) | 消息体 |
+--------+--------+
```

**Java 中的处理**：
- Netty 框架提供了 `LengthFieldBasedFrameDecoder`，基于长度字段自动处理粘包拆包
- 也可以使用 `DelimiterBasedFrameDecoder`（分隔符）或 `FixedLengthFrameDecoder`（固定长度）

### 补充：为什么客户端关闭后还能接收数据？

客户端发送 FIN 后进入 FIN_WAIT_1 状态，只是表示"我不再发送数据"，但仍可以接收服务端发送的数据。这就是 TCP 全双工的体现：关闭发送通道不影响接收通道。
