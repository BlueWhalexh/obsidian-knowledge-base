# 半连接队列与全连接队列

## 问题
什么是半连接队列和全连接队列？SYN 攻击的原理是什么？如何防御？

## 答案

### 概述

TCP 三次握手过程中，服务器维护了两个队列：
- **半连接队列（SYN Queue / Incomplete Connection Queue）**：存储收到了 SYN 但还没收到 ACK 的连接
- **全连接队列（Accept Queue / Complete Connection Queue）**：存储完成了三次握手、等待 `accept()` 系统调用取走的连接

### 三次握手与两个队列的工作流程

```
客户端                         服务端
  |                              |
  | ---- SYN (seq=x) ---------> |
  |                              | 1. 服务端收到 SYN
  |                              | 2. 创建连接放入 半连接队列（SYN Queue）
  |                              | 3. 回复 SYN+ACK
  | <--- SYN+ACK (seq=y,ack=x+1) |
  |                              |
  | ---- ACK (ack=y+1) -------> |
  |                              | 4. 服务端收到 ACK
  |                              | 5. 将连接从 半连接队列 移到 全连接队列（Accept Queue）
  |                              |
  |                              | 6. 应用程序调用 accept() 从全连接队列取出连接
  |                              |
```

**详细流程**：

1. **客户端发送 SYN**：客户端发起连接请求，发送 SYN 报文
2. **服务端收到 SYN**：
   - 服务端收到 SYN 后，创建一个连接对象
   - 将连接放入**半连接队列（SYN Queue）**
   - 设置定时器（SYN Timeout），如果超时未收到 ACK 则重传 SYN+ACK
   - 回复 SYN+ACK
3. **客户端发送 ACK**：客户端收到 SYN+ACK 后，发送 ACK 确认
4. **服务端收到 ACK**：
   - 服务端收到 ACK 后，将连接从**半连接队列**移到**全连接队列（Accept Queue）**
5. **应用层取走连接**：
   - 应用程序调用 `accept()` 系统调用，从全连接队列中取出连接
   - 取出后，全连接队列腾出一个位置

### 队列满时的行为

#### 半连接队列满

当半连接队列满时，新到达的 SYN 包会被丢弃或根据 `tcp_syncookies` 配置决定：

- **tcp_syncookies = 0**：直接丢弃 SYN 包
- **tcp_syncookies = 1（默认）**：启用 SYN Cookie 机制，不使用半连接队列
  - 服务端根据 SYN 包的信息计算出一个特殊的序列号（SYN Cookie）作为 SYN+ACK 的序列号
  - 不在半连接队列中存储连接信息
  - 收到 ACK 时，通过序列号反推出连接信息，直接放入全连接队列
  - **缺点**：丢失了 TCP 选项信息（如窗口缩放、时间戳）

#### 全连接队列满

当全连接队列满时，根据 `tcp_abort_on_overflow` 配置决定：

- **tcp_abort_on_overflow = 0（默认）**：
  - 服务端直接丢弃 ACK，不回复 RST
  - 客户端会重传 ACK（重试次数由 `tcp_retries2` 控制，通常约 15 次）
  - 如果在重传期间全连接队列腾出位置，连接仍可建立
  - **问题**：客户端不知道服务端是否收到了 ACK，会一直重传

- **tcp_abort_on_overflow = 1**：
  - 服务端回复 RST（Reset）报文，直接拒绝连接
  - 客户端收到 RST 后，连接建立失败
  - **优点**：客户端能立即知道连接失败，可以快速重试

### SYN 攻击（SYN Flood）

#### 原理

SYN 攻击是一种典型的 DoS（拒绝服务）攻击：

1. 攻击者伪造大量不同的源 IP 地址
2. 向服务器发送大量 SYN 报文
3. 服务器为每个 SYN 分配资源，放入半连接队列
4. 服务器回复 SYN+ACK，但由于源 IP 是伪造的，永远不会收到 ACK
5. 半连接队列被占满，正常的连接请求无法被处理
6. 服务器资源耗尽，无法提供服务

```
攻击者（伪造IP）               服务端
  | ---- SYN (伪造IP1) ------> |  放入半连接队列
  | ---- SYN (伪造IP2) ------> |  放入半连接队列
  | ---- SYN (伪造IP3) ------> |  放入半连接队列
  | ...                        |
  |                            |  半连接队列满！
  |                            |  正常请求无法建立连接
```

#### 危害

- 半连接队列被占满，正常用户无法建立连接
- 服务器 CPU 和内存资源被大量消耗（维护半连接需要内存和定时器）
- 服务器可能因为大量超时重传而负载过高
- 严重时导致服务器无法响应任何请求

#### 防御方案

**1. 增大半连接队列容量**
```bash
# 增大半连接队列最大值
sysctl -w net.ipv4.tcp_max_syn_backlog=65535

# 增大全连接队列最大值（listen 的 backlog 参数）
# 同时需要增大 somaxconn
sysctl -w net.ipv4.core.somaxconn=65535
```

**2. 启用 SYN Cookie**
```bash
# 启用 SYN Cookie（默认已开启）
sysctl -w net.ipv4.tcp_syncookies=1
```
- SYN Cookie 不在半连接队列中存储信息，避免队列被占满
- 通过计算序列号来验证连接合法性
- 是最有效的 SYN 攻击防御手段

**3. 减少 SYN+ACK 重传次数**
```bash
# 减少 SYN+ACK 的重传次数（默认 5 次）
sysctl -w net.ipv4.tcp_synack_retries=2
```
- 减少重传次数可以更快地释放半连接队列中的资源
- 但也可能导致正常网络环境下连接建立失败

**4. 使用防火墙或 WAF**
- 限制单个 IP 的 SYN 请求速率
- 使用 SYN 代理（SYN Proxy）在防火墙层面验证连接
- 使用 CDN 或 DDoS 防护服务

**5. 应用层优化**
- 使用连接池，减少频繁建立和关闭连接
- 调整 TCP Keep-Alive 参数，保持长连接

### 相关内核参数

| 参数 | 说明 | 默认值 | 建议值 |
|------|------|--------|--------|
| `tcp_max_syn_backlog` | 半连接队列最大长度 | 1024 | 65535 |
| `somaxconn` | 全连接队列最大长度 | 128 | 65535 |
| `tcp_syncookies` | 启用 SYN Cookie | 1 | 1（保持开启） |
| `tcp_synack_retries` | SYN+ACK 重传次数 | 5 | 2~3 |
| `tcp_abort_on_overflow` | 全连接队列满时是否发送 RST | 0 | 视情况而定 |
| `tcp_retries2` | ACK 重传次数 | 15 | 保持默认 |

### 实际排查

**查看半连接队列和全连接队列状态**：
```bash
# 查看半连接队列（SYN_RECV 状态的连接数）
netstat -nat | grep SYN_RECV | wc -l

# 查看全连接队列溢出情况
# Recv-Q：全连接队列中等待 accept 的连接数
# Send-Q：全连接队列的最大长度
ss -ltn
# State   Recv-Q  Send-Q  Local Address:Port
# LISTEN  0       128     0.0.0.0:80

# 如果 Recv-Q 接近或等于 Send-Q，说明全连接队列快满了
```

**查看内核参数**：
```bash
# 查看当前值
sysctl net.ipv4.tcp_max_syn_backlog
sysctl net.core.somaxconn
sysctl net.ipv4.tcp_syncookies

# 修改参数
sysctl -w net.ipv4.tcp_max_syn_backlog=65535
```

### Java 中的 backlog

在 Java 网络编程中，`ServerSocket` 的构造函数参数 `backlog` 就是全连接队列的大小：

```java
// backlog 参数指定全连接队列的最大长度
// 如果不指定，使用系统默认值（通常为 50 或 128）
ServerSocket serverSocket = new ServerSocket(8080, 100);
```

Tomcat 的 `server.xml` 中也可以配置：
```xml
<Connector port="8080" acceptCount="100" />
<!-- acceptCount 就是全连接队列的最大长度 -->
```
