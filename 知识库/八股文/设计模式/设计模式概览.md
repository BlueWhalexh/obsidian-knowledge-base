# 设计模式概览

## 问题
你了解哪些设计模式？单例模式有哪些实现方式？为什么 DCL 要加 volatile？策略模式和观察者模式怎么用？

## 答案

### 设计模式分类

设计模式分为三大类，共 23 种：

#### 创建型模式（5种）

关注对象的创建方式，将对象的创建与使用分离。

| 模式 | 目的 | 典型应用 |
|------|------|---------|
| **单例模式** | 保证一个类只有一个实例 | Spring Bean、Runtime、数据库连接池 |
| **工厂方法** | 定义创建对象的接口，由子类决定实例化哪个类 | SLF4J LoggerFactory、Collection.iterator() |
| **抽象工厂** | 创建一系列相关对象的接口 | JDBC Connection、Spring BeanFactory |
| **建造者模式** | 分步构建复杂对象 | StringBuilder、Lombok @Builder、Guava ImmutableList |
| **原型模式** | 通过复制创建新对象 | Object.clone()、Spring Bean 的 scope="prototype" |

#### 结构型模式（7种）

关注类和对象的组合方式。

| 模式 | 目的 | 典型应用 |
|------|------|---------|
| **适配器模式** | 将不兼容的接口转换为期望的接口 | Arrays.asList()、InputStreamReader |
| **桥接模式** | 将抽象与实现分离 | JDBC Driver、消息发送（消息类型 × 发送方式） |
| **组合模式** | 树形结构的统一处理 | 文件系统、JMenu/JMenuItem |
| **装饰器模式** | 动态添加功能 | Java IO 流、Collections.synchronizedList() |
| **外观模式** | 提供统一的高层接口 | SLF4J、JdbcTemplate |
| **享元模式** | 共享细粒度对象 | Integer.valueOf() 缓存、String 常量池 |
| **代理模式** | 控制对象访问 | Spring AOP、MyBatis Mapper、RPC 调用 |

#### 行为型模式（11种）

关注对象之间的通信。

| 模式 | 目的 | 典型应用 |
|------|------|---------|
| **责任链模式** | 请求沿链传递，直到被处理 | Servlet Filter、Netty Pipeline |
| **命令模式** | 将请求封装为对象 | Runnable、ThreadPoolExecutor.execute() |
| **迭代器模式** | 顺序访问集合元素 | Iterator、for-each |
| **中介者模式** | 用中介对象封装交互 | MVC Controller、聊天室 |
| **备忘录模式** | 保存和恢复对象状态 | 事务回滚、游戏存档 |
| **观察者模式** | 一对多的依赖关系 | Spring Event、MQ、Zookeeper Watch |
| **状态模式** | 状态改变时改变行为 | 线程状态、订单状态机 |
| **策略模式** | 定义一族算法 | Comparator、ThreadPoolExecutor 拒绝策略 |
| **模板方法** | 定义算法骨架，子类实现细节 | AbstractList、JdbcTemplate |
| **访问者模式** | 作用于对象结构的各元素 | ASM 字节码操作 |

### 常用设计模式详解

#### 1. 单例模式（Singleton）

保证一个类只有一个实例，并提供一个全局访问点。

**实现方式一：饿汉式**
```java
public class Singleton {
    // 类加载时就创建实例，线程安全
    private static final Singleton INSTANCE = new Singleton();

    private Singleton() {}

    public static Singleton getInstance() {
        return INSTANCE;
    }
}
```
- **优点**：实现简单，线程安全
- **缺点**：类加载时就创建，可能浪费资源（如果实例一直未被使用）

**实现方式二：懒汉式（线程不安全）**
```java
public class Singleton {
    private static Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {          // 第一次检查
            instance = new Singleton();  // 创建实例
        }
        return instance;
    }
}
```
- **问题**：多线程下可能创建多个实例（非线程安全）

**实现方式三：懒汉式 + synchronized**
```java
public class Singleton {
    private static Singleton instance;

    private Singleton() {}

    public static synchronized Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}
```
- **优点**：线程安全
- **缺点**：每次获取实例都要加锁，性能差

**实现方式四：双重检查锁（DCL，Double-Checked Locking）**
```java
public class Singleton {
    private static volatile Singleton instance;  // 必须用 volatile

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {                // 第一次检查（无锁）
            synchronized (Singleton.class) {
                if (instance == null) {        // 第二次检查（有锁）
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```
- **优点**：线程安全，性能好（只在第一次创建时加锁）
- **注意**：必须用 `volatile` 修饰（原因见下文）

**实现方式五：静态内部类**
```java
public class Singleton {
    private Singleton() {}

    // 静态内部类在被引用时才会加载
    private static class SingletonHolder {
        private static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return SingletonHolder.INSTANCE;
    }
}
```
- **优点**：线程安全（类加载机制保证），懒加载（只有调用 getInstance() 时才加载内部类）
- **原理**：Java 类加载过程是线程安全的，由 JVM 保证

**实现方式六：枚举**
```java
public enum Singleton {
    INSTANCE;

    public void doSomething() {
        // 业务方法
    }
}

// 使用
Singleton.INSTANCE.doSomething();
```
- **优点**：最简洁、线程安全、防止反射和反序列化攻击
- **原理**：枚举实例由 JVM 保证唯一
- **推荐**：Effective Java 推荐的最佳实现方式

#### 为什么 DCL 要加 volatile？

**问题**：`instance = new Singleton()` 不是原子操作，分为三步：
1. 分配内存空间
2. 初始化对象
3. 将 instance 指向分配的内存

**指令重排**：JVM 可能将步骤 2 和 3 重排为 1 → 3 → 2：
1. 分配内存空间
2. 将 instance 指向分配的内存（此时 instance != null，但对象未初始化）
3. 初始化对象

**多线程问题**：
```
线程A：执行 1 → 3（重排后），instance 已不为 null
线程B：第一次检查 if (instance == null) 为 false，直接返回 instance
线程B：使用未初始化的 instance，导致程序出错！
```

**volatile 的作用**：
- `volatile` 禁止指令重排（通过内存屏障）
- 写操作（赋值）前会插入 StoreStore 屏障，保证之前的写操作都已完成
- 读操作后会插入 LoadLoad 屏障，保证之后的读操作都读取最新值
- 确保 `instance = new Singleton()` 按顺序执行：1 → 2 → 3

#### 2. 策略模式（Strategy）

定义一族算法，将每个算法封装起来，使它们可以互相替换。

**核心思想**：
- 定义策略接口（Strategy）
- 实现多个具体策略（ConcreteStrategy）
- Context 持有 Strategy 接口，运行时可以切换策略

```java
// 策略接口
public interface PayStrategy {
    void pay(double amount);
}

// 具体策略
public class AlipayStrategy implements PayStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("支付宝支付: " + amount);
    }
}

public class WechatPayStrategy implements PayStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("微信支付: " + amount);
    }
}

// Context
public class PaymentContext {
    private PayStrategy strategy;

    public PaymentContext(PayStrategy strategy) {
        this.strategy = strategy;
    }

    public void setStrategy(PayStrategy strategy) {
        this.strategy = strategy;
    }

    public void executePay(double amount) {
        strategy.pay(amount);
    }
}

// 使用
PaymentContext context = new PaymentContext(new AlipayStrategy());
context.executePay(100.0);  // 支付宝支付: 100.0

context.setStrategy(new WechatPayStrategy());
context.executePay(200.0);  // 微信支付: 200.0
```

**实际应用**：
- **Java Comparator**：`Collections.sort(list, comparator)` 可以传入不同的比较策略
- **Spring Resource**：`Resource` 接口有不同的实现（ClassPathResource、FileSystemResource）
- **ThreadPoolExecutor 拒绝策略**：AbortPolicy、CallerRunsPolicy、DiscardPolicy
- **支付方式选择**：不同支付方式对应不同策略

**与 if-else 的区别**：
- if-else：硬编码，每次新增策略都要修改代码，违反开闭原则
- 策略模式：新增策略只需添加新类，不修改现有代码

#### 3. 观察者模式（Observer）

定义一对多的依赖关系，当一个对象状态改变时，所有依赖它的对象都会被通知并自动更新。

**核心思想**：
- **Subject（主题/被观察者）**：维护观察者列表，状态改变时通知所有观察者
- **Observer（观察者）**：定义更新接口，收到通知后执行相应操作

```java
// 观察者接口
public interface EventListener {
    void onEvent(String eventType, Object data);
}

// 主题/事件管理器
public class EventManager {
    private Map<String, List<EventListener>> listeners = new HashMap<>();

    public void subscribe(String eventType, EventListener listener) {
        listeners.computeIfAbsent(eventType, k -> new ArrayList<>()).add(listener);
    }

    public void unsubscribe(String eventType, EventListener listener) {
        List<EventListener> list = listeners.get(eventType);
        if (list != null) {
            list.remove(listener);
        }
    }

    public void notify(String eventType, Object data) {
        List<EventListener> list = listeners.get(eventType);
        if (list != null) {
            for (EventListener listener : list) {
                listener.onEvent(eventType, data);
            }
        }
    }
}

// 具体主题
public class OrderService {
    private EventManager eventManager;

    public OrderService() {
        eventManager = new EventManager();
    }

    public void subscribe(String eventType, EventListener listener) {
        eventManager.subscribe(eventType, listener);
    }

    public void createOrder(String orderId) {
        // 创建订单逻辑...
        System.out.println("订单创建成功: " + orderId);
        // 通知观察者
        eventManager.notify("ORDER_CREATED", orderId);
    }
}

// 具体观察者
public class SmsListener implements EventListener {
    @Override
    public void onEvent(String eventType, Object data) {
        System.out.println("发送短信通知，订单: " + data);
    }
}

public class EmailListener implements EventListener {
    @Override
    public void onEvent(String eventType, Object data) {
        System.out.println("发送邮件通知，订单: " + data);
    }
}

// 使用
OrderService orderService = new OrderService();
orderService.subscribe("ORDER_CREATED", new SmsListener());
orderService.subscribe("ORDER_CREATED", new EmailListener());
orderService.createOrder("ORD001");
// 输出：
// 订单创建成功: ORD001
// 发送短信通知，订单: ORD001
// 发送邮件通知，订单: ORD001
```

**实际应用**：
- **Spring Event**：`ApplicationEvent` + `ApplicationListener`
- **MQ（消息队列）**：发布-订阅模式，生产者发布消息，消费者订阅处理
- **Zookeeper Watch**：节点变化时通知监听者
- **Java Swing/JavaFX**：按钮点击事件监听
- **MQTT**：物联网中的发布-订阅协议

**观察者模式 vs 发布-订阅模式**：
- **观察者模式**：Subject 直接通知 Observer（耦合度较高）
- **发布-订阅模式**：通过中间件（如 MQ）解耦，发布者和订阅者互不知道对方
- 实际上发布-订阅模式是观察者模式的演进版本

### 设计模式在项目中的应用

#### Spring 中的设计模式

| 设计模式 | Spring 中的应用 |
|---------|----------------|
| 工厂模式 | BeanFactory、ApplicationContext |
| 单例模式 | Spring Bean 默认是单例 |
| 代理模式 | AOP（JDK 动态代理、CGLIB） |
| 模板方法 | JdbcTemplate、RestTemplate |
| 观察者模式 | ApplicationEvent、@EventListener |
| 策略模式 | Resource 接口、HandlerMapping |
| 责任链模式 | Filter Chain、Interceptor Chain |
| 适配器模式 | HandlerAdapter |
| 建造者模式 | BeanDefinitionBuilder |
| 装饰器模式 | BeanWrapper |

#### 下单链路中的设计模式

1. **策略模式**：选择支付方式（支付宝/微信/银行卡）
2. **观察者模式**：订单创建后通知库存服务、积分服务、消息服务
3. **责任链模式**：下单前的校验（参数校验 → 库存校验 → 风控校验）
4. **模板方法模式**：订单处理流程模板（校验 → 扣库存 → 创建订单 → 支付）
5. **工厂模式**：根据订单类型创建不同的订单处理器
6. **建造者模式**：构建复杂的订单对象
