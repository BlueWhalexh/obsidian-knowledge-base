# 项目中用了哪些设计模式 — 下单链路

## 问题

> 你在项目中用了哪些设计模式？结合具体业务场景说说。

## 回答策略

面试官问这个问题，不是让你背出 23 种设计模式的定义，而是看你**是否在实践中无意识地用到了、并且能清晰讲出来**。

最佳策略：**锚定一个核心业务链路来讲**，比如「订单下单链路」。一条链路从 Controller → Service → DAO → 第三方，天然覆盖 10+ 种设计模式，表达起来有逻辑、有层次，比零散列举强 10 倍。

---

## 下单链路中的设计模式（从前到后）

```
┌─────────────────────────────────────────────────────────┐
│                    Controller 层                          │
│  1. 依赖注入 (IoC / DI)                                    │
│  2. 门面模式 (Facade)                                      │
└───────────┬─────────────────────────────────────────────┘
            │
┌───────────▼─────────────────────────────────────────────┐
│                    Service 层                              │
│  3. 单例模式 (Singleton) — Spring Bean 默认                │
│  4. 代理模式 (Proxy) — AOP 事务 @Transactional             │
│  5. 策略模式 (Strategy) — 支付方式选择                       │
│  6. 责任链模式 (Chain of Responsibility) — 下单校验链         │
│  7. 模版方法 (Template Method) — 支付流程骨架                │
│  8. 观察者模式 (Observer) — 下单成功事件发布                  │
│  9. 建造者模式 (Builder) — 复杂订单对象构建                   │
└───────────┬─────────────────────────────────────────────┘
            │

│                    DAO / 基础设施层                         │

│ 11. 适配器模式 (Adapter) — 对接第三方支付 API                 │

└─────────────────────────────────────────────────────────┘
```

## 逐层详细解释

### 1. 单例模式 (Singleton)

**位置**：Spring 容器管理的所有 Bean 默认作用域都是 singleton。

> Spring 把 Service、Controller、Repository 统统注册为单例，由容器统一管理。目的是减少对象创建开销、保证状态一致性。底层是 ConcurrentHashMap 做的一级缓存 singletonObjects。

```java
// Spring 中的体现（你不需要写 DCL）
@Service  // 默认 @Scope("singleton")
public class OrderService { ... }
```

**关键点**：Spring 的单例 ≠ 经典 GoF 单例。Spring 利用 IoC 容器 + 三级缓存解决循环依赖，比手写 DCL 更复杂。

---

### 2. 依赖注入 (Dependency Injection / IoC)

**位置**：贯穿所有层。Controller 注入 Service，Service 注入 DAO/MQ/RPC。

> Spring 通过 DI 反转了对象的创建和依赖管理职责。好处是解耦、可测试（单测时 mock 依赖即可）。

```java
@RestController
public class OrderController {
    @Autowired
    private OrderService orderService;  // 依赖注入
}
```

**和 Spring 的关系**：DI 是 IoC 的具体实现方式。Spring 通过 `@Autowired` + `BeanFactory` + `BeanPostProcessor` 完成注入。

---

### 3. 门面模式 (Facade)

**位置**：OrderService（对外提供一个统一入口，内部协调多个子系统）。

> 下单接口看似简单，背后要协调：库存服务、支付服务、优惠券服务、风控服务、物流服务、短信服务。上层 Controller 只需要调 `orderService.placeOrder()`，不需要知道内部有多复杂。

```java
@Service
public class OrderService {
    // 对外暴露统一入口，内部编排多个子系统
    public OrderResult placeOrder(PlaceOrderRequest req) {
        // 1. 风控校验
        // 2. 库存预占
        // 3. 金额计算 + 优惠券核销
        // 4. 支付
        // 5. 订单落库
        // 6. 发送通知
    }
}
```

**关键点**：微服务架构中，门面模式的下一个形态就是 **API Gateway**（如 Spring Cloud Gateway）。

---

### 4. 代理模式 (Proxy)

**位置**：Spring AOP，最典型的就是 `@Transactional` 事务管理。

```java
@Service
public class OrderService {
    @Transactional  // Spring 生成代理对象，包裹事务开启/提交/回滚
    public void createOrder(OrderDTO dto) {
        orderDao.insert(order);
        stockService.deduct(dto.getSkuId(), dto.getQty());
        couponService.use(dto.getCouponId());
    }
}
```

**原理**：Spring 用 JDK 动态代理（有接口）或 CGLIB（无接口）生成代理对象，在方法前后织入事务逻辑。

**常见坑**：同类内部调用 `this.method()` 不走代理，事务不生效。

---

### 5. 策略模式 (Strategy)

**位置**：支付方式选择（微信支付 / 支付宝 / 银行卡 / 花呗）。

```java
// 策略接口
public interface PaymentStrategy {
    PayResult pay(PayRequest request);
}

// 具体策略
@Component("wechatPay")
public class WechatPayment implements PaymentStrategy { ... }

@Component("alipay")
public class AlipayPayment implements PaymentStrategy { ... }

// 上下文：根据支付方式动态选择策略
@Service
public class PaymentService {
    @Autowired
    private Map<String, PaymentStrategy> strategyMap;  // Spring 自动注入所有实现

    public PayResult doPay(PayRequest request) {
        PaymentStrategy strategy = strategyMap.get(request.getPayType());
        return strategy.pay(request);
    }
}
```

**关键点**：Spring 的依赖注入天然支持策略模式 —— 用 `Map<String, Interface>` 收集所有实现类，key 是 Bean name。这比手写 if-else / switch 优雅得多。

---

### 6. 责任链模式 (Chain of Responsibility)

**位置**：下单前的校验链（库存 → 余额 → 风控 → 优惠券 → 限购）。

```java
// 校验器接口
public interface OrderValidator {
    void validate(OrderContext ctx, ValidatorChain chain);
}

// 链管理器
public class ValidatorChain {
    private List<OrderValidator> validators;
    private int index = 0;

    public void doValidate(OrderContext ctx) {
        if (index < validators.size()) {
            validators.get(index++).validate(ctx, this);
        }
    }
}

// 具体校验器
@Component
public class StockValidator implements OrderValidator {
    public void validate(OrderContext ctx, ValidatorChain chain) {
        if (stock不足) throw new BizException("库存不足");
        chain.doValidate(ctx);  // 传递到下一个
    }
}
```

**关键点**：每个校验器只做一件事、失败就短路。新增校验规则只需加一个 Validator 实现类，不需要改主流程代码 —— 对扩展开放，对修改关闭（开闭原则）。

> 有经验的候选人还可以补充：在复杂项目中，责任链可以结合**流程引擎**（如 LiteFlow、Flowable）做可视化编排。

---

### 7. 模版方法 (Template Method)

**位置一**：支付流程骨架 —— 付款操作统一流程（参数校验 → 调用渠道 → 解析响应 → 更新状态），不同支付方式只覆写「调用渠道」这一步。

```java
public abstract class AbstractPayment {
    public final PayResult execute(PayRequest req) {
        validateParams(req);           // 1. 固定步骤
        ChannelResponse resp = invokeChannel(req);  // 2. 子类实现
        PayResult result = parseResponse(resp);     // 3. 固定步骤
        updateOrderStatus(result);     // 4. 固定步骤
        return result;
    }
    protected abstract ChannelResponse invokeChannel(PayRequest req);
}
```

**位置二**：Spring 内置 — `RestTemplate`、`JdbcTemplate`、`RedisTemplate`、`KafkaTemplate`。这些 Template 类把固定流程（连接管理、异常处理、资源释放）封装好，只暴露出关键步骤给调用方。

```java
// RedisTemplate 封装了连接获取/释放/序列化，你只关心读写
redisTemplate.opsForValue().set("order:123", orderDTO);
```

> **面试加分点**：提到 Spring 的 Template 系列是模版方法 + 回调（Callback）的组合，比经典模版方法更灵活。

---

### 8. 观察者模式 (Observer) / 事件驱动

**位置**：下单成功后的事件通知（发短信、发优惠券、扣库存、记日志）。

```java
// 1. 定义事件
public class OrderPlacedEvent extends ApplicationEvent {
    private Order order;
}

// 2. 发布事件
@Service
public class OrderService {
    @Autowired
    private ApplicationEventPublisher publisher;

    public void placeOrder(OrderDTO dto) {
        // ... 下单逻辑
        publisher.publishEvent(new OrderPlacedEvent(this, order));
    }
}

// 3. 监听者（业务解耦，可异步）
@Component
public class SmsListener {
    @EventListener
    @Async  // 异步执行，不阻塞主流程
    public void sendSms(OrderPlacedEvent event) { ... }
}

@Component
public class PointListener {
    @EventListener
    @Async
    public void addPoints(OrderPlacedEvent event) { ... }
}
```

**关键点**：下单主流程不需要知道「发短信」「发积分」这些旁路逻辑的存在。新增一个旁路需求（比如「下单后推送企业微信」），只需加一个 Listener，不动主流程。

**进阶**：高并发场景下，观察者模式演进为消息队列（MQ），实现更可靠的事件驱动。

---

### 9. 建造者模式 (Builder)

**位置**：构建复杂的订单对象（订单头 / 订单行 / 收货地址 / 支付信息 / 优惠信息 层层嵌套）。

```java
Order order = Order.builder()
    .orderId(IdGenerator.next())
    .buyerId(userId)
    .items(orderItems)
    .totalAmount(total)
    .discountAmount(discount)
    .shippingAddress(address)
    .paymentMethod(payType)
    .remark(remark)
    .build();
```

**关键点**：Lombok 的 `@Builder` 让建造者模式零成本使用。避免了 10+ 个参数的长构造函数或层层 setter。

---

### 10. 抽象工厂 (Abstract Factory)

**位置**：根据支付类型创建对应的支付渠道实例及其依赖对象。

```java
public interface PaymentFactory {
    PaymentStrategy createPayment();
    RefundStrategy createRefund();
    PaymentNotifyParser createParser();
}

// 微信支付工厂
@Component
public class WechatPaymentFactory implements PaymentFactory {
    public PaymentStrategy createPayment() { return new WechatPayment(); }
    public RefundStrategy createRefund() { return new WechatRefund(); }
    public PaymentNotifyParser createParser() { return new WechatParser(); }
}
```

**对比简单工厂**：简单工厂只创建单一产品，抽象工厂创建**产品族**（支付 + 退款 + 回调解析属于同一族）。

---

### 11. 适配器模式 (Adapter)

**位置**：对接第三方支付 API（每个渠道的参数格式、签名算法、返回格式完全不同）。

```java
// 微信适配器
@Component
public class WechatPayAdapter {
    public PayResponse pay(PayRequest req) {
        // 把统一的 PayRequest 转成微信 SDK 需要的格式
        WechatPayRequest wxReq = convertToWechat(req);
        // 调用微信 SDK
        WechatPayResponse wxResp = wechatSDK.unifiedOrder(wxReq);
        // 把微信响应转成统一 PayResponse
        return convertToStandard(wxResp);
    }
}
```

**关键点**：适配器屏蔽了外部系统的差异，内部代码只依赖统一接口。切换支付渠道时只需换一个 Adapter，业务代码零改动。这就是**依赖倒置原则（DIP）**的体现。

---

### 12. 装饰器模式 (Decorator)

**位置**：缓存层装饰 DAO（查订单时先查缓存，缓存未命中再查 DB）。

```java
// 装饰器实现同一接口，在不修改原有 DAO 的情况下增加缓存能力
public class CachedOrderRepository implements OrderRepository {
    private OrderRepository delegate;  // 被装饰的原始 DAO
    private RedisTemplate redis;

    public Order findById(Long id) {
        Order cached = redis.opsForValue().get("order:" + id);
        if (cached != null) return cached;
        Order order = delegate.findById(id);
        redis.opsForValue().set("order:" + id, order, 30, TimeUnit.MINUTES);
        return order;
    }
}
```

## 面试回答话术模板

> 「我在项目中用的设计模式还挺多的，我用一条核心业务链路来串一下。比如**下单流程**，从前到后至少涉及十几种模式：
>
> **框架层面**，Spring 默认就把 Service/Controller 注册为单例，通过 DI 做依赖注入，这是最基础的。AOP 事务本质是代理模式。
>
> **业务层面**，支付方式选择用了策略模式，下单前的各种校验（库存、风控、余额）用责任链串联，支付流程骨架是模版方法 —— 这个思路和 Spring 自带的 RedisTemplate、JdbcTemplate 一脉相承。下单成功后的事件通知是观察者模式。
>
> **基础设施层面**，对接微信/支付宝是不同的 Adapter，创建支付渠道实例用抽象工厂，复杂对象的构建用 Builder。
>
> 我觉得设计模式不是背出来的，而是在写代码时为了解决『怎么样不写死』、『怎么样加功能不改老代码』这些实际问题时自然用上的。」

## 进阶思考：模式之间的关系

很多人孤立地学设计模式，但实际项目中它们是**组合使用**的：

| 组合 | 例子 |
|------|------|
| **策略 + 抽象工厂** | 工厂创建策略对象，策略执行具体逻辑 |
| **责任链 + 模版方法** | 链中每个节点的执行流程用模版方法定义骨架 |
| **门面 + 观察者** | 门面封装主流程，观察者处理旁路通知 |
| **适配器 + 代理** | 适配器转换接口，代理控制访问 |
| **单例 + DI** | DI 容器管理单例的生命周期 |
