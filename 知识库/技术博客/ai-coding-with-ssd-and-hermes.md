## 背景

AI Coding 很容易被理解成"给模型一个 prompt，让它帮我写代码"。但实际做项目时，我越来越觉得 prompt 只是入口，不是系统。

这次个人技术博客的迭代就是一个典型例子：它涉及内容同步、Quartz 构建、UI 风格、nginx 反代、DNS、线上静态服务、GitHub 流程和回滚策略。如果把所有上下文都塞进一个 prompt，不出三轮对话 AI 就开始"失忆"。

问题不在 AI，而在我们的协作方式——**我们一直在用聊天工具的方式跟一个 Agent 协作，而不是用工程化的方式**。

这篇文章把这次实战中沉淀的经验整理出来，核心思想是：**AI Coding 的本质是人机多 Agent 协作系统**，你需要像设计系统一样设计 AI 的输入、记忆和执行边界。

---

## Prompt 不够：为什么需要 Agent 架构思维

Prompt 可以告诉 AI 当前要做什么，但它解决不了几个关键问题：

1. 项目长期目标是什么？
2. 哪些内容不能碰、哪些服务不能停？
3. 之前做过什么决策、踩过什么坑？
4. 什么算完成、怎么验证？
5. 出问题怎么回滚？
6. 多个会话/多个 AI 之间怎么协作？

这些问题本质上是一个 **Agent 的长期记忆和上下文管理问题**。如果只放在聊天上下文里，随着会话切换，AI 就像每次重启的进程——状态全丢。

把 AI 看作 Agent，就需要给它设计：

| 能力 | 对应设计 | 解决什么问题 |
|------|---------|------------|
| **上下文管理** | 结构化输入文档 | 每次会话不丢失项目意图和边界 |
| **记忆管理** | 持久化文档 + 会话日志 | 跨 session 不重复踩坑 |
| **规划能力** | SSD + 验收标准 | 不跑偏、不过度实现 |
| **执行能力** | Hermes 分层 + Runbook | 出问题能定位，修完能验证 |

## 上下文管理：不只是"塞更多上下文"

很多人把上下文管理理解为"prompt 写长一点"。但真正的问题是：**AI 的上下文窗口是有限的，而项目知识是无限的**。

关键是把信息分层，按需注入：

### 第一层：永久记忆（项目文档）

这些不会因为会话切换而丢失，每个新 AI 会话第一件事就是读它们：

- `SSD.md` — 需求范围和非目标（不要做什么比做什么更重要）
- `AGENTS.md` — 接手规则、项目边界、文件约定
- `HERMES_ARCHITECTURE.md` — 架构分层
- `RUNBOOK.md` — 构建、部署、回滚步骤
- `progress.md` — 历史问题和修复记录

**实用技巧**：这些文档要短。每篇控制在 50 行以内，否则 AI 读完后真正能处理代码的窗口就少了。

### 第二层：会话记忆（当前任务上下文）

每次会话的 prompt 应该包含：
- 当前任务的目标（要什么）
- 验收标准（什么算完成）
- 约束条件（不能碰什么）

会话结束时，把**新发现的问题**写回 `progress.md`，形成闭环。

### 第三层：工具记忆（可检索的知识库）

当项目知识超过 1000 行文档时，靠全量读入就不现实了。这时需要：
- 把技术决策文档化（ADR）
- 把踩坑记录打索引
- 让 AI 通过搜索（grep/find）按需获取，而不是全量注入

**这次实战的体会**：前三轮会话靠全量注入就够了。等到第 10 轮，`progress.md` 和 Runbook 的累积内容才是最有价值的上下文。

## 记忆管理：让 Agent 有"长期记忆"

AI Coding 最大的痛点是：**同样的坑，每次换会话都会再踩一次**。

解决方式是给 AI Agent 分层记忆架构：

```
永久记忆（项目文档）
  ├── SSD.md              ← 语义记忆：项目是什么、不是啥
  ├── HERMES_ARCHITECTURE.md  ← 语义记忆：系统怎么拆的
  ├── RUNBOOK.md           ← 程序记忆：怎么做、怎么验
  └── AGENTS.md            ← 语义记忆：协作规则

持久的 episod ic 记忆
  └── progress.md          ← 做了什么、踩过什么坑、怎么修的

快速更新的工作记忆
  └── 当前会话 prompt      ← 现在要做什么
```

这次博客项目里，`progress.md` 记录了中文 URL 编码问题、pretty URL 映射规则、nginx 配置变更。每次新会话 AI 读到这些，就不会再去问"为什么文章打不开"。

**实战建议**：每次 AI 修完一个 bug 或踩到一个坑，立即执行三条命令：
```
1. 把原因写进 progress.md
2. 把修复步骤写进 RUNBOOK.md
3. 在项目里留下测试/验证手段
```

## SPD 驱动：从 SSD 到验收闭环

SPD（Specification-Driven Development）不是新概念，但在 AI Coding 里特别重要——**因为 AI 不会像人一样说"我觉得这个需求有点模糊"**。它会直接开干，干完发现不是你要的。

所以需要把"规格"做得足够明确：

### SSD 不只要写范围，还要写非目标

```
第一版做：
  - 个人技术博客 + Obsidian 公开笔记站
  - 支持搜索、标签、图谱
  - 支持 RSS 和 sitemap

第一版不做：
  × 论坛、账号系统
  × 私人 vault 云同步
  × 不自动发布未确认内容
```

没有"非目标"的 SSD，AI 会在实现过程中不断"顺手加功能"。

### 验收标准要可执行

与其写"文章页面要好看"，不如写：
```
验收检查清单（RUNBOOK 的一部分）：
  [ ] 首页 200
  [ ] 文章索引 200
  [ ] RSS /sitemap.xml 200
  [ ] 中文 URL 可访问
  [ ] 构建无报错
```

这样 AI 改完可以自己跑验证，形成 **规划 → 执行 → 验证 → 记录** 的闭环。

## Agent 执行架构：Hermes 的实战价值

Hermes 六层架构不只是架构图，它在 AI Coding 里最大的价值是 **调试不慌**：

| 层 | 关注点 | 常见问题 |
| --- | --- | --- |
| Horizon | 用户看到的页面体验 | 样式错乱、链接 404 |
| Editor | Obsidian 写作 + 筛选 | 内容不同步、筛选遗漏 |
| Renderer | Quartz 4 构建 | 构建失败、pretty URL 404 |
| Middleware | nginx、域名、HTTPS | 证书、反代配置、DNS |
| Estate | 服务器目录、端口 | 端口冲突、目录权限 |
| Safeguards | 备份、验证、回滚 | 回滚不可用、验证漏项 |

当 AI 定位问题时，按层排查：

```
域名打不开？ → Middleware（DNS / nginx / HTTPS）
文章 404？   → Renderer（Quartz 构建 / pretty URL）
首页改了不生效？ → Safeguards（有没有走 Git 发布）
```

这次实战中，文章索引 404 的排查过程：
1. 首页正常 → Horizon 没问题
2. RSS、sitemap 正常 → Renderer 构建没问题
3. 具体文章 404 → 生产静态服务不支持 Quartz pretty URL
4. 修复：服务脚本加 URL 映射 → `posts/foo` → `public/posts/foo.html`
5. 中文 URL 加 `decodeURIComponent`

整个过程没有乱改代码，因为分层告诉我问题在哪一层。

## AI Coding 的正确输入：Agent 的上下文包

这次实践后，我给每个 AI 会话准备的"上下文包"是：

```
1. README 或 SSD.md         ← 10 秒让 AI 知道这个项目是啥
2. AGENTS.md                 ← 5 秒让 AI 知道协作规则
3. HERMES_ARCHITECTURE.md    ← 20 秒让 AI 知道代码怎么组织
4. RUNBOOK.md                ← 20 秒让 AI 知道怎么跑、怎么验
5. progress.md               ← 30 秒让 AI 知道历史踩坑
6. 当前任务的具体描述        ← 当前意图
```

总共约 2-3 分钟阅读量。之后 AI 的每步操作都能对号入座。

**对比**：之前用长 prompt 方式（把需求写在对话里），第一轮还可以，第三轮 AI 就开始遗漏细节。现在用文档化的上下文包，即使隔一周换会话，AI 重新进入状态的时间不超过 5 分钟。

## AI Coding 的上线流程：执行与验证分离

之前踩了一个流程问题：为了快速修线上，直接把文件同步到服务器。它能解决眼前问题，但 GitHub 没有记录，服务器状态和本地状态容易分叉。

现在的流程是 Agent 执行 + 人工验证的分离：

```text
本地修改（Agent 执行）
  -> npm.cmd run check
  -> npm.cmd run build
  -> npm.cmd run verify
  -> git diff / git status
  -> git add / commit / push

服务器部署（半自动）
  -> 服务器 git pull
  -> npx quartz build
  -> 线上抽样验证

记录（闭环）
  -> 更新 progress.md（如有踩坑）
```

Agent 负责到 commit 为止，部署和验证阶段保留人工确认环节。

## 结论

AI Coding 的重点不是"让 AI 写代码"，而是 **设计一个人机多 Agent 协作系统**：

| 维度 | 传统思路 | Agent 思路 |
|------|---------|-----------|
| 输入 | 一个 prompt | 结构化上下文包（6 份文档） |
| 上下文管理 | 靠聊天记录 | 靠文档分层 + 按需注入 |
| 记忆 | 会话结束即忘 | 永久记忆 + 情景记忆 + 工作记忆 |
| 规格 | 口头描述 | SSD + 可执行验收标准 |
| 执行 | AI 一次性生成 | 规划 → 执行 → 验证 → 记录闭环 |
| 部署 | 手工复制 | Git 流程 + 可回滚 |

当这些东西都在，AI 才不是一次性的代码生成器，而是一个可以接手项目、定位问题、执行变更、补充文档的工程协作者。
