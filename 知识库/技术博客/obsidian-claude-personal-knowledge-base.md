## 背景

我之前的笔记主要散落在 Obsidian 里：有 Java 后端八股文、场景题、AI 大模型、项目复盘、面试准备，也有 daily、todo、实习记录。

这类笔记的问题不是"没有内容"，而是内容太多之后，很难被稳定复用。临近面试时，我需要的是一个可以快速检索、可以按主题浏览、可以从一篇笔记跳到相关笔记的知识网络，而不是一堆孤立 Markdown 文件。

市面上笔记工具很多，但我花了一段时间才找到真正适合的组合。这篇文章说说我的对比思考，以及一个 10 分钟就能上手的入门方案。

---

## 传统笔记的问题

在进入工具对比之前，先看传统笔记方式的核心痛点：

| 痛点 | 表现 | 结果 |
|------|------|------|
| **信息堆积** | 记完就不看了，越积越多 | 笔记 = 数字垃圾 |
| **缺乏连接** | 每篇笔记是孤岛，没有交叉引用 | 知识不成网络 |
| **检索困难** | 靠文件夹分类，跨主题找东西靠猜 | 找到时已经放弃了 |
| **维护成本高** | 整理需要大量手动归类、打标签 | 坚持不了两周 |
| **无法发布** | 记在本地就是本地，没法变成作品 | 没有外部反馈循环 |

传统笔记最大的问题不是工具，而是 **只有输入通道，没有整理和输出通道**。输入多了不整理，就变成"数字囤积"。

## 主流笔记工具对比

### Obsidian vs Notion

| 维度        | Obsidian                   | Notion                |
| --------- | -------------------------- | --------------------- |
| **数据所有权** | 本地 Markdown 文件，你的就是你的      | 云端，服务挂了或封号就没了         |
| **离线能力**  | 全离线可用                      | 依赖网络                  |
| **搜索速度**  | 本地全文搜索，毫秒级                 | 网络请求，有延迟              |
| **知识连接**  | [[双向链接]] + Graph View，原生支持 | 依赖 Database 关联，不够直观   |
| **AI 集成** | 本地文件可被任意 AI 工具读写           | 只有 Notion AI（收费，能力受限） |
| **发布能力**  | 通过 Quartz 等可发布为静态站点        | 只能分享 Notion 页面（需登录）   |
| **扩展性**   | 插件生态 + 文件系统，你想怎么玩都行        | 受限于平台 API             |

**结论**：Notion 适合团队协作和轻量项目管理，但作为**个人知识库**，Obsidian 在数据主权、连接能力和离线体验上全面胜出。

### Obsidian vs Roam Research / Logseq

| 维度 | Obsidian | Roam / Logseq |
|------|---------|--------------|
| **核心理念** | 双向链接 + 图谱 | 大纲 + 块引用 |
| **上手曲线** | 适中，文件夹结构直观 | 陡峭，需要适应大纲思维 |
| **性能** | 上万篇笔记依然流畅 | 大库会卡顿 |
| **AI 友好度** | 纯文本文件，AI 可直接读写 | 格式特殊，AI 处理成本高 |
| **社区生态** | 庞大，插件 1500+ | 较小 |

**结论**：Roam/Logseq 的块级引用理念很先进，但对我而言，**AI Coding 需要 AI 能直接读写笔记文件**，纯 Markdown 是无缝对接的前提。

### Obsidian + Claude 的独特优势

这是其他任何笔记工具都无法复制的组合优势：

**1. AI 可以"读"你的笔记**
因为 Obsidian 的文件就是本地 Markdown，Claude（或任何 AI）可以直接读取、分析、整理你的笔记。不需要 API 集成，不需要插件，不需要导出。

**2. AI 可以"写"你的笔记**
AI 可以在 `知识库/` 下创建新的原子笔记、更新 MOC 索引、补充交叉链接。整个过程在文件系统层面完成，Obsidian 下次启动时自动识别。

**3. 知识整理可以自动化**
我的日常流程是：
- 在 DailyNotes 里写草稿
- Claude 自动识别八股文内容 → 创建原子笔记 → 更新索引 → 建交叉链接

这相当于给 Obsidian 配了一个**自动图书管理员**。

**4. 内容可以工程化发布**
Obsidian 负责写作和连接，Claude 负责整理和验证，Quartz 构建静态站点，Git 做版本控制。整个链条是工程化的，可验证、可回滚。

**5. AI 随你换**
因为笔记是纯文本 Markdown，今天用 Claude，明天换 GPT-5，后天用本地模型，笔记不受影响。你不是绑定在一个 AI 上，而是绑定在开放格式上。

## 快速入门：10 分钟搭建 Obsidian + Claude 知识库

如果你也想试试这个组合，这里是最小可行配置：

### Step 1: 安装 Obsidian

从 [obsidian.md](https://obsidian.md) 下载安装。创建一个新 vault，位置选本地目录（比如 `~/my-knowledge-base`）。

### Step 2: 建立基础目录结构

在 vault 根目录创建：

```
DailyNotes/       ← 每日日记（YYYY-MM-DD.md）
知识库/
  ├── MOC/        ← 索引页（Map of Content）
  └── *.md        ← 原子笔记
CLAUDE.md         ← 给 AI 看的规则文件
```

### Step 3: 写一篇笔记试试

`DailyNotes/2026-05-17.md`:

```markdown
## 今日学习

#### 什么是双向链接？
双向链接是 Obsidian 的核心功能。用 `[[笔记名]]` 可以链接到另一篇笔记，
Obsidian 会自动建立反向链接。
```

### Step 4: 让 Claude 帮你整理

把 vault 路径告诉 Claude，说"帮我整理今天的笔记"。Claude 会：
1. 读取 DailyNote
2. 识别可整理的内容
3. 在 `知识库/` 下创建原子笔记
4. 更新 MOC 索引
5. 添加交叉链接

### Step 5（可选）：发布为博客

如果想发布到网络，加一层静态站点构建：

```
Obsidian 写作
  -> 筛选公开内容
  -> Quartz / Hugo 构建
  -> git push
  -> 服务器部署
```

参考我的项目 [[技术博客/ai-coding-with-ssd-and-hermes|AI Coding 实战]] 中的 Hermes 架构。

## 我现在的完整工作流

```text
日常收集
  -> Obsidian 里写 daily note（草稿阶段）
  -> Claude 按 CLAUDE.md 规则整理
    -> 八股文入库（原子笔记 + MOC）
    -> 学习笔记归档（关联待学清单）
    -> 交叉链接补充
  -> 筛选可公开内容
  -> 同步到知识库源目录
  -> Quartz 构建 + Git 发布
  -> hjhxh.site 上线
```

这套流程跑了几个月，最大的感受是：**知识库的价值不取决于你写了多少，而取决于你整理了多少**。Obsidian 解决了"怎么写"，Claude 解决了"怎么整"，组合在一起之后，知识库才真正变成了一个活的系统——而不是一个越来越大、越来越乱的文件夹。

## 总结

| 你想解决什么问题 | 推荐方案 |
|----------------|---------|
| 只是记笔记 | Obsidian 就够了 |
| 需要跨设备同步 | Obsidian + iCloud/同步盘 |
| 需要团队协作 | Notion |
| 需要 AI 帮你整理 | Obsidian + Claude |
| 需要发布为博客 | Obsidian + Claude + Quartz |

如果你已经在用 Obsidian，加上 AI 就是**成本最低、效果最明显的升级**——不需要迁移，不需要重学，你现有的笔记直接就能被 AI 访问和整理。

如果你还没开始，从 Step 1-4 开始，10 分钟就能拥有一个 AI 原生的个人知识库。
