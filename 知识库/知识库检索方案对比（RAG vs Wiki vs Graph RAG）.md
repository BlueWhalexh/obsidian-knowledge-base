# 知识库检索方案对比

## 一、三种方案一览

| 维度 | 传统 RAG（向量检索） | Wiki 模式（Karpathy 路线） | Graph RAG（混合方案） |
|------|---------------------|--------------------------|----------------------|
| **核心思路** | 把笔记切成碎片 → 向量库 → 相似度检索 | LLM 预处理，写成索引+合成文章，查询时沿 `[[link]]` 导航 | 向量检索 + 知识图谱 + 重排序 |
| **适用规模** | 百万级文档 | ~100–400 篇 | 数千条笔记 |
| **基础设施** | Pinecone / ChromaDB 等向量库 | 纯 Markdown 文件 | LanceDB + 实体提取 |
| **检索质量** | 数学上最近 ≠ 语义上最有用，噪声大 | 预合成、连贯、可追溯 | 关系感知（"为什么 X 和 Y 矛盾"） |
| **可调试性** | 翻 embedding 日志 | 直接读 markdown 文件 | 检查实体和边 |
| **查询成本** | 每次检索都有 API 调用 | $0（知识已预编译） | 本地免费 / 少量 API 费用 |
| **最佳场景** | 企业文档搜索 | 个人精选知识库 | 活跃笔记系统，需要发现隐藏关联 |

## 二、为什么纯向量检索不适合个人知识库

1. **Chunking 困境**：固定大小切分要么丢上下文，要么塞进无关内容
2. **Top-K 噪声**：返回的 5 个片段可能是同一段话的 5 个相似版本
3. **无累积效应**：每次查询是"重新检索"，不像研究员会越学越懂
4. **黑盒**：你无法浏览自己的 embedding，出问题无从下手

> "向量相似度搜索返回的是数学上最近的片段，不是最有用的片段。"

## 三、Wiki 模式详解（推荐起步方案）

### 三层架构（Karpathy llm-wiki）

```
┌─────────────────────────────────────────────┐
│  raw/    源材料层    AI 只读不改             │
│  ├── 网页摘录                                │
│  ├── PDF 笔记                                │
│  └── 对话记录                                │
├─────────────────────────────────────────────┤
│  wiki/   合成层      AI 维护的 Markdown       │
│  ├── 原子文章（带 [[wikilink]]）              │
│  ├── index.md 总索引                         │
│  └── 子索引页                               │
├─────────────────────────────────────────────┤
│  CLAUDE.md + SKILL.md  规则层   AI 遵守       │
│  ├── 笔记命名规范                            │
│  ├── 入库流程                                │
│  └── 分类规则                                │
└─────────────────────────────────────────────┘
```

### 三个核心操作

| 操作 | 做什么 | 何时触发 |
|------|--------|---------|
| **Ingest（摄入）** | 读 raw → 写 wiki → 更新 index | 有新内容时 |
| **Query（查询）** | 读 index → 沿 link 导航 → 返回答案 | 提问时 |
| **Lint（检查）** | 扫描断链、孤儿页、矛盾点 | 定期或修改后 |

### 你的 Vault 已经是 Wiki 了

```
Karpathy 模型              →    你的 Vault
─────────────────────────────────────────────
raw/                      →    DailyNotes/
                                 每天的日记草稿
                                 
wiki/                     →    知识库/
                                 ├── 八股文/    原子 Q&A
                                 ├── MOC/       索引页
                                 └── *.md       学习笔记
                                 
CLAUDE.md + SKILL.md     →    CLAUDE.md
                                 .claude/skills/
```

### Wiki 模式 vs 传统 RAG 的本质区别

| | 传统 RAG | LLM Wiki |
|---|---|---|
| **知识存储** | 向量嵌入（不可读） | Markdown（可直接翻阅） |
| **累积效应** | 每次重新检索，无记忆 | 每次 ingest 后知识更丰富 |
| **查错** | 翻 embedding 日志 | `grep` 或 `Ctrl+F` |
| **迁移** | 导出向量库 + 重建索引 | `git clone` |
| **多余碎片** | chunk 可能切碎语义 | 文章粒度由 LLM 把控 |

**你已经在做的事** = Wiki 模式的核心：
- MOC 索引页（[[八股文索引]]、[[场景题索引]]）
- 原子化笔记 + 双向链接
- YAML frontmatter + 标签分类

**AI 可以帮你增强的**：
- 自动从新笔记中提取关键信息更新索引
- 发现笔记间的隐含关联并建议补充链接
- 批量读 raw 材料，生成结构化合集

**局限性**：规模上限约 400 篇 / 40 万词，超出后索引页会过于臃肿。

## 四、Graph RAG（未来进阶方向）

在向量检索之上叠加**知识图谱**，能回答关系型问题：

> ❌ "找到关于睡眠的笔记"（传统 RAG）
> ✅ "为什么我的睡眠笔记和效率系统是矛盾的？"（Graph RAG）

**支持插件**：
- [Neural Composer](https://github.com/numbpill/neural-composer) — LightRAG 集成 Obsidian，本地运行
- [Smart Connections](https://github.com/brianpetro/obsidian-smart-connections) — 笔记相似度 + AI 对话

**建议**：等你的知识库规模超过 300 条原子笔记后，再考虑引入。现阶段聚焦 Wiki 模式性价比最高。
